# V16 Release Plan: Sweep, Loft, Pattern Depth, Expression Extensions, and Mass Properties

Status: **Approved for implementation.** Slice **A1** (pattern viewport display
for extrude-family seeds) is implemented on main. Remaining slices (A2+) are
not started. Use this document as the V16 implementation source of truth.

Revision note (post-review, round 2, planning approved 2026-07-08): Product
Goal #1 is single-line Must / multi-line Stretch only; loft Must is parallel
face-attached paths (XY+XZ free base-plane diversity demoted); circular-axis
origin recompute rules; mass props per-query injection only;
`SCHEMA_UPGRADED_TO_V20` emission surface; cad-core pattern seed Must narrowed
to extrude-family; Mat4 document-space rigid convention vs protocol TRS
`Transform`. Round 1 fixes retained (Mat4 instances, V20 save migration,
derivedExactMetadata projection, topology matrix, dual-read, lineage
confidence, multi-solid policy, G0/G1/G2, Must vs Stretch DoD).

This document is a **release contract** for product release **V16**. It is
strict about product outcomes, architecture boundaries, source truth,
diagnostics, and proof. It is flexible about exact internal helper names,
slice order, and algorithms where implementation evidence should drive the
best design.

**Version naming caution:** product release V16 is not the same as document
schema `web-cad.project.v16` (already used since edge-finish for chamfer and
fillet). V16 source records that cannot be represented by
`web-cad.project.v19` introduce **`web-cad.project.v20`**.

## Product Goal

After V16, a Partbench user should be able to:

1. **Sweep** a supported profile sketch along a supported path and get a solid
   body with structured diagnostics for unsupported paths or failures.
   **Required (Must):** a single **line** entity (`pathEntityIds` length 1).
   **Optional (Stretch / prove-then-enable):** ordered connected multi-line
   chains — not release-required.
2. **Loft** across two or more supported planar profile sketches on **distinct,
   roughly parallel proven planar frames** via the face-attached authoring path
   (e.g. extrude pedestal → `sketch.createOnFace` → loft base + top profiles)
   and get a solid body with structured diagnostics for unsupported section
   sets. Free combinations of perpendicular base planes (e.g. XY + XZ alone)
   are **not** a Must green path.
3. Treat **pattern instances** as durable, addressable identities (instance
   index + rigid transform) while the manufacturing result remains a single
   fused (or multi-solid compound with diagnostics) result body.
4. Drive a **linear pattern** along a **generated-edge / named / axis-aligned
   topology edge direction**, not only global X/Y/Z.
5. Drive a **circular pattern** about a **user-defined rotation axis**
   (generated edge / named ref / axis-aligned topology linear edge with origin
   + direction), not only global axes through the document origin.
6. **Mirror** across a standard plane **with a signed offset**, or across a
   **named / topology-proven planar face** used as the mirror plane.
7. Author **parameter expressions** with trigonometric functions and simple
   conditionals (ternary and comparisons), without introducing a scripting
   language. Expressions continue to bind only through existing
   parameter→dimension channels — not directly to pattern spacing or mirror
   offsets.
8. Query **mass properties** (volume, surface area, center of mass; moments of
   inertia as stretch) for active bodies that have **derived exact metadata**
   available (including pattern/mirror/shell/imported when recipes exist, and
   sweep/loft when those features exist).

The user-facing promise is not "every curved path sweeps," "free offset
construction planes," or "every multi-body loft succeeds." The promise is:

- Supported sweep/loft matrices produce correct solids through CADOps with
  structured errors for unsupported profiles, paths, sections, or kernel
  failures.
- Pattern instance identity is durable and queryable via transforms; face/edge
  lineage catalogs are best-effort and confidence-labeled when present.
- Direction, axis, and plane references resolve through the existing
  generated-reference / named-reference / topology-anchor proof system within
  a **stated support matrix** (not all topology edges).
- Expression extensions remain pure, side-effect free, and rebuild-safe.
- Mass properties are query-only projections of kernel-derived exact metadata,
  not new source truth and not an in-core OCCT call from cad-core.

## Why V16 Comes After V15

V15 completed three capability clusters:

```text
STEP import + imported body topology identity
-> pattern / mirror / shell feature families (global axes & origin planes)
-> arithmetic parameter expressions
```

That foundation is necessary but not sufficient for practical part design
breadth:

**Sweep and loft** close the remaining basic solid-construction gap. Extrude
and revolve cover many prismatic and rotational parts; sweep covers path-driven
solids (tubes, rails, bosses along non-normal paths), and loft covers
section-driven transitions. V15 reserved sweep as an optional stretch and
explicitly deferred loft; neither was implemented.

**Pattern depth** closes the V15 identity and reference gap. V15 patterns fuse
N instances into one result body with only global axes. Practical design needs
edge-aligned patterns, custom circular axes, offset mirror planes, and a way to
address "instance i" without inventing multi-body assemblies.

**Expression extensions** close parametric design intent for angular and
conditional relationships (`pitch = diameter * sin(helix_angle)`,
`wall = thick ? 3 : 1.5`) without full scripting.

**Mass properties** close the measurement gap for general bodies. Today
`body.measurements` is analytic for simple extrudes
(`packages/cad-core/src/bodyMeasurements.ts`), and OCCT
`BRepGProp` already runs inside
`packages/occt-wasm/src/exactMetadata.ts`, but there is no general
`body.massProperties` product query. Cad-core queries are synchronous and
consume **app-supplied derived exact metadata**; V16 must extend that pipeline,
not pretend cad-core embeds the worker.

V16 is deliberately **not** assemblies, WebGPU production, collaboration,
construction/offset sketch planes, new sketch entity kinds (arc/spline), or
format breadth. It deepens single-part modeling on the V13–V15 topology and
feature stack.

## Architecture Alignment

V16 follows the same non-negotiable boundaries as `docs/architecture.md` and
the V12–V15 contracts:

- CADOps remains the center of mutation. No new feature bypasses the command
  layer (`feature.sweep`, `feature.loft`, pattern/mirror update paths, etc.).
- `cad-core` owns document source, feature records, parameter evaluation,
  commandability, reference health, rebuild planning, semantic diffs, and
  diagnostics. **Cad-core does not call OCCT.**
- `geometry-worker` / `occt-wasm` provide exact B-rep execution for sweep,
  loft, pattern transforms, mirror transforms, and GProp mass properties. They
  do not own source authority.
- React and the renderer display selections, feature status, measurements, and
  diagnostics derived from cad-core/query results. They do not infer feature
  correctness from mesh geometry.
- Agent and MCP adapters stay thin wrappers over CADOps/query paths. New V16
  commands are accessible through the same `cad.batch` interface; new queries
  through `cad.query`. Headless agents must supply derived exact metadata the
  same way the browser does (or accept unavailable mass-props diagnostics).
- renderer IDs, mesh IDs, OCCT handles, checkpoint-local topology IDs, GPU
  IDs, selection-buffer values, viewport pixels, OPFS paths, file handles,
  local paths, and export artifact IDs must never become public stable CAD
  references.
- Unsupported, stale, ambiguous, failed, or partially-proven states return
  structured diagnostics. They must not silently produce incorrect geometry.

V16 may add source data only when new features or durable identity require
source truth that cannot be reconstructed from V19 records and command history.
That is why V16 introduces `web-cad.project.v20`.

### Known V15 follow-through (in-scope early slice)

Implementation exploration found two V15 product gaps that V16 should close
early so new features are not built on broken display/reference plumbing:

1. **Pattern derived geometry sources are missing in `apps/web`.**
   `createMirrorDerivedGeometrySources` and shell sources exist in
   `apps/web/src/derivedGeometrySources.ts`, and the runtime supports
   `geometry.linearPattern` / `geometry.circularPattern` in
   `apps/web/src/derivedGeometryRuntime.browser.ts`, but
   linear/circular pattern source creators are absent. Pattern result bodies
   therefore have a **display gap**. Slice A must fix this for the **extrude-
   family seed support matrix** (see Display support matrix below).
2. **Generated reference catalogs** exist primarily for extrude / revolve /
   hole. Pattern, mirror, and shell have feature kinds in
   `packages/cad-core/src/generatedReferences.ts` but do not produce useful
   face/edge catalogs. V16 instance addressability prioritizes
   `body.patternInstances` transforms; face/edge lineage catalogs are
   **optional / best-effort** (see Decision 2).

### Display support matrix (pattern / mirror / shell seeds)

Slice A display parity matches today's mirror path:

| Seed geometry | Viewport derived source in V16 Must |
| --- | --- |
| Extrude `newBody` / boolean extrude family | **Required** (Slice A) |
| Revolve / hole / edge-finish as seed | Not required for Must; document gap |
| Imported body as seed | Not required for Must display; mass props via checkpoint path later |
| Pattern / mirror / shell as seed (recursive) | Non-goal for display recipes in V16 |
| Geometry-kernel `PatternSeedSource` | Today extrude \| booleanExtrudes only — **normative for app display, kernel requests, and cad-core validation in Must** |

**Cad-core pattern seed validation (Must — prefer narrow accept):**

- Pattern/mirror/shell **Must** commands validate seeds against the same
  extrude-family rebuild recipes that geometry-kernel and display support
  (extrude `newBody` / boolean extrude chain as today). Reject other active
  bodies at command validation with `PATTERN_SEED_BODY_UNSUPPORTED` (or the
  existing V15 seed-unsupported code), **not** a late worker crash and **not**
  a silent display-unavailable after commit.
- Broader seeds (revolve, hole, edge-finish, import, recursive pattern) are
  **non-goals for Must** even if occt-wasm can tessellate some of them
  internally. Expanding acceptance requires an explicit later slice that also
  adds kernel request types + display recipes.

## Major V16 Decisions

### 1. Product V16 Uses Schema `web-cad.project.v20`

Product release V16 introduces document schema **`web-cad.project.v20`** for
new source-of-truth records and field shapes that V19 cannot represent:

- `SweepFeature`
- `LoftFeature`
- Pattern direction / rotation-axis **reference unions** (replacing bare
  `"x" | "y" | "z"` as the only stored form in V20)
- Mirror plane **reference union** with optional signed offset
- `PatternInstanceRecord` / instance rigid transforms on pattern features

**Do not confuse** product V16 with schema `web-cad.project.v16` (chamfer /
fillet era). `.wcad` package version remains **`partbench.wcad.v2`**.

Projects use V20 only when at least one V20 source record or V20 field shape is
present after save. Load path always accepts V19.

### 2. Pattern Instances Stay Fused; Identity Is Source Transforms (Primary Model A)

V15 decision 2 remains the manufacturing model:

```text
seed body (active)
-> LinearPatternFeature / CircularPatternFeature consumes seed
-> geometry-worker creates N transforms and fuses (BRepAlgoAPI_Fuse)
-> single result body (active); may be multi-solid compound
```

V16 **does not** split pattern results into N independent public bodies by
default (that would be assemblies / multi-body compounds and breaks the
consumed-seed lifecycle).

**Primary model (A):**

1. **Source-level instance descriptors** are stored on the pattern feature.
   Each descriptor has durable `instanceIndex` in `[0, instanceCount)` and a
   **full rigid transform** that can represent rotation about an arbitrary
   axis (not only through the document origin). See `PatternInstanceRecord`.
2. **Result remains one public `bodyId`.** Prefer fuse of all instances.
3. **Instance addressability (Must):** query `body.patternInstances` returns
   indexes + rigid transforms (+ optional per-instance bounds when metadata
   exists). Instance-level **synthetic** refs (frame/bounds) may be exposed
   for measure/select.
4. **Face/edge lineage catalog (Stretch / best-effort):** optional
   generated-reference entries that map seed face stable IDs through instance
   transforms. **Never claim kernel-proven identity after fuse** unless
   checkpoint lineage proves it.

**Lineage confidence labels (normative when any lineage is emitted):**

```ts
identityConfidence: "transform-predicted" | "checkpoint-lineage" | "none"
```

- Default V16 ship: `body.patternInstances` transforms **required**; face/edge
  lineage catalog **optional** and only when tests justify it.
- Prefer **not** cloning seed face IDs after fuse. If transform-predicted face
  IDs are emitted at all, eligible operations are limited to measure / select /
  name — **never** sketch-on-face or edge-finish on `transform-predicted` IDs.
- `checkpoint-lineage` may unlock broader readiness only when V13 matching
  proves correspondence.

**Multi-solid compound policy (normative):**

- After fuse or compound fallback, detect `solidCount > 1` (via exact metadata
  topology counts or worker diagnostic).
- Always emit warning `PATTERN_MULTI_SOLID_RESULT` (and, if fuse was skipped
  for a deliberate compound path, `PATTERN_FUSE_DEGRADED_TO_COMPOUND`).
- Note: OCCT fuse of non-overlapping solids may already produce multi-solid
  compounds **without** `HasErrors()` — the multi-solid check is therefore
  **always required**, not only on explicit fallback.
- **Shell / chamfer / fillet readiness:** unsupported on multi-solid pattern
  results (`PATTERN_MULTI_SOLID_DOWNSTREAM_UNSUPPORTED` or reuse target
  unsupported codes with multi-solid detail).
- **STEP export:** multi-solid compound is a **supported export shape** with
  structured warning (export proceeds; consumer tools may see multiple solids).
  Do not silently claim single-solid.

**Why this model:**

- Keeps CADOps authority and the V15 consumed-seed contract.
- Works with V13 checkpoints without requiring assemblies.
- Avoids lying about face identity after fuse.

### 3. Sweep Uses Profile + Path Line Wire and `newBody` Lifecycle

Sweep is a first-class feature family:

```text
profile sketch entity + ordered path line entity ids
-> feature.sweep
-> geometry-worker OCCT BRepOffsetAPI_MakePipe (or proven equivalent)
-> newBody result
```

Seed lifecycle is **`newBody` only** in V16 (like a first extrude), not
add/cut. Downstream cut/add remains the extrude path.

**Profile support matrix (Must):**

- Rectangle and circle sketch entities (same profile kinds as extrude).
- Profile sketch on base plane (`XY` | `XZ` | `YZ`) or generated-face /
  topology-anchor face attachment (existing sketch attachment paths).

**Path support matrix:**

| Path form | V16 status |
| --- | --- |
| Single **line** entity (`pathEntityIds` length 1) | **Must / required** |
| Ordered chain of **connected line** entities (`pathEntityIds` length ≥ 2) | **Prove-then-enable** (connectivity validated) |
| Arc / spline / circle-as-path | **Non-goal** — no sketch arc entity exists (`SketchEntityKind` is point \| line \| rectangle \| circle only) |
| Freeform multi-branch wires | Rejected |

**Normative path field:** only `pathEntityIds: string[]` with `length >= 1`.
There is **no** separate `pathEntityId` field (avoids dual-field ambiguity).

**Wire validation (cad-core):**

- Every id resolves to a **line** entity on `pathSketchId`.
- Entities are ordered end-to-end: consecutive lines share an endpoint within
  existing sketch numeric tolerance (same tolerance family as sketch solver /
  coincidence).
- Reject branches, duplicates, zero-length lines → `SWEEP_PATH_DISCONNECTED`
  / `SWEEP_PATH_UNSUPPORTED`.
- Profile entity and path entities may **share the same sketch** or use
  different sketches — both allowed.

**Frame check:** cad-core does **not** enforce a numeric "approximately
normal" profile/path angle. Cad-core checks existence, kinds, and path length
> 0. **Kernel validates** the pipe construction; failures →
`SWEEP_GEOMETRY_FAILED` or `SWEEP_PROFILE_PATH_FRAME_INVALID` if the worker
can classify frame issues.

### 4. Loft Uses N≥2 Sections on Distinct Proven Planar Frames; Guides Out of Scope

Loft is a first-class feature family:

```text
ordered section profiles (N >= 2) on distinct proven planar frames
-> feature.loft
-> geometry-worker OCCT BRepOffsetAPI_ThruSections (solid mode)
-> newBody result
```

**Authoring reality (critical):** sketch planes are only `XY` | `XZ` | `YZ`
plus **face attachments** (`sketch.createOnFace` / topology-anchor face). There
are **no free construction/offset sketch planes** in V16. Two free sketches on
the same base plane (both at z=0) produce a **degenerate** loft and must be
rejected (`LOFT_SECTIONS_COPLANAR` / `LOFT_SECTION_FRAME_INVALID`).

**Supported authoring paths:**

| Path | Status |
| --- | --- |
| **Face-attached / parallel-offset (primary green path):** extrude pedestal → `sketch.createOnFace` on top face → loft base-plane profile + top-face profile | **Must** |
| **Imported / result body faces:** topology-anchor or generated-face attachments on **distinct, roughly parallel** planar faces with non-zero separation along the loft direction | **Must** when frames pass coplanar + parallel checks |
| Free base-plane combinations only (e.g. `XY` + `XZ` with no attachment offset) | **Not Must.** Prove-then-enable only if ThruSections tests are green; otherwise reject with `LOFT_SECTION_FRAME_INVALID` |

Solid ThruSections is reliable for **separated, roughly parallel** section
frames. Perpendicular base planes commonly fail or produce unstable solids;
listing them as Must would create flaky Slice F tests.

**Support matrix:**

- Each section is a rectangle or circle on a planar sketch with a **resolved
  3D plane frame** (base plane placement or attachment proof).
- Sections ordered; order is authorial.
- Section frames must not be coplanar (within tolerance) →
  `LOFT_SECTIONS_COPLANAR` / `LOFT_SECTION_FRAME_INVALID`.
- For Must acceptance: section normals must be **roughly parallel** (within an
  implementation angle tolerance), and projected separation along the loft
  direction must be non-zero; otherwise `LOFT_SECTION_FRAME_INVALID`.
- **Guide curves out of scope.**
- Solid loft only; **no add/cut** (`newBody` only).
- **Non-goal:** free offset construction planes; translating entities on the
  same base plane is **not** a substitute for a distinct plane frame;
  perpendicular free base-plane lofts as a product promise.

### 5. Direction, Axis, and Plane References — Explicit Support Matrix

V16 extends pattern and mirror parameters from bare enums to **reference
unions** that resolve through the same proof stack as shell open faces and
edge finish, within a **narrow V16 proof matrix**.

Existing topology command proof kinds
(`CadTopologyAnchorCommandProofKind`) are only:

- `axisAlignedPlanarFace`
- `axisAlignedLinearEdge` (with `linearAxis?: "x"|"y"|"z"`)
- `pointVertex`

There is **no** general `exactLinearEdge { origin, direction }` proof today.

**Linear pattern direction — Must support:**

| Ref | Status |
| --- | --- |
| `globalAxis` x\|y\|z | Required |
| `generatedEdge` with line proof / `curveType: "line"` + direction from catalog | Required |
| `namedReference` → such a generated edge | Required |
| `topologyAnchor` **axis-aligned** linear edge (`axisAlignedLinearEdge`) | Required |
| General (diagonal) linear topology edge | **Prove-then-enable** only if new proof kind `exactLinearEdge { origin, direction }` is implemented; otherwise `PATTERN_DIRECTION_UNSUPPORTED` |

**Circular pattern rotation axis — Must support:**

Same table, but resolution must produce **origin point + direction**:

- Global axis: direction through **document origin** (V15 behavior).
- Generated / named linear edge: direction from catalog
  (`geometricSignature.axis` / line proof). **Origin is recomputed by
  cad-core** — do **not** require a new catalog origin field for Must.
  Deterministic recompute rule (in order):
  1. Prefer **midpoint of the edge segment** reconstructed from the
     generated-edge role + source sketch entity + placement frame (same
     sources used for generated-reference geometry), or from topology /
     measurement `midpoint` / segment endpoints when already present on the
     snapshot.
  2. Else midpoint of topology anchor proof **bounds** (axis-aligned edge
     bounds center).
  3. Else entity snapshot `point` / start point if that is all that exists.
  Record which rule was used only in internal resolution logs/tests; public
  axis is always `{ origin: Vec3; direction: Vec3 }`.
- Topology axis-aligned linear edge: direction from `linearAxis`; origin via
  the same midpoint-of-bounds preference above.
- General linear edge: prove-then-enable with `exactLinearEdge` only.

Reject face-only refs and curved edges with `PATTERN_AXIS_UNSUPPORTED`.

**Mirror plane — Must support:**

| Ref | Status |
| --- | --- |
| `standardPlane` XY\|XZ\|YZ + optional signed `offset` | Required |
| `generatedFace` planar + optional offset | Required |
| `namedReference` → planar face | Required |
| `topologyAnchor` **axis-aligned** planar face | Required |
| General planar face topology proof | Prove-then-enable if proof exists; else unsupported |

**Selection filters (normative for UI / `selection.referenceCandidates`):**

- Pattern **direction** and circular **axis** picks: only edges with **line**
  proof (generated line edges; axis-aligned topology linear edges in Must).
  Reject circles, rectangle boundary as face, curved edges → structured
  unsupported diagnostics.
- Mirror plane picks: only faces with **plane** proof.
- Extend `CadGeneratedReferenceEligibleOperation` (or selection operation
  filters) with operations such as `feature.linearPatternDirection`,
  `feature.circularPatternAxis`, `feature.mirrorPlane` so the reference picker
  can filter correctly. Reuse picker UI with these filters — do not accept
  unfiltered body picks.

**Resolution authority:**

1. Cad-core validates ref kind and required fields.
2. Named refs resolve to underlying generated/topology targets.
3. Direction/plane proof loaded from catalog or topology anchor proof.
4. Unitize / apply plane offset.
5. Geometry-worker receives **resolved numeric frames only**
   (`direction: Vec3`, `axis: { origin: Vec3; direction: Vec3 }`,
   `plane: { point: Vec3; normal: Vec3 }`) — never raw document ref picks.

### 6. Expression Language Extends In-Place Without a Schema Trigger

Expression text remains `CadParameter.expression` (introduced in V19). The
**presence of trig or conditionals alone does not force `web-cad.project.v20`**.

Rationale: storage field shape is unchanged; evaluation is pure cad-core
(`packages/cad-core/src/parameterExpressions.ts`); older arithmetic remains
valid.

**Portability / health (not a schema trigger):**

When any parameter expression uses language v2 syntax (trig calls, `? :`,
comparison operators, `if(`), project health / `project.parameterEvaluation`
SHOULD report informational diagnostic
`EXPRESSION_LANGUAGE_V2_FEATURES_PRESENT` so V15-only tools and users know
the file needs a V16-capable evaluator. This is **not** stored as a required
source field and does **not** bump schema.

Document in `docs/native-format.md` notes when V16 lands: expression language
capability is product/evaluator level, independent of `web-cad.project.v*`.

### 7. Mass Properties Are Query-Only Projections of Derived Exact Metadata

Mass properties are **not** source records and **not** an OCCT call inside
cad-core.

**Normative architecture (matches existing extents/health injection):**

1. **Geometry boundary** extends exact-metadata **source recipes** (and
   checkpoint GProp for imports) for bodies in the support matrix.
   `ExactBodyMetadataSource` today is only extrude / booleanExtrudes / revolve
   / hole / edgeFinish (`packages/geometry-kernel/src/kernel.ts`) — V16 must
   extend this graph (slices G0/G2).
2. **App or agent** runs the async geometry path and supplies
   `derivedExactMetadata` as a **per-query optional field** on the request
   (same pattern as `ProjectExtentsQuery.derivedExactMetadata` /
   `project.health` in `packages/cad-core/src/index.ts`). **No ambient
   cad-core cache** of exact metadata in V16 Must — each query carries the
   snapshots it needs.
3. **`body.massProperties` in cad-core is a pure projection:** read volume /
   surfaceArea / centroid from the per-query snapshot; apply optional
   `density` (default `1`); shape response; return
   `MASS_PROPERTIES_UNAVAILABLE` / `MASS_PROPERTIES_METADATA_STALE` /
   `MASS_PROPERTIES_BODY_CONSUMED` when missing, stale, or consumed.
4. **Imported bodies:** checkpoint payload → load shape → GProp at geometry
   boundary (not rebuild-recipe). App supplies metadata snapshot keyed by body
   id.
5. **Agent/MCP / smokes:** align with V15 — command/query tests + geometry-
   boundary tests; do not assume in-core OCCT. Headless callers without
   metadata get structured unavailable, not invented numbers.
6. **Inertia stretch:** include moments if `GProp_GProps` matrix is cheap;
   same query shape, optional fields.
7. **Relationship to `body.measurements`:** keep source-analytic simple-extrude
   helper. Prefer `body.massProperties` for general bodies.

### 8. STEP XDE Colors Remain Optional Stretch

V15 ignored XDE color/layer metadata. V16 **default non-goal**. Optional late
stretch only as derived display metadata — never modeling source.

### 9. Single Normative V19→V20 Save Migration

**Normative policy (only one):**

- **Load:** accept V19 pattern/mirror enums and records; synthesize in-memory
  V20 unions + `instances[]` for session use.
- **First successful save from a V16 app** that has opened a project containing
  V15 pattern/mirror features (or any content normalized to V20 shapes):
  **always** materialize `instances[]`, normalize refs to V20 unions, export
  **`web-cad.project.v20`**, and emit informational
  `SCHEMA_UPGRADED_TO_V20`.
- There is **no** "export stays V19 until user edits" path. Session-only
  instance synthesis without durable save is **not** the product model once
  the user saves.

**`SCHEMA_UPGRADED_TO_V20` emission surface (normative for tests):**

1. **Authoritative:** include on the **save / `.wcad` package write** (or JSON
   export) result path that already carries package / export diagnostics
   (same family as package write summary / export readiness diagnostics).
2. **Mirror:** when the in-session document schema is V20 because of this
   migration, `project.health` may also report the same informational code so
   UI/agents see it without re-saving.
3. Smokes assert at least the save/export diagnostic once.

This makes instance identity **durable source** after save and avoids forked
implementations.

### 10. Dual-Read Command and Snapshot Fields (V15 Compatibility)

Breaking command compatibility is not allowed.

**Commands (`feature.linearPattern`, `feature.updateLinearPattern`, circular
and mirror counterparts):**

- **Accept forever** V15 sugar fields:
  - `axis: "x"|"y"|"z"` ≡ `direction: { kind: "globalAxis", axis }`
  - `rotationAxis: "x"|"y"|"z"` ≡ globalAxis form
  - `mirrorPlane: "XY"|"XZ"|"YZ"` ≡
    `plane: { kind: "standardPlane", plane, offset: 0 }`
- **Prefer** V20 fields in new UI: `direction`, `rotationAxis` as ref union,
  `plane` as `MirrorPlaneRef`.
- If both sugar and V20 fields are present and disagree → validation error.
- If both agree or only one present → accept.

**Snapshots / export:**

- V19 export (legacy files not yet saved in V16): old enum fields only.
- V20 export: new unions + `instances[]`; may also include deprecated sugar
  mirrors for readability **or** only unions — **decision: V20 export emits
  unions + instances only** (load still accepts both). Protocol tests cover
  dual-read on commands and load of both snapshot shapes.

## New Source Records (`web-cad.project.v20`)

Field names use `bodyId` for the result body to match implemented V15 snapshots
(`LinearPatternFeatureSnapshot.bodyId` in
`packages/cad-protocol/src/index.ts`).

```ts
// ----- Shared reference unions -----

PatternDirectionRef =
  | { kind: "globalAxis"; axis: "x" | "y" | "z" }
  | { kind: "generatedEdge"; bodyId: string; stableId: string }
  | { kind: "namedReference"; name: string }
  | { kind: "topologyAnchor"; bodyId: string; anchorId: string }

// Resolves to origin + direction. Global axes pass through document origin.
PatternRotationAxisRef =
  | { kind: "globalAxis"; axis: "x" | "y" | "z" }
  | { kind: "generatedEdge"; bodyId: string; stableId: string }
  | { kind: "namedReference"; name: string }
  | { kind: "topologyAnchor"; bodyId: string; anchorId: string }

MirrorPlaneRef =
  | {
      kind: "standardPlane"
      plane: "XY" | "XZ" | "YZ"
      offset?: number // signed; default 0; document units along plane normal
    }
  | { kind: "generatedFace"; bodyId: string; stableId: string; offset?: number }
  | { kind: "namedReference"; name: string; offset?: number }
  | {
      kind: "topologyAnchor"
      bodyId: string
      anchorId: string
      offset?: number

// Full rigid transform. Normative representation: row-major Mat4 (16 numbers)
// computed deterministically on create/update. MUST bit-match geometry-worker
// instance transforms (same construction order).
//
// Protocol note: existing `Transform { translation, rotation, scale }` TRS is
// NOT used for pattern instances (no scale; needs arbitrary-axis pivot).
// Add a `Mat4` type alias in cad-protocol during Slice A (protocol).
//
// Application convention:
//   - Mat4 is the same rigid transform the geometry worker applies to the
//     seed shape in **document space** (maps seed pose → instance pose).
//   - Instance 0 is the 4×4 identity.
//   - Rigid only: rotation + translation; **no scale** component.
//
// Equivalent construction semantics for documentation/tests:
//   linear:  T = Translate(i * spacing * unitDirection)
//   circular: T = RotateAboutAxis(origin, unitDirection, angle_i)
//             (Rodrigues / OCCT gp_Trsf rotation about axis — NOT origin-only)
//
// Do NOT use incomplete { translation, rotationAxis, rotationAngle } without
// pivot origin — that cannot represent non-origin circular axes.
PatternInstanceRecord {
  instanceIndex: number // 0 .. instanceCount-1
  transform: Mat4 // row-major 16; rigid (rotation + translation)
}

// Mat4 = readonly [
//   number, number, number, number,
//   number, number, number, number,
//   number, number, number, number,
//   number, number, number, number
// ]
```

## Pattern Depth Detail

### Instance formulas

**Linear** (instance i):

```text
T_i = Translate(i * spacing * unitDirection)
```

**Circular** (instance i):

- Arc (`totalAngleDegrees < 360`):
  `angle_i = totalAngleDegrees / (instanceCount - 1) * i`
- Full circle (`totalAngleDegrees == 360`):
  `angle_i = 360 / instanceCount * i`
- `T_i = RotateAboutAxis(axisOrigin, unitDirection, angle_i)`  
  (full rigid transform about **resolved** axis, not document-origin-only)

Stored `instances[i].transform` **must match** the transform sent to / used by
the geometry worker.

### Fuse / multi-solid policy

See Decision 2. Always detect multi-solid; warn; block shell/edge-finish
readiness; allow STEP export with warning.

## Parameter Expression Language v2

Extend `packages/cad-core/src/parameterExpressions.ts` (pure TS, no `eval`).

### Trigonometry — degrees-first CAD UX

```text
sin(deg), cos(deg), tan(deg)
asin(x), acos(x), atan(x), atan2(y, x)  // return degrees
deg(rad), rad(deg)
```

Domain errors → leave value unchanged; `EXPRESSION_DOMAIN_ERROR`; never
NaN/Infinity into source.

### Conditionals

- Comparisons produce **0 or 1**: `< > <= >= == !=`
- **Float `==` / `!=` use exact IEEE equality** (no epsilon). Document this;
  users who need tolerance write explicit ranges (`abs(a-b) < 1e-6` style)
  using existing arithmetic.
- Ternary: `condition ? a : b` (0 false; nonzero finite true).
- Builtin `if(cond, a, b)` same semantics.
- **No** `&&` / `||` / `!` — explicit non-goal. Nest ternaries or use `if`.
  Rejected tokens → `EXPRESSION_LANGUAGE_UNSUPPORTED_TOKEN` /
  `EXPRESSION_PARSE_ERROR`. Tests must reject `&&` / `||`.
- Optional later (out of V16 Must): `and(a,b)` / `or(a,b)` call builtins.

No loops, assignments, strings, side effects, multi-statement scripts, or
user-defined functions.

### Grammar extension (informal)

```text
expression  ::= ternary
ternary     ::= comparison ('?' expression ':' expression)?
comparison  ::= sum (('<'|'>'|'<='|'>='|'=='|'!=') sum)?
sum         ::= term | sum '+' term | sum '-' term
term        ::= factor | term '*' factor | term '/' factor
factor      ::= literal | paramRef | funcCall | '(' expression ')' | '-' factor
```

### Evaluation model

Unchanged V15 DAG / circular detection / `parameter.setExpression` semantics.

### Binding scope

Expressions bind through **existing parameter→sketch dimension** channels
only. **Non-goal:** direct parameter binding to pattern `spacing`,
`instanceCount`, `totalAngleDegrees`, mirror `offset`, or sweep/loft numeric
fields unless those fields already participate in a generalized binding system
(they do not in V15). Goal #7 examples apply to **parameter values and
dimensions**, not pattern feature fields.

## Definition of Done

### Must

1. `feature.sweep` creates correct solids for rectangle/circle profiles along
   a single **line** path (`pathEntityIds` length 1), with structured errors
   for unsupported paths. Rebuild/dependency parity with extrude newBody.
2. `feature.loft` creates correct solids for N≥2 supported sections on
   **distinct, roughly parallel proven planar frames** via the face-attached
   (pedestal / `sketch.createOnFace`) green path, rejecting coplanar,
   non-parallel, or degenerate frames. Rebuild parity with extrude newBody.
   Free XY+XZ base-plane-only lofts are not Must.
3. Linear and circular patterns store durable `instances[]` **Mat4** transforms
   that match geometry-worker inputs; `body.patternInstances` works; result is
   one `bodyId` (fuse or multi-solid with warnings).
4. Linear patterns accept global + generatedEdge/named + axis-aligned topology
   directions; circular patterns accept corresponding axes with origin;
   mirror accepts offset standard planes and proven planar faces. Commands
   dual-read V15 fields.
5. Pattern derived geometry sources render for **extrude-family seeds**.
6. Parameter expressions support degree-first trig, comparisons (IEEE `==`),
   and ternary/`if`, with domain-safe evaluation; no `&&`/`||`; health notes
   v2 feature presence.
7. `body.massProperties` pure projection works for bodies with available
   derived exact metadata after G0/G1 (extrude family, boolean/hole/finish,
   pattern/mirror/shell, import checkpoint). Density default 1.
8. All new commands via `cad.batch`; new queries via `cad.query` with metadata
   injection where required.
9. cad-core remains authority for validation, source mutation, diffs, health,
   rebuild planning — without embedding OCCT.
10. React/renderer consume query results; no mesh-inferred correctness.
11. V20 JSON/`.wcad` round-trips; V19 load + save upgrades with
    `SCHEMA_UPGRADED_TO_V20`.
12. Unsupported/failed/stale states use structured diagnostics without silent
    partial source mutation.
13. Focused unit tests cover protocol dual-read, Mat4 instances, expressions
    v2, migration, multi-solid policy, mass-props projection, geometry
    failures, adapter pass-through.
14. Named V16 Must smokes and release samples pass.
15. `pnpm test`, `pnpm typecheck`, `pnpm lint`, `pnpm format:check`, and
    `pnpm build` pass.

### Stretch (not required to close V16)

- Connected multi-line sweep paths (`pathEntityIds` length ≥ 2) when proven.
- Transform-predicted or checkpoint face/edge lineage catalogs.
- Moments of inertia / principal moments on mass props.
- STEP XDE color display metadata.

```text
-> verify values; edit angle; rebuild dimensions
-> asin(2) -> EXPRESSION_DOMAIN_ERROR
-> "a > 0 && b < 1" -> parse reject EXPRESSION_LANGUAGE_UNSUPPORTED_TOKEN
-> health may report EXPRESSION_LANGUAGE_V2_FEATURES_PRESENT
```

## Alternatives Considered

1. **Multi-body pattern results (N bodyIds)** — rejected (mini-assembly).
2. **Query-time-only instance transforms without source** — rejected (agents
   need durable indexes after save).
3. **Topology-tag-only lineage without source instances** — rejected as sole
   model (checkpoints optional).
4. **Incomplete TRS without pivot for circular axes** — rejected (cannot
   represent non-origin axes); use Mat4.
5. **Export stays V19 until edit** — rejected; single save-bump policy.
6. **Cad-core calls worker inside mass props query** — rejected (sync query
   model; use derived metadata injection).
7. **Sketch arc entity in V16 for sweep** — rejected (unscoped solver/UI
   explosion); single-line Must; multi-line Stretch only.
8. **Free construction planes / free XY+XZ base-plane loft as Must** — rejected;
   parallel face-attached authoring only for Must.
9. **Broad cad-core pattern seed accept with late worker fail** — rejected for
   Must; validate extrude-family at command time.
10. **Radians-first trig** — rejected for default UX.
11. **Full JS expression subset** — rejected (determinism/security).
12. **Store mass properties in source** — rejected (stale).
13. **Worker resolves named references** — rejected (document authority).
14. **Kernel-proven face lineage as Must** — rejected (fuse correspondence hard).
15. **Ambient cad-core exact-metadata cache for mass props** — rejected; per-query
    field only (match `project.extents`).
16. **Protocol TRS `Transform` for pattern instances** — rejected; rigid Mat4.
