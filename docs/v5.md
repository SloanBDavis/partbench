# V5 Target: Exact Geometry And Topology Foundation

Status: complete.

V5 should make the current authored bodies more CAD-real without broadening the
modeling surface too quickly. V4 made constrained sketches and regeneration
coherent. V5 should close the next hard architecture gap: supported authored
bodies, including the current narrow boolean result bodies, need derived exact
geometry, topology/reference health, and trustworthy measurements/extents.

V5 is not a general feature expansion milestone. It should not add revolve,
sweep, loft, fillet, chamfer, shell, patterns, assemblies, STEP, OPFS, WebGPU,
or broad boolean behavior. It should make the current V2/V3/V4 source model more
inspectable, agent-readable, and ready for future reference-consuming features.

## Product Goal

A user or structured agent can create the current supported Partbench bodies,
including narrow add/cut results, and inspect them as real CAD bodies with
generated references, exact/kernel-derived measurements where available, clear
topology health, and honest unsupported or ambiguous states.

The user-facing outcome should be:

```text
create/edit supported body -> derived exact/topology data rebuilds ->
references, measurements, extents, health, and viewport stay coherent
```

The agent-facing outcome should be:

```text
query body topology -> resolve or name generated references -> dry-run edits ->
inspect topology/measurement health -> commit source changes through CADOps
```

## Architectural Rules

V5 must keep these rules:

- CADOps remains the shared mutation protocol.
- `cad-core` remains authoritative for source document state.
- OCCT, geometry-worker, and geometry-kernel produce derived results only.
- Rendered meshes, exact geometry results, topology snapshots, and measurement
  outputs are caches/query data, not persisted source-of-truth document state.
- React UI does not import geometry internals directly.
- Agent and MCP adapters wrap CADOps and query paths. They do not define model
  semantics.
- Generated references must not expose raw OCCT, mesh, or renderer indexes as
  stable IDs.
- Ambiguous or unsupported topology must fail with structured diagnostics rather
  than silently preserving the wrong reference.
- Project format changes happen only for persisted source-of-truth data.
- Existing `web-cad.project.v1` through `web-cad.project.v13` imports remain
  supported where practical.

## V5 User Workflow Target

The V5 workflow should allow a user or structured agent to:

1. Create or open a current V13 project.
2. Create constrained or unconstrained rectangle/circle sketches.
3. Create current supported authored bodies:
   - rectangle/circle `newBody` extrudes,
   - rectangle-tool cut against rectangle target,
   - rectangle-tool cut against circle target,
   - rectangle-tool add/fuse against rectangle target.
4. Query body topology/reference data for supported bodies.
5. Name generated references where they resolve safely.
6. Inspect exact/kernel-derived measurements and extents where available.
7. See clear health for unsupported, stale, ambiguous, or failed topology.
8. Edit upstream sketches/dimensions/constraints/features and see derived
   topology, references, measurements, extents, and viewport data rebuild or
   fail honestly.
9. Use the same data through UI, agent-adapter, and MCP wrappers.

## V5 Scope

V5 should focus on derived exact geometry, topology/reference health, and
measurement confidence for the body types the app already supports.

In scope:

- a typed derived exact-geometry/topology boundary for current authored bodies;
- stable generated reference responses for supported boolean result bodies where
  the app can derive or validate them safely;
- topology/reference health status such as:
  - `healthy`,
  - `unsupported`,
  - `ambiguous`,
  - `stale`,
  - `kernel-failed`;
- exact or kernel-derived measurements/extents for supported current bodies:
  - bounds,
  - volume,
  - surface area,
  - centroid where reliable;
- source-derived fallback behavior only where it is explicitly honest;
- named references targeting supported boolean result references when the
  resolver can validate them;
- derived geometry cache keys and stale-result handling for exact/topology data;
- CADOps read/query APIs for exact/topology/measurement status;
- agent-adapter and MCP pass-through for those read paths;
- compact UI for inspecting body topology, reference health, and measurement
  confidence;
- unit/package-level tests across protocol, cad-core query behavior,
  geometry-kernel/worker, adapter/MCP, UI helpers, import/export, and stale cache
  behavior.

Out of scope:

- new modeling commands such as revolve, sweep, loft, fillet, chamfer, shell,
  pattern, mirror, or direct edits;
- broad/general booleans beyond the already supported narrow add/cut cases;
- generated references that claim stability when topology is ambiguous;
- full persistent B-rep storage;
- exact B-rep checkpointing in the project file;
- STEP/IGES import/export;
- OPFS or File System Access API;
- WebGPU renderer work;
- assemblies, mates, large-model instancing, or LOD;
- hosted collaboration or auth;
- natural-language command parsing;
- browser E2E expansion.

## Phase A: Exact Geometry And Topology Boundary

Goal: define the derived exact/topology result model before wiring it into more
queries.

Completed:

- added the read-only `body.topology` query;
- added typed status values for `healthy`, `unsupported`, `ambiguous`, `stale`,
  and `kernel-failed`;
- added structured topology issue codes for unsupported bodies, stale sources,
  ambiguous topology, empty/invalid exact results, and kernel failure;
- added source identity/cache-key metadata derived from body/feature intent;
- exposed the query through agent-adapter and MCP as thin wrappers;
- tested that querying derived topology does not mutate project JSON source
  data.

The boundary remains intentionally read-only. Missing bodies return structured
`BODY_NOT_FOUND` query errors. Primitive compatibility bodies return structured
`unsupported` topology snapshots rather than becoming a second source model.

Acceptance:

- current source document behavior is unchanged;
- exact/topology data is explicitly derived;
- no raw OCCT, mesh, or renderer indexes become stable public references.

## Phase B: Simple Extrude Exact/Topology Parity

Goal: route simple rectangle/circle `newBody` extrudes through the new
exact/topology boundary while preserving existing semantic generated reference
behavior.

Completed:

- derived topology/measurement responses for rectangle and circle newBody
  extrudes;
- semantic-source topology counts for rectangle faces/edges/vertices and circle
  faces/edges;
- deterministic source identity/cache keys that include evaluated profile
  signatures, depth, side, units, and source feature/sketch/entity IDs;
- `source-analytic` measurement confidence for the current exact-for-source
  rectangle/circle extrude formulas;
- project health entries showing topology status/model/availability without
  persisting derived topology.

Acceptance:

- simple authored extrudes do not regress;
- measurements and extents remain coherent after dimension/constraint/profile
  edits;
- existing named references keep resolving when their semantic role still
  exists.

## Phase C: Boolean Result Topology

Goal: make the current supported boolean result bodies inspectable without
claiming a full topological naming solution.

Completed:

- boolean result bodies from the current rectangle cut, circle-target cut, and
  rectangle add/fuse workflows return structured `ambiguous` topology snapshots;
- ambiguous snapshots include source identity for the result feature and target
  body, with `topologyAvailable: false` and `exactMeasurementsAvailable: false`;
- generated references and named references are not exposed for boolean result
  bodies because stable face/edge/vertex roles cannot be proven yet;
- project health reports topology status/model/availability on authored extrude
  entries without changing source dependency status.

Acceptance:

- boolean result bodies are no longer second-class when topology is supported;
- unsupported or ambiguous topology is visible and structured;
- generated references for boolean results do not silently point to the wrong
  face/edge/vertex after source edits.

## Phase D: Exact Measurements And Project Extents

Goal: make supported body measurements and project extents use the best
available derived exact data while preserving honest fallbacks.

Completed:

- `body.topology` exposes measurement confidence for current bodies;
- rectangle/circle newBody extrudes report `source-analytic` confidence for the
  existing exact-for-source measurement formulas;
- structured measurement confidence/source metadata;
- boolean result bodies continue to report unavailable exact measurements and
  extents warnings because the current kernel path returns tessellated meshes,
  not reliable mass properties or stable topology;
- unsupported and ambiguous states remain visible and do not mutate source
  document data.

Acceptance:

- measurements/extents no longer pretend boolean result bodies are unavailable
  when the kernel can produce reliable data;
- failures stay visible and do not corrupt source document state.

## Phase E: Agent/MCP And UI Integration

Goal: expose V5 exact/topology data in the same command/query-driven way as the
rest of Partbench.

Completed:

- agent-adapter and MCP pass-through for body topology queries;
- compact UI for selected body topology/reference health;
- generated/named reference behavior remains unchanged for simple generated
  references and stays unavailable for ambiguous boolean topology;
- selected-body UI displays topology status, topology model, entity counts, and
  measurement confidence;
- UI helper tests for filtering, status formatting, command/query building, and
  unsupported/ambiguous display.

Acceptance:

- users can inspect the current body/reference state without reading debug text;
- agents can query the same data and decide whether a reference-consuming edit
  is safe.

## Phase F: V5 Stabilization And Completion

Goal: declare V5 complete only when supported authored bodies and narrow boolean
result bodies have coherent exact/topology behavior, tests, and docs.

Completed:

- cad-core, protocol, adapter/MCP, storage, docs, and UI helper paths were kept
  on the read/query side of CADOps;
- V5 coverage was added for protocol shapes, cad-core topology query behavior,
  source/derived separation, project health topology summaries, adapter/MCP
  pass-through, and UI topology formatting;
- docs were updated to mark the milestone complete and call out the remaining
  exact B-rep/topological naming limitations.

Acceptance:

- no known architecture drift;
- boolean result topology/measurement status is honest;
- all required checks pass.

## Project Format Direction

V5 should not introduce a new project schema version unless it adds new
persisted source-of-truth document records. Derived exact geometry, topology
snapshots, measurement outputs, kernel statuses, cache keys, and UI selection
state should not be persisted in the JSON source format.

If V5 later needs persisted source metadata, that change should introduce
`web-cad.project.v14` with explicit V1 through V13 import compatibility and
structured validation.

Do not introduce a new format version for:

- derived exact geometry results;
- derived topology/reference query results;
- derived measurement outputs;
- mesh or tessellation cache data;
- OCCT transient identifiers;
- UI inspection state.

## Testing Strategy

V5 should continue the unit/package-test strategy. Browser checks may be used
manually for confidence, but V5 should not add browser E2E tests unless a
specific bug demands it.

Required coverage areas:

- `cad-protocol` query shapes and structured error/status shapes;
- `cad-core` query behavior, source/derived separation, undo/redo, batch
  dry-run/commit where relevant, import/export, and health;
- generated reference resolver behavior for simple and boolean result bodies;
- named reference behavior for supported/stale/ambiguous references;
- geometry-kernel and geometry-worker boolean mesh request/response behavior,
  with topology queries kept separate from mesh execution until exact metadata
  is available;
- derived geometry cache keys, invalidation, and stale async result handling;
- render scene preparation for consumed target/result bodies;
- measurement/extents confidence and fallback behavior;
- agent-adapter and MCP pass-through;
- UI helper formatting, filtering, and status display.

## V5 Acceptance Criteria

V5 is complete when:

1. The exact/topology boundary is explicit, typed, and tested.
2. Rectangle/circle newBody extrudes preserve existing generated reference
   behavior while participating in the new exact/topology path.
3. Supported rectangle cut, circle-target cut, and rectangle add result bodies
   expose generated references where stable, or structured ambiguous/unsupported
   status where not.
4. Named references remain limited to generated references that resolve safely.
   Boolean result references stay unavailable while topology is ambiguous.
5. Measurements/extents use exact/source-analytic data where available and expose
   confidence/source metadata.
6. Source edits, constraints, undo/redo, import/export, and stale async geometry
   handling keep topology and measurement data coherent.
7. Agent/MCP wrappers expose V5 read paths as thin wrappers over CADOps.
8. The UI presents topology/reference/measurement confidence without misleading
   users into thinking broad topology naming is solved.
9. Unit and package tests cover the workflow without relying on browser E2E
   tests.

## Deferred Beyond V5 Unless Explicitly Scoped

- broad stable topological naming across arbitrary feature edits;
- general booleans beyond current narrow add/cut cases;
- generated references for unsupported or ambiguous topology;
- persistent exact B-rep storage/checkpoints;
- STEP/IGES import/export;
- new feature operations such as revolve, sweep, loft, shell, fillet, chamfer,
  mirror, pattern, or direct edits;
- OPFS/File System Access;
- WebGPU renderer;
- assemblies and mates;
- hosted collaboration/auth;
- natural-language command entry.
