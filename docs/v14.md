# V14 Release Plan: Topology-Backed Downstream Modeling

Status: active. Tranche A, topology-backed command-target readiness, is
implemented as a non-mutating protocol/cad-core/agent/MCP query slice. Slice B,
topology-backed sketch attachment, is implemented as a narrow CADOps/app
product path. Slice C is partially implemented for supported topology-backed
cut/add and hole targets, including the explicit XZ side-plane circular hole
path on supported circle-origin result bodies. Slice D is partially implemented
for topology-anchor edge-finish command flow and the first result-edge class:
rectangle cut-wall longitudinal edges on supported cut result bodies.

V14 is the release that should turn the V13 topology identity foundation into
normal single-part modeling behavior. V13 made arbitrary exact topology
identifiable, checkpointed, matchable, repairable, measurable, and partly
command-ready. V14 should make supported active result bodies useful as
downstream modeling targets.

This document is intentionally a release contract, not an implementation
script. It is strict about product outcomes, architecture boundaries, source
truth, diagnostics, and proof. It is flexible about exact internal helper
names, tranche order, and algorithms where implementation evidence should drive
the best design.

## Product Goal

After V14, a Partbench user should be able to keep working on a supported
single part after the first operation creates a result body.

The core workflow should be:

1. Create a rectangle or circle sketch.
2. Extrude it as a body.
3. Create a sketch on a supported face.
4. Cut or add with that sketch.
5. Select a supported face on the resulting body.
6. Create another sketch there.
7. Cut or add again.
8. Select eligible result edges and apply chamfer or fillet where cad-core and
   the geometry boundary can prove the target.
9. Edit upstream supported features and either rebuild cleanly or receive
   precise repair/blocked diagnostics.
10. Save and open the project through `.wcad` without losing the topology
    identity needed for those commands.

The user-facing promise is not "every imported or arbitrary model can be
edited perfectly." The promise is that supported topology-backed result bodies
can participate in downstream commands through the same authoritative command
layer as source-semantic bodies, and unsupported cases explain the actual
reason.

## Why V14 Comes After V13

V13 solved the foundation problem:

```text
exact body result ->
topology snapshot ->
checkpoint payload ->
stable topology anchor ->
matching/repair evidence ->
reference health ->
command-readiness diagnostics
```

That foundation is necessary but not sufficient for a usable CAD workflow. A
body can look valid in the viewport and still be non-commandable if the command
validator cannot prove how to consume its face, edge, or body reference.

V14 should close that gap for the current supported single-part feature
families. It should bridge from stable topology identity to actual command
targets for downstream sketch attachment, cut/add, and edge finishing.

## Architecture Alignment

V14 follows the same non-negotiable boundaries as `docs/architecture.md`:

- CADOps remains the center of mutation.
- `cad-core` owns document source, topology anchor records, commandability,
  validation, reference health, rebuild planning, semantic diffs, and
  diagnostics.
- geometry-worker/OCCT may provide exact B-rep execution, topology evidence,
  checkpoints, signatures, and derived runtime results.
- React and the renderer may display selections, topology status, command
  affordances, and diagnostics, but they must not infer CAD commandability from
  meshes or visual state.
- agents and MCP stay thin wrappers over the same CADOps/query paths used by
  the UI.
- renderer IDs, mesh IDs, OCCT handles, checkpoint-local topology IDs, GPU
  IDs, selection-buffer values, viewport pixels, OPFS paths, file handles,
  local paths, and export artifact IDs must never become public stable CAD
  references.
- unsupported, stale, ambiguous, consumed, missing, deleted, low-confidence, or
  repair-needed states return structured diagnostics. They must not silently
  retarget commands.

V14 may add source data only when a command needs source truth that cannot be
reconstructed from existing V18 topology identity records and command history.
When existing V18 topology anchor/checkpoint records are enough, V14 should
reuse them.

## Command Target Promotion Policy

V14 should make normal modeling actions feel direct, but it must not create
topology source records silently in the background.

The policy is:

- passive selection, hover, inspection, measurement, project load, and ordinary
  save/open must not mint topology anchors or checkpoints;
- readiness queries may report that a selected target needs promotion,
  checkpoint evidence, or repair before a modeling command can consume it;
- if an explicit user modeling action needs a topology anchor/checkpoint first,
  the app may ask cad-core for a non-mutating promotion/readiness plan and then
  commit the required checkpoint/anchor/repair CADOps plus the modeling command
  through the normal command layer;
- automatic command-time promotion is allowed only as part of that explicit
  user/agent command transaction and only when cad-core can prove the target
  unambiguously;
- low-confidence, ambiguous, split, merged, deleted, or repair-needed targets
  must require explicit repair or remain blocked.

This preserves the V13 rule that ordinary save/open does not silently mint
anchors while still allowing V14 modeling commands to create required source
records as part of an intentional command.

## Result Outcomes

### 1. Topology Anchors Become Command Targets

V13 topology anchors are stable references. V14 should make active, healthy,
supported topology anchors usable as command targets where validators and
runtime execution can prove the path.

Supported command-target classes should include:

- active topology body anchors for downstream boolean target bodies;
- active planar face anchors for sketch attachment;
- active eligible edge anchors for chamfer and fillet;
- named references that resolve to those active topology anchors;
- selected references that can be explicitly promoted or repaired into those
  anchor-backed command targets.

The exact request and response shapes should follow existing protocol patterns.
The important outcome is that UI, agents, and MCP all ask the same cad-core
question: "is this topology-backed reference command-ready for this operation?"

### 2. Chained Boolean Modeling Works For Supported Result Bodies

The main V14 product test is chained downstream modeling:

```text
rectangle/circle new body
-> sketch on face
-> cut or add
-> sketch on result face
-> cut or add again
```

V14 should support this for the source families and topology proof levels that
cad-core and the geometry boundary can defend. It should not rely on the
viewport looking correct. The command path must prove:

- the target body is active and not consumed;
- the target body reference resolves through document source and/or a healthy
  topology anchor;
- the sketch plane is replayable from source/topology proof;
- the operation mode and tool sketch are supported;
- the geometry-worker execution path can rebuild the result;
- the resulting body lifecycle, dependency graph, reference health, and
  semantic diff are coherent.

Unsupported target families remain blocked with structured diagnostics. A
blocked result is acceptable; vague or milestone-based copy is not.

### V14 Boolean Support Matrix

V14 implementation must keep a precise support matrix instead of relying on
phrases like "rectangle/circle workflows" or "eligible target."

By V14 completion, the mandatory downstream boolean target set is:

| Target/source chain | Tool profile | Operation | Required outcome |
| --- | --- | --- | --- |
| Active rectangle `newBody` extrude or supported topology-backed rectangle-family result body | Rectangle attached sketch | Cut | Command-ready when topology/body/sketch proof is healthy |
| Active circle `newBody` extrude or supported topology-backed circle-family result body | Rectangle attached sketch | Cut | Command-ready when topology/body/sketch proof is healthy |
| Active rectangle `newBody` extrude or supported topology-backed rectangle-family result body | Rectangle attached sketch | Add | Command-ready when topology/body/sketch proof is healthy |
| Supported topology-backed circle-family result body | Circle attached sketch | Add | Command-ready when topology/body/sketch proof is healthy |
| Supported V12/V13 circle-tool result path | Circle attached sketch | Cut/add | Command-ready only where current cad-core validation and geometry runtime already prove support |

Direct unanchored circle-target add, circle-tool add beyond the proven
topology-backed/V12/V13 paths, curved-face sketch planes, arbitrary profile
recognition, and unsupported target/tool combinations must stay structured
blocked states until deliberately implemented and documented.

The first V14 implementation tranche should verify this matrix against current
cad-core behavior and update this document if the code proves a row is broader
or narrower than stated. UI affordances must be derived from the verified
cad-core matrix, not from the visible body shape.

### 3. Sketch-On-Result-Face Is A Real Source Workflow

V13 introduced a narrow topology-anchor face attachment path. V14 should make
that path usable in normal modeling flows for supported result-body planar
faces.

The persisted sketch attachment must be replayable. It should store public
source/topology proof only, such as topology anchor/checkpoint references and
validated planar frame evidence. It must not store checkpoint-local face IDs,
OCCT handles, mesh indexes, renderer IDs, or viewport data.

Implementation may reuse existing generated-face attachment records, extend
the V18 topology-anchor attachment shape, or add a narrowly scoped source
shape if needed. The release doc does not prescribe that choice. The outcome is
that downstream feature validation and rebuild can re-create the sketch plane
without treating the renderer as authority.

### 4. Edge Finishing Works On Proven Result Edges

V14 should extend chamfer and fillet commandability from source-semantic
generated edges into topology-backed result edges where the target is proven.

The minimum useful result is:

- active result-body edges that are stable, healthy, and eligible can be used by
  chamfer/fillet;
- named references to those edges work through the same command path;
- consumed, stale, ambiguous, curved-unsupported, low-confidence, missing, or
  repair-needed edges are blocked with structured diagnostics;
- committed features produce semantic diffs and transaction history that
  preserve topology-anchor provenance.

The exact matching or validator strategy should be chosen during
implementation. The doc requires proof and boundary safety, not a specific
algorithm.

At minimum, V14 should make one real result-edge class command-ready rather
than leaving "eligible" undefined: active axis-aligned linear edges on
V14-supported result bodies, backed by topology anchor evidence and validated
by cad-core against exact snapshot descriptors and command/runtime support.
Curved edges, circular cap edges, seam edges, and ambiguous/split/replaced
edges remain blocked unless a later V14 tranche proves them explicitly.

### 5. Rebuild And Reference Health Stay Honest

V14 should not just make a command run once. It should keep the document
understandable after changes.

After supported chained operations and upstream edits, these query surfaces
should agree:

- project structure;
- reference health;
- dependency graph;
- rebuild plan/body lifecycle;
- feature editability;
- selection/reference candidates;
- transaction history;
- project health;
- topology identity readiness.

Healthy topology-backed references should remain active when matching proves
survival. Replaced/split/merged/ambiguous/deleted/low-confidence cases should
be explicit repair or blocked states. Automatic retargeting is only allowed
when the implementation can prove it is unambiguous and the docs are updated
to describe that policy. Silent low-confidence retargeting is out of scope.

### 6. Product UI Stops Exposing Internal Milestones

V14 should clean up user-facing topology and commandability copy while adding
the new workflows.

The UI should say what the user can do:

- "Ready for sketch, cut, add, measure"
- "Select a planar face to create a sketch"
- "This edge is not eligible for fillet"
- "Reference needs repair"
- "Target body has been consumed"

The UI should not say:

- tranche names;
- release numbers as explanations;
- "this was deferred in V10/V13";
- implementation milestone labels;
- private topology/checkpoint IDs;
- raw command internals as primary copy.

Detailed diagnostics can remain inspectable, but normal modeling surfaces
should stay compact and action-oriented.

## Suggested Implementation Slices

These slices are guidance, not a mandated order. If implementation evidence
shows a better sequence, update this document and proceed with the safer path.

### Slice A: Command Target Readiness Contract

Status: implemented.

Add or refine shared query/protocol support for topology-backed command target
readiness. The query should answer which operations are available for a
selected topology-backed body, face, edge, named reference, or generated
reference, and why unsupported targets are blocked.

Expected proof:

- typed request/result/diagnostic shapes;
- cad-core derives readiness from document source and topology records;
- UI, agent, and MCP wrappers stay thin;
- no renderer/kernel/cache/file IDs leak into public references.

Implemented proof:

- `topology.commandTargetReadiness` accepts body, generated-reference,
  named-reference, and topology-anchor targets, an optional desired operation,
  optional topology match results, and optional topology-anchor snapshot proof.
- Responses report `ready`, `needs-promotion`,
  `needs-checkpoint-evidence`, `needs-repair`, blocked, missing, stale,
  ambiguous, consumed, unsupported, and non-commandable states with supported
  operation summaries and structured selection/topology diagnostics.
- The operation vocabulary now includes `feature.extrudeCutTarget`,
  `feature.extrudeAddTarget`, and `feature.holeTarget` for topology-backed
  body target readiness, while existing face/edge/name/measure/inspect
  operations remain unchanged.
- Active topology body anchors can report downstream cut/add/hole target
  readiness; active planar topology face anchors can report sketch-on-face
  readiness; and active axis-aligned linear topology edge anchors can report
  chamfer/fillet readiness where existing cad-core proof supports them.
- The query is non-mutating: passive selection/readiness does not mint
  anchors, checkpoints, or repairs, does not change `web-cad.project.v18`, and
  does not change `.wcad` package layout.
- Agent and MCP pass-through use the same cad-core query and preserve private
  ID boundaries.

### Slice B: Topology-Backed Sketch Attachment Product Path

Status: implemented.

Make supported planar topology-anchor faces usable from current selection and
viewport/modeling flows for sketch creation.

Expected proof:

- replayable source attachment;
- downstream validation understands the attachment;
- `.wcad` save/open preserves required topology identity;
- unsupported face types remain blocked with clear diagnostics.

Implemented proof:

- `sketch.createOnFace` accepts topology-anchor face targets through the normal
  CADOps path. Source-semantic topology anchors resolve to existing replayable
  generated-face attachments; exact-only planar face anchors can commit a
  `topologyAnchorFace` attachment when the caller supplies public
  `axisAlignedPlanarFace` proof from checkpoint evidence.
- App command helpers, viewport contextual commands, Inspector, and the
  Modeling panel carry selected topology anchor IDs into the create-sketch
  form when the user has an explicit stable topology reference selected.
- The web command form can carry public topology-anchor proof to CADOps without
  exposing renderer, mesh, OCCT, checkpoint-local, GPU, OPFS, file, local path,
  viewport, or export artifact IDs.
- `.wcad` save/open preserves `topologyAnchorFace` sketch attachments through
  the existing `web-cad.project.v18` / `partbench.wcad.v2` topology identity
  foundation without a schema or package migration.
- Current viewport/modeling UI support is limited to anchor-backed generated
  face selections. Arbitrary exact-only topology faces still require a caller
  that can supply validated checkpoint proof and are not exposed as a new
  viewport picking class in this slice.

### Slice C: Chained Cut/Add On Result Bodies

Status: partially implemented for topology-anchor `feature.extrude` add/cut
targets and topology-anchor `feature.hole` target bodies.

Make `feature.extrude` cut/add consume supported active topology-backed result
bodies as target bodies.

Expected proof:

- second cut/add after a first cut/add works for supported rectangle/circle
  workflows;
- target consumed/active lifecycle is correct;
- upstream edit/rebuild behavior is coherent;
- failed paths do not mutate source.

Implemented proof:

- Active body topology anchors now resolve forward through prior add/cut
  extrude features that consumed the same anchor, and active topology body
  anchors on supported result bodies can target that result body directly. A
  second rectangle cut can target the first result body through
  `targetTopologyAnchorId` while retaining a public resolved `targetBodyId` in
  the committed feature.
- Cad-core command validation, project health, generated-reference creation,
  and project import validation share the same conservative support rule:
  source body targets remain supported as before, and topology-backed result
  targets must trace back through the same active body anchor to a supported
  original rectangle/circle target.
- `selection.referenceCandidates`, `reference.health`,
  `topology.anchorCommandReadiness`, and `topology.commandTargetReadiness` now
  use the same target-family support matrix as command validation. Circle-origin
  topology body anchors stay cut/add/hole-ready where supported, while
  unsupported unanchored or non-commandable targets remain blocked instead of
  being over-advertised and failing later.
- The Sketch panel target list can surface active topology-backed result bodies
  and submits anchor-backed CADOps without mixing `targetBodyId` and
  `targetTopologyAnchorId`; unanchored result bodies remain absent from the
  target list.
- App-level workflow proof now composes promoted result-body face sketches for
  rectangle and circle profiles submitted as topology-backed `feature.extrude`
  add/cut commands where the support matrix permits them, plus circle profiles
  submitted as `feature.hole`. The browser smoke proves rectangle-family
  result-body second cuts, circle-family result-body rectangle cuts,
  rectangle-family result-body adds, result-face holes, and explicit side-plane
  circular holes while preserving the replayable face attachment, target
  anchor, project health, `.wcad`, and JSON source records. Focused
  core/app/geometry tests and deterministic release samples prove
  circle-profile `feature.extrude` cuts on supported rectangle/circle result
  targets and circle-profile adds on supported topology-backed circle-origin
  result targets.
- `feature.hole` can target supported active topology-backed result bodies
  through the same body-anchor proof path, and `feature.holeTarget` readiness
  is now advertised through the shared `selection.referenceCandidates` /
  `topology.commandTargetReadiness` path. Cad-core stores the resolved public
  `targetBodyId` plus `targetTopologyAnchorId`, keeps unsupported, stale,
  missing, ambiguous, consumed, and repair-needed anchors blocked with
  structured diagnostics, and preserves the source boundary through project
  structure, health, dependency graph, import/export, generated references,
  topology source identity, app command helpers, derived geometry, and exact
  metadata. Generated references and JSON import validation also accept the
  same active result-body anchor proof so replayed add/cut result bodies remain
  understandable after save/open.
- Agent-adapter and MCP batch proof now covers the same topology body-anchor
  add and hole commit path, verifying that external callers receive the
  resolved public target body plus anchor proof through project feature and
  health queries without gaining adapter-only semantics.
- A deterministic `smoke:v14-release-samples` runner now covers the command,
  readiness, lifecycle, JSON/`.wcad`, and source-boundary proof for supported
  result-body cut/add/hole target paths, explicit circle side-plane holes, and
  the rectangle cut-wall result-edge chamfer subset through both named
  generated references and direct topology edge anchors without relying on
  browser or geometry-worker state.
- JSON import replay can seed saved source topology identity when the topology
  records are source state rather than transaction-created records, preserving
  anchor-backed feature history without storing renderer, mesh, kernel,
  viewport, file, or checkpoint-local identifiers in feature source.
- Explicit side-plane circular holes are supported for the proven XZ sketch
  workflow against a supported circle-origin topology-backed result body. The
  app keeps this as a normal `feature.hole` command with a public sketch plane
  and body topology anchor; it does not make curved circular side faces
  sketch-attachable or infer tangent workplanes from viewport picks.
- UI-created holes now run the same derived-geometry runtime conversion as the
  display pipeline against a projected document before committing source when
  derived geometry is enabled. Runtime failure blocks the UI command with
  action-oriented copy and leaves the authoritative document unchanged.
- Current support is still limited to `feature.extrude` add/cut where the
  verified rectangle/circle target/tool matrix permits it, `feature.hole`
  result-body target chaining, and the Slice D rectangle cut-wall longitudinal
  edge-finish path. Curved side-wall face attachment, arbitrary profile
  recognition, direct unanchored circle-target add, and broader
  topology-backed edge finishing remain later V14 work.

### Slice D: Topology-Backed Edge Finish

Status: partially implemented for active topology-anchor generated edges on
supported active rectangle/circle new-body extrude targets and for rectangle
cut-wall longitudinal generated result edges on supported cut result bodies.

Make eligible topology-backed result edges usable for chamfer and fillet.

Expected proof:

- supported result edges can be chamfered/filleted;
- command validators prove the target;
- geometry-worker remains the execution boundary;
- unsupported edge classes remain structured diagnostics.

Implemented proof:

- The web product path now carries an enriched selected topology edge anchor
  from viewport/modeling selection into the shared edge-finish CADOps form, so
  Chamfer/Fillet commands submit `topologyAnchorId` instead of renderer,
  mesh, viewport, or generated-ID-only state.
- `feature.chamfer` and `feature.fillet` command builders preserve sanitized
  public `topologyAnchorProof` when supplied with a topology edge anchor, while
  cad-core remains the validator that proves the edge and resolves the
  generated edge used by geometry execution.
- The edge-finish UI no longer applies a stale rectangle-only product gate for
  circle new-body edge targets; unsupported, consumed, stale, missing, and
  non-commandable selections still block through structured readiness and
  validation diagnostics.
- Rectangle cut-wall profile edges with stable IDs such as
  `generated:edge:<body>:longitudinal:uMin:vMin` now advertise
  `feature.chamfer` and `feature.fillet` through generated references,
  selection/reference candidates, and reference health. The viewport/modeling
  flow routes those edges through the shared CADOps edge-finish form.
- The V14 browser workflow now selects command-ready rectangle cut-wall
  longitudinal generated edges, saves named references for those edges, creates
  both chamfer and fillet features through the Modeling panel using the
  named-reference option, and verifies the exported JSON keeps only public
  `targetBodyId`, `namedReference`, and named-reference generated-edge target
  source data.
- Cad-core accepts those active cut-wall longitudinal edges as chamfer/fillet
  targets through direct generated-edge and named-reference command sources,
  records the public replay source, preserves dry-run/no-mutation behavior, and
  emits normal semantic diffs for committed result bodies. Project health now
  reports the same rectangle cut-wall result-edge chamfer/fillet features as
  healthy instead of applying the older source-body-only edge-finish gate.
- Derived geometry, exact metadata, geometry-kernel validation, and the OCCT
  WASM adapter now pass the public boolean target source tree for this
  edge-finish path. The replay source remains command/document topology data,
  not renderer IDs, mesh IDs, checkpoint-local IDs, OCCT handles, viewport
  data, OPFS paths, or file handles.
- Result-edge support is intentionally limited to rectangle cut-wall
  longitudinal edges in this slice. Curved edges, circular cap/rim edges, seam
  edges, ambiguous or repaired topology, and unsupported target/tool chains
  remain blocked with structured diagnostics.
- Edge-finish readiness/status copy now uses product wording for selected
  edges, active result bodies, downstream results, and new result creation
  instead of generated-reference internals, raw body/feature IDs, or derived
  geometry wording.

### Slice E: Reference Repair And Rebuild Hardening

supported result-edge chamfer chains and supported result-body hole chains.

Harden repair/rebuild behavior after chained operations and upstream edits.

Expected proof:

- references survive when matching proves survival;
- replacement/ambiguous/deleted states are clear;
- repair actions remain explicit;
- project/query surfaces agree after undo/redo and save/open.

Implemented proof:

- Cad-core and the web Inspector now agree when an upstream source extrude is
  blocked by a downstream result-body chamfer chain: the structured
  `FEATURE_EDIT_CONSUMED_BODY` diagnostic names the downstream consuming
  feature, blocks mutation, and preserves source/history/query state through
  JSON and `.wcad` round trips.
- The required V14 browser workflow now selects the original source after the
  result cut-wall chamfer path and proves the edit controls stay disabled with
  action-oriented repair guidance rather than internal milestone/debug copy.
- The V14 browser workflow also selects the original source after the
  result-body hole `.wcad` round trip and proves the same disabled edit controls
  and action-oriented downstream hole diagnostic survive save/open.
- Cad-core now explicitly proves that the supported result-edge chamfer chain
  keeps project structure, project health, dependency graph, rebuild plan,
  reference health, selection candidates, topology command-target readiness,
  feature editability, and transaction history aligned across undo, redo, JSON
  round trips, and `.wcad` round trips. The deterministic release smoke also
  proves the same rectangle cut-wall edge subset through a direct public
  topology edge anchor.
- Cad-core also proves the supported result-body hole chain blocks upstream
  source edits without mutating source/history/query state, and preserves the
  hole target proof across JSON and `.wcad` round trips.

## Storage And Schema Position

V14 should prefer the existing V13 storage foundation:

- `web-cad.project.v18` topology identity source records;
- `partbench.wcad.v2` checkpoint payload preservation;
- V18 checkpoint metadata, topology anchors, repair records, and topology
  identity settings;
- command history and semantic diffs.

Do not introduce a new project schema only because a query result changes, a
diagnostic changes, or a derived topology snapshot gains more evidence.

A new schema after V18 is justified only if V14 adds new source-of-truth data
that cannot be represented by current V18 records and command history. If that
happens, the implementation must:

- document the exact new source record;
- migrate older projects explicitly;
- keep `.wcad` package semantics lossless;
- avoid storing raw kernel, renderer, mesh, GPU, selection-buffer, OPFS, file,
  local path, viewport, or export artifact identifiers;
- update `docs/native-format.md`.

JSON remains explicit import/export for source interchange. `.wcad` remains
the lossless native project workflow for checkpoint-backed topology identity.

- split;
- merged;
- low-confidence;
- repair-needed;
- runtime-unavailable;
- rebuild-failed;
- derived-rebuild-pending.

User-facing copy should explain the current action problem, not the internal
release history. For example:

```text
This face needs repair before it can host a sketch.
```

is better than:

```text
V13 does not claim command-ready arbitrary topology for this face.
```

## Non-Goals

V14 should not include these unless explicitly re-scoped:

- STEP import or imported-body healing;
- assemblies, mates, instance selection, or large-assembly LOD;
- production WebGPU renderer or selection-buffer replacement;
- broad new solid feature families such as shell, sweep, loft, pattern,
  mirror, or direct edit;
- parameter expressions;
- hosted collaboration;
- natural-language command entry;
- arbitrary automatic low-confidence topology retargeting;
- proprietary CAD import/export;
- treating renderer, mesh, OCCT, GPU, selection-buffer, OPFS, path,
  file-handle, export, or viewport identifiers as public stable CAD IDs.

V14 may touch geometry-worker/OCCT to support required exact downstream
execution, but only through isolated geometry boundaries.

## Definition Of Done

V14 is done when:

1. Supported topology-backed result bodies can be used for chained downstream
   cut/add workflows.
2. Supported topology-backed planar faces can host replayable attached
   sketches.
3. Supported topology-backed result edges can drive chamfer/fillet where
   validators and runtime execution prove the target.
4. cad-core remains the authority for commandability, validation, source
   mutation, semantic diffs, reference health, and rebuild state.
5. React and renderer surfaces consume cad-core/query results and do not infer
   topology stability from visual geometry.
6. `.wcad` save/open preserves the topology identity needed by supported V14
   workflows.
7. JSON import/export either preserves source safely or reports loss of
   checkpoint payloads clearly.
8. User-facing UI does not mention internal milestone/tranche limitations as
   explanations.
9. Unsupported, stale, consumed, ambiguous, missing, deleted, low-confidence,
   or repair-needed cases return structured diagnostics and do not mutate
   source.
10. Agent and MCP surfaces use the same cad-core command/query paths as the UI.
11. Focused unit tests cover protocol, cad-core command validation, semantic
    diffs, rebuild/reference health, storage boundaries, adapter pass-through,
    and source/derived/session separation.
12. Browser smokes cover at least one long chained modeling workflow through
    create, select, sketch-on-result-face, cut/add again, edge finish where
    supported, upstream edit, save/open, and no console errors.
13. `pnpm test`, `pnpm typecheck`, `pnpm lint`, `pnpm format:check`, and
    `pnpm build` pass.

## Release Acceptance Workflows

The release should include deterministic and browser-level proof for workflows
like these:

### Chained Cut Workflow

```text
rectangle sketch
-> extrude new body
-> select planar face
-> create attached rectangle sketch
-> cut body
-> select planar face on result body
-> create second attached rectangle sketch
-> cut result body again
-> measure/inspect final body
-> save/open .wcad
-> verify references and body lifecycle remain coherent
```

### Result Edge Finish Workflow

```text
create body
-> cut/add to create result body
-> select eligible result edge
-> chamfer or fillet
-> verify final result body and reference health
-> edit an upstream supported parameter
-> rebuild or show explicit repair-needed diagnostics
```

### Repair Workflow

```text
create stable reference
-> change upstream supported source
-> match topology
-> show active/replaced/ambiguous/deleted state
-> explicitly repair when safe
-> prove no silent low-confidence retargeting
```

## How To Use This Document

Implementation prompts should pick one coherent slice, keep it independently
testable, and update this document if implementation evidence changes the
best path.

Do not treat suggested slice names or internal helper ideas as hard
requirements. Treat the product outcomes, source/derived boundaries,
diagnostics, and proof requirements as the contract.
