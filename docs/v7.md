# V7 Release Draft: Real CAD Alpha

Status: draft, active planning.

V7 is a major release target, not a single narrow implementation milestone.
V6 completed the practical solid-modeling milestone: constrained sketch
workflows, extrude, revolve, hole, chamfer, fillet, derived OCCT meshes, derived
exact metadata, topology/measurement health, compact modeling workflows, and
agent/MCP pass-through over CADOps. V7 should turn that foundation into a more
complete local CAD product.

V7 should be implemented through scoped tranches, but the release itself is
broad. Each tranche should be independently testable and should preserve the
core architecture:

```text
authoritative CADOps document -> derived exact geometry/cache ->
stable references and selection -> local project workflow ->
browser-native CAD product surfaces -> release hardening
```

Important naming note: V7 release planning is not the same as
`web-cad.project.v7`, which is an older saved-project schema identifier for V3
parameter and sketch-dimension source data. If V7 release work adds new
source-of-truth project data beyond current `web-cad.project.v16`, the next
saved schema should be `web-cad.project.v17` with explicit migrations.

## Product Goal

A user can build and revise a small mechanical part in the browser with a
credible CAD workflow:

```text
create/open project -> sketch/constrain/dimension ->
extrude/revolve/hole/chamfer/fillet -> select meaningful faces/edges ->
inspect and repair references/health -> save locally -> export/share
```

A structured agent can operate on the same project through CADOps:

```text
query structure/topology/health -> resolve stable references ->
dry-run edits -> inspect semantic diffs and diagnostics ->
commit auditable transactions -> preserve user-visible model intent
```

The release should make Partbench feel less like a set of proven engine slices
and more like an early usable CAD application.

## Architectural Rules

V7 must keep these rules:

- CADOps remains the center of all mutations.
- Human UI, scripts, tests, and agent/MCP adapters use the same command/query
  layer.
- `cad-core` remains authoritative for source document state.
- Rendered meshes, exact-kernel results, topology snapshots, thumbnails, and
  caches remain derived unless a tranche deliberately adds persisted source
  records.
- React UI does not import geometry internals directly.
- Renderer, geometry-kernel, geometry-worker, protocol, storage, and MCP
  boundaries remain separate.
- MCP remains an adapter over CADOps and query paths.
- Generated references must not expose raw OCCT, mesh, renderer, or selection
  buffer indexes as stable IDs.
- Ambiguous topology or stale references fail with structured diagnostics
  instead of silently targeting the wrong entity.
- New project formats are introduced only for new source-of-truth data.
- V7 implementation tranches must include focused tests and should not rely on
  browser E2E tests as the only durable signal.

## Release Pillars

### Stable Topology, References, And Selection

V6 intentionally reports ambiguous topology for many result bodies. V7 should
make reference-consuming workflows safer and more useful.

Release outcomes:

- durable face/edge/vertex reference model for a larger but still explicit set
  of supported bodies;
- selection that resolves to semantic references, not mesh or renderer indexes;
- named reference creation, validation, stale detection, and repair flows;
- topology health that explains unsupported, stale, ambiguous, kernel-failed,
  and repaired states;
- CADOps/query support for reference resolution, eligibility, diagnostics, and
  dry-run feature targeting;
- UI integration with the existing feature tree and modeling workflow so
  selected references are inspectable and usable for commands.

Likely first implementation tranche:

- selected generated edges/faces for simple extrude bodies and the most
  defensible V6 result subsets;
- a structured selection/reference query contract shared by UI and agents;
- feature tree and inspector affordances for stale or ambiguous references;
- unit/package coverage for reference stability across upstream edits.

### Production Viewport And Interaction

The current canvas viewport is sufficient for milestone work. V7 should move
toward a production CAD viewport without letting rendering become source of
truth.

Release outcomes:

- renderer-level selection and hover feedback that maps back to semantic
  references through query-layer data;
- clearer visual states for active bodies, consumed bodies, pending derived
  geometry, failed derived geometry, and selected references;
- measurement and inspection overlays driven by CADOps/query results;
- viewport performance baselines for representative V7 sample parts;
- a WebGPU path if explicitly scoped after the reference model is ready.

The WebGPU renderer should be a release pillar, not a prerequisite for every
other V7 tranche. If it is started, it must stay behind the renderer boundary
and preserve the same derived mesh/reference contract.

### Native Project Workflow

V7 should make local project handling feel like product behavior rather than a
debug JSON panel, while preserving the current source-of-truth model.

Release outcomes:

- clear open/save/save-as workflow for current project JSON and future native
  package work;
- explicit decision on when to introduce File System Access and OPFS;
- project metadata, thumbnails, and recovery behavior if scoped;
- migration rules for any new source records;
- no persisted exact metadata, OCCT topology IDs, or tessellated mesh caches
  unless a specific storage tranche deliberately makes them rebuildable cache
  artifacts outside source truth.

If a native `.wcad` package begins in V7, it should be introduced as a storage
milestone with its own format tests, migration tests, and cache/source
separation checks.

### Interop

STEP/IGES import/export remains deferred after V6, but V7 is the first release
where interop can reasonably become product-facing if scoped.

Release outcomes:

- STEP export for supported current bodies before broad STEP import;
- GLB or lightweight visualization export only as display output, never as
  source truth;
- clear unsupported diagnostics for bodies/features that cannot export yet;
- geometry-kernel/worker boundaries that keep OCCT exchange APIs isolated;
- tests around units, body identity, and unsupported feature reporting.

STEP import is higher risk than export and should be a later V7 tranche unless
the release explicitly prioritizes interoperability over modeling breadth.

### Product UI Hardening

The app now has a first feature tree and improved modeling workflow. V7 should
harden and deepen those surfaces instead of treating UI as a blank slate.

Release outcomes:

- feature tree actions that correspond to CADOps commands or explicit queries;
- inspector depth for bodies, sketches, features, references, health,
  measurements, and transactions;
- coherent selection state between viewport, feature tree, inspector, and
  modeling panels;
- workflow affordances for reference eligibility, target body state, consumed
  body chains, and unsupported commands;
- command history and semantic diffs that humans can read and agents can audit;
- keyboard and command-palette ergonomics if scoped;
- browser verification of representative workflows as a release checklist.

### Agent And MCP Productization

V6 proves agent/MCP pass-through. V7 should make the agent surface useful for
larger workflows and safer for real projects.

Release outcomes:

- stable project summary, query, batch, measurement, topology, and reference
  tools;
- dry-run review data shaped for human approval and automated callers;
- permission/audit concepts for destructive or broad agent actions;
- semantic diffs that remain compact but explain meaningful CAD consequences;
- examples that demonstrate an agent creating, editing, inspecting, and
  repairing a small part without visual guessing.

### Release Hardening

V7 should add repeatable release confidence beyond unit/package checks.

Release outcomes:

- representative sample projects for bracket-like, revolved, hole, and
  edge-finished workflows;
- smoke checks for OCCT worker load, derived geometry, metadata, and export
  paths;
- manual browser release checklist for the upgraded modeling workflow;
- docs for supported and unsupported behavior;
- performance and bundle-size tracking where it affects product decisions.

## Candidate Implementation Tranches

These tranches are intentionally ordered by dependency. The release can adjust
order, but each tranche should keep a clear scope and definition of done.

### Tranche A: Reference And Selection Contract

Goal: define the semantic selection/reference contract that V7 UI, agents, and
future renderer work can share.

In scope:

- typed CADOps/query shapes for selection candidates and reference resolution;
- body/feature/reference eligibility summaries;
- stable references for the supported simple-body subset already defensible from
  source semantics;
- diagnostics for stale, ambiguous, missing, unsupported, and consumed targets;
- UI helpers that map current tree/viewport selections to command-ready
  references;
- adapter/MCP wrappers over the same query layer;
- tests proving no raw mesh, OCCT, or renderer indexes become stable public IDs.

Out of scope:

- broad arbitrary topology naming;
- full selection-buffer implementation;
- WebGPU renderer replacement;
- STEP import/export;
- native package storage.

Acceptance:

- a selected supported face/edge can be represented as a semantic reference;
- unsupported selections produce structured diagnostics;
- named references can be validated after upstream edits;
- agents can query the same reference eligibility that the UI uses.

Initial implementation note:

- `selection.referenceCandidates` is the first query-level contract. It accepts
  semantic body, generated-reference, and named-reference selections and returns
  command-ready generated-reference candidates when the current authoritative
  document can prove them.
- The supported subset is intentionally narrow: authored rectangle/circle
  `newBody` extrude bodies and generated references already derived from that
  source semantics. Boolean, revolve, hole, chamfer, and fillet result topology
  remains ambiguous or unsupported unless a later tranche proves stable
  semantics.
- Consumed source bodies can still resolve semantically when their authored
  generated references are available, but candidates are marked
  non-commandable with a consumed-body diagnostic.
- The contract is query-only/derived and does not introduce a new
  `web-cad.project.v17` schema.

### Tranche B: Feature Tree And Inspector Integration

Goal: connect the upgraded feature tree and modeling workflow to V7 reference
and health semantics.

In scope:

- tree rows for sketches, features, bodies, consumed result chains, generated
  references, and named references;
- inspector panels for topology health, exact metadata confidence, dependency
  health, generated-reference measurements, and selection/reference candidate
  diagnostics;
- command-ready target summaries for selected semantic bodies, generated
  references, and named references;
- stale, ambiguous, missing, unsupported, consumed, and non-commandable
  reference presentation;
- command wiring for safe tree/modeling actions such as target selection,
  generated-reference naming, sketch-on-face creation, and chamfer/fillet edge
  affordances where already supported;
- focused UI-helper/component tests and cad-core/query tests for any new
  command/query behavior.

Out of scope:

- wholesale redesign of the modeling workflow;
- UI-only source state;
- feature operations that bypass CADOps.

Acceptance:

- tree, inspector, and modeling panels agree on selected entity identity;
- supported selections display command-ready state from
  `selection.referenceCandidates`;
- unsupported actions are not offered as valid commands and explain the
  structured diagnostic that blocked them;
- health and reference diagnostics are visible without reading raw JSON.

Initial implementation note:

- The feature tree, inspector, and modeling actions now consume
  `selection.referenceCandidates` responses for current semantic UI selections:
  selected body rows, generated-reference rows, selected generated references,
  and named-reference rows.
- React formats and routes query responses only. Commandability stays derived
  from `cad-core`/CADOps candidate results and existing command builders.
- Reference-consuming UI flows prefer the candidate result where it applies:
  sketch-on-face options, generated-reference naming, and chamfer/fillet edge
  affordances are blocked by structured diagnostics for consumed, stale,
  ambiguous, unsupported, missing, or non-commandable selections.
- This tranche remains query-only/derived. It does not add viewport picking,
  selection buffers, new modeling commands, storage migration, or
  `web-cad.project.v17`.

### Tranche C: Viewport Selection And Visual State

Goal: make viewport interaction precise enough for CAD workflows while keeping
the renderer derived.

In scope:

- hover/selection feedback for supported bodies and references;
- visual states for active, consumed, hidden, failed, pending, and selected
  derived geometry;
- viewport-to-reference query plumbing;
- measurement/inspection overlays driven by query data;
- performance baseline fixtures for representative V7 sample parts.

Out of scope:

- using mesh triangle IDs as durable references;
- broad WebGPU rewrite unless separately accepted;
- large assembly renderer architecture.

Acceptance:

- clicking a supported reference selects the same semantic target the tree and
  agents can resolve;
- renderer state can be rebuilt from document/query state;
- failed or stale derived geometry is visible and diagnosable.

Initial implementation note:

- C1 connects the current viewport status overlay to the same
  `selection.referenceCandidates` contract already used by the feature tree,
  inspector, modeling workflow, agents, and MCP.
- The status overlay derives command-ready summaries, supported operation
  labels, structured diagnostics, consumed-body state, and derived geometry
  ready/pending/failed/fallback state from app/query state.
- Selected generated references and named references after resolution still
  highlight at body/object granularity in the current canvas viewport. Exact
  face/edge/vertex highlighting remains future Tranche C work.
- C2 adds current-canvas body/object pick intent in `apps/web`: renderer pick
  IDs are resolved against current project structure and scene objects, then
  body-backed selections consume `selection.referenceCandidates` status and
  structured diagnostics. Primitive object picks keep object-editing behavior
  while viewport status can report the object-backed body candidate diagnostic.
- C2 also adds a compact viewport-adjacent generated-reference action list for
  the selected supported body. The list is query-derived from
  `selection.referenceCandidates`; choosing an item is the only viewport
  generated-reference selection path in this slice.
- Renderer, mesh, OCCT, and future selection-buffer identifiers remain derived
  implementation details and are not promoted to public stable IDs.
- This slice is query-only/derived and does not introduce persistent selection
  state, exact face/edge/vertex viewport picking, WebGPU, new modeling
  commands, storage migration, or `web-cad.project.v17`.
- C3 adds transient current-canvas hover status for body/object render IDs using
  the same semantic resolution and `selection.referenceCandidates` query
  summaries as pick intent. Sketch display and unknown render targets produce
  structured unsupported or missing hover diagnostics and do not become
  generated-reference selections.
- C3 also adds a compact selected-target measurement/inspection overlay. Body
  rows come from `body.measurements`; generated-reference rows and structured
  missing/stale/unsupported text come from
  `body.generatedReferenceMeasurements` through the same app state used by the
  inspector.
- C3 includes one deterministic helper-level viewport baseline check for a
  rectangle extrude body with generated references and derived geometry status.
  The check records semantic body/reference IDs and labels only, excluding raw
  mesh, OCCT, cache, and selection-buffer implementation IDs.
- C4 consolidates C1-C3 status, reference actions, hover feedback, and
  measurement display into one compact viewport interaction surface. Selected
  target status remains primary; body/reference measurements attach to that
  selected summary; hover is subordinate and suppressed when it would duplicate
  the selected render target; reference actions are grouped, prioritized, and
  capped so they stay available without swamping the viewport.
- C4 keeps unsupported, missing, stale, consumed, and non-commandable states as
  structured diagnostics in the viewport surface and continues to keep raw mesh,
  OCCT, cache, and future selection-buffer identifiers out of visible text.
- C3/C4 remain query-only/derived and do not add exact face/edge/vertex picking,
  persistent hover or selection state, WebGPU, new modeling commands, storage
  migration, or `web-cad.project.v17`.

### Tranche D: Local Project Workflow

Goal: make save/open behavior credible for local product use.

In scope:

- current project JSON open/save polish;
- migration display and validation diagnostics;
- project metadata and thumbnails if stored as cache/display data;
- explicit File System Access and OPFS decisions;
- format tests if new source data or package metadata is introduced.

Out of scope:

- STEP/IGES import/export;
- hosted collaboration;
- persisted B-rep checkpoints unless explicitly scoped as source or cache.

Acceptance:

- users can save, reopen, and validate current V16 projects confidently;
- any new source records use `web-cad.project.v17` with migration tests;
- cache artifacts are clearly rebuildable and not required source truth.

Initial implementation note:

- D1 polishes the existing local JSON workflow in `apps/web`. The Project panel
  now summarizes the authoritative current export, draft source state, schema
  migration status, structured validation issues, replacement/history impact,
  and same-document-source detection before import.
- This slice remains ordinary `web-cad.project.v16` JSON import/export through
  cad-core. It does not add File System Access, OPFS, `.wcad` packages,
  thumbnails, cache storage, native metadata, or `web-cad.project.v17`.
- D2 makes the Project panel honest about local save/open capability. It reports
  ordinary JSON import/export as the active mode, detects whether the current
  browser exposes the existing download/upload primitives and File System
  Access picker functions, and displays OPFS, thumbnails, cache storage, and
  native `.wcad` packages as deferred.
- D2 is status-only for native storage. It does not call File System Access
  picker APIs, request file permissions, write OPFS data, create thumbnails or
  cache files, implement `.wcad` packages, or introduce
  `web-cad.project.v17`.

### Tranche E: Export And Sharing

Goal: provide the first product-facing interop path for supported models.

In scope:

- STEP export for supported exact bodies if OCCT exchange bindings are available
  in the chosen package/build;
- GLB or mesh export for visualization if useful;
- structured unsupported export diagnostics per body/feature;
- command/query or worker boundaries for export jobs;
- tests for units, empty/invalid bodies, unsupported results, and worker errors.

Out of scope:

- broad STEP import;
- proprietary CAD formats;
- making exported meshes authoritative.

Acceptance:

- a representative supported V7 sample part can export successfully;
- unsupported bodies fail with actionable diagnostics;
- export does not mutate source document state.

E1 implementation note:

- `project.exportReadiness` is the first Tranche E contract. It reports STEP
  and Mesh/GLB visualization formats as not file-exportable yet while exposing
  deterministic source-body readiness, units, source/derived boundary notes, and
  structured diagnostics.
- The query classifies candidate bodies from authoritative project structure:
  empty projects, active rectangle/circle `newBody` extrudes, consumed bodies,
  primitive compatibility bodies, and deferred V6 result bodies are
  distinguished without reading display output or export job state.
- Agent and MCP surfaces pass the same query result through CADOps. The Project
  panel displays the same readiness status for users.
- E1 does not implement STEP or GLB writers, export jobs, storage changes,
  import paths, OPFS/File System Access flows, or `web-cad.project.v17`.

E2 implementation note:

- the web app now provides the first product-facing visualization export path:
  `partbench-visualization.glb`, generated as a transient GLB display artifact
  from ready derived visualization meshes;
- the supported E2 subset is intentionally narrow: active authored rectangle
  and circle `newBody` extrude bodies whose current derived mesh entries are
  ready and valid. Empty projects, consumed bodies, primitive compatibility
  bodies, deferred V6 result bodies, missing/pending/stale/failed meshes, and
  invalid mesh buffers report structured diagnostics instead of attempting a
  vague export;
- the GLB writer is dependency-free and serializes existing app-level mesh DTOs
  into glTF 2.0 binary output. The artifact metadata reports visualization
  body counts plus vertex/triangle/byte counts, and user-visible summaries do
  not expose renderer, OCCT, cache, selection-buffer, or internal mesh IDs;
- `project.exportReadiness` remains the shared CADOps source-readiness
  contract used by agent and MCP pass-throughs. The Project panel overlays that
  source contract with app-derived mesh availability before enabling the GLB
  download action;
- STEP remains unavailable/deferred until an exact exchange writer binding
  exists. E2 does not add STEP/IGES import, OPFS/File System Access handles,
  native package storage, persisted export jobs, new modeling commands, or
  `web-cad.project.v17`.

### Tranche F: Agent/MCP Release Surface

Goal: make external automation useful for V7 product workflows.

In scope:

- project summary shaped around tree, health, references, and export readiness;
- dry-run previews for multi-step edits;
- audit metadata for agent transactions;
- examples and tests for agent-driven part creation, reference repair, and
  export-readiness checks;
- MCP wrappers that remain thin over agent-adapter/CADOps.

Out of scope:

- natural-language parsing as source behavior;
- production hosted auth;
- MCP-specific semantics that bypass the internal protocol.

Acceptance:

- an agent can perform a representative V7 workflow without inspecting pixels;
- human review can inspect the same semantic diff before commit;
- MCP output remains compact and deterministic.

#### Implemented Tranche F1: Agent/MCP Release Project Summary

F1 upgrades `project.summary` and `cad.project_summary` into a compact
release-oriented summary for agents:

- the legacy `units`, `objectCount`, and `objects` fields remain available for
  compatibility;
- new summary blocks report source-derived structure counts, project health
  status and issue count, semantic reference capability counts, export
  readiness counts/statuses, and concise workflow hints;
- reference capability counts are aggregate-only. They do not expose raw
  generated-reference, renderer, OCCT, mesh, or selection-buffer IDs;
- export readiness is still the shared source contract. The summary may report
  deferred STEP/GLB writers from `project.exportReadiness`, but browser-derived
  GLB artifact availability remains outside cad-core;
- no project JSON/schema change is introduced. The summary is query-derived and
  not persisted.

#### Implemented Tranche F2: Agent Workflow Preview And Audit Surface

F2 adds a compact deterministic review surface to agent/MCP batch execution:

- `cad.batch` responses include a `review` block with requested mode, effective
  intent, operation count, entity-change counts, operation labels, audit
  summary, commit-gate state, hints, and blockers;
- dry-runs remain previews only and do not mutate document state or create
  transaction ids. Commits still require explicit adapter permission;
- refused commits, validation errors, empty batches, destructive delete
  operations, and CADOps warnings produce structured review blockers or hints
  suitable for human inspection before commit;
- committed transactions preserve normalized audit metadata in transaction
  history;
- MCP and stdio transport stay thin over the agent-adapter/CADOps path. Review
  data summarizes request/result shape and does not introduce renderer, mesh,
  OCCT, cache, or selection-buffer identifiers as stable public IDs;
- no project JSON/schema change is introduced. Preview/review data is derived
  response data and is not persisted outside existing committed transaction
  audit metadata.

### Tranche G: Release Samples, Smokes, And Documentation

Goal: turn V7 from implemented slices into a repeatable release.

In scope:

- sample projects for each supported workflow;
- smoke scripts for derived geometry, exact metadata, project load/save, and
  export paths where implemented;
- manual browser checklist for feature tree, modeling workflow, selection,
  reference repair, and save/open;
- documentation updates for capabilities, limitations, and troubleshooting.

Acceptance:

- required checks pass;
- representative workflows are documented and manually verified;
- known limitations are explicit in docs and UI diagnostics.

#### Implemented Tranche G1: Release Samples And Acceptance Fixtures

G1 adds a typed, source-only release-sample fixture layer in `cad-core`:

- `V7_RELEASE_SAMPLE_FIXTURES` defines deterministic fixture metadata and
  CADOps batches for representative V7 acceptance projects;
- `createV7ReleaseSampleBatch(id)` lets tests or future release scripts build
  each sample through the existing CADOps batch path, then use ordinary current
  project JSON import/export for round-trip checks;
- current samples cover an authored rectangle `newBody` extrude with generated
  face/edge references and named references, an authored circle `newBody`
  extrude with generated references and export/reference readiness coverage,
  and a consumed-body diagnostics sample that documents non-commandable
  consumed source references plus ambiguous unsupported result topology;
- each fixture records release acceptance metadata: id, title, description,
  units, supported workflow tags, expected source counts, expected project
  health status/issue count, expected generated/named reference behavior,
  expected `selection.referenceCandidates` behavior, expected
  `project.summary` reference/export counts, expected `project.exportReadiness`
  source readiness, and known limitations;
- tests prove each sample imports/exports as current `web-cad.project.v16`,
  does not introduce `web-cad.project.v17`, reports expected structured health,
  resolves supported body/generated/named references, reports honest STEP and
  source Mesh/GLB readiness, appears in `project.summary`, and keeps raw
  renderer, mesh, OCCT, cache, and selection-buffer identifiers out of public
  sample metadata and project JSON.

G1 is an acceptance fixture layer, not browser automation. Manual browser
checks, smoke scripts, app-derived GLB availability, and broader sample coverage
remain separate Tranche G work.

#### Implemented Tranche G2: Release Sample Smoke Runner And Checklist Seed

G2 adds the first repeatable V7 release acceptance smoke over the G1 fixtures:

- `pnpm smoke:v7-release-samples` runs a deterministic non-browser CLI smoke;
- the smoke builds each fixture through `createV7ReleaseSampleBatch(id)`,
  exports/imports current project JSON, and verifies current schema only,
  `project.health`, `selection.referenceCandidates`, `project.exportReadiness`,
  `project.summary`, and source/derived separation;
- `scripts/v7-release-samples.mjs` exposes the pure helper used by tests, while
  `scripts/smoke-v7-release-samples.mjs` provides human output and optional
  `--json` stdout;
- `docs/v7-release-checklist.md` seeds the manual release checklist for feature
  tree/modeling workflow, generated/named references, viewport selection/status,
  project JSON workflow, and GLB download behavior.

G2 remains non-browser and query/source focused. Browser release checks are
documented but not automated yet, and no metrics, sample JSON, derived meshes,
topology caches, export artifacts, persistent selection state, or
`web-cad.project.v17` are introduced.

#### Implemented Tranche G3: Browser Release Workflow Smoke

G3 adds the first repeatable browser smoke for the implemented V7 product
workflow:

- `pnpm smoke:v7-browser-workflow` builds the app, serves `apps/web/dist`, and
  drives the browser through Chromium/CDP using the existing smoke helpers;
- the smoke creates a deterministic sketch, rectangle profile, `newBody`
  extrude, result body, and named generated face reference;
- required checks cover app load, feature tree/body selection, inspector and
  modeling `selection.referenceCandidates` status, viewport semantic reference
  actions, generated/named-reference routing, Project/File JSON and storage
  capability status, export readiness, STEP deferred status, and Mesh/GLB
  status;
- optional GLB download readiness is separated from required workflow checks.
  When a build does not expose a ready derived visualization mesh, the smoke
  records a skipped GLB check with the observed readiness text instead of
  failing vaguely;
- `scripts/v7-browser-workflow.mjs` keeps deterministic result/summary
  formatting covered by script tests.

G3 complements, but does not replace, the G2 source/query smoke. It does not add
new CAD features, broad browser E2E infrastructure, screenshots, tracked
downloads or metrics, persistent selection state, native storage, STEP, WebGPU,
assemblies, exact face/edge viewport picking, or `web-cad.project.v17`.

## V7 Scope Guardrails

V7 is broad, but each implementation tranche should avoid mixing unrelated
architecture risks.

Do not combine these in a single tranche unless explicitly approved:

- WebGPU renderer replacement and topology/reference model changes;
- native package storage and STEP import/export;
- broad sketch solver expansion and general feature-graph expansion;
- assembly/LOD architecture and local project format migration;
- natural-language command parsing and production MCP auth.

Do not introduce another saved project format unless source-of-truth data
requires it. Query results, topology summaries, derived exact metadata,
selection state, thumbnails, mesh caches, and renderer display state should stay
rebuildable by default.

## Deferred Unless Explicitly Scoped Into V7

- full arbitrary topological naming for every OCCT result shape;
- general boolean trees beyond currently supported scoped behavior;
- shell, sweep, loft, mirror, patterns, and direct modeling;
- broad sketch constraint families such as tangent, concentric, equal,
  symmetry, spline, and curvature;
- dragging solver UX;
- arbitrary parameter expressions;
- persistent B-rep checkpoints as source truth;
- broad STEP import with healing and assembly reconstruction;
- assemblies, mates, large-model LOD, and instancing;
- hosted collaboration;
- production MCP auth;
- natural-language command entry.

## Definition Of Done For V7 Tranches

A V7 tranche is done only when:

1. The tranche scope is implemented through CADOps/query-layer behavior.
2. Source-of-truth state remains in `cad-core`.
3. Derived geometry, renderer state, exact metadata, and caches remain derived.
4. Package boundaries remain clean.
5. Relevant protocol, cad-core, geometry/worker, adapter/MCP, UI-helper, and
   serialization tests are added where the tranche touches those surfaces.
6. `pnpm test` passes.
7. `pnpm typecheck` passes.
8. Relevant build, lint, and format checks pass.
9. Docs are updated for new capabilities, limitations, scripts, or format
   behavior.
10. Manual browser checks are recorded for user-facing workflow changes.

  and which should remain read-only until their CADOps behavior is explicit?
