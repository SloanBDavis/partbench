# V1 Foundation

Status: complete.

V1 is the completed browser-native CAD foundation after the initial setup work.
It is not a full CAD product, but it is a coherent local-first modeling workflow
and a stable baseline for the next architecture phase.

## What V1 Proves

V1 proves that:

- CADOps can be the shared mutation protocol for UI, tests, scripts, agents, and
  MCP wrappers.
- `cad-core` can remain authoritative while renderer meshes stay derived.
- Browser workers can isolate command and geometry execution.
- OCCT/WASM can run behind a typed geometry boundary without being imported by
  `cad-core` or the React UI.
- The app can save and load a versioned source-of-truth project JSON format.
- Structured agent/MCP access can wrap CADOps instead of defining the internal
  CAD architecture.

## Supported V1 Workflow

A user or structured agent can:

1. Open the web app.
2. Create boxes, cylinders, spheres, cones, and tori through CADOps.
3. View worker-derived tessellated geometry when the geometry path is enabled.
4. Fall back to simple primitive rendering if derived geometry is disabled,
   loading, unavailable, or failed.
5. Select objects.
6. Inspect IDs, names, types, dimensions, transforms, units, measurements, and
   geometry status.
7. Update transforms, dimensions, names, and document units through CADOps.
8. Delete objects through CADOps.
9. Undo and redo transactions.
10. Export and import a versioned project file for the current document model.
11. Query project summary, object details, measurements, extents, primitive
    feature summaries, and transaction history through CADOps.
12. Use MCP tools for structured project summary, measurements, extents,
    transaction history, and batch dry-run/commit.
13. Run browser-worker geometry smoke checks and track timing metrics over time.

## Implemented Capabilities

### Command and Document Model

- Typed and versioned CADOps command schemas.
- Transactional command application.
- Semantic diffs for committed commands.
- Batch dry-run and commit.
- Undo and redo.
- Query/read APIs separate from mutation APIs.
- Optional transaction actor metadata:
  `human | agent | script | system`.
- Adapter-originated audit metadata: source, request ID, tool name, intent, and
  operation count.
- Document units with metadata-only relabeling and preserve-physical-size
  conversion.
- Optional object display names.
- Structured validation responses with operation, path, expected, received,
  operation index, and object ID context where practical.
- Read-only measurements and project extents for supported primitives.
- Read-only transaction history.
- Read-only primitive feature summaries as a migration bridge toward future
  feature/body concepts.

### Geometry Pipeline

- Asynchronous browser Worker geometry path.
- Typed and serializable geometry request/response shapes.
- Isolated OCCT/WASM primitive tessellation.
- Geometry-kernel facade separate from `cad-core` and the renderer.
- App-layer derived geometry service with cache keys and stale result handling.
- Derived geometry invalidation after create, transform update, dimension
  update, delete, undo, redo, and project import/load.
- Structured worker diagnostics for WASM load, unsupported primitive,
  kernel/tessellation, runtime, and transport failures.
- Non-gating smoke metrics for OCCT load time, tessellation time, worker time,
  round-trip time, mesh sizes, and OCCT asset size.

### Renderer and UI

- Canvas viewport with primitive fallback and mesh-backed display.
- Orbit, pan, zoom, fit all, fit selected, and reset controls.
- Selection and selected-object display for primitive and mesh-backed objects.
- Object list, inspector, dimension editor, transform editor, measurement panel,
  geometry status panel, transaction history panel, batch panel, and project
  import/export panel.
- Narrow viewport layout that keeps the viewport reachable before long panels.
- UI mutations go through CadEngine/CADOps rather than direct document edits.

### Project Format and Persistence

- V1 source-of-truth JSON schema:
  `web-cad.project.v1`.
- Project export/import for objects, dimensions, transforms, units, display
  names, transaction history, redo history, actor metadata, and audit metadata.
- Import validation for schema version, document shape, object dimensions,
  transforms, units, names, transaction shape, redo stack shape, duplicate
  transaction IDs, generated object ID collisions, and transaction replay where
  practical.
- Derived meshes and geometry status are excluded from saved project data.
- Future native package direction documented in `docs/native-format.md`.

### MCP and Agent Use

- Agent adapter over CADOps batch/query APIs.
- MCP adapter over the structured agent adapter.
- Minimal stdio JSON-RPC MCP server.
- Read/query and dry-run are allowed by default at the adapter boundary.
- Commit requires explicit caller permission.
- No natural-language parsing.

## Acceptance Checks

The V1 foundation is considered complete when the standard check suite passes:

```sh
pnpm test
pnpm typecheck
pnpm build
pnpm lint
pnpm format:check
pnpm smoke:occt-browser
```

Browser workflow smoke should also cover:

- app load,
- create/select/edit/delete for supported primitives,
- undo/redo,
- unit/name/dimension/transform editing,
- project export/import round trip,
- invalid project import feedback,
- derived mesh status,
- primitive fallback mode, and
- desktop plus narrow layout sanity.

## V1 Non-Goals

V1 intentionally does not include:

- Full feature graph.
- Authoritative B-rep topology in `cad-core`.
- Sketch solver.
- Stable topological naming.
- Fillets, chamfers, booleans, shell, patterns, lofts, or sweeps.
- STEP import/export.
- OPFS or File System Access API.
- Final `.wcad` package implementation.
- WebGPU renderer.
- Large-assembly LOD/instancing.
- Hosted collaboration.
- Natural-language command parsing.
- Production MCP auth/permission system.

## Known Limitations

- The authoritative document is still a primitive scene model.
- Measurements are approximate primitive-derived values, not exact kernel/B-rep
  measurements.
- Derived geometry currently covers box, cylinder, sphere, cone, and torus
  tessellation.
- OCCT uses the full OpenCascade.js WASM in the main path.
- Production builds keep derived geometry disabled unless explicitly enabled.
- Object names are optional and may duplicate.

## Next Target

V1 should not keep absorbing new architecture work. The next implementation
target is `docs/v2.md`, which starts the move from primitive scene modeling
toward a feature/body CAD foundation aligned with `docs/architecture.md`.
