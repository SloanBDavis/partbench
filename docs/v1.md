# V1 Foundation

V1 is the current browser-native CAD foundation after the completed MVP. It is
useful as a small local-first modeling workflow without pretending to be the
full long-term architecture.

V1 keeps the current architectural proof as a coherent local-first CAD workflow:

- CADOps remains the only mutation path.
- `cad-core` remains authoritative.
- Geometry work runs asynchronously behind typed boundaries.
- Rendered meshes remain derived views/caches.
- MCP remains an adapter over CADOps.
- The app remains browser-native and can later run local or hosted from the same
  bundle.

## V1 Product Goal

A user can open the browser app, create simple parts, see worker-derived
geometry, inspect and edit the model structurally, save/load the project, and
optionally let an external structured agent inspect and modify the model through
CADOps dry-run and commit.

This is still not a full CAD product. It is the first version that should feel
like a coherent CAD foundation instead of a set of isolated adapters.

## V1 Supported Workflow

The V1 workflow allows a user or structured agent to:

1. Open the web app.
2. Create boxes, cylinders, spheres, cones, and tori through CADOps.
3. View worker-derived tessellated geometry when the geometry path is enabled.
4. Fall back to simple primitive rendering if derived geometry is unavailable.
5. Select objects.
6. Inspect IDs, names, types, dimensions, transforms, units, and geometry status.
7. Update transforms, dimensions, names, and document units through CADOps.
8. Delete objects through CADOps.
9. Undo and redo transactions.
10. Export and import a versioned project file for the current document model.
11. Query project summary and object details through CADOps.
12. Query current primitive measurements and project extents through CADOps.
13. Query transaction history/audit summaries through CADOps.
14. Use MCP tools for structured project summary, measurements, extents,
    transaction history, and batch dry-run/commit.
15. Run browser-worker geometry smoke checks and track timing metrics over time.

## V1 System Requirements

### Command and Document Model

V1 should keep the current CADOps design and expand it carefully.

Required:

- CADOps command schemas remain typed and versioned.
- Every mutation is transactional.
- Every committed command returns a semantic diff.
- Batch dry-run and commit remain supported.
- Undo and redo continue to work through transactions.
- Query/read APIs remain separate from mutation APIs.
- Commands must not depend on React, renderer, MCP, or OCCT internals.

Implemented V1 additions:

- Dimension update commands for supported primitives.
- Actor metadata on transactions. Current implementation stores optional
  `human | agent | script | system` metadata with optional actor id/name.
- Audit metadata on adapter-originated commits. Current implementation can store
  generic source, request ID, tool name, intent, and operation count on
  committed transactions.
- Units on the project/document. Current implementation supports explicit
  metadata-only relabeling or preserve-physical-size conversion for current
  primitive dimensions and transform translations.
- Optional object display names. Current implementation does not require unique
  names.
- Structured validation responses for supported operations. Current batch
  validation errors include stable codes plus operation, path, expected, and
  received details where practical.
- Read-only measurement queries for supported primitives. Current implementation
  derives local bounds, transformed world bounds, and approximate volume for
  boxes, cylinders, spheres, cones, and tori from the authoritative document.
- Read-only transaction history queries. Current implementation summarizes the
  existing transaction model with IDs, undo/redo status, actor metadata,
  audit metadata, operation summaries, and semantic diff summaries.
- A clearer path from scene object to future feature/body concepts. Current
  implementation adds a read-only `project.features` query that derives
  primitive feature summaries for boxes, cylinders, spheres, cones, and tori
  from the current scene objects plus active transaction history. It is not a
  persisted feature graph.

### Geometry Pipeline

V1 should turn the current OCCT-backed primitive path into a small but real
derived geometry pipeline.

Required:

- Geometry execution remains asynchronous.
- Browser Worker path remains typed and serializable.
- `cad-core` does not import OCCT or renderer internals.
- The renderer receives renderer-facing mesh data only.
- Meshes are invalidated when document objects change.
- If OCCT fails to load, the app still opens and the command layer still works.
- Timing metrics continue to be recorded without becoming pass/fail thresholds.
- Worker failures are reported with structured codes/stages for WASM load,
  unsupported primitives, kernel/tessellation, runtime, and transport failures.

Implemented V1 additions:

- Object revision or cache keys.
- Mesh derivation service outside `cad-core`.
- Derived mesh status in UI.
- Hardened geometry worker lifecycle/error handling.
- Browser smoke metrics for load time, tessellation time, round-trip time, mesh
  sizes, and OCCT asset size.

### Renderer and UI

V1 should improve the current viewport enough to support the geometry workflow,
but it should not jump to WebGPU without a scoped renderer milestone.

Required:

- Current primitive renderer remains usable.
- Derived mesh display works for supported primitives.
- Selection, inspector, delete, transform update, undo, redo, and batch controls
  remain command-layer driven.
- UI does not mutate document state directly.
- UI does not import OCCT.

Implemented V1 additions:

- Geometry status per object.
- Clear visual distinction between primitive fallback and derived mesh.
- Better mesh selection/highlight.
- Better viewport fit/focus controls.
- Focused error messages for failed geometry jobs.

### Project Format and Persistence

V1 should keep the current JSON path deliberate and versioned, but it does not
need the final OPFS/native package system yet.

Required:

- Versioned source-of-truth project data.
- Objects, display names, dimensions, transforms, units, and
  command/transaction history where practical.
- Tests for round-trip and migration behavior.
- Derived meshes remain optional and rebuildable.

Implemented V1 additions:

- Documented native package target in `docs/native-format.md`.
- Current JSON export/import with visible schema, project summary, and import
  validation preview.
- Initial migration/version checks.

Not required for V1 unless explicitly scoped:

- OPFS.
- File System Access API.
- Final `.wcad` package implementation.

### MCP and Agent Use

V1 should keep agent access structured and deterministic.

Required:

- MCP tools remain thin wrappers over CADOps.
- `cad.batch` remains the main write tool.
- Query before mutation remains supported.
- Dry-run before commit remains supported.
- No natural-language parser is required.

Implemented V1 additions:

- Structured query tools for project summary, measurements, extents, features,
  and transaction history.
- Actor metadata for agent commits. Current implementation threads actor
  metadata through CADOps, the agent adapter, and the MCP wrapper.
- First audit metadata for agent commits. Current implementation records source,
  request ID, tool name, intent, and operation count; fuller tool input/output
  hashes remain future work.
- Clearer validation and error codes.
- Permission defaults for read/query/dry-run versus commit. Current
  implementation allows read/query and dry-run by default, while adapter/MCP
  commits require explicit caller permission.

## V1 Non-Goals

Do not include these in V1 unless a later prompt explicitly changes scope:

- Full feature graph.
- Sketch solver.
- Stable topological naming system.
- Fillets, chamfers, booleans, shell, patterns, or lofts.
- STEP import/export.
- OPFS or File System Access API.
- WebGPU renderer.
- Large-assembly LOD/instancing.
- Hosted collaboration.
- Natural-language command parsing.
- Production MCP auth/permission system.

## V1 Exit Criteria

V1 is complete when:

1. Boxes, cylinders, spheres, cones, and tori can be created through CADOps and
   displayed through a reliable derived geometry path.
2. The app remains usable if OCCT fails or is unavailable.
3. Derived geometry invalidates correctly after document changes.
4. The project can be saved and loaded through a versioned format for the current
   model.
5. Structured CADOps query and batch APIs work from UI, tests, and MCP adapter.
6. Undo/redo remain reliable across UI and batch operations.
7. Browser-worker geometry metrics are easy to collect and compare over time.
8. `pnpm test`, `pnpm typecheck`, `pnpm build`, `pnpm lint`,
   `pnpm format:check`, and relevant smoke checks pass.

Current status: the repository is ready to treat as a V1 foundation when the
standard check suite and browser workflow smoke pass on the release candidate.
This does not mean the app is a full CAD system; it means the current
primitive-scene model, derived geometry path, project interchange, command
history, and structured agent surfaces are coherent enough to build the next
architecture phases on top.

## V1 Quality Bar

V1 should not add dead architecture. Every new boundary should have a current
caller and tests.

Prefer:

- Small typed APIs.
- Explicit validation.
- Structured diffs and errors.
- Browser-worker smoke tests for real WASM behavior.
- Clear fallback behavior.

Avoid:

- Making OCCT part of `cad-core`.
- Letting React own CAD state.
- Treating meshes as authoritative.
- Adding natural-language features before structured operations are solid.
- Starting WebGPU before renderer requirements are grounded in real geometry
  workload.
