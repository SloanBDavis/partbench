# V17 Release Record: Composite Sketch Profiles, Arcs, and Curved Sweep Paths

Status: **Completed.** Slices A–I and every Must support-matrix row are
implemented on main. This document is the completed V17 product, architecture,
storage, validation, and acceptance contract and release record. V16 remains
the compatibility baseline in `docs/v16.md`.

Implementation evidence includes canonical circular arcs and construction
state, the complete arc solver matrix, analytic oriented profile/path
validation, composite extrude and revolve, corrected-Frenet circular-arc/G1
chain sweeps, dependency and stale-state handling, exact metadata/topology/
checkpoint/STEP readiness, UI and agent/MCP parity, V1–V20 migration coverage,
and minimum-triggered V21 JSON/CBOR/`.wcad` round-trips. The six named V17 smoke
commands recorded below provide release-level workflow evidence.

This record remains decision-complete. Maintenance may use private helper names
and equivalent algorithms, but may not weaken a Must row, silently infer
ambiguous source intent, omit a required boundary, or advertise a geometry path
outside the completed support matrix.

**Version naming caution:** product release V17 is not document schema
`web-cad.project.v17`, which is already used by V11 advanced sketch constraint
records. Product V17 introduces **`web-cad.project.v21`**. Native package
version remains **`partbench.wcad.v2`**.

## Product Goal

V17 delivers the ability for a Partbench user or agent to:

1. Author and edit source-backed circular arcs through CADOps.
2. Mark sketch geometry as construction geometry without removing it from the
   solver or making UI state authoritative.
3. Select an explicit, stable, ordered closed wire made from line and arc
   entities and use it as an extrude profile.
4. Extrude that wire as `newBody`, `add`, or `cut` within the existing
   topology-backed target matrix.
5. Revolve a supported line/arc wire as a `newBody`.
6. Sweep the existing rectangle/circle profile kinds along a supported circular
   arc or a smooth ordered line/arc chain.
7. Edit any referenced line or arc and get deterministic dependency, rebuild,
   failure, topology-health, display, and mass-property behavior.
8. Discover profile and path candidates structurally through queries instead
   of asking the UI, an agent, or the kernel to guess authorial intent.
9. Save, load, dry-run, commit, undo, redo, inspect, and automate every V17
   source mutation through the same CADOps boundary.

The release demonstration is:

```text
Create a slotted plate from line/arc composite profiles.
Create a revolved line/arc part.
Sweep a circle or rectangle along a tangent line/arc elbow path.
Edit an arc radius and observe dependent rebuild, topology health,
display geometry, exact metadata, and mass properties update.
Round-trip the result through JSON and .wcad and repeat through agent/MCP.
```

V17 does not promise a general-purpose 2D wire editor, arbitrary planar region
recognition, splines, trimming, multiple profile regions, loft guides, datum
planes, or assemblies.

## Why This Release Comes After V16

V16 completed the basic solid feature families but retained two deliberate
authoring constraints:

- solid profiles are individual rectangle or circle entities;
- sweep Must accepts exactly one line path entity.

Those constraints now prevent common parts such as slotted plates, rounded
brackets, hooks, and bent tubes. V17 removes that ceiling by adding the missing
source abstraction: an explicit ordered profile/path reference whose members
retain sketch-entity identity.

An arc entity alone would be an incomplete release. Without an ordered
composite profile, the arc could render and solve but could not participate in
ordinary solid construction. Conversely, a kernel-side wire assembled from
whatever curves happen to touch would bypass source intent and make rebuilds,
agents, and topology identity ambiguous. V17 therefore delivers arcs, explicit
wire references, consumers, and proof as one coherent tranche.

## Non-Negotiable Architecture

V17 preserves all rules in `docs/architecture.md`, `docs/implementation-plan.md`,
and `docs/v16.md`:

- CADOps owns every source mutation. UI convenience tools translate gestures
  into CADOps; they do not edit documents directly.
- `cad-core` owns sketch entities, construction state, ordered profile/path
  references, feature intent, dependency/rebuild state, and diagnostics.
- `cad-core` does not call OCCT and does not validate a profile by tessellating
  it.
- `sketch-solver` remains pure TypeScript and has no document, renderer, worker,
  or OCCT dependency.
- `geometry-kernel`, `geometry-worker`, and `occt-wasm` receive resolved,
  validated geometric inputs, not public document references.
- Rendered curves and meshes, candidate lists, solver residuals, OCCT wires,
  exact metadata, topology snapshots, and mass properties remain derived.
- Renderer IDs, mesh IDs, OCCT handles/indexes, checkpoint-local topology IDs,
  selection-buffer IDs, pixels, file handles, OPFS paths, and local paths never
  become public CAD references.
- Agent and MCP surfaces remain thin typed wrappers over `cad.batch` and
  `cad.query`.
- Unsupported or ambiguous states fail with structured diagnostics. A kernel
  success may not override a cad-core support-matrix rejection.

### Complete vertical-slice rule

A V17 support-matrix row is enabled only when all applicable layers exist:

```text
protocol type + validator
-> cad-core command/dry-run/diff/undo/redo
-> dependency/rebuild/editability/health
-> geometry request + real OCCT result
-> derived display source
-> exact metadata + mass-properties path
-> topology/reference-health behavior
-> JSON/.wcad migration and round-trip
-> UI and agent/MCP parity
-> named unit/integration/release smoke proof
```

Returning a mesh alone, storing an unvalidated array of IDs, adding a UI-only
tool, or accepting a command that later produces `display unavailable` does
not count as implementation.

## Normative Decisions

### 1. V17 uses schema `web-cad.project.v21`

V21 is required because V20 cannot represent:

- `SketchArcEntity`;
- explicit `construction` state on sketch entities;
- oriented ordered composite `SketchProfileRef` records;
- oriented ordered `SketchPathRef` records;
- normalized profile/path fields on profile-consuming V21 features.

V21 uses the existing top-level project document and transaction model. It does
not change `.wcad` layout, checkpoint entries, source hashing rules, or package
version. See Storage and Migration below.

### 2. Arcs have one canonical source representation

Persisted arcs use center, radius, start angle, and signed sweep angle:

```ts
interface SketchArcEntity {
  readonly id: string;
  readonly kind: "arc";
  readonly center: Vec2;
  readonly radius: number;
  readonly startAngleDegrees: number; // canonical [0, 360)
  readonly sweepAngleDegrees: number; // signed; bounded away from 0 and 360 by policy
  readonly construction: boolean;
}
```

Positive sweep is counterclockwise in sketch-local coordinates; negative sweep
is clockwise. A full circle is always a `circle`, never an arc with a 360°
sweep. Endpoints are derived from the canonical fields and are not duplicated
in source.

Create/update command validation requires finite values, radius greater than
`linearTolerance`, and
`angularToleranceDegrees <= abs(sweepAngleDegrees) <= 360 - angularToleranceDegrees`.
Commands normalize finite start angles modulo 360 and canonicalize negative
zero to positive zero. V21 import requires the already-canonical `[0, 360)`
start range and rejects non-canonical stored values rather than silently
rewriting signed source. Radius and sweep bounds use the shared policy in
Decision 6.

`sketch.addArc` may accept either canonical center/angle input or three-point
input. Three-point input is deterministically converted to the unique finite
circle and the signed arc from start to end that passes through `pointOnArc`.
Collinear or coincident inputs are rejected. Updates and import/export use only
the canonical representation.

No endpoint-plus-bulge, start/mid/end, kernel curve, or tessellated point list is
stored as an alternative source shape.

### 3. Construction is explicit source intent

Every V21 sketch entity stores `construction: boolean`. V20 and older entities
migrate as `false`. Creation commands default omitted construction state to
`false`; V21 export writes the boolean explicitly.

Construction geometry:

- remains visible, selectable, editable, constrainable, dimensionable, and
  available to the sketch solver;
- is excluded from automatic profile candidate discovery;
- is rejected as a member of a solid `SketchProfileRef`;
- may be used explicitly as a revolve axis or sweep path;
- is rendered with the existing restrained construction style and is not
  inferred from color, dash pattern, name, layer, or selection state.

The only toggle command is `sketch.setEntityConstruction`. Delete/recreate is
not an acceptable substitute because it would destroy entity identity.

### 4. Composite profiles are explicit oriented source references

The source union is:

```ts
interface SketchEntityProfileRef {
  readonly kind: "entity";
  readonly sketchId: string;
  readonly entityId: string; // rectangle | circle only
}

interface OrientedSketchSegmentRef {
  readonly entityId: string;
  readonly orientation: "forward" | "reverse";
}

interface SketchWireProfileRef {
  readonly kind: "wire";
  readonly sketchId: string;
  readonly segments: readonly OrientedSketchSegmentRef[];
}

type SketchProfileRef = SketchEntityProfileRef | SketchWireProfileRef;
```

For an arc, `forward` follows its signed authored sweep; `reverse` traverses
the same authored curve in the opposite direction. For a line, `forward`
follows start to end. Orientation changes traversal only; it never mutates the
source entity.

A wire profile stores authorial selection, not a cached discovered loop. The
feature owns a copy of the ordered references. Candidate queries are derived
and may be regenerated; they are never source authority.

On command commit, cad-core normalizes every valid wire profile to
counterclockwise traversal in sketch-local coordinates. If input traversal is
clockwise, cad-core reverses segment order and flips every orientation before
storing it. The semantic diff reports `profileOrientationNormalized: true`.
The operation is deterministic and preserves entity IDs.

### 5. Paths are a separate type from profiles

Paths are not profiles with a missing closure:

```ts
interface SketchEntityPathRef {
  readonly kind: "entity";
  readonly sketchId: string;
  readonly entityId: string; // line | arc
  readonly orientation: "forward" | "reverse";
}

interface SketchChainPathRef {
  readonly kind: "chain";
  readonly sketchId: string;
  readonly segments: readonly OrientedSketchSegmentRef[];
}

type SketchPathRef = SketchEntityPathRef | SketchChainPathRef;
```

An entity path has explicit orientation so path start, frame initialization,
and sweep result are stable. A chain preserves authorial order. Cad-core never
chooses a path start by coordinate sorting or kernel wire order.

### 6. One shared sketch geometry policy controls validation

V17 adds a pure shared policy module owned by `cad-core` and mirrored as typed
resolved settings at the geometry boundary:

```ts
interface SketchGeometryPolicy {
  readonly linearTolerance: 1e-7;       // current document units
  readonly angularToleranceDegrees: 0.1;
  readonly minimumProfileArea: 1e-12;   // square document units
}
```

The literal defaults above are normative. Algorithms may use scale-aware
intermediate math, but may not expose different product decisions. Unit-change
commands continue to scale geometry; the policy is evaluated in current
document units, matching existing sketch-solver behavior.

OCCT calls use the greater of the converted linear tolerance and the kernel's
own confusion tolerance. Kernel tolerance must not be persisted back into
source or used to make cad-core accept a source-invalid wire.

### 7. Profile validation is exact and deterministic

A `wire` profile is Must-supported only when all conditions hold:

- all segments resolve on the declared sketch;
- every member is a non-construction line or arc;
- every entity ID occurs exactly once;
- at least two segments are present;
- every segment has finite, valid geometry and non-zero length;
- oriented consecutive endpoints coincide within `linearTolerance`;
- the last endpoint coincides with the first within `linearTolerance`;
- non-adjacent segments do not intersect or overlap;
- adjacent segments meet only at their shared endpoint;
- the enclosed absolute signed area is at least `minimumProfileArea`;
- the wire has no branch, gap, repeated edge, nested loop, or inner loop.

Intersection and signed-area validation uses analytic line/line, line/circle,
and circle/circle math clipped to each finite arc interval. Tessellation is not
authoritative for either test.

Endpoint proximity does not rewrite source coordinates or change analytic
curve parameters. If endpoints are within tolerance but not bit-equal, the
resolved geometry request supplies one shared topological join whose vertex
tolerance covers the measured discrepancy, capped at `linearTolerance`; each
edge retains its exact line/circle support. The profile query reports
`coincidentWithinTolerance: true` for that join. A discrepancy above the cap is
never healed by the worker.

Two complementary arcs may form a valid two-segment closed profile. Two
opposing line segments are rejected as overlapping/zero-area. A standalone
circle or rectangle continues to use `kind: "entity"`; it is not exploded into
synthetic wire segments.

### 8. Profile candidate discovery is query-only

`sketch.profileCandidates` discovers simple closed loops from non-construction
line/arc entities and returns normalized `SketchProfileRef` candidates plus
health. Rectangle and circle entities are returned as entity candidates.

Determinism rules:

1. Build connectivity using the shared linear tolerance.
2. Reject connected components containing a branch vertex with degree other
   than two from automatic discovery.
3. Traverse each simple component from its lexicographically smallest entity
   ID, trying `forward` before `reverse`.
4. Normalize the resulting loop counterclockwise.
5. Sort candidates by the lexicographically joined ordered entity IDs.

The query never mutates the sketch or silently selects a candidate for a
feature. UI and agents must submit the exact returned profile reference.

### 9. Path validation is deliberately narrower than arbitrary wires

V17 sweep paths support:

| Path | Status |
| --- | --- |
| One line, either orientation | Required; V16 compatibility |
| One circular arc, either orientation | Required |
| Open ordered line/arc chain with G1 joins | Required |
| Closed chain | Rejected |
| Repeated entity | Rejected |
| Disconnected or branching selection | Rejected |
| Self-intersecting chain | Rejected |
| G0 corner / non-tangent internal join | Rejected in V17 |
| Circle entity as path | Rejected |
| Spline/ellipse/imported curve | Non-goal |
| Cross-sketch or 3D edge chain | Non-goal |

A chain must contain at least two segments. Every oriented consecutive endpoint
must coincide within linear tolerance. At each internal join, outgoing and
incoming unit tangents must differ by at most `angularToleranceDegrees`. For a
line/line join this means same-direction collinearity; a reversal is rejected.
For line/arc and arc/arc joins, tangent direction includes authored and traversal
orientation.

Construction lines/arcs are permitted because a path is explicit guide intent,
not a solid region. Candidate discovery for paths is provided by
`sketch.pathCandidates`; it follows the same deterministic ordering rules and
returns only V17-supported open G1 chains plus individual lines/arcs.

An entity rectangle/circle profile is also rejected when its entity is marked
construction. Construction state never changes the geometry kind into a solid
profile implicitly.

### 10. Feature support is an explicit matrix

| Consumer | Entity rectangle/circle | Line/arc wire | V17 operation modes |
| --- | --- | --- | --- |
| Extrude | Existing behavior retained | **Must** | `newBody`, `add`, `cut` |
| Revolve | Existing behavior retained | **Must** | `newBody` only |
| Sweep profile | Existing behavior retained | Not in V17 | `newBody` only |
| Sweep path | One line retained | line/arc `SketchPathRef` **Must** | `newBody` only |
| Loft section | Existing behavior retained | Not in V17 | `newBody` only |
| Hole | Circle entity retained | Not applicable | Existing modes |

There is no prove-then-enable row in this matrix. A non-Must row stays rejected
for all V17 builds even if a private OCCT experiment succeeds.

Composite extrude `add`/`cut` uses exactly the target-body readiness matrix
already shared by V14/V16. V17 expands tool profile geometry, not target body
breadth. Primitive bodies, consumed bodies, unsupported compounds, and targets
outside that matrix remain rejected before commit.

### 11. Sweep orientation is fixed, not user-configurable

V17 curved sweeps use a corrected-Frenet/parallel-transport frame selected once
at the OCCT boundary. The public source stores no kernel enum. The profile's
authored sketch frame initializes the section orientation; the path start must
lie in the profile plane within linear tolerance and its starting tangent must
be normal to that plane within `angularToleranceDegrees` for a new V17 curved
path. The path start must also coincide with the rectangle/circle profile
center in resolved 3D space within linear tolerance. Eccentric profile placement
is outside the V17 support matrix.

V20 single-line sweeps retain their V16 compatibility behavior when loaded and
rebuilt. Retargeting such a feature to an arc or chain is a V17 edit and must
pass the stricter start-frame rule.

No binormal control, twist law, guide surface, scaling law, corner transition,
closed-path seam, or add/cut sweep is in V17. Geometry that passes cad-core
validation but fails pipe construction produces a structured derived-geometry
failure after the source command commits; no partial or stale shape is
published as the current body.

### 12. Invalid downstream state is preserved honestly

Feature creation and feature retargeting require a currently valid source
profile/path. Cad-core commands remain synchronous and never call OCCT. They
commit valid source intent; the async geometry pipeline then either publishes a
matching successful result or records a structured failed derived state. UI and
adapter orchestration may request geometry readiness before offering an action,
but worker availability is not a hidden command-validity condition.

Editing a referenced sketch entity is allowed even if it makes an existing
profile or path invalid. The edit commits as source intent; dependency and
rebuild queries mark affected features `failed` or `blocked` with the precise
profile/path diagnostic, retain their authored source records, and never show
the previous mesh as current. Deleting a referenced entity follows the same
existing unresolved-dependency policy.

Cad-core must not auto-remove invalid feature segments, substitute another
candidate, reorder a path, or restore an old radius to keep geometry green.

### 13. Source identity drives topology roles

For a composite `newBody` extrude, Must generated-reference semantics are:

- start and end cap planar faces;
- one side face per ordered profile segment, keyed by source entity ID;
- one start-cap and one end-cap boundary edge per profile segment;
- one longitudinal edge per profile join, keyed by the adjacent ordered source
  entity IDs;
- join vertices where the existing generated-reference model exposes vertices.

A line-derived side face is planar. An arc-derived side face is cylindrical.
Roles are derived from source entity identity and normalized traversal, never
from an OCCT face index. If exact kernel correspondence disagrees or cannot be
proven, the reference is unavailable/ambiguous rather than guessed.

Composite add/cut results continue through V12–V14 checkpoint lineage and
topology-anchor proof. V17 does not claim source-semantic identity for split or
merged result faces merely because their tool segment is known.

Composite revolutions and curved sweeps must provide body topology health and
checkpoint-backed anchors. Rich source-semantic face catalogs are optional only
where correspondence cannot be proven; an empty catalog must be paired with an
explicit `generatedReferences: unavailable` diagnostic, not an omitted status.

### 14. Exact metadata is required for every new geometry recipe

The same resolved recipe used for display must support exact-metadata rebuild
for:

- composite profile extrude `newBody`;
- supported composite add/cut result bodies;
- composite profile revolve;
- arc/chain sweep.

`body.massProperties`, `project.extents`, `project.health`, topology checkpoint
generation, and STEP export readiness consume matching derived exact metadata
under the V16 source-identity rules. No analytic polygon/arc approximation may
be labeled exact. No V17 source stores mass properties.

V17 does not introduce automatic checkpoint creation. Existing explicit or
policy-driven checkpoint flows may rebuild every V17 recipe and persist payload
bytes under the unchanged `.wcad` v2 rules.

### 15. Public convenience tools are command sugar only

Slot and rounded-rectangle tools may be added only as convenience producers of
ordinary line/arc entities and constraints in one CADOps batch. They do not get
new entity kinds, new feature kinds, hidden composite objects, or special
kernel paths. They are Stretch and cannot substitute for the Must authoring UI.

## CADOps Mutation Contract

### New commands

```ts
interface SketchAddArcOp {
  readonly op: "sketch.addArc";
  readonly sketchId: string;
  readonly id?: string;
  readonly construction?: boolean; // default false
  readonly definition:
    | {
        readonly kind: "centerAngles";
        readonly center: Vec2;
        readonly radius: number;
        readonly startAngleDegrees: number;
        readonly sweepAngleDegrees: number;
      }
    | {
        readonly kind: "threePoint";
        readonly start: Vec2;
        readonly pointOnArc: Vec2;
        readonly end: Vec2;
      };
}

interface SketchSetEntityConstructionOp {
  readonly op: "sketch.setEntityConstruction";
  readonly sketchId: string;
  readonly entityId: string;
  readonly construction: boolean;
}
```

`sketch.updateEntity` retains its existing whole-entity shape. For an arc it
accepts the canonical `SketchArcEntity`, including explicit construction state;
three-point input is create-tool sugar only. The command cannot change entity
kind. Replacing a line with an arc requires delete and add with normal
dependency consequences; the implementation must not reinterpret an ID across
kinds. Existing point/line/rectangle/circle create commands add optional
`construction?: boolean` with the same default and persist it through their
canonical entity snapshots.

### Extended commands

The following commands add normalized V21 fields:

```text
feature.extrude / feature.updateExtrude -> profile: SketchProfileRef
feature.revolve / feature.updateRevolve -> profile: SketchProfileRef
feature.sweep / feature.updateSweep     -> profile: SketchEntityProfileRef; path: SketchPathRef
feature.loft / feature.updateLoft       -> normalized entity profiles only
```

Create commands allocate IDs only after validation succeeds, matching current
dry-run/commit determinism. Update commands preserve feature and result-body
IDs. A source-validation failure leaves the complete prior source unchanged. A
later async geometry failure follows Decision 12: the valid source edit remains
committed and the derived body is marked failed.

### Required semantic diffs

Every new or extended command reports structured diffs sufficient for agents
and history UI:

```ts
interface SketchEntitySemanticDiff {
  readonly sketchId: string;
  readonly entityId: string;
  readonly action: "added" | "updated" | "deleted";
  readonly entityKind: "point" | "line" | "rectangle" | "circle" | "arc";
  readonly changedFields: readonly string[];
  readonly constructionBefore?: boolean;
  readonly constructionAfter?: boolean;
}

interface FeatureInputReferenceSemanticDiff {
  readonly featureId: string;
  readonly inputKind: "profile" | "path";
  readonly before?: SketchProfileRef | SketchPathRef;
  readonly after: SketchProfileRef | SketchPathRef;
  readonly profileOrientationNormalized?: boolean;
  readonly affectedSketchIds: readonly string[];
  readonly affectedEntityIds: readonly string[];
}
```

Feature diffs also retain existing created/updated/consumed/result-body and
rebuild impact fields. They do not contain OCCT handles, resolved wire objects,
mesh IDs, or tessellation.

### Transactions and history

- Dry-run executes the same source validation and deterministic normalization
  as commit but does not mutate counters, history, source, caches, or UI state.
- Undo/redo restores exact arc, construction, ordered reference, feature,
  consumed-body, and counter state.
- A batch that creates entities and consumes them in a profile may reference
  explicit caller IDs in later operations in the same batch.
- Automatic generated IDs remain deterministic under existing transaction
  rules; agents should use explicit IDs for multi-op profile batches.
- Historical V1–V20 command events remain readable. V21 package export does not
  rewrite old transaction payloads merely to normalize document source.

## Queries

### New queries

```text
sketch.profileCandidates
  input: sketchId
  output: entity and wire candidates, ordered oriented refs, area, bounds,
          join health, construction exclusions, structured rejected components

sketch.profileReadiness
  input: SketchProfileRef + intended consumer/operation
  output: normalized ref, valid/blocked status, area/orientation, dependencies,
          target compatibility when applicable, diagnostics

sketch.pathCandidates
  input: sketchId
  output: oriented line/arc entity paths and supported open G1 chains

sketch.pathReadiness
  input: SketchPathRef + optional sweep profile ref
  output: connection/tangent/self-intersection/frame status and diagnostics
```

All are pure and non-mutating. Candidate queries may include rejected
components to explain why visible curves are not selectable, but only `ready`
candidates may be submitted by the UI without an explicit user correction.

### Extended queries

The following must understand arc entities, construction state, and ordered
profile/path dependencies where relevant:

- `sketch.get`, `sketch.evaluation`, `sketch.solverStatus`,
  `sketch.editReadiness`, and dimension/constraint queries;
- `project.structure`, `project.dependencyGraph`, `project.rebuildPlan`,
  `project.health`, `project.exportReadiness`, and parameter evaluation;
- `feature.editability` and feature summaries;
- `body.topology`, generated-reference, topology-anchor, extents,
  measurement, exact-metadata health, and mass-properties queries;
- `selection.referenceCandidates` for displayed arc and generated result
  references.

Queries report source refs, roles, statuses, and diagnostics. They never expose
private analytic intersection graphs or OCCT wire/edge indexes.

## Sketch Solver Contract

### Arc variables and evaluation

The solver model adds arc center `(cx, cy)`, radius, start angle, and signed
sweep angle. Radius is positive and sweep sign cannot cross zero during one
solve. Evaluated endpoints are derived. Solver output may normalize the start
angle but must preserve traversal sign.

An edit or solve that would produce a non-finite radius, radius at/below linear
tolerance, zero sweep, or full-circle sweep is rejected and leaves prior source
unchanged.

### Point and curve targets

V21 keeps legacy point-target records unchanged for point, line, rectangle, and
circle entities. An arc point target is a distinct union member and must store
`entityKind: "arc"`:

```ts
interface SketchArcPointTarget {
  readonly entityId: string;
  readonly entityKind: "arc";
  readonly role: "center" | "start" | "end";
}
```

Omitting `entityKind` for a target that resolves to an arc, or storing
`entityKind: "arc"` for another entity kind, is invalid. Curve targets add
`entityKind: "arc"`; existing line/circle curve-target meanings do not change.

V21 normalizes concentric and equal-radius records onto explicit curve targets:

```ts
interface SketchRadiusCurveTarget {
  readonly entityId: string;
  readonly entityKind: "circle" | "arc";
}

interface SketchConcentricConstraintV21 {
  readonly kind: "concentric";
  readonly primaryTarget: SketchRadiusCurveTarget;
  readonly secondaryTarget: SketchRadiusCurveTarget;
  // existing id, name, sketchId, and entityId fields are retained
}

interface SketchEqualRadiusConstraintV21 {
  readonly kind: "equalRadius";
  readonly primaryTarget: SketchRadiusCurveTarget;
  readonly secondaryTarget: SketchRadiusCurveTarget;
  // existing id, name, sketchId, and entityId fields are retained
}
```

The legacy `primaryCircleEntityId` and `secondaryCircleEntityId` fields are not
stored alongside these V21 targets. V20 normalization maps each legacy circle
ID to `{ entityId, entityKind: "circle" }`. V21 import rejects legacy-only,
partial, or mixed target forms with exact paths; export to a lower schema is
permitted only when both targets are circles and writes the legacy fields.

Arc dimensions use:

```ts
interface SketchArcDimensionTarget {
  readonly entityKind: "arc";
  readonly role: "radius" | "sweep";
}
```

`SketchDimensionTarget` adds this union member. V21 import requires it whenever
the dimension's `entityId` resolves to an arc and rejects a circle-radius target
used for an arc or an arc target used for another entity kind.

Radius and sweep dimension values are positive magnitudes. A sweep dimension
sets `abs(sweepAngleDegrees)` and preserves the arc's current authored sign; it
cannot reverse traversal. Reversal is an explicit arc entity edit. Literal and
parameter-backed values must remain inside the same radius/sweep bounds used by
arc commands or evaluation fails without mutating source.

### Must constraint matrix

| Constraint/dimension | Arc support |
| --- | --- |
| fixed | center, start, end point targets |
| coincident | arc center/start/end with any supported point target |
| horizontal/vertical | not directly on an arc; use endpoint points |
| midpoint | arc targets are rejected |
| parallel/perpendicular | arcs are rejected |
| tangent | line–circle retained; line–arc, arc–circle, and arc–arc Must; line–line and circle–circle rejected |
| concentric | arc–arc and arc–circle Must |
| equal radius | arc–arc and arc–circle Must |
| equal length | arcs rejected in V17 |
| angle | line–line retained; arc sweep is a dimension, not this constraint |
| symmetry | arc center/start/end point targets supported through existing point semantics; whole-arc symmetry is not supported |
| radius dimension | circle retained; arc Must; literal or existing parameter value source |
| arc sweep dimension | Must in degrees; literal or existing parameter value source |

Tangency is valid only if the solved contact lies on the finite arc interval.
The solver must report `SKETCH_TANGENCY_OUTSIDE_ARC` instead of treating the
underlying infinite circle as sufficient. Multiple mathematical tangent
solutions are resolved from the current source geometry; a solve may not jump
to another branch unless the current branch becomes invalid, in which case it
fails rather than flipping silently.

The solver continues to report under-defined, fully-defined, over-defined, and
conflicting states with residual evidence. No solver residual, Jacobian, branch
seed, or iteration cache becomes project source.

## Geometry Boundary

### Resolved primitives

Cad-core/app orchestration resolves document refs into serializable geometry:

```ts
interface ResolvedLineSegment2d {
  readonly kind: "line";
  readonly sourceEntityId: string;
  readonly start: Vec2;
  readonly end: Vec2;
}

interface ResolvedArcSegment2d {
  readonly kind: "arc";
  readonly sourceEntityId: string;
  readonly center: Vec2;
  readonly radius: number;
  readonly startAngleDegrees: number;
  readonly sweepAngleDegrees: number;
}

interface ResolvedPlanarWire {
  readonly frame: ResolvedPlaneFrame;
  readonly closed: boolean;
  readonly segments: readonly (ResolvedLineSegment2d | ResolvedArcSegment2d)[];
  readonly sourceIdentity: string;
}
```

The geometry worker receives segments already ordered and traversal-adjusted.
It revalidates finite numbers and gross connectivity defensively, but it does
not resolve sketch IDs, choose candidates, reorder segments, or mutate source.

### OCCT construction

- Lines use exact line edges.
- Arcs use exact circular edges from canonical center/radius/angle data; they
  are not tessellated polylines.
- Closed profile wires are checked with the kernel wire analyzer before face
  construction.
- Extrude uses the existing prism/boolean pipeline with the exact wire face.
- Revolve uses the existing revolution pipeline with the exact wire face.
- Sweep uses one exact ordered path wire and the fixed V17 frame behavior.
- Every operation checks builder completion, null shape, shape validity, solid
  count, and exact metadata before returning success.

The worker response remains serializable mesh/exact metadata/topology evidence.
Raw OCCT objects are disposed within the isolated boundary on success and every
failure path.

### Solid policy

Composite extrude and revolve Must return exactly one solid for `newBody`.
Composite add/cut follows existing single-result/compound diagnostics. Curved
sweep Must return exactly one solid. A shell, face, wire, null shape, or
unexpected multi-solid sweep/revolve is failure, not a successful body with a
warning.

## Storage and Migration

### V21 trigger and minimum-schema export

A project requires V21 if any of the following is present:

- an arc entity;
- any entity with `construction: true`;
- a wire `SketchProfileRef`;
- any path other than one forward-oriented line (an arc, reverse line, or
  chain);
- any other V17-only source record defined in this contract.

When a project requires V21, export writes all sketch entities with explicit
construction booleans and all extrude/revolve/sweep/loft profile/path inputs in
the normalized V21 shapes. When no V17 source is present, export retains the
existing lowest-sufficient-schema behavior; opening and saving a V20 project
does not upgrade it solely because the app supports V21.

An entity-only rectangle/circle profile ref and a forward single-line path are
losslessly representable in V20 and do not trigger V21 by themselves. When a
lower schema is selected, export deterministically writes their legacy
`sketchId`/`entityId` or `pathSketchId`/`pathEntityIds` fields. No wire, arc,
reverse path, chain, or true construction state may be down-converted.

### Load and migration

- V1–V19 migration remains unchanged through their existing steps.
- V19→V20 migration remains exactly the V16 contract.
- V20→V21 in-memory normalization sets every construction flag to `false`,
  maps rectangle/circle feature inputs to `SketchEntityProfileRef`, maps a
  V20 single-line sweep path to an oriented `SketchEntityPathRef` with
  `orientation: "forward"`, and maps loft sections to entity profile refs.
- V20 source does not become V21 on export unless a V21 trigger exists.
- V21 import requires normalized fields and rejects duplicated legacy fields.
- Unknown entity kinds, invalid arcs, invalid construction values, dangling
  segments, mixed-sketch wires, invalid orientations, and unsupported consumer
  combinations are structured import errors with exact JSON paths.
- Import never discovers or repairs a missing profile/path automatically.

### History, hashes, and packages

- `partbench.wcad.v2` is unchanged.
- Canonical CBOR and source-identity hashing include arc fields, construction
  booleans, normalized refs, order, and orientation.
- JSON and `.wcad` round-trips preserve signed sweep, normalized profile order,
  path order, names, IDs, counters, history, redo, topology metadata, and
  checkpoint payloads.
- JSON retains the existing warning that authoritative checkpoint bytes require
  `.wcad`.
- OPFS meshes and exact metadata remain cache entries keyed by new source
  identity; old cache entries may miss and rebuild but are never migrated into
  source.

## Feature-Specific Validation

### Composite extrude

- `profile.kind` may be entity rectangle/circle or a valid wire.
- Depth must satisfy existing finite positive rules.
- Sketch placement uses the existing base/attached planar frame resolver.
- `newBody` has no target; `add`/`cut` require exactly one ready active target.
- The tool wire must produce one planar face before prism construction.
- Add/cut uses the existing target lifecycle: target consumed, result active,
  no source-body mutation outside the transaction.
- A `newBody` result must be one valid solid.
- Existing rectangle/circle behavior and stable IDs must not change.

### Composite revolve

- Axis remains a non-zero sketch line resolved through existing rules.
- Axis may be construction or regular geometry.
- A composite wire and its axis line must belong to the same sketch. Existing
  rectangle/circle revolves retain their V16 compatibility matrix.
- Axis entity cannot also appear in the profile.
- The profile must not cross the axis. Touching the axis at one or more profile
  vertices is allowed only when the kernel produces one valid solid; an edge
  overlapping the axis is rejected.
- Angle remains finite, `0 < angleDegrees <= 360`.
- V17 supports `newBody` only and exactly one result solid.

### Curved sweep

- Profile remains one rectangle or circle entity on a resolved planar frame.
- Path is a valid oriented line, arc, or open G1 line/arc chain.
- Path and profile may belong to different sketches.
- New arc/chain paths must meet the fixed start point/normal frame rule.
- Profile and path references may not identify the same `(sketchId, entityId)`
  pair.
- The profile must not have zero area and the path must have non-zero length.
- Self-intersection of the resulting solid, excessive curvature relative to
  profile size, and other pipe failures are kernel failures; no partial derived
  shape is published and the committed source feature reports failed geometry.
- Exactly one valid result solid is required.

## Dependency, Editability, and Failure Semantics

- A wire consumer depends on its sketch and every ordered entity ID.
- A path consumer depends on its sketch and every ordered entity ID.
- Dependency graphs preserve order in detailed input metadata while node/edge
  sets remain de-duplicated.
- Rebuild planning includes feature consumers transitively through bodies and
  downstream references.
- `feature.editability` returns proposed-ref diagnostics without mutation.
- Toggling a profile member to construction invalidates dependent solid
  profiles; toggling a path member remains valid if all path rules hold.
- Renaming an entity never invalidates a ref because identity is ID-based.
- Reordering sketch storage never reorders a stored profile or path.
- Failed rebuilds clear current derived geometry/exact metadata for affected
  bodies and surface stale checkpoint/reference health. They do not delete
  source or silently retain an old body as current. A source-validation failure
  consumes no target. Once a valid add/cut source command commits, its target
  remains consumed by the authored feature graph even if async geometry later
  fails; undo, delete, or a valid retarget follows the existing lifecycle.

## STEP Export and Interchange

Exact STEP export must support active V17 bodies whose exact-metadata/source
recipe is healthy: composite new-body extrudes, supported composite boolean
results, composite revolves, and curved sweeps. Export uses the exact B-rep,
not display tessellation.

IGES, DXF/SVG sketch export, proprietary CAD, STEP curve import into editable
sketches, XDE color expansion, and mesh-format expansion are not V17 work.

## Security and Resource Rules

- Arc/profile/path input arrays use the existing batch and document size limits;
  V17 adds no hidden unbounded recursion.
- Candidate discovery is iterative and detects visited entities; it must
  terminate on malformed graphs.
- Import validation completes before OCCT execution.
- No expression text is evaluated as JavaScript.
- Worker cancellation and stale response rejection follow existing source
  identity/request token rules.
- Failed or cancelled OCCT builders release allocated objects.

## Explicit Non-Goals

- B-splines, NURBS, ellipses, conics other than circular arcs.
- General trim, extend, split, offset-curve, project/intersect geometry tools.
- Polyline as a distinct entity; use ordered line entities.
- Full circles encoded as arcs.
- Multiple outer regions, inner loops, islands, nested profiles, thin features,
  or open-profile extrusion.
- Composite profiles for sweep or loft.
- G0 sweep corners, closed sweep paths, 3D paths, guide rails, twist/scaling
  laws, or sweep add/cut.
- Loft guides or changes to the V16 loft support matrix.
- Free datum, offset, angled, tangent, or midplanes; datum axes/points.
- Direct parameter binding to feature fields such as pattern spacing/count,
  mirror offset, or sweep controls.
- New pattern seed breadth or independent pattern bodies.
- General unrestricted booleans or direct edit features.
- Assemblies, mates, instance hierarchy, materials, drawings, or GD&T.
- Production WebGPU, exact subentity highlight rewrite, selection-buffer
  overhaul, or renderer authority.
- IGES, proprietary CAD, editable STEP sketch curves, DXF/SVG drawing exchange.
- Hosted collaboration, natural-language parsing, production MCP auth.
- New production dependencies unless separately explained and approved. V17 is
  expected to use existing TypeScript math and OCCT capabilities.

## Rejected Alternatives

1. **Store tessellated arc points.** Rejected: loses exactness and stable intent.
2. **Store three points as the arc source.** Rejected: solver and edit semantics
   become branch-sensitive; three points remain command input sugar.
3. **Let OCCT order touching curves.** Rejected: kernel order is not authorial
   source and cannot support deterministic agents or topology roles.
4. **Infer the largest closed region.** Rejected: wrong in multi-loop sketches
   and silently changes intent after edits.
5. **Persist candidate IDs.** Rejected: candidates are derived; features store
   their exact oriented entity references.
6. **Use one type for profiles and paths.** Rejected: closure, construction,
   orientation, and consumer semantics differ.
7. **Allow arbitrary G0 pipe corners because OCCT sometimes succeeds.**
   Rejected: output and transition behavior would be under-specified.
8. **Explode rectangle/circle into hidden line/arc entities.** Rejected: breaks
   compatibility and existing stable IDs.
9. **Support composite loft/sweep profiles opportunistically.** Rejected: two
   new geometry dimensions at once weaken the testable matrix.
10. **Upgrade every save to V21.** Rejected: violates minimum-schema export and
    causes needless source churn.
11. **Rewrite historical commands to V21.** Rejected: damages audit/history
    identity; compatibility readers remain deliberate.
12. **Treat an old mesh as current after a failed edit.** Rejected: rendered
    caches cannot override document/rebuild truth.
13. **Call OCCT from cad-core to validate loops.** Rejected: violates package
    boundaries and synchronous command/query determinism.
14. **Use source entity names as topology keys.** Rejected: names are editable
    labels, not identity.
15. **Combine V17 with datum planes or assemblies.** Rejected: those are
    separate architecture and product risk clusters.

## Risk Controls

| Risk | Required control |
| --- | --- |
| Arc solver jumps branches | Seed from current geometry; preserve sweep sign; fail instead of silent branch flip |
| Near-tolerance loops vary by layer | One normative policy; analytic cad-core validation; worker only defensive |
| Kernel wire order changes topology | Send ordered edges; key roles from source IDs; prove correspondence or report unavailable |
| Boolean splits destroy segment identity | Reuse checkpoint lineage; never claim source-semantic split-face identity |
| Pipe twists unexpectedly | Fixed corrected-Frenet/parallel-transport behavior and strict start frame |
| UI chooses wrong loop | Query candidates; explicit selection; no hidden largest/nearest choice |

| Feature breadth outruns recipes | Complete vertical-slice gate per matrix row |
| Stretch displaces Must | Stretch forbidden until full Must validation is green |
