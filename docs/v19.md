# V19 Release Plan: Production Sketching and Multi-Region Profiles

Status: **Reviewed proposed specification; not implemented**

Product release V19 is the first capability release after the completed V18
Precision CAD UI overhaul. Its center is production-grade 2D sketch editing
and explicit material-region profiles: deterministic trim, extend, split, and
offset tools; completed access to the existing solver-backed constraint
families; broader driving dimensions; slots and rounded rectangles as command
sugar; and exact profile regions with inner loops for a deliberately bounded
extrude/revolve matrix.

This document is the reviewed proposed V19 scope. It does not mark any V19
capability as implemented. Until implementation begins,
`docs/implementation-plan.md`, `docs/v17.md`, and `docs/v18.md` remain the
current implementation source of truth.

## Product Goal

V17 established canonical arcs, construction geometry, explicit ordered
line/arc wires, analytic profile validation, and curved paths. V18 made those
capabilities available through one professional workbench without expanding
the modeling matrix. V19 should make that Sketch workbench capable of authoring
the profiles used in common prismatic and revolved parts without requiring
manual deletion/recreation of every curve.

After V19, a user must be able to:

1. trim, extend, and split supported lines, arcs, and circles through typed
   CADOps mutations with deterministic identity and dependency behavior;
2. offset supported individual curves and explicit line/arc chains without
   storing kernel curves or hidden sketch entities;
3. create slots and rounded rectangles as ordinary constrained line/arc
   geometry;
4. create and structurally edit the exact V19 constraint matrix through UI,
   CADOps, agents, and MCP, while retaining explicit compatibility for legacy
   angle-constraint records;
5. drive point distances, horizontal/vertical distances, point-to-line
   distances, radii, and diameters with literal or parameter values, and drive
   line angles with sense-stable literal values;
6. discover, inspect, and explicitly select whole-loop bounded material regions
   formed by rectangle, circle, line, and arc loops;
7. extrude one region with holes as a new body, or apply one or more disjoint
   regions in the supported add/cut target matrix;
8. revolve one region with holes as a new body;
9. edit source geometry and receive honest dependent-feature health rather than
   an automatically substituted region or stale mesh;
10. save, open, undo, redo, dry-run, inspect, and automate every V19 source
    shape through the same architecture used by human tools; round-trip that
    source through JSON/`.wcad`; and export each supported resulting body
    through STEP when exact readiness is green.

V19 is a large vertical capability release, not a general sketch-curve or
arbitrary-profile project. Its Must matrix remains limited to lines, circles,
canonical circular arcs, rectangles, one-level region holes, and the consumer
rows stated below.

## Why V19 Comes After V18

The current product can construct exact composite line/arc loops, but editing
those loops still requires low-level entity operations. It can consume one
rectangle, circle, or simple wire as a profile, but it cannot represent a
material region with an explicit internal void. Those omissions are visible in
ordinary plates, brackets, flanges, rings, and turned sections.

The completed releases already provide the required foundations:

- V11 provides source-backed dimensions, constraints, numerical solver
  diagnostics, and command/query transaction behavior.
- V13 and V14 provide topology identity, checkpoints, anchors, repair, and the
  bounded downstream-target readiness matrix.
- V16 provides exact sweep/loft recipes, expression language v2, exact metadata,
  and mass properties.
- V17 provides canonical line/arc geometry, analytic intersection and area
  policy, oriented profile/path references, deterministic candidate discovery,
  and source-semantic topology roles.
- V18 provides one Sketch mode, shared action availability, explicit draft and
  commit behavior, command search, accessible interaction primitives, and
  performance gates.

V19 extends those contracts. It must not replace the solver, topology system,
command engine, geometry boundary, renderer, native package, or V18 workbench
composition.

## Non-Negotiable Architecture

1. CADOps remains the only mutation authority. Pointer gestures, contextual
   tools, convenience tools, scripts, agents, MCP, tests, and import helpers all
   submit the same typed commands.
2. `cad-core` owns sketch source, profile-region source, validation,
   transactions, semantic diffs, dependency health, undo, redo, and queries.
3. `packages/sketch-solver` remains a pure numerical package. It receives a
   normalized solve model and writes no document source.
4. Analytic line/circle/arc editing and region validation are pure TypeScript
   policy. `cad-core` must not call OCCT to decide command validity.
5. OCCT builds exact wires, faces, prisms, booleans, and revolutions only after
   synchronous source validation succeeds.
6. Renderer IDs, tessellated points, pixels, mesh edges, OCCT indexes, and
   candidate IDs never become sketch or profile source authority.
7. A committed feature stores the exact submitted region membership and
   outer/hole roles after the documented orientation, cyclic-start, hole-order,
   and region-order normalization. Candidate discovery remains derived and
   query-only.
8. Existing V17 source IDs and generated-reference semantics are preserved.
   Curve-edit commands expose explicit identity mappings; they never disguise a
   delete/recreate operation as identity preservation.
9. Constraint and dimension removal is never an implicit side effect of a
   curve-edit command. Any required deletion must be named exactly in the
   command and semantic diff.
10. A retained-ID source edit may invalidate a dependent feature under the
    completed V17 policy. The feature record remains; rebuild health becomes
    blocked/failed; an old mesh is never shown as current. The completed
    `SKETCH_ENTITY_IN_USE` invariant continues to block deletion of a feature
    input ID.
11. V19 uses the existing V18 action registry, editor state machine,
    diagnostic translation, selection projection, accessibility rules, bundle
    budgets, and performance measurement approach.
12. V19 adds no production dependency by default. Any implementation proposal
    that needs one must explain its bundle, license, security, and maintenance
    cost and receive separate approval.
13. `.wcad` remains `partbench.wcad.v2`. V19 source changes do not justify a
    package-layout version.
14. V19 does not expand assemblies, WebGPU, hosted collaboration, native
    storage, STEP import breadth, MCP transports, or authentication.

### Complete vertical-slice rule

A V19 row is supported only when all of the following are complete:

```text
protocol and validation
-> cad-core dry-run/commit and semantic diff
-> dependency, undo, and redo behavior
-> solver or analytic policy where applicable
-> exact geometry and derived display where applicable
-> exact metadata, topology health, and STEP readiness where applicable
-> V18 UI action/editor/gesture
-> agent and MCP parity
-> JSON/.wcad round-trip when source changes
-> focused and release-workflow proof
```

A helper, type, private OCCT experiment, query-only prototype, hidden button, or
mocked mesh does not make a row product-supported. Unsupported rows remain
absent or visibly blocked through shared readiness.

## Normative Decisions

### 1. Product V19 uses minimum-triggered schema `web-cad.project.v22`

Product release numbers and project schema numbers are independent. Product
V15 already introduced `web-cad.project.v19`; product V17 introduced
`web-cad.project.v21`; product V18 added no schema. V19 source shapes that
cannot be represented in V21 introduce:

```text
web-cad.project.v22
```

V22 is required only when source contains at least one:

- `SketchRegionsProfileRef`;
- `entityScalar` diameter dimension target;
- `pointPair`, `pointLineDistance`, or `lineAngle` dimension target;
- V19-only command shape or V19-only semantic-diff shape in committed history
  or the redo stack;
- future source shape explicitly listed as a V22 trigger by an approved
  amendment to this document.

The resulting ordinary V21 line/arc/circle/rectangle source from trim, extend,
split, explode, offset, slot, or rounded rectangle does not by itself require
V22. Their V19-only operation and diff records do require V22 while they remain
in history or redo. Lower-schema export is permitted only when the live
document, history, and redo stack contain no V22 trigger. V19 does not rewrite
those operations into older primitive command/diff shapes.

V22 changes neither the top-level project document nor `.wcad` layout. It
participates in canonical CBOR, source identity, hashes, history, redo, package
validation, JSON diagnostics, and minimum-schema export.

### 2. Curve editing creates only ordinary existing entity kinds

V19 adds no trim entity, offset entity, polyline entity, slot entity, rounded
rectangle entity, intersection entity, or kernel-curve entity.

The source result of every curve edit is composed from the existing canonical
entity union:

```text
point | line | rectangle | circle | arc
```

Trim, extend, split, and offset are explicit commands that create, update, or
delete those entities in one transaction. Slot and rounded rectangle are
convenience commands that create ordinary line/arc entities and supported
constraints in one transaction.

No operation stores tessellated points, OCCT handles, inferred adjacency,
candidate IDs, cursor pixels, or a hidden parent composite.

### 3. One analytic policy controls intersections, projection, and regions

V19 reuses V17's `SketchGeometryPolicy`:

```ts
interface SketchGeometryPolicy {
  readonly linearTolerance: 1e-7;
  readonly angularToleranceDegrees: 0.1;
  readonly minimumProfileArea: 1e-12;
}
```

V19 extends the pure policy implementation with:

- finite line/line, line/circle, and circle/circle intersections;
- finite-arc interval filtering;
- closest-point projection to line, circle, and finite arc;
- curve parameter ordering with tolerance-aware duplicate collapse;
- infinite-support intersection for extend and offset joins;
- signed containment and boundary-distance tests;
- loop/loop intersection, touching, containment, and material-area tests.

The literal tolerances remain unchanged. V19 must not introduce a second
viewport tolerance, solver tolerance, geometry-worker tolerance, or
pixel-derived source tolerance. UI hit testing may use screen-space affordance,
but the submitted model-space point must pass the shared analytic policy.

### 4. Curve edits use explicit geometry and deterministic disambiguation

Each edit command contains all source IDs and model-space choices needed to
recompute its result. The command engine does not consume an opaque query
candidate token.

Circle editing uses one canonical parameter: sketch-local polar angle in
`[0, 360)` from `+X`, increasing counterclockwise. Intersections/split points
sort by that parameter; within-tolerance cyclic duplicates collapse, including
across the `360 -> 0` seam. Circle-derived arcs use canonical start angles,
positive counterclockwise sweeps, and increasing start-parameter order, with
the seam-spanning arc last. Trim removes the picked adjacent cyclic interval
and creates its one complementary arc; split creates one arc for every
adjacent parameter interval. Every generated arc must satisfy the complete V17
radius and signed-sweep domain:

```text
radius > linearTolerance
angularToleranceDegrees <= abs(sweep) <= 360 - angularToleranceDegrees
```

Trim:

- supports a target line, arc, or circle;
- requires one or more explicit boundary line/circle/arc IDs;
- permits regular or construction boundaries on the same sketch;
- rejects the target as its own boundary, missing boundaries, coincident line
  supports, coincident circles, and overlapping arc intervals;
- intersects each boundary as finite authored geometry; only the target is
  partitioned along its finite canonical interval;
- projects `pickPoint` onto the target;
- removes the finite target interval containing the projected pick;
- rejects a pick farther than `linearTolerance` from the target;
- requires at least one distinct interior intersection for a line/arc and at
  least two for a circle;
- treats a tangent contact as one distinct intersection and collapses duplicate
  contacts within tolerance;
- rejects a plan that would retain zero pieces or only zero-length pieces;
- rejects a pick on an intersection or a tie between intervals as ambiguous;
- never treats an unlisted sketch entity as a boundary.

Extend:

- supports one endpoint of a line or arc;
- requires explicit boundary IDs;
- permits regular or construction finite boundary geometry on the same sketch;
- rejects the target as its own boundary, coincident/overlapping supports, and
  an invalid or zero-length target;
- uses an outward target ray from the selected line endpoint or an unwrapped
  angular ray continuing the authored signed arc sweep;
- intersects that target ray with each finite boundary entity, not the
  boundary's infinite support;
- chooses the closest positive extension within tolerance;
- rejects equal-distance ambiguity, zero extension, arc sweep at or above the
  V17 full-circle bound;
- reconstructs an arc by preserving center/radius and updating
  `startAngleDegrees` and signed `sweepAngleDegrees` according to whether the
  authored start or end was extended;
- does not support circle or rectangle targets.

Split:

- supports line and arc targets at one or more projected model-space points;
- supports a circle only with at least two distinct split points;
- sorts points by canonical curve parameter;
- collapses within-tolerance duplicates;
- rejects the entire command if any submitted line/arc split point projects
  within tolerance of an existing endpoint, even when other points are valid;
- rejects zero-length or V17-domain-invalid result pieces;
- rejects coincident/overlapping cutter semantics because split consumes
  explicit points rather than implicit boundaries.

Rectangle entities are not silently exploded when an edge is clicked. A
separate explicit rectangle-explode command creates four ordinary lines in
counterclockwise boundary order:

```text
vMin: (uMin, vMin) -> (uMax, vMin)
uMax: (uMax, vMin) -> (uMax, vMax)
vMax: (uMax, vMax) -> (uMin, vMax)
uMin: (uMin, vMax) -> (uMin, vMin)
```

The `lineEntityIds` tuple follows exactly `vMin, uMax, vMax, uMin`.
Circle-to-arc conversion during trim/split is reported as delete/create because
entity kind cannot change under one ID.

Every trim, split, extend, circle-to-arc, and rectangle-explode result inherits
the edited target's `construction` value. No curve edit changes construction
intent implicitly.

### 5. Entity identity follows authored-parameter continuity

When an edit can preserve entity kind:

- extend preserves the target ID;
- a one-piece trim preserves the target ID;
- a multi-piece line/arc trim or split preserves the target ID on the result
  interval containing the original canonical parameter zero when that interval
  survives;
- otherwise, a multi-piece line/arc trim preserves the target ID on the
  retained interval with the lowest canonical start parameter;
- additional pieces receive caller-supplied or deterministically allocated new
  IDs in increasing source-parameter order.

When an edit changes kind, including circle-to-arc conversion or rectangle
explosion, no output inherits the deleted source ID. The semantic diff contains
an explicit replacement map:

```ts
interface SketchEntityReplacement {
  readonly sourceEntityId: string;
  readonly disposition: "modified" | "deleted";
  readonly resultEntityIds: readonly string[];
  readonly preservedResultEntityId?: string;
}
```

Replacement maps are transaction evidence and query output, not persistent
aliases. Named references, features, constraints, and dimensions still resolve
through their real source IDs.

A curve edit that would delete an entity referenced by a persisted feature
profile, path, or axis is rejected with `SKETCH_ENTITY_IN_USE`. V19 never
commits a dangling feature input. The caller may first delete the dependent
feature through an ordinary explicit feature command; kind-changing retarget
plus delete is not hidden inside a curve-edit operation. A kind-preserving edit
that retains the referenced ID may make its profile/path analytically invalid
under Decision 15.

### 6. Constraint and dimension consequences are explicit

Curve-edit readiness uses this normative pipeline:

```text
current committed-revision evaluated sketch geometry
-> analytic edit plan
-> deterministic target retargeting
-> residual evaluation without a free solve
-> complete invalid-record sets
-> evaluated preview and dependency effects
```

If the sketch has constraints/dimensions, readiness requires a deterministic
evaluation at the current source revision with status `solved`,
`fully-defined`, `under-defined`, `over-defined`, or `redundant`. Conflicting,
failed, unsupported, missing-target, stale, or non-converged input blocks the
edit. A sketch with no solver records uses its canonical committed source
geometry. Uncommitted drag/solver previews are never eligible.

Apply rechecks both source revision and solver-evaluation identity. The edit
plan is a hard geometric result and is materialized into authored entity
coordinates. Readiness does not run a free solve that may move the requested
intersection, restore an old length, or move unrelated entities.

Every affected preserved/retargeted constraint or dimension uses the same
normalized residual evaluator and family-specific satisfaction threshold as
`sketch.solverStatus`. A failing residual is `invalid` and requires explicit
deletion; there is no single raw tolerance applied to heterogeneous residual
families.

Readiness classifies every affected constraint and dimension as:

```text
preserved
retargeted
invalid
deleted-by-request
unaffected
```

Safe retargeting is limited to a unique geometric continuation:

- an original start/end point target may move to the unique result entity that
  still owns that authored endpoint;
- fixed/coincident endpoint intent may follow that unique endpoint;
- center targets survive only when the same circle/arc ID and center survive;
- curve-wide length, radius, sweep, tangent, equal, symmetry, and angular
  relationships are not cloned across result pieces;
- a curve-wide record on a one-piece edit is preserved only if its residual
  remains satisfied by the planned geometry;
- a curve-wide record on a multi-piece or kind-changing edit is invalid unless
  its exact target has one unambiguous surviving meaning and remains satisfied;
- no constraint is moved to a merely nearby point or curve.

Commit and ordinary command dry-run require `deleteConstraintIds` and
`deleteDimensionIds` to equal the complete corresponding invalid-record sets:
no omissions, extras, duplicates, or unrelated IDs. A missing/mismatched list
returns a typed validation error with the complete impact evidence and no
semantic diff. Callers may use `sketch.curveEditReadiness` first to obtain the
same sets and evaluated preview. A successful dry-run/commit records each
explicit deletion and the post-edit solver classification in its semantic
diff.

After commit, normal solver-status queries evaluate the new source but do not
silently rewrite the curve-edit geometry. Successful commit permits
`not-run` only when no solver records exist and otherwise permits `solved`,
`fully-defined`, `under-defined`, `over-defined`, or `redundant`.
Conflicting, failed, unsupported, missing-target, or domain-invalid results
cannot reach commit. Over-defined/redundant is allowed but reported as a
warning rather than relabeled.

The UI must show the complete consequence before Apply. Agents and MCP receive
the same readiness structure. There is no `"deleteAllInvalid": true` shortcut.

### 7. Offset is exact command sugar, not an associative feature

V19 offset creates independent ordinary sketch geometry. It does not add an
offset entity, offset constraint, parent-child link, or persistent distance
relationship, and it creates no constraints.

Must-supported sources:

| Source | Required result |
| --- | --- |
| One line | Parallel line at positive finite distance |
| One circle | Concentric circle with positive resulting radius |
| One arc | Concentric arc with preserved signed sweep |
| Rectangle entity | Rectangle when the selected side preserves positive width/height |
| Ordered open line/arc chain | Offset open chain with analytic natural joins |
| Ordered closed line/arc loop | Offset closed loop when it remains simple and non-collapsed |

The command takes a positive distance and explicit `side`:

```text
individual line/arc or open chain -> left | right
circle, rectangle, or closed loop -> inward | outward
```

Left/right is relative to authored direction for an individual entity and
submitted oriented traversal for a chain. A pointer workflow may also supply a
model-space `referencePoint` as evidence. It projects that point to the unique
closest finite source point and canonical curve parameter, rejects multiple
equal closest parameters (including a circle/arc-center point), equal-distance
segment ambiguity, or a point on source, and classifies the side from the
oriented unit tangent at that unique projection. The derived side must equal
the explicit command side. Closed source evidence uses strict containment for
inward/outward. Keyboard/agent callers may omit the evidence point and submit
the explicit side directly.

Chain validation requires:

- every segment is a line or arc on the declared sketch;
- every entity ID occurs once;
- oriented consecutive endpoints coincide within `linearTolerance`;
- `closed: true` exactly matches last-to-first endpoint closure;
- the source chain has no self-intersection, overlap, or branch;
- G0 joins are allowed; tangency is not required;
- an individual entity inherits its `construction` value;
- every member of a chain has the same construction value, which every output
  inherits; mixed regular/construction chains are rejected.

Offset produces exactly one entity per source segment in traversal order.
Rectangle input produces one rectangle entity. Circle input produces one circle
entity. An individual entity output preserves authored line direction or arc
sweep sign. Each chain output follows submitted traversal direction, so every
output line start-to-end and arc signed sweep forms a forward-connected chain.
Caller-supplied output IDs follow that exact order.

V19 has one join policy:

- intersect adjacent offset analytic supports;
- choose the unique join consistent with source traversal and local side;
- when adjacent supports are coincident because the source contains collinear
  consecutive lines or consecutive arcs on one circle, map their authored
  shared endpoint by the same translation/radial offset and validate it against
  both offset curves;
- retain exact line/circle support;
- reconstruct every offset arc on one unwrapped submitted-traversal interval
  and reject reversal, an extra wrap, or any radius/sweep outside the complete
  V17 arc domain;
- reject no-solution, multiple-solution, reversal, zero-length, self-
  intersection, or a miter farther than ten times the offset distance from
  either unjoined endpoint.

No bevel, round, variable-distance, chained associative, spline, 3D, face-edge,
or kernel-wire offset is implied. An implementation must not publish an
approximate polyline when the analytic result is unsupported.

### 8. Slots and rounded rectangles are atomic CADOps sugar

V19 promotes the V17 slot and rounded-rectangle Stretch concepts to Must.

`sketch.addSlot` accepts centerline start/end and radius. It creates:

- `sidePositive`: line from `start + normal * radius` to
  `end + normal * radius`;
- `endCap`: arc centered at `end`, starting at the positive-normal point with
  signed sweep `-180`;
- `sideNegative`: line from `end - normal * radius` to
  `start - normal * radius`;
- `startCap`: arc centered at `start`, starting at the negative-normal point
  with signed sweep `-180`.

`normal` is the left normal of the normalized centerline start-to-end unit
direction. The stable entity-ID tuple order is exactly the order above.
The stable minimal constraint order is:

1. four coincident constraints at consecutive loop joins;
2. four line/arc tangent constraints at those joins;
3. one equal-radius constraint from `endCap` to `startCap`.

The command does not add a redundant parallel constraint.

`sketch.addRoundedRectangle` accepts center, width, height, and corner radius.
Its stable entity tuple traverses clockwise from the top edge:

```text
top
topRightArc
right
bottomRightArc
bottom
bottomLeftArc
left
topLeftArc
```

Lines stop one corner radius before each nominal rectangle corner. Each arc has
signed sweep `-90`. The stable minimal constraint order is:

1. eight coincident constraints at consecutive loop joins;
2. eight line/arc tangent constraints at those joins;
3. three equal-radius constraints from `topRightArc` to each other arc;
4. horizontal constraints on `top` and `bottom`;
5. vertical constraints on `left` and `right`.

It adds no parallel or perpendicular constraints already implied by that
minimal graph.

Inputs must leave every line and arc above tolerance. A rounded rectangle with
zero radius is rejected in this command; callers use `sketch.addRectangle`.
A slot with coincident centerline endpoints is rejected; callers use a circle.

The command commits all entities and constraints atomically, produces one
semantic diff, and undoes/redoes as one transaction. Callers may supply every
entity/constraint ID in the normative order for deterministic automation. The
omitted/generated arrays are fully materialized into committed history. The
`construction` input applies to every created entity. A default result must be
under-defined or fully-defined but never redundant, over-defined, or
conflicting. No hidden group or special feature identity is stored.

### 9. Material regions are explicit loop sets

V22 adds loop-local reference types without duplicating `sketchId`:

```ts
interface SketchEntityLoopRef {
  readonly kind: "entity";
  readonly entityId: string; // rectangle | circle only
}

interface SketchWireLoopRef {
  readonly kind: "wire";
  readonly segments: readonly OrientedSketchSegmentRef[];
}

type SketchLoopRef = SketchEntityLoopRef | SketchWireLoopRef;

interface SketchProfileRegionRef {
  readonly outer: SketchLoopRef;
  readonly holes: readonly SketchLoopRef[];
}

interface SketchRegionsProfileRef {
  readonly kind: "regions";
  readonly sketchId: string;
  readonly regions: readonly [
    SketchProfileRegionRef,
    ...SketchProfileRegionRef[],
  ];
}

type SketchProfileRefV22 =
  | SketchEntityProfileRef
  | SketchWireProfileRef
  | SketchRegionsProfileRef;
```

A region is material inside its outer loop and outside each listed hole. A
regions profile is the union of its listed material regions. The feature owns
the complete explicit list. A query result, shaded cell, inferred containment
tree, or cursor hit is not source authority.

Legacy entity and wire profile refs remain valid and retain V17 behavior. They
are not rewritten to one-region records merely because a V22 project exists.

### 10. Region validation and normalization are deterministic

Every loop must satisfy the V17 closed-profile rules:

- one rectangle/circle entity or at least two non-construction line/arc
  segments;
- finite non-zero geometry;
- closed consecutive endpoints;
- no repeated entity within the loop;
- no self-intersection, overlap, branch, or area below
  `minimumProfileArea`.

Additional region rules:

- all loops belong to the declared sketch;
- an entity ID appears in at most one loop in the submitted regions profile;
- every hole is strictly inside its region outer;
- hole boundaries do not touch or intersect the outer boundary;
- holes do not touch, overlap, or contain one another;
- each hole boundary remains more than `linearTolerance` from its outer and
  sibling-hole boundaries;
- material area after subtracting every hole is at least
  `minimumProfileArea`;
- material areas of distinct regions do not overlap;
- an outer loop of one region may lie inside a hole of another only when their
  material areas remain disjoint;
- boundary contact between distinct regions is rejected in V19;
- nesting within a single region stops at outer-plus-holes; a hole cannot
  contain another hole or island in the same region record.

Cad-core normalizes:

1. every wire outer to counterclockwise traversal;
2. every wire hole to clockwise traversal;
3. rotates each oriented wire cyclically to the lexicographically smallest
   sequence of `(entityId, orientation)`;
4. holes by canonical loop key;
5. regions by canonical outer-loop key.

Entity rectangle/circle loops have canonical analytic orientation during
resolution and do not gain stored segment lists. Normalization preserves IDs
and separately reports orientation, cyclic-start, hole-order, and region-order
changes in the feature input-reference semantic diff.

The canonical loop key is derived from loop kind plus ordered source entity IDs
and orientations after normalization. It is not a persisted candidate ID.

### 11. Candidate discovery describes whole-loop containment cells

V19 adds `sketch.profileRegionCandidates`. It analytically discovers simple
closed loops from eligible non-construction rectangles, circles, lines, and
arcs, computes strict containment, and returns deterministic material cells
formed only from complete valid loops.

Discovery accepts:

- one complete rectangle or circle entity; or
- a whole-entity endpoint-connected line/arc loop satisfying the V17 simple
  wire rules.

Interior crossings, overlaps, touching curves, branches, and partial curve
intervals never create virtual split vertices or subcurve refs. They are
reported as invalid discovery components. A user must first commit explicit
trim/split operations that create whole source entities.

The response includes:

- exact `SketchProfileRegionRef` data;
- canonical loop keys for display correlation only;
- outer/hole entity IDs;
- signed and material area;
- containment depth;
- validity and structured diagnostics;
- a source revision/fingerprint for cache invalidation, not command authority.

For every node in the strict containment tree, one candidate represents that
complete loop's interior minus all of its direct child loops. This rule applies
at every nesting depth and does not enumerate hole subsets. A child candidate
therefore uses that child as outer and its own direct children as holes.

Candidate ordering is:

1. increasing containment depth;
2. decreasing absolute material area;
3. canonical outer-loop key.

The UI may hit-test a cursor against returned analytic cells and may combine
only candidates whose material and boundaries are disjoint. A parent-minus-
child cell and that child cell share a boundary and cannot be combined in one
V19 profile. The feature command still submits the exact region refs. If the
sketch changes between display and Apply, command validation recomputes from
source and rejects stale/invalid geometry; it does not trust the old candidate.

### 12. V19 has one bounded profile-consumer matrix

| Consumer | Legacy entity/wire | One region with holes | Multiple regions | V19 operation modes |
| --- | --- | --- | --- | --- |
| Extrude | V17 behavior retained | **Must** | **Must for add/cut only** | `newBody`, `add`, `cut` |
| Revolve | V17 behavior retained | **Must** | Rejected | `newBody` only |
| Sweep profile | V17 behavior retained | Rejected | Rejected | `newBody` only |
| Loft section | V17 behavior retained | Rejected | Rejected | `newBody` only |
| Hole | Circle entity retained | Not applicable | Not applicable | Existing behavior |

Additional requirements:

- region extrude `newBody` accepts exactly one region and must produce one
  connected solid with zero or more through voids;
- region extrude add/cut accepts one or more disjoint material regions and the
  existing V14/V17 target-body readiness matrix;
- region tools execute sequentially in canonical region-key order;
- each add region must intersect the current result by positive volume, and
  each intermediate fuse must remain exactly one connected valid solid;
- each cut region must remove positive volume from the current result;
- a no-op region, complete target removal, disconnected intermediate/final
  result, or compound/multi-solid result is a structured derived-geometry
  failure;
- region revolve accepts exactly one region and inherits the full V17 axis

  no-crossing, and one-sided-material contract across every loop;
- V17-supported outer vertex-only axis contact is analytically source-valid;
  every hole must remain strictly separated from the axis by more than
  `linearTolerance`;
- the async worker must produce one valid solid for a vertex-contact case or
  record structured derived failure; an optional UI geometry preflight is
  orchestration readiness, never synchronous command validity;
- region revolve must produce one connected valid solid;
- a region profile never opts a feature into composite sweep or loft support;
- geometry failure after source-valid commit follows the existing derived-state

The normalized region order is source and recipe order. It participates in
source identity, exact-metadata identity, checkpoint evidence, and repeatable

No private OCCT success expands this table.

V22 validators recheck that each referenced entity has the declared kind and

V22 dimension semantics:

- Euclidean distance uses
  `length(secondary - primary) = value`.
- Horizontal/vertical uses the ordered component
  `secondary - primary = signed(value)`, where `direction` stores the sign.
- Point-to-line uses the infinite analytic support of the line. With
  `d = unit(line.end - line.start)`, its residual is
  `cross(d, point - line.start) = sideSign * value`, where left is `+1` and
  right is `-1`. Every accepted state retains that sign.
- Line angle uses authored start-to-end unit directions and
  `atan2(cross(primary, secondary), dot(primary, secondary))`, reversing the
  sign for clockwise sense before normalization. Creation and every accepted
  solver state must remain strictly within the stored sense's
  `(angularToleranceDegrees, 180 - angularToleranceDegrees)` branch. Crossing
  0/180 returns `SKETCH_DIMENSION_ANGLE_SENSE_INVALID`.
- Linear values are positive finite current-document lengths greater than
  `linearTolerance`.
- Radius values are greater than `linearTolerance`; diameter values are greater
  than `2 * linearTolerance`. Radius and diameter target the same solver radius
  variable, and diameter display/input is exactly twice the radius.
- Arc sweep retains its signed V17 source/branch rules.
- Rectangle width/height retains its completed direct source-backed update
  behavior; it is normalized for storage but is not misrepresented as a new
  V19 numerical solver residual.
- New V19 `lineAngle` dimensions accept a literal value source only. Untyped
  parameters cannot safely distinguish angular and linear scaling during
  `document.updateUnits`; typed quantities are outside V19. Existing
  parameter-backed V17 arc-sweep dimensions retain their compatibility
  behavior.
- `preservePhysicalSize` scales new linear literal dimensions and does not
  scale new angular literal dimensions. `metadataOnly` changes unit metadata
  while preserving every stored literal number. Tests cover both modes.

When V22 is emitted, every live dimension uses the normalized V22 snapshot.
V1-V21 entity-local dimensions normalize in memory. Entity scalar
width/height/length/radius/sweep targets down-convert when no other V22 trigger
exists. Diameter display intent and every multi-entity target remain V22.

- V21-representable entity-scalar dimensions continue to appear through the

### 15. Invalid downstream state remains honest

Feature creation and retargeting require a currently valid region profile.
Sketch edits are allowed to make an existing profile invalid under the V17
source-intent policy.

After such an edit:

- the original feature record and exact submitted region refs remain source;
- dependency health names the intersecting, non-closed, or containment failure;
- every persisted feature input ID still resolves; a delete-producing edit on
  an in-use entity was blocked by Decision 5;
- rebuild state becomes blocked or failed;
- matching old derived geometry is invalidated;
- the UI offers candidate repair/retargeting only as an explicit feature edit;
- no candidate is auto-selected based on area, proximity, or prior shading.

Undo restores the prior source and derived-readiness identity. Redo reapplies
the same semantic edit. File load recomputes candidate and dependency health
from source; it does not persist a previously green candidate cache.

### 16. Topology roles extend only for the V19 result matrix

### 18. Resource limits are explicit

V19 validators enforce finite bounded work before analytic traversal or OCCT:

- at most 4,096 sketch entities per edited sketch;
- at most 256 explicit boundary IDs in one trim/extend command;
- at most 1,024 split points in one command;
- at most 1,024 source segments in one offset;
- at most 256 regions, 512 total loops, and 4,096 total segment references in
  one regions profile;
- at most 512 discovered candidate regions per query;
- at most 250,000 analytic pair/edge visits during candidate discovery;
- at most 100,000 analytic predicate visits during synchronous submitted-
  profile command validation;
- no duplicate IDs in an explicit command list;
- no non-finite points, distances, angles, or radii.

Candidate discovery uses a deterministic spatial index/bounded pair generator.
In the browser it executes through the command worker as a cancellable,
revision-keyed query. Direct non-browser callers receive the same pure
synchronous result bounded by the predicate limits. Exceeding a candidate,
pair, or visit cap returns `SKETCH_REGION_COMPLEXITY_LIMIT`; it never returns an
incomplete set as if complete.

Responses page at most 100 candidates. Pagination uses the source revision plus
the last canonical candidate key; a revision mismatch is stale and restarts
the query. Optional explicit entity-ID narrowing is validated before
discovery. Limits are protocol constants with tests, not hidden OCCT or UI
thresholds.

submits that operation without reconstructing it from viewport state.

`expectedSourceRevision` has this exact form:

```text
partbench-source-v1:<lowercase sha256>
```

The digest is the current
`createCadProjectSourceIdentity(exportCadProject(engine)).sha256`, computed
over the committed document plus authoritative history and redo by the existing
canonical-CBOR source-identity algorithm. Consequently every authoritative
document, schema, units, history, or redo change invalidates it. Draft form
state, selection, camera, viewport scale, label placement, meshes, region
caches, worker cache eviction, and recomputation of identical derived results
do not.

`expectedSolverEvaluationIdentity` is the canonical literal `"none"` only when
the sketch has no solver records. Otherwise it has this exact form:

```text
partbench-sketch-solver-evaluation-v1:<lowercase sha256>
```

Its digest is SHA-256 of canonical CBOR over
`{ sourceRevision, sketchId, solverStatus, evaluatedEntities,
orderedConstraintResiduals, orderedDimensionResiduals }`. Entity, constraint,
and dimension records sort by ID; evaluated
point/line/arc/circle/rectangle geometry uses the canonical source shapes and
number normalization already used by the solver/query boundary; residual
records include ID, normalized family, status, and canonical numeric residual.
Presentation diagnostics and timing are excluded. The identity therefore
changes whenever the committed source revision or any evaluated
geometry/status/residual used by edit policy changes, but not when a worker
recomputes byte-identical evidence.

Readiness returns ready only when that evaluated evidence belongs to the same
current source revision and satisfies the Decision 6 status policy. Commit and
dry-run validate both identities at batch entry, before prospective ID
allocation, analytic planning, or mutation, and return
`SKETCH_EDIT_SOURCE_REVISION_STALE` or
`SKETCH_EDIT_SOLVER_STATE_BLOCKED` on mismatch.

Committed history retains the precondition as audit input. Undo/redo restores
the materialized transaction state. Package import validates the precondition
shape but canonical replay recomputes the operation/diff from serialized source
state and does not depend on transient worker state or reapply the optimistic
check.

Curve-edit commands use the normal CADOps batch envelope and dry-run/commit
modes, but a batch containing any `SketchCurveEditOp` must contain exactly that
one operation. A mixed or multi-edit batch fails before precondition validation
with `SKETCH_EDIT_BATCH_MULTIPLE_UNSUPPORTED`. This V19 boundary avoids hidden
intermediate readiness and staged-solver semantics; automation sequences
multiple edits as separate readiness/command transactions.

Counts and provided IDs must match the analytically determined result exactly
or the command fails without mutation. Commands return ordinary transaction,
audit, and actor metadata. When output IDs are omitted by a direct caller
rather than readiness, successful commit materializes the full ordered arrays
into the operation snapshot stored in history/redo before it is hashed. Replay
never regenerates multi-output IDs from mutable counters. Dry-run computes the
same prospective IDs without mutating counters.

## Solver Contract

V19 reuses the V17 normalized entity variables and Levenberg-Marquardt
diagnostics. New normalized residual inputs are:

```text
point-pair euclidean distance
point-pair horizontal distance
point-pair vertical distance
point-to-infinite-line perpendicular distance
directed line-to-line angle
circle/arc diameter mapped to radius
```

Required numerical behavior:

- seed from current evaluated geometry;
- preserve V17 signed arc sweep branches;
- preserve horizontal/vertical direction, point-line side, and line-angle
  sense exactly as stored;
- reject zero-length target lines before solve;
- resolve point, line endpoint, rectangle/circle center, and arc
  center/start/end targets through the normalized generalized point-target
  mapping;
- report rank, under-definition, redundancy, conflict, domain, and iteration
  failure through structured existing solver status;
- never commit solver residuals, matrices, iteration traces, or evaluated
  coordinates as hidden source.

Curve-edit commands operate on the deterministic current-revision evaluated
geometry defined by Decision 6, never an uncommitted solver preview. A sketch
with solver records and an unresolved/conflicting/stale/non-converged
evaluation blocks with a precise diagnostic; it never falls back to different
screen or raw-source geometry. Decision 6's residual check is not a free solve
and cannot move the analytic result.

Dimension application follows the existing source-backed solve/update model.
V19 does not introduce continuous background source mutation during drag.

## Geometry Boundary

`geometry-kernel`, `geometry-worker`, and `occt-wasm` add typed resolved region
recipes. Requests contain resolved finite analytic curves and loop roles, not
source candidate IDs or UI state.

Required construction order:

1. build exact outer and hole edges from resolved line/circle supports;
2. build ordered wires using the normalized traversal;
3. create one planar face per region with holes;
4. extrude or revolve according to the consumer row;
5. for add/cut, apply one prism at a time in normalized region-key order;
6. after every add, prove positive-volume intersection and one connected valid
   fused solid;
7. after every cut, prove positive volume removal and one non-empty connected
   valid remaining solid;
8. validate final non-null shape, solid count, connectivity, validity, and
   operation-specific policy;
9. tessellate and publish only for the matching source identity;
10. compute matching exact metadata and topology/checkpoint evidence.

OCCT sewing/healing must not make source-invalid gaps, overlaps, touching
regions, or self-intersections valid. Worker diagnostics distinguish source
validation failure, wire construction failure, face construction failure,
multi-solid result, boolean failure, invalid shape, and tessellation failure.

## Storage and Migration

### V22 load and in-memory normalization

The load path accepts V1 through V22.

V1-V20 follow their existing migrations. V21 source loads unchanged. In memory:

- legacy dimensions normalize to `SketchDimensionSnapshotV22`;
- legacy entity/wire profiles retain their existing public form;
- legacy constraints retain the V21 normalized target behavior;
- no region profile is inferred from old simple profiles;
- no command history event is rewritten.

Malformed V22 source is rejected before replacing the current document.
Import must reject:

- mixed legacy and V22 fields in one dimension;
- a regions profile with cross-sketch, missing, construction, repeated, open,
  intersecting, touching, overlapping, or invalidly nested loops;
- non-canonical loop orientation/order where V22 import requires canonical
  source;
- unsupported consumer/profile combinations;
- duplicate or non-finite normalized targets.

### Minimum-schema export

Export uses V22 only when a V22 trigger exists.

If no V22 trigger exists:

- normalized entity-scalar dimensions down-convert to their lowest lossless
  historical form;
- legacy entity/wire profiles follow the completed V21 trigger rules;
- arcs and true construction still trigger V21;
- V20 and older minimum-schema behavior remains unchanged.

Once V22 is required, all live dimensions are written in normalized V22 shape
and all region profiles are canonical. Lower-schema export is permitted only
after the last live-source trigger and the last V19-only operation/diff in both
history and redo are absent. Merely undoing an operation into redo does not
remove its trigger.

There is no session flag that upgrades a document because a V19 tool was once
used, but authoritative retained history is a real schema requirement.

### JSON, CBOR, `.wcad`, and checkpoints

V22 source participates in:

- JSON validation and round-trip;
- canonical CBOR encoding;
- document and command hashes;
- source identity;
- history and redo;
- `.wcad` v2 manifests and package validation;
- topology checkpoint source identity.

`.wcad` remains `partbench.wcad.v2`. Existing V13 checkpoint payload entries
remain the only authoritative binary checkpoint path. JSON continues to warn
when checkpoint payload bytes cannot be carried.

Candidate regions, intersection graphs, edit previews, dimension label
positions, solved coordinates, residuals, meshes, exact metadata, and region
fill tessellation remain derived and are never package source.

## Must Definition of Done

V19 is complete only when:

1. Trim, extend, split, rectangle explode, and offset implement exactly their
   analytic support matrix through typed CADOps.
2. Curve edits provide deterministic identity replacements and never silently
   delete or approximately retarget constraints, dimensions, references, or
   features; post-edit residual checks cannot move the requested analytic
   result.
3. Readiness accepts a precondition-free proposal, returns the complete
   prepared operation, and each curve edit commits alone under the exact
   source/solver identity algorithms without staged-batch ambiguity.
4. Slots and rounded rectangles create only ordinary line/arc/constraint
   source in one atomic, undoable transaction.
5. Every Decision 14 constraint row has its exact create/update/rename/delete
   product behavior, including the legacy-angle compatibility boundary.
6. Every V19 dimension target is source-backed, follows its literal/parameter
   value-source matrix, is numerically solved where Decision 13 claims a
   residual (with legacy rectangle direct-update behavior stated honestly),
   round-trips, and diagnoses under-definition, redundancy, conflict,
   direction/side/sense, and invalid domain.
7. Region refs are explicit, canonical, source-owned, and independent from
   query candidate identity.
8. Region validation is analytic and deterministic for the stated
   rectangle/circle/line/arc, containment, hole, and disjoint-material matrix.
9. Whole-loop candidate discovery is query-only, bounded, cancellable in the
   browser, cached/paginated by source revision, and never invents subcurves or
   substitutes a candidate during feature creation or rebuild.
10. Region extrude new/add/cut and region revolve new-body pass the exact
   consumer matrix through real OCCT.
11. Every new exact recipe has current display, exact metadata, mass
    properties, topology health, checkpoint behavior, and STEP readiness.
12. Source-semantic topology roles exist for the required new-body extrude
    boundaries, while unproven boolean/revolve correspondence remains honestly

13. Editing a referenced region curve preserves source intent, invalidates

16. Agents and MCP can dry-run, commit, query, validate, and inspect every V19

17. V22 loads and round-trips through JSON, canonical CBOR, `.wcad`,
    shape-compatible canonical history/diff replay, redo, source hashes, and

18. Minimum-schema export emits V22 for every listed live-source, history, and
    redo trigger, refuses lowering while one remains, and losslessly retains

19. Candidate caches, solved coordinates, label positions, region fills,

20. Predicate/candidate limits, the representative near-limit case, and

## Stretch Goals

Stretch begins only after every Must item is green:

- associative offset constraints for the supported individual-curve matrix,
  if a separately reviewed source and solver contract is approved;
- additional conservative constraint inference for equal radius, equal length,
  and symmetry;
- region selection by window and explicit remove-from-selection interaction;
- richer source-semantic generated references for region revolve where real
  OCCT correspondence proves them;
- a broader stress-model benchmark catalog beyond the mandatory representative
  near-limit release case.

Stretch may not add a new entity kind, curve family, feature family, consumer
row, package version, or schema after V22.

## Explicit Non-Goals

- Ellipses, B-splines, NURBS, Bézier curves, general conics, or imported
  editable curve entities.
- Project/intersect geometry, silhouette curves, face-edge conversion, 3D
  sketch curves, or surface curves.
- Associative offset in Must, variable offsets, bevel/round offset joins, or
  kernel-wire offset as source authority.
- Fillet-as-sketch-edit, chamfer-as-sketch-edit, general curve replacement, or
  arbitrary rectangle-edge editing without explicit explosion.
- Persistent polyline, slot, rounded-rectangle, trim, or region-candidate
  entity kinds.
- Constraint inference that commits without explicit user/agent intent.
- Parameter binding for new V19 line-angle dimensions or typed dimensional
  parameters.
- Region boundaries that touch or overlap, self-intersecting loops, more than
  outer-plus-holes within one region, or automatic inference of arbitrary
  nesting/islands.
- Multi-region `newBody` features, public multi-solid bodies, independent body
  creation per selected region, or a body-result schema expansion.
- Composite region profiles for sweep or loft, composite loft sections, sweep
  add/cut, thin features, open-profile extrusion, ribs, or guide curves.
- Datum planes, datum axes/points, free/offset/angled/tangent/midplane
  authoring.
- General unrestricted booleans, direct modeling, or arbitrary-result topology
  commandability.
- Exact face/edge/vertex highlight rewrite, selection-buffer overhaul,
  persisted measurements, section view, clipping, or WebGPU.
- Assemblies, mates, instances, materials, drawings, GD&T, or manufacturing
  toolpaths.
- Broader STEP/IGES/DXF/SVG/proprietary interchange or editable STEP sketches.
- Hosted collaboration, cloud projects, recent-file services, natural-language
  parsing, plugin UI, production MCP auth, or new MCP transport.
- Full dark mode, arbitrary panel docking, mobile, or touch-first redesign.
- A new `.wcad` package version, OPFS source data, or persistent derived
  intersection/candidate/solver caches.
- New production dependencies without separately documented approval.

## Rejected Alternatives

1. **Store selected shaded cells as source.** Rejected: pixels and derived
   tessellation cannot be deterministic command authority.
2. **Let OCCT infer and heal profile regions.** Rejected: source validity would
   vary by kernel behavior and violate cad-core determinism.
3. **Infer the largest outer loop and every contained loop.** Rejected: it
   silently chooses design intent and fails for nested selectable material.
4. **Represent holes by loop orientation alone.** Rejected: explicit outer/hole
   roles are clearer for agents, validation, migration, and repair.
5. **Rewrite every V21 profile to `regions`.** Rejected: needless source churn
   and a violation of minimum-triggered schema policy.
6. **Persist candidate IDs.** Rejected: candidates change with source and are
   derived query results.
7. **Make trim a viewport-only delete gesture.** Rejected: automation, dry-run,
   semantic diffs, dependency evidence, and undo require a real command.
8. **Always use every sketch curve as a trim boundary.** Rejected: unrelated
   future curves would change command meaning.
9. **Silently delete broken constraints during trim.** Rejected: destructive
   source consequences must be explicit and auditable.
10. **Keep one entity ID while changing circle to arc or rectangle to lines.**
    Rejected: IDs cannot change semantic kind without corrupting references.
11. **Store an offset parent link without solver support.** Rejected: it would
    imply associativity that rebuild cannot honor.
12. **Approximate unsupported offsets with polylines.** Rejected: it loses exact
    source geometry and changes downstream topology.
13. **Allow multiple disjoint new-body solids under one existing body record.**
    Rejected: that is a separate public body-model decision.
14. **Combine V19 with datum geometry or arbitrary topology expansion.**
    Rejected: both are distinct source/support-matrix risk clusters.
15. **Use solver previews as curve-edit source.** Rejected: only authoritative
    source coordinates may be mutated.
16. **Upgrade every save to V22.** Rejected: project schema remains
    minimum-triggered.
17. **Lower V19 history into older primitive operations.** Rejected: it would
    rewrite authoritative audit intent and require a second diff/replay
    contract.

## Risk Controls

| Risk | Required control |
| --- | --- |
| Trim chooses the wrong interval | Explicit boundaries plus model-space pick, analytic interval ordering, ambiguity rejection |
| Edit silently changes design intent | Replacement map, dependency classification, exact deletion lists, dry-run before UI Apply |
| Constraint target jumps to nearby geometry | Retarget only unique authored endpoint continuity; otherwise block |
| Offset produces unstable corner geometry | One analytic natural-join rule, miter bound, reject ambiguity/self-intersection |
| Region inference becomes hidden source | Explicit submitted refs; candidate query remains derived and revalidated |
| Nested loops create exponential candidates | One whole-loop containment cell per tree node, no hole-subset enumeration, explicit complexity limits |
| Multi-region feature creates disconnected solids | Canonical sequential booleans, positive-effect checks, and one-connected-solid validation after every step |
| Hole topology receives unstable IDs | Loop and boundary-member source keys where proven; checkpoint/ambiguous status elsewhere |
| New dimensions jump solution branches | Stored direction/side/sense, exact residual conventions, deterministic conflict tests |
| V22 causes needless file churn | Enumerated source/history/redo triggers, normalized in-memory compatibility, lossless lowering only after all triggers disappear |
| Heavy sketch queries block pointer work | Revision cache, command-worker cancellation, predicate/candidate caps, V19 performance gate |
| UI and automation diverge | One CADOps/query surface and adapter/browser parity tests |
| V18 shell regresses | Existing action ownership, accessibility, bundle, performance, and browser compatibility gates |
| Scope expands into general curves/features | Fixed entity and consumer matrices plus explicit non-goals |

## Final Compatibility Statement

V19 is intended to expand what users can author in a sketch and how explicitly
selected material regions feed existing solid features. It does not replace
the completed topology system or broaden every downstream result.

V1-V21 projects and commands retain their documented behavior. V17 entity/wire
profiles, arcs, construction geometry, paths, solver branches, topology roles,
and feature matrix remain valid. V18 remains the only production workbench and
its accessibility/performance contracts remain gates.

V22 is emitted only for its enumerated region/dimension source or retained
V19-only history/redo triggers. `.wcad` remains v2. Meshes, region candidates,
intersection graphs, solver results, exact metadata, and viewport state remain
derived.

The V19 promise is:

```text
ordinary exact sketch curves
-> deterministic command-backed editing
-> explicit constraints and dimensions
-> explicit whole-loop material regions
-> bounded exact extrude/revolve consumption
-> honest topology, rebuild, storage, UI, and agent behavior
```
