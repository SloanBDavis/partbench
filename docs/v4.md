# V4 Target: Constrained Sketch Solving And Regeneration

Status: planned active milestone.

V4 should be a larger product step than V3. V3 added source-of-truth
parameters, driving sketch dimensions, narrow line orientation constraints, and
direct evaluator behavior. V4 should turn that foundation into a usable,
agent-readable constrained sketch workflow that can reliably regenerate current
authored bodies.

V4 is still not the full long-term CAD architecture. It should deliberately
take on one hard area, constrained sketch solving, while avoiding unrelated
systems such as STEP, native storage, WebGPU, fillets, chamfers, or broad
topology naming.

## Product Goal

A user or structured agent can define a constrained sketch profile, drive it
with dimensions and parameters, extrude it, edit the sketch intent later, and
see downstream bodies, measurements, extents, references, health, and derived
meshes update coherently.

The user-facing outcome should be:

```text
edit constrained sketch intent -> solver/evaluator updates sketch geometry ->
features regenerate -> viewport, measurements, health, and history stay honest
```

The agent-facing outcome should be:

```text
query sketch constraints/dimensions -> dry-run an edit -> inspect solver status,
semantic diffs, affected features/bodies, and health -> commit through CADOps
```

## Architectural Rules

V4 must keep these rules:

- CADOps remains the shared mutation protocol.
- `cad-core` remains authoritative for document source data.
- Sketch entities, parameters, dimensions, constraints, and solver status are
  typed and queryable.
- Solver/evaluator outputs are derived results, not hidden React state.
- React UI does not own model authority or geometry internals.
- OCCT remains outside `cad-core`.
- Rendered meshes remain derived views/caches.
- MCP and agent adapters wrap CADOps and do not define the internal API.
- Project format changes happen only for persisted source-of-truth data.
- Existing V1 through V10 project compatibility remains importable where
  practical through explicit migration.
- New structures must have current callers and tests.

## V4 User Workflow Target

The V4 workflow should allow a user or structured agent to:

1. Create or open a current Partbench project.
2. Create or select a sketch on a base plane or eligible generated planar face.
3. Add points, lines, rectangles, and circles.
4. Apply supported constraints to sketch points and entities.
5. Add or edit supported dimensions, optionally bound to document parameters.
6. Query sketch solve/evaluation status before committing risky changes.
7. Create or update extrude features from solved rectangle/circle profiles.
8. Edit dimensions, parameters, or supported constraints later.
9. See downstream authored bodies and supported boolean results rebuild or fail
   with structured status.
10. Save/load the V4 project data.
11. Query and mutate the same model through agent/MCP wrappers.

## V4 Scope

V4 should focus on constrained sketch solving and regeneration.

In scope:

- a clear internal sketch solver/evaluator boundary inside `cad-core`;
- source-of-truth sketch point/reference records where needed to make
  constraints durable;
- additional supported constraints with explicit behavior:
  - point coincident,
  - fixed point,
  - horizontal line,
  - vertical line,
  - line length,
  - rectangle width/height,
  - circle radius;
- parameter-bound dimensions continuing to work through the same CADOps path;
- structured solver status for healthy, under-defined, over-defined,
  inconsistent, missing-target, unsupported, and invalid-value states;
- semantic diffs that identify source records changed and affected downstream
  sketch/entity/feature/body data where practical;
- downstream regeneration for current authored extrudes, attached-sketch
  extrudes, and supported narrow add/cut results;
- unit/package-level tests at protocol, core, serialization, adapter/MCP,
  derived-geometry, render-preparation, and UI-helper boundaries;
- compact UI for inspecting and editing supported sketch constraints and
  dimensions without turning the interface into debug output.

Out of scope:

- full general geometric constraint solving;
- dragging solver UX unless explicitly scoped later in V4;
- arbitrary parameter expressions;
- tangent, concentric, equal, symmetry, offset, curvature, spline, and NURBS
  constraints unless a later V4 slice deliberately adds one;
- automatic profile recognition beyond current rectangle/circle paths;
- new CAD feature operations such as revolve, sweep, loft, shell, fillet, or
  chamfer;
- generated references for boolean result bodies unless separately scoped;
- broad topology naming guarantees;
- exact B-rep persistence;
- STEP import/export;
- OPFS or File System Access API;
- WebGPU production renderer;
- hosted collaboration or auth;
- natural-language command parsing;
- browser E2E expansion.

## Phase A: Solver Boundary And Sketch Reference Model

Goal: make the solver/evaluator architecture explicit before adding more
constraint behavior.

Status: implemented for current V3 behavior. The existing rectangle
width/height, circle radius, line length, horizontal line, vertical line, and
`sketch.evaluation` behavior now flows through an internal `cad-core` sketch
solver/evaluator boundary. This did not add new persisted source data and does
not change the project format.

Deliverables:

- identify current direct evaluator logic and isolate it behind a small
  `cad-core` internal boundary;
- define typed solve/evaluation input and output shapes;
- define durable references for sketch points/entity roles where required for
  constraints;
- preserve existing V3 commands and query shapes where practical;
- ensure `sketch.evaluation` reports current V3 behavior through the new
  boundary;
- add focused tests proving existing rectangle width/height, circle radius,
  line length, and horizontal/vertical behavior is unchanged;
- update docs/native-format.md only if new persisted source records require a
  new project version.

Acceptance:

- no behavior regression from V3;
- solver/evaluator results are derived query data;
- no React, renderer, worker, or OCCT coupling is introduced.

## Phase B: Coincident And Fixed Point Constraints

Goal: add the first genuinely solver-shaped constraints.

Status: implemented for the first fixed and coincident point constraints. V4
now supports source-of-truth fixed constraints for point position, line start,
line end, rectangle center, and circle center targets. It also supports
source-of-truth coincident constraints between two explicit point targets using
the same target model.

Deliverables:

- typed source records for point-level constraints if the current V8/V9
  constraint shape is not sufficient;
- commands for creating, renaming where consistent, and deleting supported
  constraints through CADOps;
- supported targets:
  - point fixed to authored coordinates,
  - line endpoint fixed,
  - rectangle center fixed,
  - circle center fixed,
  - point/endpoint/center coincident where target roles are explicit;
- validation for missing targets, unsupported target roles, duplicate
  constraints, and obvious conflicts;
- structured status for under-defined, over-defined, inconsistent, and
  unsupported cases;
- undo/redo, batch dry-run/commit, semantic diffs, project import/export, and
  adapter/MCP pass-through;
- tests covering solver inputs/outputs and failure states.

Acceptance:

- users and agents can express fixed/coincident intent as source records;
- unsupported role combinations fail clearly;
- no UI-only constraint state exists.

Current deterministic coincident behavior is intentionally narrow. If one
target is fixed, the other target moves to the fixed coordinate. If neither
target is fixed, the secondary target moves to the primary target coordinate. If
both targets are fixed to different coordinates, the command/evaluation reports
an inconsistent constraint. This is not yet a general constraint solver.

## Phase C: Solver-Backed Dimensions And Orientation Constraints

Status: implemented for the current V10 dimension/constraint set. No project
format change was required because the work changed derived evaluator behavior,
not persisted source records.

Goal: move V3 direct dimension/orientation behavior through the solver boundary
and make interactions between supported constraints coherent.

Deliverables:

- line length, rectangle width/height, circle radius, horizontal, and vertical
  constraints evaluated through the common solver/evaluator path;
- deterministic resolution rules for supported combinations;
- explicit conflict detection where supported constraints disagree;
- structured issues for invalid, under-defined, over-defined, and unsupported
  sketches;
- semantic diffs that show affected entity geometry and source records;
- tests for combined constraint cases such as:
  - fixed endpoint plus line length,
  - coincident endpoints plus horizontal/vertical,
  - parameter-bound rectangle dimensions,
  - fixed circle center plus radius dimension.

Acceptance:

- supported combined constraints behave predictably;
- unsupported combinations report status instead of silently mutating geometry.

## Phase D: Regeneration And Health Hardening

Goal: prove constrained sketches correctly drive the current feature/body
pipeline.

Deliverables:

- downstream authored extrudes consume evaluated sketch geometry where solver
  behavior applies;
- attached-sketch extrudes continue resolving through generated face frames;
- supported add/cut results rebuild or report honest unsupported status;
- project extents, object/body/reference measurements, generated references,
  named references, dependency health, and transaction history stay coherent;
- `project.health` reports sketch dimension and sketch constraint status,
  affected features/bodies, and structured issues for inconsistent or
  unsupported constrained sketch state;
- derived geometry cache keys include solved/evaluated sketch changes;
- stale async geometry results cannot overwrite newer solved state;
- tests for parameter/constraint edits through newBody, attached extrude,
  rectangle cut, circle-target cut, and rectangle add workflows.

Acceptance:

- editing supported sketch intent later changes downstream model output
  predictably;
- consumed targets and unsupported boolean result references remain honest.

## Phase E: Human And Agent Workflow UI

Goal: make constrained sketching understandable without adding hidden model
state.

Deliverables:

- compact UI for supported dimensions and constraints;
- clear selected sketch/entity constraint status;
- concise solver health display;
- no walls of instructional/debug text;
- UI command-building helpers for constraint creation/edit/delete;
- UI filtering so unsupported constraint targets are not presented as available;
- agent/MCP query examples in docs where useful;
- tests for formatting, filtering, command builders, stale status, and
  unsupported messages.

Acceptance:

- the user can discover the current constrained sketch workflow from the UI;
- the UI reflects CADOps/query state and does not invent separate state.

## Phase F: V4 Stabilization And Completion

Goal: declare V4 complete only after the constrained sketch workflow is
coherent, tested, documented, and ready for the next major architecture step.

Deliverables:

- subagent or human review of architecture boundaries, core model behavior,
  derived geometry, adapter/MCP wrappers, storage, docs, and UI helpers;
- focused fixes for must-fix findings;
- broad unit/package-level coverage for V4 workflows;
- docs updated to mark V4 complete;
- implementation plan updated with the next recommended milestone.

Acceptance:

- no known architecture drift;
- no unsupported sketch/solver state is presented as working;
- all required checks pass.

## Project Format Direction

V4 introduced `web-cad.project.v9` when fixed point constraints added persisted
target and coordinate source records that cannot be represented by V8. It then
introduced `web-cad.project.v10` when coincident constraints added two durable
point targets to one source record that cannot be represented by V9.

Likely future version triggers:

- new persisted constraint records beyond V10 horizontal/vertical/fixed/coincident
  constraints;
- new solver source records that are not query-only status;
- source-level constraint grouping or identity data needed for migration and
  agent workflows.

Do not introduce a new format version for:

- query-only solver/evaluation status;
- derived solved geometry snapshots;
- renderer display frames;
- derived mesh/cache data;
- UI selection state.

Any later format version must include explicit import compatibility from older
accepted versions where practical and structured validation for new source
records.

## Testing Strategy

V4 should continue the current unit/package-test strategy rather than expanding
browser E2E tests.

Required coverage areas:

- `cad-protocol` command/query shapes;
- `cad-core` validation, semantic diffs, undo/redo, batch dry-run/commit, and
  transaction history;
- solver/evaluator input/output behavior and failure states;
- project JSON import/export and migration;
- project health, structure, measurements, and extents;
- generated and named reference behavior where constrained edits affect source
  geometry;
- derived geometry cache keys, invalidation, and stale async result handling;
- render scene preparation and fallback/mesh selection;
- agent-adapter and MCP pass-through;
- UI helper command builders, filtering, formatting, and status display.

Browser checks may be used manually for confidence, but V4 should not add
fragile browser E2E tests unless a specific bug demands it.

## V4 Acceptance Criteria

V4 is complete when:

1. The sketch solver/evaluator boundary is explicit and tested.
2. Existing V3 dimensions and horizontal/vertical constraints still work
   through the V4 path.
3. Fixed and coincident constraints are supported for explicitly scoped sketch
   point/entity roles.
4. Supported constraints and dimensions can be combined predictably.
5. Unsupported, under-defined, over-defined, inconsistent, missing-target, and
   invalid-value states produce structured query/validation output.
6. Parameter and constraint edits regenerate current authored extrudes and
   supported narrow boolean results where supported.
7. Project JSON import/export/migration is explicit and tested.
8. Agent/MCP wrappers expose V4 commands/queries as CADOps wrappers.
9. The UI makes current constrained sketching usable without hidden model state
   or misleading affordances.
10. Unit and package tests cover the workflow without depending on brittle
    browser E2E tests.

## Deferred Beyond V4 Unless Explicitly Scoped

- full general sketch solving;
- complex geometric constraints such as tangent, concentric, equal, symmetry,
  spline, and curvature constraints;
- dragging solver UX;
- arbitrary parameter expressions;
- generated references for boolean result bodies;
- broad stable topological naming;
- exact B-rep persistence;
- general booleans/add, fillets, chamfers, shell, patterns, lofts, sweeps,
  revolve, and direct edits;
- STEP import/export;
- OPFS cache implementation and File System Access open/save;
- native `.wcad` package implementation;
- local launcher with cross-origin isolation headers;
- WebGPU production renderer;
- assemblies, mates, large-model LOD, and instancing;
- hosted collaboration;
- production MCP auth/permission system;
- natural-language command entry.
