# V2 Target: Feature/Body CAD Foundation

Status: complete as of 2026-05-22.

V2 should move Partbench from a primitive-scene CAD foundation to the first real
feature/body CAD foundation while preserving the architectural boundaries proven
in V1.

V2 is not the full long-term architecture. It is the next practical product
target.

## Product Goal

A user can create a simple part using source-of-truth sketch and feature data,
derive body meshes behind the worker boundary, save/load the project, and
inspect the model structurally through UI and CADOps queries. V2 also includes
the first narrow rectangle-tool add/cut boolean slices, but it does not claim a
general boolean or exact topology solution.

The key transition is:

```text
V1: scene primitives -> derived primitive meshes
V2: parts/features/bodies -> worker-derived geometry -> derived body meshes
```

## Architectural Rules

V2 must keep these rules:

- CADOps remains the shared mutation protocol.
- `cad-core` remains authoritative.
- OCCT remains outside `cad-core`.
- React does not import geometry internals.
- Renderer meshes remain derived views/caches.
- MCP and agents wrap CADOps.
- V1 projects remain loadable or migratable.
- New model structures must have current callers and tests.

## V2 User Workflow Target

The V2 workflow should allow a user or structured agent to:

1. Open the app.
2. Create or open a project.
3. Work with at least one part.
4. Create a simple sketch on a plane.
5. Create the first feature from that sketch: a rectangle or circle extrude.
6. Produce a source-authored body with rebuildable worker-derived mesh data.
7. Display the derived mesh for that body.
8. Inspect the feature/body structure.
9. Query features, bodies, measurements, and transaction history.
10. Save/load the V2 project data.
11. Continue opening V1 project JSON through migration or compatibility.

## Phase A: Document Model Foundation

Goal: introduce the smallest useful part/feature/body model.

Status: complete for V2. Current V1 scene primitives still
own source-of-truth object data, while CADOps read queries expose a derived
default part, primitive features, solid bodies, and object source mappings.

Deliverables:

- Define typed IDs and snapshots for:
  - project/root document,
  - part,
  - feature,
  - body,
  - body instance if needed.
- Preserve current scene primitives through compatibility or migration.
- Decide how current primitive commands map into V2:
  - compatibility scene objects,
  - primitive features,
  - primitive bodies, or
  - a staged migration path.
- Add CADOps read queries for feature tree, parts, bodies, and source metadata.
- Add tests for migration, undo/redo, serialization, and query behavior.

Current mapping:

- The document has one derived default part: `part:default`.
- Each current scene primitive maps to one primitive feature:
  `feature:<objectId>`.
- Each primitive feature maps to one derived solid body: `body:<objectId>`.
- `project.structure` lists parts, features, bodies, and object source mappings.
- `project.features` remains available and now includes `partId` and `bodyId`.
- Primitive-derived structures are not persisted separately in project JSON.
- Body instances are intentionally deferred until there is an assembly or
  instancing caller.

Non-goals:

- Do not implement sketches or extrude in this phase unless explicitly scoped.
- Do not create a large unused feature graph.

## Phase B: Project Format and Storage Direction

Goal: prepare source-of-truth storage for feature/body data.

Status: complete for V2. The Phase A bridge remains query-only
and deterministic. Phase C sketches are source-of-truth data and introduced
`web-cad.project.v2`. Phase D authored extrude features are source-of-truth data
and introduced `web-cad.project.v3`. Phase E/F attached face sketches are
source-of-truth sketch metadata and introduced `web-cad.project.v4`. Phase E
named generated references are source-of-truth user/agent metadata and
introduced `web-cad.project.v5`. Phase G explicit authored extrude operation
mode introduced `web-cad.project.v6`. Imports continue to accept V1, V2, V3,
V4, and V5 projects through explicit migration.

Deliverables:

- Update `docs/native-format.md` for the V2 source model.
- Define the next project format version and migration from
  `web-cad.project.v1`.
- Decide the first native package shape for:
  - manifest,
  - source document,
  - command log,
  - exact geometry checkpoints,
  - derived mesh/cache data,
  - metadata.
- Keep JSON import/export available for debugging and interchange.
- Scope OPFS and File System Access separately.

Current decision:

- No format change was needed for the derived `project.structure` bridge alone.
- `web-cad.project.v2` was introduced when source-of-truth sketches were added.
- `web-cad.project.v3` was introduced when authored extrude feature records and
  generated body IDs were added.
- Current JSON stores objects, sketches, attached sketch metadata, extrude
  features, units, object names, dimensions, transforms, transaction history,
  redo history, actor metadata, and audit metadata.
- V1 JSON still imports with empty sketches/features and fresh sketch,
  feature, and body counters.
- V2 JSON still imports with existing sketches plus empty authored features and
  fresh feature/body counters.
- V3 JSON still imports with existing sketches/features and no attached sketch
  metadata.
- Primitive-derived part/feature/body IDs are rebuilt after load:
  `part:default`, `feature:<objectId>`, and `body:<objectId>`.
- Authored extrude features and their generated body IDs are persisted because
  they are source-of-truth rebuild inputs.
- A later project format should start when constraints, sketch profiles,
  explicit parts, additional feature kinds that require new source data, exact
  body checkpoints, durable persisted topological references, assemblies, or
  other data cannot fit cleanly in the current V6 shape.

Non-goals:

- Do not add OPFS and File System Access in the same unbounded change.
- Do not persist derived meshes as source of truth.

## Phase C: Sketch Model First Slice

Goal: add a minimal sketch model that can feed the first feature.

Status: complete for V2. Sketches are authored document
data in `cad-core`, mutated through CADOps, saved in the current project format,
and queryable through core, agent adapter, and MCP wrappers.

Implemented:

- Sketch containers on `XY`, `XZ`, and `YZ` planes.
- Minimal sketch entities:
  - point,
  - line,
  - rectangle,
  - circle.
- CADOps commands for create, rename, delete, add/update/delete entities.
- Semantic diffs, undo/redo, batch dry-run/commit, validation, and transaction
  summaries for sketch edits.
- `project.sketches` and `sketch.get` read queries.
- Agent adapter and MCP sketch query exposure as thin wrappers.
- Compact UI panel for creating sketches and adding/editing/deleting sketch
  entities.
- V1 project import compatibility and current project export with sketches.
- Tests for command validation, diffs, undo/redo, serialization, queries, and
  adapter/MCP wrappers.

Non-goals:

- No full sketch solver unless explicitly scoped.
- No arbitrary spline/NURBS sketch work yet.
- No profile recognition beyond direct rectangle/circle entity references.

## Phase D: First Feature Operation

Goal: create the first source-authored body and rebuildable derived mesh from
source-of-truth feature data.

Status: complete for V2 with `feature.extrude`.

Implemented:

- Typed CADOps command: `feature.extrude`.
- Source profiles: rectangle sketch entity and circle sketch entity.
- Simple extrusion model: positive finite depth and sketch-plane-normal side
  behavior (`positive`, `negative`, or `symmetric`).
- Authored extrude feature records in `cad-core` with generated feature/body
  IDs, source sketch ID, source entity ID, profile kind, depth, side, explicit
  operation mode, optional target body ID for boolean modes, and optional
  feature name.
- Validation for sketch existence, entity existence, supported profile kind,
  positive finite depth, supported side, supported operation mode, and unique
  feature/body IDs.
- Operation mode is now explicit as `newBody`, `add`, or `cut`. `newBody`
  remains the default standalone behavior. The first narrow `cut` slices support
  a rectangle sketch-extrude tool cutting one active authored rectangle or
  circle `newBody` target body and rebuild the result through the OCCT-derived
  geometry path where that runtime path is enabled. The first narrow `add` slice
  supports rectangle tool plus active authored rectangle `newBody` target in the
  authoritative command model and rebuilds through the same derived-geometry/UI
  boolean result path. Circle-tool booleans, circle-target add, and broader
  boolean cases remain unsupported and fail with structured errors.
- Semantic diffs for feature/body creation.
- Undo/redo, batch dry-run/commit, transaction history summaries, and project
  round trip.
- `feature.updateExtrude` support for authored sketch-extrude features, limited
  to depth and side updates. The source sketch/entity reference is not editable
  yet.
- Rectangle and circle source profile values can be edited through
  `sketch.updateEntity`; dependent authored extrude features and bodies are
  marked modified so derived geometry rebuilds from the updated sketch source.
- Project `web-cad.project.v6` export with authored extrude feature source
  data, attached sketch metadata, named generated references, and explicit
  operation mode; V1, V2, V3, V4, and V5 imports still migrate.
- `project.structure` and `project.features` expose primitive-derived features
  plus authored sketch-extrude features and bodies.
- Agent/MCP query wrappers continue to expose structure through CADOps.
- Geometry-kernel and geometry-worker support rectangle/circle extrude
  tessellation behind the existing worker boundary.
- Renderer display remains derived: ready extrude meshes are preferred, while a
  simple box/cylinder fallback is used while geometry is disabled, pending, or
  unavailable.
- Extrudes from sketches attached to generated planar faces derive their
  rebuildable mesh/fallback display from the current attached face frame. The
  feature still references the same sketch/entity source data; no new command or
  persisted topology is introduced.
- Compact UI support for selecting a rectangle/circle sketch entity, entering
  extrusion depth and optional IDs/name, creating the extrude through CADOps,
  selecting the resulting body, and editing authored extrude depth.
- `feature.delete` support for authored sketch-extrude features, including
  semantic diffs, undo/redo, batch behavior, UI deletion, save/load regression
  coverage, derived-geometry cleanup, and agent/MCP batch pass-through.

Non-goals:

- No arbitrary sketch profile recognition yet.
- No broad feature edit command yet. Primitive-derived compatibility features
  cannot be deleted through `feature.delete` or edited through
  `feature.updateExtrude`.
- No general booleans beyond the narrow rectangle-tool add/cut slices, fillets,
  chamfers, shell, loft, sweep, or direct edits yet.
- No topology mutation features before stable references exist.

## Phase E: Topological Reference First Slice

Goal: avoid building future features on unstable face/edge indexes.

Status: complete for V2. First read-only semantic reference slices exist for
authored rectangle/circle sketch-extrude bodies.

Implemented:

- CADOps query `body.generatedReferences` for one authored sketch-extrude body.
- CADOps query `body.resolveGeneratedReference` for resolving one generated
  stable ID on an authored sketch-extrude body.
- A generated body reference with stable semantic ID, owning part ID, source
  feature ID, source sketch ID, source sketch entity ID, profile kind, depth,
  side, sketch plane, and source profile signature.
- Generated face references for current authored extrudes:
  - rectangle extrude: start cap, end cap, `side:uMin`, `side:uMax`,
    `side:vMin`, and `side:vMax`;
  - circle extrude: start cap, end cap, and `side:circular`.
- Generated edge references for current authored extrudes:
  - rectangle extrude: start-cap profile edges, end-cap profile edges, and four
    longitudinal corner edges;
  - circle extrude: start and end circular edges.
- Generated vertex references for current authored rectangle extrudes:
  `start:uMin:vMin`, `start:uMin:vMax`, `start:uMax:vMin`,
  `start:uMax:vMax`, `end:uMin:vMin`, `end:uMin:vMax`, `end:uMax:vMin`, and
  `end:uMax:vMax`.
- Circle extrudes return `vertexCount: 0` and `vertices: []` because the current
  circular profile has no stable discrete semantic vertices.
- Lightweight geometric signatures with cap normals, side normals or cylinder
  axis role, edge curve type, edge axis role, adjacent face and edge roles,
  sketch plane, extrude side, profile kind, depth, current source profile
  dimensions, and rectangle vertex profile points where practical.
- Deterministic read-only labels and descriptions for generated body, face,
  edge, and vertex references. These make references easier to inspect through
  agents and future UI panels without creating a persisted naming subsystem.
- Deterministic read-only eligibility metadata for planned future operations:
  `feature.attachSketchPlane`, `feature.measureReference`, and
  `feature.selectReference`. Planar faces advertise sketch-plane attachment;
  circular side faces, bodies, edges, and vertices do not. This is planning
  metadata only, not an implemented operation.
- Internal generated-reference validation helpers resolve references by body ID
  and stable ID, then check expected kind and operation eligibility for future
  reference-consuming commands.
- Source-of-truth named generated references through `reference.nameGenerated`
  and `reference.deleteName`. Names store the user/agent label, target body ID,
  generated stable ID, and resolved reference kind at creation time.
- Read/query support through `reference.listNamed` and
  `reference.resolveNamed`. Listing marks stale targets clearly, while resolving
  returns the current generated reference object or a structured stale/missing
  target error.
- App UI support for naming the selected generated reference, deleting named
  references, listing document named references, and marking stale named targets.
- `sketch.createOnFace` creates a source-of-truth sketch attached to an eligible
  generated planar face reference from an authored sketch-extrude body. The
  command accepts either the generated body ID plus face stable ID or a named
  reference that resolves to an eligible generated planar face. The sketch stores
  resolved attachment metadata, but generated references themselves remain
  derived from feature source data rather than persisted topology.
- The app UI lists generated face references for the selected authored body,
  shows labels, descriptions, and eligibility, and offers
  `sketch.createOnFace` only for eligible planar faces.
- The viewport displays attached sketches on a derived frame for their current
  generated planar face reference. Stale or unresolved attachments fall back to
  the saved sketch plane with a clear sketch-panel status.
- Existing `feature.extrude` can derive rectangle/circle extrude geometry from
  attached sketches by resolving the sketch's generated face attachment to a
  current face frame. Stale or unresolved attachment references produce a clear
  derived-geometry fallback status rather than drawing the body at the wrong
  base plane.
- Agent adapter and MCP wrapper exposure as thin read-only wrappers over the
  CADOps query.
- Structured missing/stale reference errors through
  `GENERATED_REFERENCE_NOT_FOUND`.
- Tests for rectangle/circle roles, missing and unsupported bodies, depth/side
  edits, source profile edits, generated reference resolution, feature delete
  undo/redo, import/export round-trip, and adapter/MCP responses.

Current limits:

- Only authored sketch-extrude bodies are supported. Primitive-derived bodies
  return a structured unsupported-reference error.
- Generated references are semantic and derived from source feature data; they
  are not exact B-rep topology and are not persisted in project JSON. Named
  references are persisted source-of-truth metadata that point at those derived
  generated references.
- Attached sketches persist only the source generated-face attachment metadata.
  Their viewport placement is derived from current generated reference data.
  There is no sketch solver, source-level face-local coordinate remapping, or
  extrude-from-attached-sketch feature graph beyond resolving the current
  generated face frame for rectangle/circle extrudes.
- Vertex references are implemented only for authored rectangle extrudes.
  Broader vertex references for curves or arbitrary B-rep topology remain
  deferred.

Deliverables:

- Typed entity references for bodies, faces, edges, vertices, sketches, and
  features.
- Feature lineage for generated entities where practical.
- Lightweight geometric signatures for disambiguation.
- Clear failure behavior when references are ambiguous.

Non-goals:

- Do not claim a complete topological naming solution in the first slice.
- Do not expose raw kernel indexes as durable public API.

## Phase F: Source-Derived Measurements and Health Queries

Goal: make current measurements and health queries use authoritative source
data rather than renderer meshes.

Current status: complete for V2. The current slices add `body.measurements` for
authored rectangle/circle sketch-extrude bodies, include those authored bodies in
`project.extents`, and add `body.generatedReferenceMeasurements` for measuring
one semantic generated body, face, edge, or rectangle vertex reference.
Measurements and extents are derived from source-of-truth
sketch/profile/depth/side data, including attached-sketch face placement when
the attachment resolves. Stale or unresolved attached-sketch body sources are
excluded from project extents with structured warnings. These are not
renderer-mesh measurements, do not read raw OCCT indexes, and are not persisted.
`project.health` adds read-only dependency and health reporting for authored
extrudes, attached sketches, generated bodies, and named generated references.
It marks healthy, stale, missing-source, and unsupported dependency states, but
does not mutate or regenerate the model.

Implemented for V2:

- Source-derived body measurement query for current authored extrude bodies.
- Project extents that union V1 primitive object bounds with source-derived
  authored rectangle/circle extrude body bounds.
- Source-derived generated-reference measurement query for current authored
  rectangle/circle extrude body, face, edge, and rectangle vertex references.
- Source-derived project health query for authored extrudes, attached sketches,
  and named generated references.
- Source-derived bounding box, volume, surface area, centroid, and related
  measurements for current rectangle/circle extrude source data where
  practical.
- CADOps read/query shapes for source-derived measurements.
- Primitive-derived measurements remain fallback for V1-compatible objects.
- Agent/MCP wrappers remain thin query wrappers.

Non-goals:

- No renderer-mesh measurements as source of truth.
- No exact B-rep/topological measurement guarantees from generated-reference
  measurements yet. They are analytic measurements derived from current feature
  source data and semantic roles.

## Phase G: Renderer and Narrow Boolean Direction

Goal: improve rendering based on real feature/body workloads.

Status: complete for V2. The geometry layer has a
`geometry.booleanExtrudes` request behind the existing
geometry-kernel/geometry-worker/OCCT boundary. That path now backs the first
narrow authoritative `feature.extrude` cut slices: a rectangle tool cutting one
active authored rectangle or circle `newBody` target body. Rectangle add/fuse
with an active authored rectangle `newBody` target now rebuilds through the same
derived mesh/cache path and is available through the existing extrude UI.
Boolean geometry still returns serializable derived mesh data only and does not
produce topology naming or generated-reference invalidation maps. Circle-tool
booleans, circle-target add, and general booleans remain unsupported.

Deliverables:

- Better edge display and selected-face/body highlighting in the current
  renderer where practical.
- Benchmark scenes that represent actual V2 workloads.
- Harden the narrow rectangle-tool cut slices against real workloads before
  expanding boolean scope.
- WebGPU requirements before implementation:
  - buffer ownership,
  - picking,
  - instancing,
  - culling,
  - LOD,
  - large-model behavior.

Non-goals:

- Do not start WebGPU as a cosmetic rewrite.
- Do not expand beyond the narrow rectangle-tool boolean slices until topology
  naming, empty-result behavior, tolerance handling, performance, and
  generated-reference invalidation are deliberately scoped.

## Deferred Beyond V2 Unless Explicitly Scoped

- Full sketch constraint solver.
- Stable topological naming across broad feature edits.
- Fillets, chamfers, general booleans/add, shell, patterns, lofts, and sweeps.
- STEP import/export.
- OPFS cache implementation.
- File System Access open/save.
- Local launcher with cross-origin isolation headers.
- WebGPU production renderer.
- Assemblies, mates, large-model LOD, and instancing.
- Hosted collaboration.
- Production MCP auth/permission system.
- Natural-language command entry.

## V2 Done Means

V2 is done. The app can create sketch-driven authored bodies through CADOps,
display them through derived meshes, save/load them through a versioned project
format, expose them through structured queries, preserve named/generated
reference and attached-sketch workflows, support the first narrow rectangle-tool
add/cut boolean slices, and keep the V1 workflow and project compatibility
intact.
