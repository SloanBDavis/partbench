# V2 Target: Feature/Body CAD Foundation

Status: active implementation target.

V2 should move the app from a primitive-scene CAD foundation to the first real
feature/body CAD foundation while preserving the architectural boundaries proven
in V1.

V2 is not the full long-term architecture. It is the next practical product
target.

## Product Goal

A user should be able to create a simple part using source-of-truth feature
data, derive exact geometry behind the worker boundary, display the resulting
body as a mesh, save/load the project, and inspect the model structurally
through UI and CADOps queries.

The key transition is:

```text
V1: scene primitives -> derived primitive meshes
V2: parts/features/bodies -> exact geometry -> derived body meshes
```

## Architectural Rules

V2 must keep these rules:

- CADOps remains the shared mutation protocol.
- `cad-core` remains authoritative.
- OCCT remains outside `cad-core`.
- React does not import geometry internals.
- Renderer meshes remain derived views/caches.
- MCP and agents wrap CADOps.
- V1 projects remain loadable or migratable.
- New model structures must have current callers and tests.

## V2 User Workflow Target

The V2 workflow should allow a user or structured agent to:

1. Open the app.
2. Create or open a project.
3. Work with at least one part.
4. Create a simple sketch on a plane.
5. Create the first feature from that sketch: a rectangle or circle extrude.
6. Produce an exact kernel-backed body behind the worker boundary.
7. Display the derived mesh for that body.
8. Inspect the feature/body structure.
9. Query features, bodies, measurements, and transaction history.
10. Save/load the V2 project data.
11. Continue opening V1 project JSON through migration or compatibility.

## Phase A: Document Model Foundation

Goal: introduce the smallest useful part/feature/body model.

Status: first compatibility slice implemented. Current V1 scene primitives still
own source-of-truth object data, while CADOps read queries expose a derived
default part, primitive features, solid bodies, and object source mappings.

Deliverables:

- Define typed IDs and snapshots for:
  - project/root document,
  - part,
  - feature,
  - body,
  - body instance if needed.
- Preserve current scene primitives through compatibility or migration.
- Decide how current primitive commands map into V2:
  - compatibility scene objects,
  - primitive features,
  - primitive bodies, or
  - a staged migration path.
- Add CADOps read queries for feature tree, parts, bodies, and source metadata.
- Add tests for migration, undo/redo, serialization, and query behavior.

Current mapping:

- The document has one derived default part: `part:default`.
- Each current scene primitive maps to one primitive feature:
  `feature:<objectId>`.
- Each primitive feature maps to one derived solid body: `body:<objectId>`.
- `project.structure` lists parts, features, bodies, and object source mappings.
- `project.features` remains available and now includes `partId` and `bodyId`.
- Primitive-derived structures are not persisted separately in project JSON.
- Body instances are intentionally deferred until there is an assembly or
  instancing caller.

Non-goals:

- Do not implement sketches or extrude in this phase unless explicitly scoped.
- Do not create a large unused feature graph.

## Phase B: Project Format and Storage Direction

Goal: prepare source-of-truth storage for feature/body data.

Status: decision implemented and evolved. The Phase A bridge remains query-only
and deterministic. Phase C sketches are source-of-truth data and introduced
`web-cad.project.v2`. Phase D authored extrude features are source-of-truth data
and introduced `web-cad.project.v3`. Imports continue to accept V1 and V2
projects through explicit migration.

Deliverables:

- Update `docs/native-format.md` for the V2 source model.
- Define the next project format version and migration from
  `web-cad.project.v1`.
- Decide the first native package shape for:
  - manifest,
  - source document,
  - command log,
  - exact geometry checkpoints,
  - derived mesh/cache data,
  - metadata.
- Keep JSON import/export available for debugging and interchange.
- Scope OPFS and File System Access separately.

Current decision:

- No format change was needed for the derived `project.structure` bridge alone.
- `web-cad.project.v2` was introduced when source-of-truth sketches were added.
- `web-cad.project.v3` was introduced when authored extrude feature records and
  generated body IDs were added.
- Current JSON stores objects, sketches, extrude features, units, object names,
  dimensions, transforms, transaction history, redo history, actor metadata, and
  audit metadata.
- V1 JSON still imports with empty sketches/features and fresh sketch,
  feature, and body counters.
- V2 JSON still imports with existing sketches plus empty authored features and
  fresh feature/body counters.
- Primitive-derived part/feature/body IDs are rebuilt after load:
  `part:default`, `feature:<objectId>`, and `body:<objectId>`.
- Authored extrude features and their generated body IDs are persisted because
  they are source-of-truth rebuild inputs.
- A later project format should start when constraints, sketch profiles,
  explicit parts, additional feature kinds that require new source data, exact
  body checkpoints, topological references, assemblies, or other data cannot fit
  cleanly in the current V3 shape.

Non-goals:

- Do not add OPFS and File System Access in the same unbounded change.
- Do not persist derived meshes as source of truth.

## Phase C: Sketch Model First Slice

Goal: add a minimal sketch model that can feed the first feature.

Status: first source-of-truth slice implemented. Sketches are authored document
data in `cad-core`, mutated through CADOps, saved in the current project format,
and queryable through core, agent adapter, and MCP wrappers.

Implemented:

- Sketch containers on `XY`, `XZ`, and `YZ` planes.
- Minimal sketch entities:
  - point,
  - line,
  - rectangle,
  - circle.
- CADOps commands for create, rename, delete, add/update/delete entities.
- Semantic diffs, undo/redo, batch dry-run/commit, validation, and transaction
  summaries for sketch edits.
- `project.sketches` and `sketch.get` read queries.
- Agent adapter and MCP sketch query exposure as thin wrappers.
- Compact UI panel for creating sketches and adding/editing/deleting sketch
  entities.
- V1 project import compatibility and current project export with sketches.
- Tests for command validation, diffs, undo/redo, serialization, queries, and
  adapter/MCP wrappers.

Non-goals:

- No full sketch solver unless explicitly scoped.
- No arbitrary spline/NURBS sketch work yet.
- No profile recognition beyond direct rectangle/circle entity references.

## Phase D: First Feature Operation

Goal: create the first exact body from source-of-truth feature data.

Status: first slice implemented with `feature.extrude`.

Implemented:

- Typed CADOps command: `feature.extrude`.
- Source profiles: rectangle sketch entity and circle sketch entity.
- Simple extrusion model: positive finite depth and positive-side extrusion.
- Authored extrude feature records in `cad-core` with generated feature/body
  IDs, source sketch ID, source entity ID, profile kind, depth, side, and
  optional feature name.
- Validation for sketch existence, entity existence, supported profile kind,
  positive finite depth, and unique feature/body IDs.
- Semantic diffs for feature/body creation.
- Undo/redo, batch dry-run/commit, transaction history summaries, and project
  round trip.
- `feature.updateExtrude` support for authored sketch-extrude features, limited
  to positive finite depth updates. The source sketch/entity reference and
  extrusion side are not editable yet.
- Rectangle and circle source profile values can be edited through
  `sketch.updateEntity`; dependent authored extrude features and bodies are
  marked modified so derived geometry rebuilds from the updated sketch source.
- Project `web-cad.project.v3` export with authored extrude feature source data;
  V1 and V2 imports still migrate.
- `project.structure` and `project.features` expose primitive-derived features
  plus authored sketch-extrude features and bodies.
- Agent/MCP query wrappers continue to expose structure through CADOps.
- Geometry-kernel and geometry-worker support rectangle/circle extrude
  tessellation behind the existing worker boundary.
- Renderer display remains derived: ready extrude meshes are preferred, while a
  simple box/cylinder fallback is used while geometry is disabled, pending, or
  unavailable.
- Compact UI support for selecting a rectangle/circle sketch entity, entering
  extrusion depth and optional IDs/name, creating the extrude through CADOps,
  selecting the resulting body, and editing authored extrude depth.
- `feature.delete` support for authored sketch-extrude features, including
  semantic diffs, undo/redo, batch behavior, UI deletion, save/load regression
  coverage, derived-geometry cleanup, and agent/MCP batch pass-through.

Non-goals:

- No arbitrary sketch profile recognition yet.
- No broad feature edit command yet. Primitive-derived compatibility features
  cannot be deleted through `feature.delete` or edited through
  `feature.updateExtrude`.
- No fillets, chamfers, booleans, shell, loft, sweep, or direct edits yet.
- No topology mutation features before stable references exist.

## Phase E: Topological Reference First Slice

Goal: avoid building future features on unstable face/edge indexes.

Deliverables:

- Typed entity references for bodies, faces, edges, vertices, sketches, and
  features.
- Feature lineage for generated entities where practical.
- Lightweight geometric signatures for disambiguation.
- User/agent-readable named references.
- Clear failure behavior when references are ambiguous.

Non-goals:

- Do not claim a complete topological naming solution in the first slice.
- Do not expose raw kernel indexes as durable public API.

## Phase F: Exact Measurements and Kernel Queries

Goal: make measurement queries use exact geometry when exact bodies exist.

Deliverables:

- Kernel-backed bounding box, volume, surface area, centroid, and related
  measurements where practical.
- CADOps read/query shapes for exact measurements.
- Primitive-derived measurements remain fallback for V1-compatible objects.
- Agent/MCP wrappers remain thin query wrappers.

Non-goals:

- No renderer-mesh measurements as source of truth.

## Phase G: Renderer and Performance Direction

Goal: improve rendering based on real feature/body workloads.

Deliverables:

- Better edge display and selected-face/body highlighting in the current
  renderer where practical.
- Benchmark scenes that represent actual V2 workloads.
- WebGPU requirements before implementation:
  - buffer ownership,
  - picking,
  - instancing,
  - culling,
  - LOD,
  - large-model behavior.

Non-goals:

- Do not start WebGPU as a cosmetic rewrite.

## Deferred Beyond V2 Unless Explicitly Scoped

- Full sketch constraint solver.
- Stable topological naming across broad feature edits.
- Fillets, chamfers, booleans, shell, patterns, lofts, and sweeps.
- STEP import/export.
- OPFS cache implementation.
- File System Access open/save.
- Local launcher with cross-origin isolation headers.
- WebGPU production renderer.
- Assemblies, mates, large-model LOD, and instancing.
- Hosted collaboration.
- Production MCP auth/permission system.
- Natural-language command entry.

## V2 Done Means

V2 is done when the app can create at least one simple sketch-driven exact body
through CADOps, display it through derived meshes, save/load it through a
versioned project format, expose it through structured queries, and keep the V1
workflow and project compatibility intact.
