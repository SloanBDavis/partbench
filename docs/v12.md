# V12 Planning Record: Stable Boolean Topology And Result References

Status: in progress as of 2026-06-20. V12 is the stable boolean topology and
result-reference release. Tranches A-D1 implemented the first typed boolean
topology readiness contract, command-ready rectangle cut-wall face references,
command-ready rectangle add wall/cap face references, command-ready rectangle
cut-wall profile edge references, and command-ready rectangle add-cap profile
edge references for selection, naming, and measurement. The circle tool slice
adds command-ready circle cut-wall/rim references and circle add-wall/cap/
cap-edge references for selection, naming, measurement, and planar add-cap
sketch attachment. Tranche E1 reports supported V12 cut-wall face/edge generated
and named references as active across scoped upstream source extrude edits while
keeping unproven result topology repair-needed. The add-face/edge source/query
slice extends that same survival behavior to supported rectangle add wall/cap
faces and add cap profile edges, including repair coverage for missing names.
The current circle query-alignment slice keeps circle add cap edge health,
rebuild, and editability consistent with the same shared reference contract.
The current edit-effects slice extends survival reporting across supported
source sketch profile and driving-dimension edits, plus source-affecting
solver-backed constraint and dimension creation paths. Tranche F1 now keeps the
browser product surface compact by showing only the command actions advertised
through the semantic reference contract, and Tranche G2 adds browser smoke
coverage for both the original cut-result face/edge workflow and the first
add-result face/edge workflow.
It follows the completed V10 editable feature history/stable reference health
release and the completed V11 full sketch solver/parametric sketch UX release.

V12 is a modeling-authority release, not a renderer, storage, or broad new
feature-family release. The release center is:

```text
authored source features -> boolean result body lifecycle ->
semantic result topology roles -> command-ready generated references ->
reference health/repair -> viewport and UI consumption
```

## Product Goal

Partbench should stop treating supported cut/add boolean result bodies as
visually valid but semantically non-commandable.

A user should be able to:

- extrude a rectangle or circle as a new body;
- create a sketch on a supported generated planar face;
- cut or add with a supported rectangle/circle tool profile;
- select the resulting supported faces and edges;
- create another sketch on a supported planar result face;
- name supported boolean result faces and edges;
- measure and inspect supported result references;
- edit upstream source parameters and see each downstream reference reported as
  active, replaced, stale, repair-needed, ambiguous, deleted, or unsupported;
- use the same behavior through UI, scripts, agents, MCP, tests, and future
  viewport tools.

The specific user-visible issue V12 should fix is the current diagnostic:

```text
Boolean result body body_2 does not yet have stable command-ready generated topology.
```

The viewport can already render the body because derived geometry exists. V12
should make the supported generated topology command-ready where source
semantics prove identity.

## Why This Is The Next Major Release

V7 made semantic references useful. V8 made project files, cache, and exact
export practical. V9 made viewport interaction semantic. V10 made feature
edits, rebuild impact, reference health, and repair explicit. V11 made sketch
design intent and solver status first-class.

The next bottleneck is result-body topology. Current Partbench can create and
display scoped boolean cut/add result bodies, but those result bodies mostly
remain diagnostic-only for generated references. That limits:

- sketch-on-face after a cut/add;
- naming faces and edges created or modified by booleans;
- downstream chamfer/fillet/hole/sketch workflows on boolean result bodies;
- reliable feature edits after booleans;
- agent-authored multi-step CAD workflows;
- future imported-body healing, assemblies, and production selection buffers.

V12 should remove the first meaningful slice of that bottleneck without
pretending to solve arbitrary OCCT topological naming.

## Architecture Alignment

V12 follows `docs/architecture.md`:

- the command protocol remains the center of the system;
- `cad-core` remains the authority for document source, boolean result
  lifecycle, generated-reference commandability, reference health, repair
  eligibility, and semantic diffs;
- React and the renderer may display semantic state but must not infer stable
  topology or commandability;
- geometry-worker/OCCT may compute derived exact geometry, topology counts,
  bounds, measurements, and opaque signatures, but raw OCCT topology indexes do
  not become public stable CAD IDs;
- renderer IDs, mesh indexes, GPU buffers, selection-buffer colors, viewport
  pixels, OPFS paths, file handles, STEP/GLB artifacts, and browser state do not
  enter stable generated-reference identity;
- agents and MCP remain thin wrappers over CADOps/query paths;
- unsupported or ambiguous cases return structured diagnostics instead of
  silently exposing unstable references.

V12 should make the topological naming problem narrower and more explicit. It
should not move CAD authority into the renderer or the geometry kernel.

## Non-Goals

V12 should not implement:

- arbitrary stable topological naming for every OCCT result shape;
- persistent exact B-rep checkpoints as source truth by default;
- STEP import, imported-body healing, or assembly reconstruction;
- WebGPU or production selection-buffer rendering;
- broad new modeling commands such as shell, sweep, loft, pattern, mirror, or
  direct edit;
- general boolean trees beyond the existing scoped cut/add source model unless
  a tranche explicitly scopes one additional narrow case;
- stable generated topology for revolve, hole, chamfer, or fillet result bodies
  unless the release explicitly adds a separate scoped tranche after boolean
  topology is proven;
- automatic silent retargeting of named references;
- persisted viewport camera, hover, selection, active tool, or measurement
  session state;
- natural-language command entry or hosted collaboration.

## Major V12 Decisions

### 1. V12 Is A Stable Boolean Result Topology Release

The first priority is current authored boolean result bodies produced by
`feature.extrude` with `operationMode: "cut"` or `operationMode: "add"`.

The supported starting subset is:

- a target body whose source topology is already command-ready from current
  source semantics, initially rectangle/circle `newBody` extrude bodies and any
  later source-semantic bodies explicitly proven by V12 tests;
- a tool profile from an authored rectangle or circle sketch entity;
- an attached sketch placement that resolves through an existing command-ready
  generated planar face or a base-plane sketch where the operation is already
  supported;
- one direct boolean result body, not an arbitrary boolean tree.

V12 can expand within the release only after the first subset is tested and
still keeps raw kernel topology private.

### 2. Stable IDs Come From Semantic Roles, Not Kernel Indexes

V12 generated-reference IDs for supported boolean results must be deterministic
from public source semantics:

- result body ID;
- creating feature ID;
- operation mode;
- target source body/reference role;
- tool sketch/profile entity ID and role;
- face/edge/vertex semantic role;
- source profile segment role or source entity role when needed;
- optional source-derived role ordinal when the ordinal is defined by source
  entity order, not by OCCT topology order.

Stable IDs must not encode:

- OCCT shape pointers, topology indexes, face indexes, edge indexes, vertex
  indexes, or generated kernel IDs;
- renderer object IDs, mesh IDs, triangle indexes, selection-buffer IDs, GPU
  buffers, or pixel coordinates;
- OPFS paths, file handles, local file paths, cache keys, export artifact IDs,
  or viewport/session state.

Opaque geometric signatures may be reported as derived validation metadata, but
they are not public stable IDs and are not source-of-truth by themselves.

### 3. Source Semantics First, Checkpoints Later

V12 should first prove stable references from authored source semantics and
derived validation. It should not add persistent exact B-rep checkpoints as
source truth unless a later tranche proves source semantics are insufficient
for a clearly scoped command-ready reference.

The default V12 plan does not require a new saved-project schema. Query-derived
boolean topology, generated-reference labels, reference health, measurements,
and derived signatures do not introduce `web-cad.project.v18`.

`web-cad.project.v18` is reserved only if a V12 tranche adds new
source-of-truth document data, such as explicit user-authored topology anchors,
manual topology repair records, or persisted exact checkpoint source records.
That is not part of the default V12 scope.

The `.wcad` package version remains `partbench.wcad.v1` unless the package
layout changes. A schema V18 source document can still be stored in the same
package layout if V18 ever becomes necessary.

### 4. Boolean Result Roles Must Be Explicit

V12 should introduce a boolean result topology role vocabulary before expanding
command support. The vocabulary should be understandable by UI, agents, MCP,
tests, and future selection buffers.

For supported cut results, candidate roles include:

- `booleanResultBody`;
- `targetCarriedFace`;
- `targetModifiedFace`;
- `targetSplitFace`;
- `cutWallFace`;
- `cutStartRimEdge`;
- `cutTerminalFace`;
- `cutTerminalRimEdge`;
- `cutExitRimEdge`;
- `cutWallProfileEdge`;
- `targetCarriedEdge`;
- `targetSplitEdge`;
- `intersectionVertex`.

For supported add results, candidate roles include:

- `booleanResultBody`;
- `targetCarriedFace`;
- `targetModifiedFace`;
- `addedWallFace`;
- `addedCapFace`;
- `addSeamEdge`;
- `addProfileEdge`;
- `targetCarriedEdge`;
- `intersectionVertex`.

The implementation should add only the roles it can prove. Unsupported roles
must stay diagnostic rather than inferred from index order.

### 5. Command-Ready Means Eligible Operations Are Proven

V12 should not make every generated reference command-ready just because it has
a label. Each supported boolean result reference must carry explicit operation
eligibility, such as:

- `reference.nameGenerated`;
- `feature.attachSketchPlane` for planar result faces;
- `feature.chamfer` or `feature.fillet` for supported linear/circular result
  edges only after those edge roles are proven;
- measurement/inspect operations where source or derived exact data can support
  the value with a clear authority label.

If a reference is selectable but not command-ready, the response should say why
using existing reference-health categories.

### 6. Reference Health Uses Existing V10 Categories

V12 should extend, not replace, the V10 vocabulary:

- active;
- replaced;
- stale;
- consumed;
- ambiguous;
- missing;
- unsupported;
- repair-needed;
- deleted.

Boolean result topology should participate in:

- `selection.referenceCandidates`;
- `reference.health`;
- `project.dependencyGraph`;
- `project.rebuildPlan`;
- `feature.editability`;
- `reference.repairName`;
- transaction semantic diffs;
- agent/MCP query and batch responses.

### 7. Target Consumption Remains A Source Lifecycle Concept

Cut/add operations continue to consume target bodies and create result bodies.
V12 should make the result body's supported generated topology command-ready;
it should not resurrect consumed target bodies as command targets.

Consumed target references remain useful for lineage, rebuild, and diagnostics,
but downstream commands should target active result-body references unless a
specific command explicitly accepts consumed source lineage as input.

### 8. UI Integration Must Stay Compact

V12 should improve the existing Tree, Selection, Inspector, Modeling, and
compact viewport surfaces without reintroducing viewport-blocking panels.

Supported boolean result references should appear naturally in the existing
selection/reference lists and actions. Detailed ambiguity or repair diagnostics
belong in the left Selection/Inspector/Modeling surfaces, not as large overlays.

Exact face/edge highlighting may remain owning-body-level until renderer
support improves. Commandability must still come from cad-core, not visual
highlighting.

## Proposed Tranche Sequence

### Implemented Tranche A: Boolean Topology Contract And Readiness

V12 Tranche A adds the shared boolean result topology readiness vocabulary
without making unsupported references command-ready.

Implemented behavior:

- protocol/core types now define `partbench.boolean-topology.v1` readiness,
  boolean result roles, role status, derived exact validation status, source
  inputs, and structured diagnostics;
- `body.topology` returns an optional `booleanTopology` readiness block for
  current authored add/cut extrude result bodies;
- readiness reports source semantic inputs from authoritative document state:
  feature ID, result body ID, operation mode, target body ID, tool sketch ID,
  tool sketch entity ID, and tool profile kind;
- add and cut result bodies expose readiness entries with every result role
  still `commandReady: false`;
- matching derived exact metadata updates only the validation status to
  `available`; it does not make topology available or command-ready;
- stale, unsupported, failed, unavailable, or absent derived exact metadata
  remains structured validation diagnostics;
- no generated-reference resolver, selection, naming, attach-sketch, repair,
  schema, `.wcad`, renderer, or UI commandability behavior changed.

Tranche A deliberately leaves `topologyAvailable: false` for boolean result
bodies. It is a contract/readiness slice only.

Tests cover protocol shapes, add/cut readiness, derived exact metadata as
validation-only evidence, and preservation of existing ambiguous topology
diagnostics.

### Implemented Tranche B: Source-Semantic Boolean Role Derivation

V12 Tranche B derives the first source-semantic boolean result role set from
current supported source features without promoting those roles to generated
references.

Implemented behavior:

- boolean result body identity is reported as a proven source-backed role with a
  deterministic `boolean-role:body:*` readiness ID;
- direct cut results from command-supported rectangle tool profiles derive four
  proven
  source-semantic `cutWallFace` roles (`side:uMin`, `side:uMax`, `side:vMin`,
  and `side:vMax`);
- direct cut results from command-supported circle tool profiles derive one
  proven cylindrical `cutWallFace` role (`side:circular`) plus proven circular
  start/terminal rim edge roles (`start:circular` and `end:circular`);
- proven cut-wall roles include deterministic readiness IDs, labels, and source
  role names;
- `booleanTopology.status` becomes `partial` when at least one non-body result
  role is source-proven or command-ready;
- command-supported rectangle add results derive proven `addedWallFace` roles
  (`side:uMin`, `side:uMax`, `side:vMin`, and `side:vMax`) plus a proven
  `addedCapFace` role (`endCap`);
- command-supported rectangle add results also derive four proven
  `addProfileEdge` roles on the added cap (`end:uMin`, `end:uMax`, `end:vMin`,
  and `end:vMax`);
- command-supported circle add results derive one cylindrical `addedWallFace`
  role (`side:circular`), one planar `addedCapFace` role (`endCap`), and one
  circular added-cap profile edge role (`end:circular`);
- carried/modified/split target faces, add seam/base edges, terminal faces, rim
  edges, split edges, and intersection vertices remain ambiguous or deferred;
- Tranche C promotes the command-supported rectangle cut-wall, circle cut-wall,
  rectangle add wall/cap, and circle add wall/cap roles to command-ready
  generated references;
- derived exact metadata remains validation-only evidence and does not affect
  source identity or commandability;
- no repair, schema, `.wcad`, renderer, or exact-kernel authority behavior
  changed.

Tests cover:

- protocol shapes for partial role readiness;
- add/cut readiness diagnostics;
- rectangle cut-wall and add wall/cap role derivation;
- derived exact metadata as validation-only evidence;
- preservation of ambiguous top-level body topology and deferred result roles.

### Implemented Tranche C: Command-Ready Boolean Face References

V12 Tranche C makes the first supported boolean result planar faces usable by
the existing generated-reference consumers.

Implemented behavior:

- `body.generatedReferences` and `body.resolveGeneratedReference` now expose
  command-ready generated face references for direct command-supported rectangle
  cut-wall roles on supported cut result bodies and rectangle add wall/cap roles
  on supported add result bodies;
- supported cut-wall generated references use existing stable IDs scoped to the
  result body, for example `generated:face:body_cut_1:side:uMin`;
- supported add generated face references use the same result-body-scoped stable
  ID family, for example `generated:face:body_add_1:side:uMin` for added wall
  faces and `generated:face:body_add_1:endCap` for the added cap;
- generated signatures include the extrude operation mode and target body ID as
  derived context, while keeping renderer, mesh, OCCT, and selection-buffer IDs
  out of public references;
- `selection.referenceCandidates` resolves those generated references through
  the existing shared query path;
- `reference.nameGenerated` can name supported cut-wall and add wall/cap face
  references;
- `sketch.createOnFace` can attach sketches to supported rectangle cut-wall and
  add wall/cap faces through the existing generated-face validator;
- the boolean result body reference is present as a container for generated
  references but has no command-ready operations;
- add seam/base edges, carried/split target faces, terminal faces, rim edges,
  and intersection vertices remain non-commandable diagnostics.

Tests cover:

- generated-reference creation and resolution for supported rectangle cut-wall
  faces and supported rectangle add wall/cap faces;
- selection candidate commandability for supported boolean result faces;
- naming and `sketch.createOnFace` through the same CADOps path;
- preservation of the current project schema and no raw renderer/kernel ID
  leakage.

### Tranche D: Boolean Edge And Rim References

Goal: add command-ready edge references only where source semantics prove them.

Status: D1 is implemented for rectangle cut-wall profile edges. D2 is
implemented for rectangle added-cap profile edges.

D1 implemented:

- direct command-supported rectangle cut results now expose four command-ready
  cut-wall profile edges:
  - `generated:edge:<resultBodyId>:longitudinal:uMin:vMin`;
  - `generated:edge:<resultBodyId>:longitudinal:uMin:vMax`;
  - `generated:edge:<resultBodyId>:longitudinal:uMax:vMin`;
  - `generated:edge:<resultBodyId>:longitudinal:uMax:vMax`;
- `body.topology.booleanTopology.roleReadiness` reports those entries as
  `cutWallProfileEdge` with matching generated stable IDs;
- `body.generatedReferences`, `body.resolveGeneratedReference`,
  `selection.referenceCandidates`, `reference.nameGenerated`, and
  `body.generatedReferenceMeasurements` support the D1 edge subset;
- D1 edge eligibility is intentionally limited to `feature.measureReference`
  and `feature.selectReference`, with naming supplied through the existing
  selected-reference path;
- chamfer/fillet on boolean result edges remains deferred because the current
  derived geometry source/runtime only supports edge-finish targets backed by
  active `newBody` extrudes.

D2 implemented:

- direct command-supported rectangle add results now expose four command-ready
  added-cap profile edges:
  - `generated:edge:<resultBodyId>:end:uMin`;
  - `generated:edge:<resultBodyId>:end:uMax`;
  - `generated:edge:<resultBodyId>:end:vMin`;
  - `generated:edge:<resultBodyId>:end:vMax`;
- `body.topology.booleanTopology.roleReadiness` reports those entries as
  `addProfileEdge` with matching generated stable IDs;
- `body.generatedReferences`, `body.resolveGeneratedReference`,
  `selection.referenceCandidates`, `reference.nameGenerated`, and
  `body.generatedReferenceMeasurements` support the D2 add-edge subset;
- D2 edge eligibility is intentionally limited to `feature.measureReference`
  and `feature.selectReference`, with naming supplied through the existing
  selected-reference path;
- add seam/base edges, target-carried edges, split/intersection edges, and
  chamfer/fillet on add result edges remain deferred.

Circle tool slice implemented:

- direct command-supported circle cut results now expose one command-ready
  cylindrical cut-wall face (`generated:face:<resultBodyId>:side:circular`) and
  two command-ready circular rim edges:
  - `generated:edge:<resultBodyId>:start:circular`;
  - `generated:edge:<resultBodyId>:end:circular`;
- direct command-supported circle add results now expose one command-ready
  cylindrical added-wall face (`generated:face:<resultBodyId>:side:circular`),
  one planar added-cap face (`generated:face:<resultBodyId>:endCap`), and one
  command-ready added-cap circular edge
  (`generated:edge:<resultBodyId>:end:circular`);
- circle cylindrical faces and circle boolean edges are intentionally limited to
  select/name/measure/inspect behavior; only the planar added cap is eligible
  for `sketch.createOnFace`;
- add seam/base topology, target-carried/split topology, arbitrary
  intersections, and boolean edge-finish commands remain deferred.

Scope:

- derive supported cut wall/profile edge roles from source profile segments and
  target face roles;
- expose supported edge generated references through the same resolver and
  `selection.referenceCandidates`;
- enable naming and measurement/inspect for proven supported edge roles;
- enable chamfer/fillet affordances only after the command validator and
  derived geometry pipeline can support boolean-result edge-finish targets;
- keep complex split/intersection edge cases ambiguous or repair-needed.

Definition of done:

- supported boolean result edges can be selected and named;
- supported D1 boolean result edges can be measured through source-analytic
  generated-reference measurements;
- chamfer/fillet affordances remain hidden or non-commandable until the
  edge-finish runtime can consume boolean result targets;
- unproven edge topology remains diagnostic.

### Tranche E: Rebuild, Edit, And Reference Health Integration

Goal: make V12 references survive or fail explicitly across supported upstream
edits.

Status: E1 is implemented for upstream source-extrude edits that revalidate a
direct rectangle cut result. The current add source/query slice also reports
active generated and named reference effects for supported rectangle add
wall/cap faces and add cap profile edges after the upstream source extrude is
edited. E2 is implemented for supported source sketch/profile edits and driving
sketch dimension edits that revalidate the same direct downstream boolean
result subset. The source/query repair slice is implemented for explicit
`reference.repairName` retargeting from missing names to command-ready V12 cut
wall face/edge, add wall face, add cap face, and add cap edge targets. Browser
smoke now proves the existing compact repair affordance for repairing a missing
name to a command-ready V12 cut-wall face. The current query-alignment slice
makes `feature.editability`, `project.dependencyGraph`, and
`project.rebuildPlan` agree with those supported V12 result references,
including circle add cap edge health/editability, while keeping the broad result
body lifecycle repair-needed/ambiguous. Broader browser repair polish remains
deferred.

E1 implemented:

- `feature.updateExtrude` semantic diffs now include active generated-reference
  effects for command-ready V12 cut-wall face and cut-wall profile edge stable
  IDs on a revalidated rectangle cut result;
- named references pointing at those command-ready V12 stable IDs also receive
  active reference effects;
- the broader result body still reports `AMBIGUOUS_RESULT_TOPOLOGY` /
  repair-needed replacement semantics for topology V12 has not proven;
- `reference.health` continues to resolve supported named V12 cut-wall edge
  references as active and commandable after the upstream edit;
- supported named rectangle add wall/cap face references also remain active and
  commandable after the upstream source-extrude edit;
- `sketch.updateEntity` source profile edits now emit the same consumed source,
  active generated/named result-reference, and repair-needed broad result
  topology effects for supported direct rectangle cut chains;
- `sketch.dimension.update` source driving-dimension edits now emit the same
  active generated/named result-reference effects for supported direct
  rectangle add wall/cap faces;
- source-affecting solver-backed `sketch.constraint.create` and
  `sketch.dimension.create` commit paths are covered for the current supported
  direct downstream boolean subset: midpoint constraint creation keeps V12
  cut-wall face/edge refs active, and fresh driving dimension creation keeps
  V12 add wall/cap face refs active;
- `reference.repairName` can explicitly repair missing named references onto
  command-ready V12 cut-wall face, cut-wall profile edge, and added-cap face
  targets when the target kind and `selection.referenceCandidates` contract
  validate the replacement;
- `feature.editability` now reports active supported V12 generated and named
  result-reference changes for direct cut/add downstream consumers, while also
  retaining the broad `AMBIGUOUS_RESULT_TOPOLOGY` replacement diagnostic for
  unproven result topology;
- `project.dependencyGraph` and `project.rebuildPlan` are covered for supported
  V12 cut/add refs: generated/named reference nodes stay active, result body
  containers remain non-command-ready, and body lifecycle stays conservative;
- the deterministic V12 release smoke covers cut/add -> select/name/sketch ->
  feature edit -> source sketch/dimension edit -> solver-backed source create
  edit -> reference effects -> reference health -> explicit repair ->
  graph/rebuild/editability alignment -> JSON/`.wcad`.

Scope:

- continue expanding solver-backed sketch edit coverage only when a path
  actually changes source geometry or downstream rebuild state; non-geometric
  rename/delete operations should not invent result-topology effects;
- update `reference.health`, `project.dependencyGraph`, `project.rebuildPlan`,
  and `feature.editability` for V12 boolean result topology;
- preserve existing named references when stable IDs remain valid;
- require explicit `reference.repairName` when a named result reference can no
  longer be proven or must move to a replacement target;
- keep failed rebuilds non-mutating.

Definition of done:

- supported source edits update boolean result lifecycle/reference effects in
  semantic diffs;
- stale or ambiguous named references can be diagnosed and explicitly repaired;
- tests cover edit -> rebuild -> reference health -> repair -> JSON/`.wcad`
  round-trip.
