# V15 Release Plan: STEP Import, Expanded Feature Families, and Parameter Expressions

Status: planned.

V15 is the next large release following the completed V14 topology-backed
downstream modeling release. V14 turned V13 topology anchors into real
downstream command targets for chained single-part modeling. V15 should
significantly expand what Partbench can do by adding three major capability
clusters:

1. **STEP import and imported body integration** — bringing real-world geometry
   into Partbench and connecting it to the V13 topology identity system so
   imported bodies can participate in downstream modeling.
2. **Expanded solid feature families** — linear pattern, circular pattern,
   mirror, and shell features, which are required for practical part design and
   have been deferred since V6.
3. **Parameter expression language** — arithmetic expressions that reference
   other parameters, making the parametric dimension system genuinely useful
   for design intent.

This document is a release contract, not an implementation script. It is strict
about product outcomes, architecture boundaries, source truth, diagnostics, and
proof. It is flexible about exact internal helper names, slice order, and
algorithms where implementation evidence should drive the best design.

## Product Goal

After V15, a Partbench user should be able to:

1. Import a STEP file and see the imported body in the feature tree.
2. Select faces and edges on an imported body where topology proof exists.
3. Create sketches on imported planar faces and perform cut/add operations.
4. Create linear or circular patterns of an existing body along a global axis.
5. Mirror an existing body across a standard plane.
6. Hollow a solid body using the shell operation, removing specified faces.
7. Define parameters with arithmetic expressions that reference other named
   parameters, and see downstream dimensions rebuild automatically.

The user-facing promise is not "every STEP file opens perfectly" or "every
feature chain rebuilds without error." The promise is:

- Supported STEP files (single-body and simple multi-body AP203/AP214/AP242)
  import as authored imported body records connected to the V13 topology
  system, with healing diagnostics and clear structured errors for failures.
- Pattern, mirror, and shell features work for the supported target body types
  with structured diagnostics for unsupported or failed cases.
- Parameter expressions evaluate in dependency order and trigger downstream
  rebuild, with circular reference detection and structured diagnostics.

## Why V15 Comes After V14

V14 completed the topology-backed downstream modeling foundation:

```text
topology anchors ->
chained sketch-on-result-face ->
cut/add on result bodies ->
edge finish on result edges ->
rebuild/reference health
```

That foundation is necessary but not sufficient for a broadly useful CAD
application. V15 addresses the three most impactful remaining gaps:

**STEP import** closes the gap between Partbench and the real-world CAD
ecosystem. Without import, Partbench can only operate on geometry it created
itself. V13's checkpoint/anchor system was built to support imported geometry
— V15 is the release that delivers on that investment.

**Expanded feature families** close the gap in modeling breadth. Current
Partbench can extrude, revolve, hole, chamfer, and fillet. Every practical
part design needs pattern, mirror, and shell. These have been deferred since
V12 explicitly listed them as non-goals.

**Parameter expressions** close the gap in parametric utility. Currently a
user can bind a dimension to a parameter, but parameters are independent. A
mounting plate where `bolt_pitch = plate_width / 4` is not expressible without
manual recalculation. Expressions unlock genuine design intent capture.

## Architecture Alignment

V15 follows the same non-negotiable boundaries as `docs/architecture.md`:

- CADOps remains the center of mutation. No new feature bypasses the command
  layer.
- `cad-core` owns document source, feature records, parameter evaluation,
  commandability, reference health, rebuild planning, semantic diffs, and
  diagnostics.
- geometry-worker/OCCT provides STEP reading, healing, B-rep execution for new
  feature families, topology snapshots for imported bodies, and checkpoint
  payload bytes. It does not own source authority.
- React and the renderer display selections, feature status, and diagnostics
  derived from cad-core/query results. They do not infer feature correctness
  from mesh geometry.
- Agent and MCP adapters stay thin wrappers over CADOps/query paths. New V15
  commands are accessible through the same `cad.batch` interface as existing
  commands.
- renderer IDs, mesh IDs, OCCT handles, checkpoint-local topology IDs, GPU
  IDs, selection-buffer values, viewport pixels, OPFS paths, file handles,
  local paths, and export artifact IDs must never become public stable CAD
  references.
- Unsupported, stale, ambiguous, failed, or partially-healed states return
  structured diagnostics. They must not silently produce incorrect geometry.

V15 may add source data only when new features require source truth that cannot
be reconstructed from existing V18 topology identity records and command
history. New feature families and parameter expressions require new source
records and therefore introduce `web-cad.project.v19`.

## Major V15 Decisions

### 1. STEP Import Uses V13's Topology Identity System

Imported bodies are not a new category of second-class geometry. They are
authored `ImportedBodyFeature` source records connected to the same V13
topology checkpoint and anchor system that backs authored feature bodies.

The import pipeline should:

- Read STEP bytes through the isolated geometry boundary (OCCT STEP reader).
- Apply a healing pass for common import issues (open shells, tolerance gaps,
  degenerate geometry) and record healing diagnostics.
- Extract exact B-rep topology snapshots for each imported body.
- Produce topology checkpoint payload bytes for the V13 checkpoint system.
- Return checkpoint metadata and body summaries to cad-core.

Cad-core should:

- Create `ImportedBodyFeature` source records with checkpoint references.
- Store checkpoint metadata in `web-cad.project.v18` topology identity records
  (same as V13-authored checkpoints).
- Allow users and agents to mint topology anchors for imported body entities
  through the same explicit `topology.anchor.create` command path as V13.
- Report clear structured diagnostics when import fails or healing is partial.

Imported body downstream commands should reuse the V13/V14 command-target
readiness contract. An imported face that has a topology anchor with
axis-aligned planar proof can host a sketch through the same
`sketch.createOnFace` path. An imported body that has an active topology anchor
can be targeted by cut/add through the same anchor-backed `feature.extrude`
path. The command validators must prove the target from checkpoint evidence —
they must not silently accept imported geometry just because it exists.

### 2. Pattern Features Produce Merged Single Result Bodies

Linear and circular patterns should produce a single merged B-rep result body
that contains all instances fused together. This is the simplest model that
maps correctly onto the existing consumed-body lifecycle:

```text
seed body (active)
-> LinearPatternFeature consumes seed body
-> merged result body (N instances fused via OCCT BRepAlgoAPI_Fuse)
```

The seed body is consumed by the pattern feature and becomes a rebuild input.
The result body is the new active body. This is consistent with how
`feature.chamfer` and `feature.fillet` consume their target and produce a result.

For V15, pattern instances are not individually addressable as separate bodies.
Individual instance reference/topology remains a V16 concern. Pattern result
bodies do NOT auto-create topology checkpoints; checkpoints require an explicit
`topology.checkpoint.create` call. Topology anchors can be explicitly minted for
result topology after a checkpoint is manually created.

Pattern direction for V15 is limited to the three global axes (X, Y, Z). Named
edge references as pattern directions are a V16 extension.

### 3. Mirror Produces a Single Result Body

Mirror should produce a single result body. The source body is optionally
included in the result (controlled by `includeOriginal`). The mirror plane is
one of the three standard planes (XY, XZ, YZ).

The mirror plane for V15 is limited to XY, XZ, YZ. Named plane references are
a V16 extension.

Mirror uses OCCT BRepBuilderAPI_Transform or equivalent mirroring APIs.

### 4. Shell Removes Specified Faces and Hollows the Body

Shell takes an active body, a wall thickness, and optional open-face references
(faces to remove). The result is a hollow solid.

Open face references for V15 are limited to:
- Generated face stable IDs on the target body (same as sketch-on-face refs).
- Named references that resolve to generated faces on the target body.
- Topology anchor IDs for faces that have V13 proof.

Unsupported face references, faces that cannot be opened without geometry
failure, and too-thin walls that cause collapse are structured errors.

Shell uses OCCT BRepOffsetAPI_MakeThickSolid.

### 5. Parameter Expressions Are Pure Arithmetic

The expression language for V15 is deliberately narrow:

- Numeric literals (integer or decimal): `10`, `3.14`
- Named parameter references: the parameter's exact `name` field
- Arithmetic operators: `+`, `-`, `*`, `/`
- Grouping: `( )` 
- Built-in functions: `max(a, b)`, `min(a, b)`, `abs(x)`, `sqrt(x)`,
  `round(x)`, `floor(x)`, `ceil(x)`

No scripting, no conditionals, no trigonometric functions in V15. Those are V16
extensions.

Expressions are evaluated in topological parameter dependency order. Circular
references are detected before evaluation and reported as a structured
`PARAMETER_CIRCULAR_REFERENCE` diagnostic without evaluating any part of the
cycle.

The `CadParameter` source record gains an optional `expression?: string` field.
When an expression is present, the parameter's `value` is derived from
evaluating the expression. When no expression is present, `value` is the
authoritative literal value (existing behavior).

### 6. V19 Is the New Source Schema for V15 Records

V15 introduces `web-cad.project.v19` to cover all new source-of-truth records
that cannot be represented by `web-cad.project.v18`:

- `ImportedBodyFeature` source records
- `LinearPatternFeature` source records
- `CircularPatternFeature` source records
- `MirrorFeature` source records
- `ShellFeature` source records
- `CadParameter.expression` optional expression field

Projects use V19 only when at least one of these records is present. Projects
without V15 source records continue to export as their minimum required version
(V16, V17, or V18).

`.wcad` package version remains `partbench.wcad.v2`. Imported body checkpoint
payloads use the same checkpoint entry structure as V13 authored bodies.

### 7. Sweep Is a Stretch Goal

`SweepFeature` (sweeping a profile sketch along a path sketch) is listed in
`docs/architecture.md` as a planned Feature variant. It is scoped as a V15
stretch goal:

- If implementation of Slices A–H proves the feature families and STEP import
  cleanly within the release, a Sweep slice may be added before the release
  closes.
- If implementation evidence shows sweep is better served by a separate focused
  release, it defers to V16.

The V15 definition of done does not require Sweep. The non-goals section lists
it explicitly as out of scope unless a stretch slice is added.

## New Source Records (web-cad.project.v19)

The following source records require V19 and must not appear in older schemas:

```ts
ImportedBodyFeature {
  id: string
  kind: "importedBody"
  name?: string
  sourceFileName: string          // original filename for display; not a path
  sourceFormat: "step"
  bodyId: string
  checkpointId: string            // V13 topology checkpoint ID; required for commit — import fails with STEP_CHECKPOINT_UNAVAILABLE if a checkpoint cannot be produced
  healingApplied: boolean
}

LinearPatternFeature {
  id: string
  kind: "linearPattern"
  name?: string
  seedBodyId: string              // active body consumed by this pattern
  axis: "x" | "y" | "z"
  spacing: number                 // positive finite distance between instances
  instanceCount: number           // integer >= 2
  resultBodyId: string
}

CircularPatternFeature {
  id: string
  kind: "circularPattern"
  name?: string
  seedBodyId: string
  rotationAxis: "x" | "y" | "z"
  totalAngleDegrees: number       // positive, <= 360
  instanceCount: number           // integer >= 2
  resultBodyId: string
}

MirrorFeature {
  id: string
  kind: "mirror"
  name?: string
  seedBodyId: string
  mirrorPlane: "XY" | "XZ" | "YZ"
  includeOriginal: boolean
  resultBodyId: string
}

ShellFeature {
  id: string
  kind: "shell"
  name?: string
  targetBodyId: string            // active body consumed by this shell
  wallThickness: number           // positive finite
  openFaceRefs: OpenFaceRef[]     // faces to remove (may be empty)
  resultBodyId: string
}

OpenFaceRef =
  | { kind: "generatedFace"; bodyId: string; stableId: string }
  | { kind: "namedReference"; name: string }
  | { kind: "topologyAnchor"; bodyId: string; anchorId: string }

// Extended from current CadParameter
CadParameter {
  id: string
  name: string
  value: number                   // evaluated value (literal or expression result)
  expression?: string             // optional arithmetic expression
  description?: string
}
```

pattern and return semantic diffs.

## STEP Import Pipeline Detail

The import pipeline is the most architecturally significant addition in V15.
The pipeline must maintain strict source/derived boundaries at every step.

### Phase 1: Geometry Boundary (geometry-kernel/geometry-worker)

The geometry boundary receives STEP file bytes as a transient input (never
stored in source). It should:

1. **Read STEP bytes** using OCCT STEPControl_Reader. Report structured errors
   for corrupt or unreadable files.

2. **Transfer shapes** from the STEP session to OCCT topology. Detect whether
   the result is a single compound, multiple solids, or an assembly tree.

3. **Heal** the transferred shapes using OCCT ShapeFix_Shape or
   BRepBuilderAPI_Sewing where appropriate. Record what healing actions were
   applied. Healing is not optional and must always run; partial healing results
   are valid with structured diagnostics.

4. **Extract metadata**: body name (from STEP product name or entity label),
   shape type, bounding box, face count, edge count, vertex count.

5. **Extract topology snapshots** using the same V13 topology snapshot
   extraction path (exact descriptors: surface class, area, normal, axis,
   radius, curve class, length, loop/coedge structure).

6. **Produce checkpoint payload bytes** using the same B-rep checkpoint writer
   as V13 (OCCT BRepTools.Write_3 format).

7. Return: `ImportedBodyPayload[]` containing body name, bounding box,
   face/edge/vertex counts, topology snapshot, checkpoint payload bytes, and
   structured healing diagnostics. Private OCCT shape handles, traversal order
   indexes, and topology maps must not appear in the response.

For V15, the supported STEP subset is:
- Single solid bodies and simple compounds of solids.
- Multi-body STEP: each top-level solid becomes a separate `ImportedBodyFeature`.
- Assembly STEP (STEP AP242 product structures): flatten to leaf solid bodies.
  Assembly hierarchy metadata (parent names, transforms) may be recorded in
  `ImportedBodyFeature.name` as a display hint but does not become a modeling
  construct in V15.
- Color and layer metadata from XDE: ignored in V15 (surfaced as display
  metadata in V16).

### Phase 2: cad-core

Cad-core receives `ImportedBodyPayload[]` from the geometry boundary and should:

1. **Validate** that each payload has the expected shape. Reject payloads with
   corrupt checkpoint bytes, empty topology snapshots, or missing source
   identity before creating any source records.

2. **Create checkpoint metadata records** (`TopologyCheckpointRecord`) in
   `web-cad.project.v18` topology identity for each imported body, referencing
   the checkpoint payload entries in the `.wcad` package. Checkpoint creation
   here is part of the explicit `project.importStep` transaction, not an
   auto-checkpoint side effect — imported bodies have no rebuild recipe (no
   sketch or feature history), so a checkpoint is the only way to restore their
   topology identity after a reload.

3. **Create `ImportedBodyFeature` source records** in `web-cad.project.v19`.
   The record stores the body ID, checkpoint ID, source filename, source format,
   and healing status. It does not store STEP bytes, OCCT handles, or mesh data.

4. **Emit a semantic diff** containing created bodies, checkpoint records, and
   imported body features.

5. **Preserve `.wcad` v2 package entries** for checkpoint payloads on the next
   Save/Save As, using the same mechanism as V13-authored checkpoints.

The command must support dry-run mode. A dry-run import should report:
- How many bodies would be imported.
- Estimated bounding boxes (from payload bounding box data).
- Any healing diagnostics that would be applied.
- Whether checkpoint payload bytes are available.
Without committing any source records.

### Phase 3: Downstream Command Eligibility

Imported body topology follows the V13 anchor/checkpoint command-eligibility
path. After import:

- Users and agents can call `topology.anchor.create` to mint stable anchors
  for specific imported body faces, edges, or vertices, using the same explicit
  command as V13-authored bodies.
- Active planar face topology anchors on imported bodies can be used by
  `sketch.createOnFace` through the `topologyAnchorFace` attachment path
  (same as V13/V14 exact-only face anchors).
- Active body topology anchors on imported bodies can be used as cut/add
  targets through `feature.extrude`'s `targetTopologyAnchorId` path (same as
  V14's topology-backed result body targets), subject to the same support
  matrix.
- Active edge topology anchors on imported bodies can be used by
  `feature.chamfer` and `feature.fillet` through the `topologyAnchorProof`
  edge-finish path (same as V13/V14 edge-finish proof).

No new command eligibility paths are added for imported bodies — they flow
through exactly the same V13/V14 command target paths as topology-anchored
authored bodies. This is intentional: imported bodies are topology identity
records, not a special command category.

## Pattern Feature Detail

Linear and circular patterns share the same lifecycle model:

```text
active seed body
-> pattern feature consumes seed body
-> geometry-worker creates N instances and fuses them (OCCT BRepAlgoAPI_Fuse
   or BRepBuilder compound + ShapeFix)
-> merged result body (one OCCT TopoDS_Solid)
-> result body is new active body (no auto-checkpoint; use topology.checkpoint.create explicitly)
```

The seed body is consumed by the pattern feature. This matches the
`feature.chamfer`/`feature.fillet` lifecycle: the seed body becomes a
rebuild input, and the result body is the new active body. Downstream commands
on the merged result body use V13/V14 topology anchor paths.

Active imported body records (`ImportedBodyFeature` with an active body ID)
qualify as seeds for pattern, mirror, and shell features on the same basis as
any other active authored body.

### Linear Pattern

`feature.linearPattern` accepts:
- `seedBodyId`: must be an active authored body (not already consumed).
- `axis`: `"x"`, `"y"`, or `"z"`.
- `spacing`: positive finite number in document units. Distance between
  instance centers. Minimum: greater than zero.
- `instanceCount`: integer >= 2. The total number of instances, including
  the original at position 0.

The N instance positions along the axis are:
```
position_i = i * spacing, for i in [0, instanceCount - 1]
```

The first instance is centered at the seed body's current location. Each
subsequent instance is offset by `spacing` along the chosen axis.

The result is the BRep union of all N instances. If the union fails (e.g.,
overlapping instances that cannot be cleanly fused), the command fails with a
structured geometry error. The failure must not mutate source state.

### Circular Pattern

`feature.circularPattern` accepts:
- `seedBodyId`: must be an active authored body.
- `rotationAxis`: `"x"`, `"y"`, or `"z"`. The axis of rotation passes through
  the document origin.
- `totalAngleDegrees`: positive finite, > 0, <= 360. Arc covered by the
  pattern.
- `instanceCount`: integer >= 2. Total instances.

Instance rotation angle formula depends on whether the pattern covers a full
circle:
- **Arc** (`totalAngleDegrees < 360`): `angle_i = totalAngleDegrees / (instanceCount - 1) * i`
  for i in [0, instanceCount - 1]. The first instance is at 0° and the last is
  at `totalAngleDegrees`.
- **Full circle** (`totalAngleDegrees == 360`): `angle_i = 360 / instanceCount * i`
  for i in [0, instanceCount - 1]. Instances are evenly distributed without a
  duplicate at the wrap-around point.

The result is the BRep union of all N rotated instances.

### Edit Behavior

`feature.updateLinearPattern` and `feature.updateCircularPattern` allow
changing spacing, count, and axis on existing features. The result body is
regenerated. If the updated parameters create geometry conflicts, the command
returns structured errors and does not mutate source.

Reference health for downstream features that depend on the pattern result body
is re-evaluated after edit. Topology anchors on the old result body should be
marked stale or repair-needed if the geometry changed.

## Mirror Feature Detail

Mirror does not consume the seed body. The seed body remains active in both
modes.

```ts
feature.mirror accepts:
- seedBodyId: must be an active authored body
- mirrorPlane: "XY" | "XZ" | "YZ"
- includeOriginal: boolean
  - true: result body = fused union of original + mirrored copy; seed remains active
  - false: result body = mirrored copy only; seed remains active
```

If `includeOriginal = true`, the result body is an independent BRep union of
the original body fused with its mirror image. The seed body remains active
alongside the result body. This is the most common use: creating a symmetric
part by mirroring one half.

If `includeOriginal = false`, the result body is just the mirror image. The
seed body remains active (not consumed).

The mirror plane always passes through the document origin (aligned with the
chosen standard plane). V16 can extend this with offset planes.

Mirror uses OCCT BRepBuilderAPI_Transform with a reflection matrix across the
chosen plane.

`feature.updateMirror` allows changing `mirrorPlane` and `includeOriginal`.

## Shell Feature Detail

Shell hollows a solid body by removing material to create walls of uniform
thickness, optionally removing specified faces to create openings.

```ts
feature.shell accepts:
- targetBodyId: must be an active authored body (not already consumed)
- wallThickness: positive finite number in document units
- openFaceRefs: array of OpenFaceRef (may be empty, creates fully enclosed shell)
```

Open face references must resolve to active faces on `targetBodyId`. For V15,
supported reference paths are:
- `{ kind: "generatedFace"; bodyId; stableId }` — generated face references on
  the target body, resolved through the same V12/V14 generated-reference path.
- `{ kind: "namedReference"; name }` — named references that resolve to
  generated faces on the target body.
- `{ kind: "topologyAnchor"; bodyId; anchorId }` — active topology face anchors
  on the target body with valid checkpoint proof.

Shell uses OCCT BRepOffsetAPI_MakeThickSolid with the resolved open faces as
the "faces to remove" list.

Common failure cases that must return structured errors without mutating source:
- `wallThickness` >= minimum wall thickness of the body (shell would collapse).
- Open face references that cannot be resolved or are stale.
- Geometry that OCCT cannot shell (very thin features, non-manifold geometry).
- Shell failures due to topology conflicts with the chosen faces.

`feature.updateShell` allows changing `wallThickness` and `openFaceRefs`. The
target body identity is fixed at creation; changing the target body requires
deleting and recreating the shell feature.

## Parameter Expression Language

### Expression Syntax

Expressions are plain-text arithmetic strings stored in `CadParameter.expression`.

Grammar (informal):
```
expression  ::= term | expression '+' term | expression '-' term
term        ::= factor | term '*' factor | term '/' factor
factor      ::= literal | paramRef | funcCall | '(' expression ')' | '-' factor
literal     ::= [0-9]+ ('.' [0-9]*)?
paramRef    ::= identifier       (matches CadParameter.name exactly)
funcCall    ::= identifier '(' expression (',' expression)* ')'
identifier  ::= [a-zA-Z_][a-zA-Z0-9_]*
```

Supported built-in functions: `max`, `min`, `abs`, `sqrt`, `round`, `floor`,
`ceil`.

Division by zero leaves the parameter's current `value` unchanged and reports a
structured `EXPRESSION_DIVISION_BY_ZERO` error. The value must NOT be set to
`Infinity` — `JSON.stringify(Infinity)` produces `"null"`, causing silent data
corruption on save.

Unresolvable parameter references (name not found) return a structured
`PARAMETER_REF_NOT_FOUND` error and leave the parameter's current `value`
unchanged.

Unsupported identifiers (not a known parameter name and not a built-in function)
return a structured `EXPRESSION_UNKNOWN_IDENTIFIER` error.

### Evaluation Model

The evaluation model is a directed acyclic graph (DAG) over parameters:

- An edge `A → B` means parameter B references parameter A in its expression.
- Evaluation processes parameters in topological order (dependencies first).
- If the DAG contains a cycle, no parameters in the cycle are evaluated.
  Instead, all parameters in the cycle receive a
  `PARAMETER_CIRCULAR_REFERENCE` diagnostic and retain their previous `value`.

When a parameter's value changes (either by updating a literal value or by
evaluating an expression), all parameters that directly or transitively depend
on it are re-evaluated in topological order. All sketch dimensions that
reference an affected parameter are rebuilt. All features that reference those
sketch dimensions are flagged for rebuild.

### `parameter.setExpression` Command

This command extends the existing `parameter.update` behavior:

- Accepts `parameterId` and an optional `expression` string.
- Setting `expression` to `null` or omitting it clears the expression (returns
  the parameter to literal-value mode).
- Setting `expression` to a non-empty string registers the expression.
- After the expression is registered, cad-core re-evaluates the parameter DAG.
- The resulting `value` for each affected parameter is updated in source.
- Semantic diffs report: which parameters changed value, which sketch
  dimensions were rebuilt, which features were affected.

The expression string is validated at command time:
- Parse errors are returned as structured validation errors before commit.
- Circular reference detection runs before evaluation.
- A `parameter.setExpression` that would create a circular reference is
  rejected with `PARAMETER_CIRCULAR_REFERENCE`.

### Storage

`CadParameter.expression` is a V19 source field. When any parameter has a
non-null expression, the project exports as `web-cad.project.v19`. When no
parameters have expressions, the project continues to export at its minimum
required version.

The `value` field continues to store the last evaluated numeric value. On
project load, cad-core re-evaluates all expressions in DAG order to confirm
consistency. If re-evaluation produces a different value than the stored `value`,
cad-core reports a structured `EXPRESSION_VALUE_INCONSISTENCY` diagnostic (not a
fatal import error) and uses the re-evaluated value.

## Implementation Slices

These slices are guidance. If implementation evidence shows a better sequence,
update this document and proceed with the safer path.

### Slice A: STEP Import Protocol and Readiness Contract

Goal: establish the STEP import command shape and capability readiness without
performing any import.

Scope:
- `project.importStep` CADOps command type and dry-run/commit shape.
- `project.importReadiness` query: reports geometry worker STEP reader
  capability, healing support, and current imported body count.
- `ImportedBodyPayload` geometry boundary response type (received from
  geometry worker, never persisted as source).
- `ImportedBodyFeature` source record type for V19.
- Structured diagnostic codes for import failure states:
  - `STEP_FILE_CORRUPT`
  - `STEP_READER_UNAVAILABLE`
  - `STEP_NO_SOLID_FOUND`
  - `STEP_HEALING_FAILED`
  - `STEP_CHECKPOINT_UNAVAILABLE`
  - `STEP_BODY_LIMIT_EXCEEDED`
  - `IMPORTED_BODY_CHECKPOINT_MISSING`
  - `IMPORTED_BODY_TOPOLOGY_UNAVAILABLE`
- `body.importedBodyStatus` query type.
- Agent/MCP stub pass-through for new query.
- Tests proving public protocol shapes exclude renderer, mesh, OCCT, GPU,
  selection-buffer, OPFS, file-handle, path, viewport, and export IDs.

Non-goals:
- No STEP bytes processed yet.

### Slice F: Mirror Feature

Goal: implement `feature.mirror` with full source/derived/rebuild/health
behavior.

Scope:
- `feature.mirror` command: validates seed body, mirrorPlane, includeOriginal;
  geometry-worker creates mirrored copy (OCCT BRepBuilderAPI_Transform with
  reflection matrix); optionally fuses original + mirror (OCCT
  BRepAlgoAPI_Fuse); cad-core creates V19 `MirrorFeature` + result body.
- `feature.updateMirror`: edits mirrorPlane, includeOriginal; regenerates result
  body.
- Mirror seed lifecycle is asymmetric and mirrors the linear/circular pattern
  family (whose result subsumes its seed instance and therefore consumes the
  seed):
  - `includeOriginal = true`: the result body is the union of seed + mirror,
    which geometrically subsumes the seed. The seed is consumed by the mirror
    feature (marked `consumedByFeatureId`) so the viewport and STEP export hold
    a single merged solid rather than two overlapping bodies.
  - `includeOriginal = false`: the result body is the mirrored copy alone, which
    does not contain the seed. The seed body stays active as an independent
    original alongside the mirror result.
  Consumption is derived from the current feature set, so toggling
  `includeOriginal` through `feature.updateMirror` flips it automatically:
  turning it on consumes the seed, turning it off releases the seed back to
  active. Creating a mirror on an already-consumed seed is rejected
  (`MIRROR_SEED_BODY_CONSUMED`).
- Project structure, health, dependency graph, rebuild plan, feature
  editability, reference health updated for mirror features.
- V19 import/export validation.
- Agent/MCP batch pass-through.
- UI: Mirror panel in Modeling; plane selector; includeOriginal toggle.
- Tests: mirror creation (both includeOriginal modes), edit, seed body survival
  when `includeOriginal = false`, seed body consumption when
  `includeOriginal = true`, includeOriginal toggle, reference health.

### Slice H: Parameter Expression Language

Goal: implement `CadParameter.expression`, expression parser, DAG evaluator,
and downstream rebuild effects.

Scope:
- Pure TypeScript expression parser (no external dependency) in
  `packages/cad-core` or a new `packages/expression-evaluator` sub-package.
  Parser validates syntax and reports structured parse errors before any
  evaluation. No eval() or Function constructor — expression safety is
  required.
- Parameter DAG: cad-core extracts parameter references from expression strings
  at command time and maintains a dependency graph.
- Circular reference detection: before any evaluation, detect cycles in the
  DAG and report `PARAMETER_CIRCULAR_REFERENCE` diagnostics for all parameters
  in each cycle.
- Topological evaluation order: evaluate parameter expressions in reverse
  topological order (dependencies first).
- `parameter.setExpression` command: set or clear expression; trigger DAG
  re-evaluation; update `value` fields; emit semantic diffs for all changed
  parameter values plus affected sketch dimensions and features.
- `parameter.update` (existing command) extended: when a parameter has an
  expression, `parameter.update` changing the literal value is blocked with a
  structured `PARAMETER_HAS_EXPRESSION` diagnostic. Users must use
  `parameter.setExpression` to clear the expression first.
- `project.parameterEvaluation` query: returns current parameter DAG as a
  readable structure (parameter IDs, names, expression strings, current values,
  parse errors, circular cycles, evaluation order).
- V19 schema: `CadParameter.expression` optional field; migration from V18
  adds null expression for all parameters.
- Import validation: on project load, re-evaluates all expressions in DAG
  order; reports `EXPRESSION_VALUE_INCONSISTENCY` as a diagnostic (not hard
  failure); uses re-evaluated values.
- Downstream rebuild: when a parameter value changes due to expression
  evaluation, all sketch dimensions bound to that parameter are updated
  (existing `parameter.update` rebuild chain). All affected features flag for
  rebuild.
- UI: Parameters panel expression editor — small input field beside the value
  for entering an expression; clear expression button; expression status
  (valid/invalid/circular); dependency indicators.
- Agent/MCP: `parameter.setExpression` accessible via `cad.batch`;
  `project.parameterEvaluation` accessible via `cad.query`.
- Tests: expression parse, circular detection, DAG evaluation order,
  expression update downstream rebuild, expression clear, project round-trip
  with expressions, inconsistency diagnostic on load.

## Sweep Feature (Stretch Goal)

If all Slices A–I complete within V15 scope with implementation quality that
meets the V15 definition of done, a Sweep slice may be added before the release
closes.

Minimal Sweep scope:
- Profile: a single rectangle or circle sketch entity on a base-plane or
  generated-face attached sketch.
- Path: a single line entity on a different base-plane sketch. For V15 the path
  must be a straight line (sweep along a linear path = equivalent to an
  off-axis extrude, but along the line direction rather than the sketch normal).
- `feature.sweep` command: validates profile sketch/entity, path sketch/entity;
  geometry-worker uses OCCT BRepOffsetAPI_MakePipe or equivalent; cad-core
  creates `SweepFeature` V19 source record + result body.
- If the path is not a straight line, the command is rejected with a structured
  `SWEEP_PATH_UNSUPPORTED` diagnostic (curved path sweeps defer to V16).
- `feature.updateSweep`: edits profile or path references.

If sweep is not implemented in V15, `SweepFeature` is reserved in the V19
schema shape but not instantiable until V16.

## Storage and Schema Position

V15 introduces `web-cad.project.v19` for all new V15 source records.

Projects without V15 source records continue to export at their minimum
required version (V16 for ordinary features, V17 when V11 advanced constraints
are present, V18 when V13 topology identity records are present).

### V19 Triggers

Projects export V19 when any of the following are present:
- Any `ImportedBodyFeature` record.
- Any `LinearPatternFeature` record.
- Any `CircularPatternFeature` record.
- Any `MirrorFeature` record.
- Any `ShellFeature` record.
- Any `CadParameter` with a non-null `expression` field.
- (If sweep is implemented) Any `SweepFeature` record.

### Migration from V18 to V19

V19 is additive over V18. Migration adds empty imported body, pattern, mirror,
and shell feature lists. Existing parameters gain `expression: null` by default.

### JSON Export for V19 Projects

V19 projects may contain imported body checkpoint payloads (same as V18 V13
topology checkpoints). JSON export must clearly report that checkpoint payloads
require `.wcad` — the same lossy-export warning as V13. Imported body topology
anchor health is marked as `IMPORTED_BODY_CHECKPOINT_MISSING` in JSON-only
loaded projects; the parametric source and feature tree still load correctly.

### `.wcad` Package Version

`.wcad` package version remains `partbench.wcad.v2`. Imported body checkpoint
payloads are stored as `checkpoints/<checkpointId>.brep` etc., exactly as V13
authored body checkpoints.

### Do Not Introduce V19 for Query-Only Changes

Pattern result body topology, expression evaluation results, import
health diagnostics, and healing outcomes are derived query data. They do not
justify a schema version by themselves. V19 is only for the new source records
listed above.

## Non-Goals

V15 should not implement the following unless a slice explicitly scopes one:

- Assembly system: mates, instance hierarchies, large-assembly LOD, and
  multi-part workflows.
- Production WebGPU renderer or selection-buffer-based picking.
- Exact face/edge/vertex highlighting in the viewport (deferred to renderer
  architecture work).
- Proprietary format import: SolidWorks, CATIA, NX, Creo, DWG.
- IGES import (scoped separately from STEP; Phase 1 architecture mentions it,
  but V15 limits import to STEP).
- Natural-language command entry.
- Hosted collaboration.
- Loft feature (V16).
- Sweep feature (V16 if not implemented as stretch in V15).
- Direct edit features (not yet scoped).
- Assembly-level pattern or mirror (no assembly system yet).
- Pattern instances as individually addressable sub-bodies (V16).
- Mirror with offset plane (V16).
- Circular pattern with user-defined rotation axis (V16).
- Parameter scripting or conditionals (V16).
- Trigonometric functions in expressions (V16).
- Broad automatic low-confidence topology retargeting.
- Treating renderer, mesh, OCCT, GPU, selection-buffer, OPFS, path,
  file-handle, viewport, or export artifact identifiers as public stable IDs.

## Definition of Done

V15 is done when:

1. Supported STEP files (AP203/AP214/AP242 single-body and simple compound)
   import through `project.importStep`, creating `ImportedBodyFeature` source
   records and V18 topology checkpoint entries.
2. Imported body checkpoint payloads are preserved in `.wcad` v2.
3. Topology anchors can be explicitly created on imported body faces, edges,
   and vertices using the existing `topology.anchor.create` path.
4. Topology-anchor-backed imported faces can host attached sketches through
   `sketch.createOnFace`.
5. Topology-anchor-backed imported bodies can be targeted by `feature.extrude`
   cut/add through the V14 `targetTopologyAnchorId` path.
6. `feature.linearPattern` creates correct merged result bodies for supported
   seed body types, with structured errors for unsupported/failed cases.
7. `feature.circularPattern` creates correct merged result bodies for supported
   seed body types.
8. `feature.mirror` creates correct result bodies in both `includeOriginal`
   modes, without consuming the seed body.
9. `feature.shell` creates correct hollow result bodies with and without open
   face references.
10. Pattern, mirror, and shell features are editable through their update
    commands with correct rebuild effects.
11. `parameter.setExpression` sets arithmetic expressions on parameters and
    triggers correct downstream rebuild of dependent parameters and sketch
    dimensions.
12. Circular reference detection rejects expression cycles with structured
    diagnostics.
13. All new V15 features are accessible through `cad.batch` (agent/MCP).
14. cad-core remains the authority for commandability, validation, source
    mutation, semantic diffs, reference health, and rebuild state.
15. React and renderer surfaces consume cad-core/query results and do not infer
    topology or feature correctness from visual geometry.
16. V19 JSON and `.wcad` round-trips preserve all new V15 source records
    without storing raw kernel, renderer, mesh, GPU, selection-buffer, OPFS,
    file, local path, viewport, or export artifact identifiers.
17. JSON export for V19 projects clearly reports that checkpoint payloads
    require `.wcad`.
18. User-facing UI does not expose internal diagnostic codes, release numbers,
    or implementation milestone labels as explanations.
19. Unsupported, failed, stale, consumed, ambiguous, missing, or
    repair-needed states return structured diagnostics without mutating source.
20. Focused unit tests cover: protocol shapes, cad-core command validation,
    semantic diffs, rebuild/reference health, V19 storage boundaries, adapter
    pass-through, source/derived/session separation, expression evaluation,
    and pattern/mirror/shell geometry failure paths.
21. `pnpm smoke:v15-step-import-workflow`, `pnpm smoke:v15-feature-families-workflow`,
    `pnpm smoke:v15-expressions-workflow`, and `pnpm smoke:v15-release-samples`
    pass.
22. `pnpm test`, `pnpm typecheck`, `pnpm lint`, `pnpm format:check`, and
    `pnpm build` pass.

```text
create rectangle extrude (body_1)
```

