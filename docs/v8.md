# V8 Release Plan: Local CAD Foundation And Exact Interop

Status: active major-release plan as of 2026-06-12. Tranche A storage
protocol/package contract, Tranche B `.wcad` package read/write helpers,
Tranche C File System Access project workflow, Tranche D1 OPFS cache
status/clear foundation, and Tranche D2 derived visualization mesh cache
population are implemented. Tranche E1 exact STEP export contract/readiness and
Tranche E2 minimal exact STEP writer execution are implemented for supported
active rectangle/circle `newBody` extrude source bodies through the geometry
boundary. Tranche F Agent/MCP package/export surface and Tranche G V8 release
package/export smokes are implemented.

V8 is the next major release after the V7 Real CAD Alpha. V7 made the supported
CAD workflow coherent: semantic references, feature tree, selection tab,
current canvas interaction, JSON project workflow, visualization export
readiness, agent/MCP review surfaces, and repeatable release smokes.

V8 should now cut into the architecture roadmap by making Partbench feel like a
real local CAD application:

```text
native project package -> local open/save/save-as -> OPFS derived cache ->
exact STEP export -> agent-readable project/export workflows ->
release-grade package/export smokes
```

This release deliberately does not attempt to finish the whole architecture.
WebGPU, broad arbitrary topological naming, broad sketch solving, assemblies,
hosted collaboration, production MCP auth, and broad STEP import remain separate
major risks unless a later V8 tranche explicitly scopes a narrow preparatory
slice.

Important naming note: the V8 release is not the same as
`web-cad.project.v8`, which is an old saved-project schema identifier for the
first horizontal/vertical sketch-constraint source slice. The first native
package format is a storage envelope named `partbench.wcad.v1`. The underlying
saved project document remains `web-cad.project.v16` until V8 adds new
source-of-truth document data. If that happens, the next project schema is
`web-cad.project.v17`, with explicit migrations.

## Product Goal

A user can work with a Partbench project like a local CAD project rather than a
debug JSON file:

```text
new project -> create/edit supported bodies -> save as .wcad ->
close/reopen .wcad -> keep command history/audit metadata ->
reuse derived cache when safe -> request exact STEP export with precise status ->
share JSON/GLB when useful as secondary artifacts
```

An agent or script can use the same command/query layer:

```text
inspect project/package state -> dry-run or commit CADOps ->
inspect export readiness -> request supported exports ->
receive structured package/export diagnostics -> avoid renderer/cache IDs
```

The user-visible product outcome is simple: Partbench gets a credible native
project workflow and the first exact CAD interchange path, without making
derived meshes, browser handles, or export artifacts authoritative.

## Architecture Alignment

V8 preserves these architecture decisions from `docs/architecture.md`:

- CADOps remains the center of mutations.
- `cad-core` owns authoritative source document state.
- Human UI, scripts, tests, agent adapter, and MCP adapter consume the same
  command/query layer.
- Rendered meshes, exact metadata, topology summaries, thumbnails, OPFS
  entries, and visualization exports are derived unless explicitly promoted to
  source data by a scoped schema change.
- React UI does not import geometry internals directly.
- Geometry, renderer, command engine, protocol, storage, import/export, and MCP
  boundaries stay separate.
- OPFS is used for browser-private cache data.
- File System Access API is used for user-visible local open/save when the
  browser supports it.
- The native package is documented and open. It is not a mystery binary blob.
- STEP is the primary exact CAD interchange target. Proprietary CAD formats are
  not in scope.
- MCP wraps CADOps and storage/export queries. MCP does not define internal
  architecture.
- Local and hosted modes should converge on the same static web app and command
  engine, even if V8 does not yet implement the launcher or hosted storage.

## Why V8 Is Not The Whole Remaining Architecture

The remaining architecture contains several high-risk subsystems:

- WebGPU renderer replacement, selection buffer, and exact viewport picking.
- Broad topological naming across arbitrary OCCT results.
- General sketch solver and drag/diagnostic UX.
- Assemblies, mates, instancing, LOD, and large-model performance.
- STEP import with healing, units, metadata, and assembly reconstruction.
- Production MCP authentication, permissions, and hosted collaboration.

Bundling all of those into V8 would create a release that is hard to test and
easy to overclaim. V8 instead chooses storage and exact export as the primary
roadmap cut because they unblock larger projects, real local workflows, future
assemblies, exact interchange, and cache discipline.

## V8 Scope Summary

V8 is in scope for:

- native `.wcad` package format v1;
- File System Access open/save/save-as for `.wcad` where available;
- download/upload fallback for browsers without File System Access;
- explicit dirty state, saved source identity, and package health UI;
- OPFS content-addressed derived cache for rebuildable artifacts;
- thumbnail and derived metadata cache rules, if implemented as cache;
- exact STEP export contracts and readiness for supported bodies, with real
  bytes only if the OCCT/WASM exchange writer is available through the geometry
  boundary;
- structured diagnostics when STEP export is unavailable or unsupported;
- agent/MCP pass-through for package status, project summary, export readiness,
  and export requests where safe;
- release samples and smokes for package round-trip, migration, source/derived
  separation, and exact export behavior.

V8 is not in scope for:

- STEP import, except perhaps package-level diagnostics for unsupported import
  attempts;
- IGES import/export;
- proprietary CAD formats;
- `.wcad` as a hosted collaboration format;
- local launcher implementation;
- WebGPU renderer replacement;
- exact face/edge/vertex viewport picking;
- persistent selection or hover state;
- broad arbitrary topological naming;
- broad sketch solver expansion;
- assemblies, mates, instancing, and LOD;
- new modeling commands such as shell, sweep, loft, mirror, patterns, or direct
  edits;
- production MCP auth or natural-language command parsing.

## Current V8 Implementation Status

### Implemented In Tranche A

V8 now has the first typed package/storage contract:

- package version `partbench.wcad.v1`;
- extension `.wcad`;
- typed `manifest.json` shape;
- required source entries `document.cbor` and `commands.cbor`;
- source identity algorithm `partbench-source-v1`;
- structured validation issue codes for malformed manifests, unsupported
  versions, missing source entries, invalid package paths, byte-length/hash
  mismatch, unsupported document schemas, source identity mismatch, and stale or
  unsupported cache entries;
- pure source-identity helpers over encoded document/command bytes plus schema
  and units metadata;
- read-only `project.packageReadiness` query in `cad-core`;
- thin agent-adapter and MCP pass-through via `cad.project_package_readiness`;
- tests proving source identity excludes filenames, file handles, OPFS paths,
  viewport/selection state, thumbnails, meshes, export artifacts, cache-only
  data, renderer IDs, OCCT IDs, and selection-buffer IDs.

### Implemented In Tranche B

V8 now has minimal deterministic `.wcad` package read/write helpers in
`cad-core`:

- package bytes are ZIP-compatible and deterministic for the same project
  source and metadata options;
- `manifest.json`, `document.cbor`, and `commands.cbor` are written as the
  required source entries;
- `document.cbor` stores the current authoritative document snapshot;
- `commands.cbor` stores committed transaction history and redo stack data;
- `partbench-source-v1` is computed from the encoded document and command
  bytes plus schema/units metadata;
- manifest byte lengths, hashes, and source identity are validated on read;
- malformed packages, missing entries, invalid CBOR payloads, hash/length
  mismatch, unsupported schemas, and source identity mismatch return structured
  diagnostics;
- optional cache index data can be reported as stale or unsupported warnings
  without blocking source load;
- packages round-trip through the same current project importer used by JSON
  import/export.

No ZIP or CBOR production dependency was added for Tranche B. The implemented
helpers support the narrow stored-ZIP and canonical JSON-compatible CBOR subset
needed for current project source packages. If later tranches need compression,
streaming, broad CBOR data models, or third-party ZIP compatibility hardening,
that dependency decision should be made explicitly in that tranche.

Tranche B does not call File System Access APIs, add upload/download UI, write
OPFS data, add thumbnails/mesh caches, export STEP, or change the saved project
schema from `web-cad.project.v16`.

### Implemented In Tranche C

V8 now has the first app-level `.wcad` project workflow:

- the Project/File panel exposes `Open .wcad`, `Save`, and `Save As`;
- supported browsers can use File System Access handles for direct open/save;
- browsers without direct handles, or denied picker access, fall back to
  `.wcad` upload/download;
- project state reports storage mode, dirty/saved state, package version,
  document schema, source identity, last result, and package diagnostics;
- JSON import/export remains available as explicit debug/interchange;
- browser file handles remain app-only permission state and are not written
  into `.wcad`, JSON, cad-core, agents, or MCP.

Tranche C does not write OPFS cache artifacts, add STEP export, introduce
`web-cad.project.v17`, or remove JSON import/export.

### Implemented In Tranche D1

V8 now has the first OPFS cache foundation in the web app:

- app-layer detection for `navigator.storage.getDirectory`;
- a Partbench V8 cache root and typed `partbench.opfs-cache.v1` index contract;
- cache entries keyed by source identity, artifact kind/version, document
  schema, units, kernel/worker versions, and settings;
- structured diagnostics for unavailable OPFS, permission or handle failure,
  missing/invalid indexes, stale source identity, unsupported artifact version,
  byte length/hash mismatch, clear failure, and ignored cache state;
- Project/File cache status, health counts, diagnostics, refresh, and Clear
  cache action;
- tests proving cache-only state does not change `.wcad` source identity or
  expose file handles, OPFS paths, renderer IDs, mesh IDs, OCCT IDs, or
  selection-buffer IDs.

Tranche D1 does not populate derived mesh, thumbnail, package-unpack, or export
intermediate artifacts. OPFS remains browser-private rebuildable cache only;
project load through `.wcad` or JSON must work when OPFS is missing, stale,
invalid, denied, or cleared.

### Implemented In Tranche D2

V8 now populates the first narrow OPFS artifact class:

- artifact kind `derivedMesh`;
- artifact version `partbench-derived-mesh.v1`;
- deterministic cache keys using source identity, document schema, units,
  derived-geometry source key, kernel version, worker version, and tessellation
  settings key;
- JSON-encoded rebuildable app artifact payloads containing the renderer-ready
  derived visualization mesh and derived-geometry metrics;
- cache read before worker generation for clean saved/opened `.wcad` source
  identity;
- cache write after successful derived mesh generation;
- structured diagnostics for missing, stale, corrupt, unsupported, unavailable,
  denied, read-failed, and write-failed artifact paths;
- Project/File cache status refreshed after derived cache reads and writes;
- tests proving cache hit/miss/fallback behavior and that OPFS paths, browser
  handles, renderer IDs, OCCT IDs, viewport state, selection-buffer IDs, and
  cache-only data do not enter project source identity.

Tranche D2 is intentionally narrow. OPFS is still optional browser-private
rebuildable cache. Unsaved or dirty projects generate derived visualization
meshes through the existing runtime path without using OPFS. Clearing cache does
not mutate document source, command history, current file handles, `.wcad`
bytes, JSON export, selection state, or viewport state. D2 does not add package
thumbnails, package-unpack cache, export intermediates, STEP export, WebGPU,
viewport picking, new modeling commands, or `web-cad.project.v17`.

## Current Bottlenecks V8 Relieves

### Local Project Bottleneck

Current Partbench can round-trip `web-cad.project.v16` JSON, but that is still a
debug/interchange workflow. V8 makes a user-visible project file the normal
path while keeping JSON export/import available for debugging, tests, and MCP.

### Cache Bottleneck

Current derived geometry can be rebuilt, but there is no durable browser-private
cache policy. V8 introduces OPFS cache rules so derived meshes, thumbnails, and
exact export preparation can be reused when source identities match and safely
discarded when stale.

### Interop Bottleneck

V7 `project.exportReadiness` honestly reported STEP as deferred. V8 E1
separates exact STEP readiness from Mesh/GLB visualization readiness, identifies
eligible source bodies from authoritative project state, and exposes a
`project.exportExact` request/result contract. V8 E2 proves the OCCT/WASM STEP
writer and produces AP242 STEP bytes for the supported exact source subset
through the geometry boundary. GLB remains visualization output only.

### Product Workflow Bottleneck

The UI still exposes JSON as the practical save/load path. V8 should move the
primary File/Project workflow to native project actions:

```text
New
Open .wcad
Save
Save As
Export STEP
Export GLB visualization
Export JSON
Import JSON
```

JSON stays available, but it should not be the normal product mental model once
`.wcad` exists.

### Agent Workflow Bottleneck

Agents can already inspect and edit the model through CADOps. V8 gives agents a
real project/export surface to inspect and request without reading browser UI or
raw files.

## Major V8 Decisions

This section records decisions that are not already fully fixed by
`docs/architecture.md`.

### 1. Native Package Name

The user-visible package extension is:

```text
.wcad
```

The format name is:

```text
Partbench WCAD Package
```

The first package version string is:

```text
partbench.wcad.v1
```

This is separate from `web-cad.project.v16` and any future
`web-cad.project.v17` document schema.

### 2. Physical Package Representation

V8 uses a ZIP-compatible package file for user-visible `.wcad` files.

The logical package must also be directory-compatible so tests, future local
launcher code, and future hosted storage can operate on the same internal
layout without depending on a single file abstraction.

Required V8 package entries:

```text
manifest.json
document.cbor
commands.cbor
```

Reserved package directories:

```text
parts/
assemblies/
brep/
meshes/
edge-display/
thumbnails/
exports/
metadata/
debug/
```

Reserved directories may be absent in a minimal V8 package. Loaders must not
require cache directories to exist.

### 3. Source Of Truth In The Package

For V8, the authoritative package source is:

```text
manifest.json
document.cbor
commands.cbor
```

`document.cbor` contains the current authoritative document source model.
`commands.cbor` contains committed transaction history, redo stack if retained
by the current project model, actor metadata, audit metadata, and semantic
diffs needed for undo/redo, auditability, replay checks, and future macro work.

V8 does not persist OCCT IDs, renderer IDs, mesh IDs, selection-buffer IDs,
geometry-worker timing metrics, viewport camera state, selection state, hover
state, or query output as source truth.

### 4. CBOR Versus JSON

The native package uses CBOR for `document.cbor` and `commands.cbor`, matching
the architecture direction for compact internal storage.

JSON remains supported for:

- explicit `Export JSON`;
- explicit `Import JSON`;
- tests and deterministic fixtures;
- MCP/debug interchange where text readability matters;
- optional package debug mirrors under `debug/`, if a tranche explicitly adds
  them as non-authoritative artifacts.

A package loader must ignore debug mirrors for source reconstruction. Debug
mirrors must not be required for load.

### 5. Project Schema Relationship

V8 package work does not automatically introduce `web-cad.project.v17`.

The current document source can remain equivalent to `web-cad.project.v16` if
V8 only changes the storage envelope and splits document/history into CBOR
entries. A new project schema is required only when V8 adds source data that
cannot be represented by V16, such as:

- project-level materials or named views;
- authored part definitions beyond the current derived default part;
- command-log representation that cannot round-trip current transactions;
- source B-rep checkpoints required to rebuild imported/external exact bodies;
- durable topological references beyond current named generated references.

If V8 adds such data, it must introduce:

```text
schemaVersion: web-cad.project.v17
```

with explicit migrations from all accepted older schemas.

### 6. Manifest Shape

`manifest.json` is small, human-readable, and stable enough for package
inspection without decoding CBOR.

V8 manifest fields:

```ts
WcadManifestV1 {
  packageVersion: "partbench.wcad.v1"
  product: "Partbench"
  createdBy: {
    app: "partbench"
    version?: string
  }
  createdAt: string
  modifiedAt: string
  units: "mm" | "cm" | "m" | "in"
  document: {
    schemaVersion: "web-cad.project.v16" | "web-cad.project.v17"
    path: "document.cbor"
    byteLength: number
    sha256: string
  }
  commands: {
    path: "commands.cbor"
    byteLength: number
    sha256: string
  }
  sourceIdentity: {
    sha256: string
    algorithm: "partbench-source-v1"
  }
  cache?: {
    entriesPath?: "metadata/cache-index.json"
    policy: "optional-rebuildable"
  }
  thumbnail?: {
    path: string
    mimeType: "image/png" | "image/webp"
    byteLength: number
    sha256: string
  }
}
```

Manifest hashes protect package integrity and give storage/export code a stable
source identity. They are not security signatures.

### 7. Source Identity

The V8 source identity is a deterministic hash over:

- package source version;
- document schema version;
- canonical `document.cbor`;
- canonical `commands.cbor`;
- project units;
- any future source B-rep checkpoint entries if they become required source.

The source identity must not include:

- file name;
- File System Access handle identity;
- OPFS path;
- viewport camera;
- selection state;
- hover state;
- derived mesh content;
- thumbnail content;
- export artifact content;
- timing metrics.

Changing only cache files must not change the source identity.

### 8. File System Access Behavior

When File System Access API is available, V8 uses it for:

- `Open .wcad`;
- `Save`;
- `Save As`;
- preserving the current file handle while permission remains valid;
- detecting permission loss and prompting for `Save As` when needed.

When File System Access API is unavailable or permission is denied, V8 falls
back to:

- upload/select `.wcad` file for open;
- download `.wcad` file for save/save-as;
- clear diagnostics explaining that direct save is unavailable.

File handles are browser permission objects, not project data. They must not be
written into `.wcad`, JSON export, transaction history, agent summaries, or MCP
responses. If V8 stores recent file handles in browser-local state, that state
is app preference data only and must degrade cleanly when a handle becomes
invalid.

### 9. OPFS Cache Behavior

OPFS is browser-private cache storage. It is not the project file.

V8 Tranche D1 implements the app-layer cache contract, status, diagnostics, and
clear behavior. V8 Tranche D2 implements the first narrow artifact population
path for `partbench-derived-mesh.v1` derived visualization mesh artifacts. This
is not a broad performance cache.

V8 may store OPFS entries for:

- derived visualization meshes;
- derived exact metadata snapshots;
- thumbnails;
- package unpack cache;
- export preparation intermediates;
- cache indexes keyed by source identity, kernel version, tessellation settings,
  and app/cache version.

V8 must not require OPFS entries to open a project. Missing OPFS data should
cause regeneration, not project-load failure.

OPFS cache keys must include enough versioning to prevent stale reuse:

```text
sourceIdentity
artifactKind
artifactVersion
kernelVersion
geometryWorkerVersion
tessellationSettings
units
```

The app must provide a way to report cache health and clear cache data. Clearing
cache must not mutate the source document or `.wcad` package.

### 10. Package Cache Entries

V8 `.wcad` packages may include rebuildable cache artifacts, but source load
must not depend on them.

Allowed V8 package cache entries:

- `thumbnails/*`
- `meshes/*`
- `edge-display/*`
- `metadata/cache-index.json`
- `exports/*` only if explicitly written as user-requested attached artifacts
  and marked non-source

Cache entries inside a package must be validated against the package
`sourceIdentity`. Stale cache entries are ignored with structured diagnostics.
They must not silently override source-derived rebuild output.

### 11. B-Rep Checkpoints

V8 does not persist B-rep checkpoints as source truth for the existing authored
feature subset unless a tranche explicitly adds source B-rep checkpoint
semantics.

Reason: current supported bodies are rebuilt from authored sketches/features,
and V8 does not implement STEP import. Without imported exact bodies, B-rep
checkpoints are performance/cache artifacts, not required rebuild inputs.

The `brep/` directory is reserved for future source or cache checkpoint work.
If a future V8 tranche makes any B-rep checkpoint source-authoritative, it must
also define:

- checkpoint file format;
- checkpoint source identity;
- kernel version compatibility;
- project schema impact;
- migration behavior;
- failure behavior when the checkpoint cannot load.

### 12. STEP Export Target

V8 targets STEP export before STEP import.

Default exact export target:

```text
STEP AP242
```

If the available OCCT/WASM writer only supports AP203 or AP214 in the current
binding, V8 may expose those as fallback export variants, but the product-level
goal remains "STEP export" and exported metadata must report the actual
application protocol used.

V8 STEP export must:

- use exact geometry through the geometry-kernel/geometry-worker boundary;
- preserve project units;
- preserve body names when the writer supports names;
- export only active commandable bodies that can produce exact shapes;
- reject unsupported bodies with structured diagnostics;
- never export tessellated GLB/STL-style mesh data as fake STEP;
- not mutate source document state;
- not persist export jobs into project source by default.

### 13. STEP Import Deferral

STEP import is deferred beyond the initial V8 release scope.

Reason: high-quality import requires healing, units, colors, names, XDE
metadata, assembly reconstruction, imported-body source records, persistent
exact bodies or checkpoints, and new topology/reference decisions. That is a
separate architecture-risk cluster.

V8 may include UI/agent diagnostics that say STEP import is unsupported. It
must not add a partial import path that creates mesh-only source bodies.

### 14. Export Job Model

V8 export is a job-like operation, but not a source mutation.

The shared readiness contract remains:

```text
project.exportReadiness
```

V8 may add a typed export request through the app/agent/MCP surfaces, such as:

```ts
ProjectExportRequest {
  format: "step" | "glb" | "json" | "wcad"
  scope: "project" | { bodyIds: string[] }
  options?: Record<string, unknown>
}
```

The response should include:

```ts
ProjectExportResult {
  ok: boolean
  artifact?: {
    fileName: string
    mimeType: string
    byteLength: number
    sha256?: string
  }
  diagnostics: StructuredDiagnostic[]
}
```

Artifacts are delivered through browser download/File System Access or adapter
transport. They are not committed CADOps transactions.

### 15. JSON Role In V8

JSON remains required, but demoted from primary product storage.

V8 keeps:

- Export JSON;
- Import JSON;
- current JSON migration tests;
- release fixture JSON round-trips where useful;
- JSON compatibility docs.

V8 product UI should make `.wcad` the default project action once implemented.
JSON controls should be presented as interchange/debug actions rather than the
main save/open workflow.

### 16. UI Decisions

V8 should not reintroduce the Advanced tools bloat removed after V7.

The primary project UI should live in the existing File/Project surface and
show:

- current project name;
- storage mode: unsaved, `.wcad` file, uploaded fallback, JSON fallback;
- dirty/saved state;
- package version;
- document schema version;
- source identity;
- last save result;
- direct-save availability;
- OPFS cache health;
- export readiness;
- structured package/export diagnostics.

The viewport remains unobstructed. Package/export details belong in panels,
dialogs, or bottom status surfaces, not large viewport overlays.

### 17. Agent And MCP Decisions

Agent/MCP wrappers remain thin over CADOps/query/export paths.

V8 may expose:

- project/package summary;
- native package readiness;
- export readiness;
- safe export request for supported formats;
- package validation diagnostics;
- source identity and schema summaries.

V8 must not expose:

- browser file handles;
- local filesystem paths by default;
- OPFS paths;
- arbitrary file write access;
- raw package binary content unless a transport explicitly supports artifact
  return and permissions allow it;
- renderer, mesh, OCCT, cache, or selection-buffer IDs as stable public IDs.

Default agent export behavior should be preview/readiness first. Any operation
that writes a user-visible file should require an explicit caller capability or
human UI action.

### 18. Security And Permissions

V8 is not the production MCP auth release, but local file behavior still needs
safe defaults:

- opening a user-selected file is a user action;
- saving to an existing handle requires valid browser permission;
- failed permissions produce structured diagnostics;
- app-local recent handles are preferences only;
- no command or MCP response should leak local absolute paths by default;
- OPFS cache clearing is allowed because it deletes rebuildable cache, not
  source project data;
- destructive source edits still go through existing CADOps transaction rules.

### 19. Dependency Policy

V8 may need new production dependencies for:

- ZIP read/write;
- CBOR encode/decode;
- hashing if platform APIs are insufficient for the target surface.

Every new dependency must be justified at implementation time with:

- why browser/platform APIs are insufficient;
- package size impact;
- license compatibility;
- maintenance status;
- whether it runs in browser, worker, Node scripts, or all of them;
- why it does not collapse storage, geometry, or UI boundaries.

No dependency should be added for STEP itself unless it is part of the explicit
OCCT/WASM exchange boundary decision.

## Package Validation

V8 package import must validate before loading:

- package container can be read;
- required entries exist;
- no required entry is duplicated;
- `manifest.json` is valid UTF-8 JSON;
- package version is supported;
- manifest paths are relative package paths with no traversal;
- `document.cbor` and `commands.cbor` byte lengths match manifest metadata;
- hashes match when present;
- document schema is supported;
- document and command data validate through cad-core import/replay rules;
- source identity matches reconstructed source data;
- cache entries are optional and stale-safe;
- unknown entries are preserved only if the package writer explicitly supports
  preservation, otherwise they are ignored with diagnostics.

Invalid packages should fail with structured issues, not vague parse errors.

Suggested issue codes:

```text
WCAD_MISSING_MANIFEST
WCAD_INVALID_MANIFEST
WCAD_UNSUPPORTED_PACKAGE_VERSION
WCAD_MISSING_DOCUMENT
WCAD_MISSING_COMMANDS
WCAD_HASH_MISMATCH
WCAD_INVALID_DOCUMENT_CBOR
WCAD_INVALID_COMMANDS_CBOR
WCAD_UNSUPPORTED_DOCUMENT_SCHEMA
WCAD_SOURCE_IDENTITY_MISMATCH
WCAD_STALE_CACHE_ENTRY
WCAD_UNSUPPORTED_ENTRY
WCAD_PERMISSION_DENIED
WCAD_FILE_SYSTEM_ACCESS_UNAVAILABLE
```

## Storage Package Boundary

V8 should add or evolve a storage-focused package rather than putting package
code directly into React components.

Preferred boundary:

```text
packages/storage
  package manifest types
  package read/write
  CBOR encode/decode helpers
  package validation
  source identity hashing
  OPFS cache index helpers
  File System Access abstractions that can be mocked
```

If the existing repo shape makes a separate package too much for the first
tranche, app code may start with focused helpers, but the API must be designed
so storage can move behind a package boundary without changing CADOps or
cad-core authority.

Cad-core should provide import/export of authoritative project snapshots and
validation. It should not know about browser file handles, OPFS, DOM downloads,
or ZIP containers.

## Import/Export Package Boundary

STEP export should live behind the geometry/import-export boundary, not in app
UI code.

Preferred request shape:

```ts
GeometryRequest =
  | { type: "exportStep"; scope: ExportScope; options: StepExportOptions }
```

The app/project surface coordinates the request and file delivery. The geometry
worker produces bytes or a transferable artifact handle plus structured
diagnostics. Cad-core contributes source structure and export readiness, but it
does not become a file writer.

## V8 Release Pillars

### Pillar 1: Native Package

Deliver `.wcad` package v1 with deterministic read/write, validation, source
identity, current project round-trip, and corruption diagnostics.

### Pillar 2: Local Save/Open

Deliver direct local open/save/save-as through File System Access where
available, with download/upload fallback where not available.

### Pillar 3: Rebuildable Cache

Deliver OPFS cache indexing and source-identity-based invalidation for
rebuildable artifacts. The project must load without the cache.

### Pillar 4: Exact Export

Deliver STEP export for supported exact bodies, or a clear kernel-capability
diagnostic if the exchange writer is unavailable in the current OCCT/WASM
boundary. Do not fake exact export from meshes.

### Pillar 5: Product UI

Make `.wcad` the normal project workflow, keep JSON as debug/interchange, and
show package/cache/export status without crowding the viewport.

### Pillar 6: Agent/MCP Surface

Expose package and export state through thin adapter/MCP pass-throughs over the
same query/export contracts used by the UI.

### Pillar 7: Release Confidence

Add deterministic package/export fixtures, focused unit tests, source/derived
separation tests, and browser smokes for local project workflows.

## Release Gate

V8 is release-ready only when:

1. `.wcad` package v1 is documented, typed, tested, and implemented for the
   selected tranche scope.
2. Current V7/V8 representative projects round-trip through `.wcad`.
3. JSON import/export remains available and compatible.
4. Direct local open/save/save-as works where File System Access is available.
5. Fallback upload/download works where File System Access is unavailable.
6. OPFS cache data is optional, rebuildable, and clearable.
7. STEP export either works for the supported exact subset or reports a precise
   kernel-capability blocker. The release must not claim STEP success without a
   real exact writer.
8. Agent/MCP package/export surfaces remain thin over shared query/export
   contracts.
9. No renderer, mesh, OCCT, cache, OPFS, File System Access, or
   selection-buffer IDs become stable public CAD IDs.
10. Existing V7 modeling, reference, selection, Project/File JSON, GLB, and
    agent/MCP behavior does not regress.

## Implementation Guidance

Keep the first implementation tranche small enough to verify. A good first V8
implementation prompt should start with Tranche A, not with File System Access
and STEP at the same time.

The first concrete implementation goal should establish the storage/package
contract:

- manifest types;
- package validation diagnostics;
- source identity helper;
- current project snapshot split into document and commands;
- tests that prove package metadata is source-aware but cache/file handles are
  not source.

Only after that contract is typed and tested should V8 add actual package
read/write, then File System Access UI, then OPFS cache, then STEP export.
