# V10 Planning Record: Editable Feature History And Stable Modeling References

Status: in progress as of 2026-06-17. Tranche A is implemented as the
feature editability and rebuild-readiness contract/query slice. Tranche B is
implemented as the dependency graph and reference-health query slice. Tranche
C1 is implemented as the first authored extrude edit commit/rebuild-effects
slice. Tranche C2 is implemented as the conservative authored revolve, hole,
chamfer, and fillet parameter edit commit slice. Tranche D1 is implemented as
the first rebuild-plan/body-lifecycle contract and semantic-diff slice. Tranche
D2 is implemented as the first transactional source-model rebuild slice for
direct supported source-to-consumer chains. Tranche E1 is implemented as the
first source-semantic generated-reference expansion for authored hole result
bodies. Tranche E2 is implemented as the query-only generated axis reference
slice for source-semantic revolve and hole axes. Tranche F1 is implemented as
the first sketch edit readiness and rebuild-impact query slice. Tranche G1 is
implemented as the first explicit named-reference repair contract and retarget
command slice.

V10 follows the completed V8 Local CAD Foundation And Exact Interop release and
the completed V9 Viewport-Native CAD Interaction release candidate. V8 made
Partbench a local CAD application with `.wcad`, OPFS cache behavior, and exact
STEP export for supported source bodies. V9 made the viewport a direct semantic
interaction surface for the currently defensible body, face, edge, reference,
measure, inspect, and contextual-action workflows.

V10 should make Partbench feel like an editable parametric CAD modeler instead
of a direct-selection prototype with shallow history. The user should be able to
change earlier feature intent, rebuild downstream results, understand which
references survived, and get precise diagnostics when a model cannot update
safely.

The release center is:

```text
source feature intent -> dependency/rebuild graph -> stable semantic reference
health -> editable feature parameters -> downstream rebuild diagnostics ->
viewport/inspector/modeling actions that stay command-ready
```

V10 is intentionally a large modeling-depth release, but it must not become
arbitrary topological naming, production WebGPU, assemblies, STEP import, or a
general sketch-solver rewrite. Its job is to deepen the CAD semantics that V7,
V8, and V9 made visible.

## Product Goal

The normal supported workflow should feel more like real parametric CAD:

```text
create/open .wcad -> create sketches/features -> select bodies/faces/edges in
viewport -> edit earlier feature intent -> rebuild -> see survived, replaced,
or broken references -> repair or continue -> save/export
```

The visible product differences should be:

- feature tree rows feel editable, not just historical;
- supported features expose focused edit controls for their real source
  parameters;
- downstream bodies update after source sketch or feature edits when the
  dependency graph can prove the rebuild safely;
- named/generated references show whether they are active, replaced, stale,
  consumed, ambiguous, or broken;
- viewport contextual actions work on more post-feature geometry, or clearly
  explain why a target is not command-ready;
- rebuild failures are structured CAD diagnostics, not generic UI errors;
- agents and MCP callers can dry-run edits and see the same rebuild/reference
  diagnostics as the UI.

## Why This Is The Next Major Release

V8 and V9 removed two major product bottlenecks: Partbench can now save/open
real local projects and users can interact with the model directly in the
viewport. The next bottleneck is model depth.

The architecture identifies stable entity naming, parametric history, exact
model authority, and sketch solver diagnostics as foundational CAD problems.
Current Partbench has the beginnings of those systems, but the supported stable
reference set is still narrow:

- healthy generated references are limited mostly to simple authored
  rectangle/circle `newBody` extrudes;
- boolean, revolve, hole, chamfer, and fillet result bodies are often
  diagnostic-only or ambiguous for generated references;
- feature editing is not yet a broad first-class workflow across the existing
  authored feature set;
- sketch dimensions and constraints exist, but they are not yet a general
  solver/editing foundation;
- named references can exist, but they need better survival, replacement, and
  repair semantics across edits.

WebGPU, assemblies, and STEP import are important architecture items, but they
become much more valuable after Partbench has stronger source, rebuild, and
reference semantics. V10 should make later renderer, assembly, import, and AI
work easier rather than building a more scalable shell around shallow modeling.

## Architecture Alignment

V10 preserves these decisions from `docs/architecture.md`:

- CADOps remains the center of CAD mutations.
- `cad-core` owns authoritative document state, feature source intent, command
  validation, dependency health, and stable semantic reference decisions.
- Human UI, scripts, tests, agents, and MCP use the same command/query layer.
- Rendered meshes, exact metadata snapshots, topology summaries, viewport hit
  candidates, OPFS cache data, and export artifacts are derived/session data
  unless a tranche explicitly adds source-of-truth data.
- React may present edit controls and route commands, but React does not infer
  topology stability or commandability from renderer or geometry internals.
- Geometry-kernel/worker/OCCT may provide exact rebuilds, generated/modified/
  deleted maps, and geometric signatures through typed boundaries, but raw OCCT
  topology indexes never become public stable CAD IDs.
- Stable references are semantic. When ambiguity remains, commands and queries
  must fail with structured diagnostics instead of silently editing the wrong
  face or edge.
- Sketch solving must report degrees of freedom, under-constrained entities,
  conflicting constraints, and redundant constraints where implemented.
- MCP wraps CADOps. MCP does not become a separate CAD authority.

V10 should explicitly strengthen the architecture's "exact model first, mesh
second" rule. Visualization meshes may help users see the result, but they must
not become the source of feature edits, stable IDs, measurements, or command
eligibility.

## Scope Summary

V10 is in scope for:

- typed feature edit/rebuild request, result, semantic diff, and diagnostic
  protocol shapes;
- source-backed dependency/rebuild graph behavior for current parts, sketches,
  features, bodies, named references, and generated references;
- command/query support for feature editability, dependency health, rebuild
  previews, and reference survival;
- source-semantic stable reference expansion for existing authored feature
  families where defensible:
  - authored rectangle/circle `newBody` extrudes;
  - authored `feature.revolve` where analytic source semantics prove faces,
    edges, or axes;
  - authored `feature.hole` where created hole wall, cap, axis, and circular
    edge references can be safely named from source intent;
  - scoped rectangle-tool add/cut results only where source lineage and
    generated/modified/deleted evidence are unambiguous;
  - chamfer/fillet result references only where source edge lineage and kernel
    evidence prove the replacement safely;
- explicit consumed/replacement body and reference lifecycle semantics;
- feature edit commands for current feature source records where source data
  already exists or where a scoped schema migration adds needed source fields;
- feature suppression/unsuppression and rollback preview only if implemented
  as explicit source state with structured rebuild diagnostics;
- sketch constraint/dimension expansion only where it supports feature editing,
  rebuild reliability, and clear solver diagnostics;
- named-reference health, replacement, repair, and delete behavior;
- viewport/Selection/Inspector/Modeling UI integration for editability,
  rebuild, reference health, and repair workflows;
- agent/MCP dry-run and commit paths for the same edit/rebuild operations;
- release fixtures and browser smokes that prove edit -> rebuild -> reference
  survival or structured failure.

V10 is not in scope for:

- arbitrary stable topology naming for every OCCT result shape;
- production WebGPU renderer replacement;
- production WebGPU selection buffer;
- assemblies, mates, LOD, instancing, or assembly document source;
- STEP import, IGES import/export, or proprietary CAD import/export;
- broad shell, sweep, loft, pattern, mirror, or direct-edit modeling features
  unless explicitly added as a separate scoped tranche after the rebuild
  foundation is healthy;
- storing B-rep checkpoints as source truth unless a tranche explicitly adds a
  source checkpoint format and migration;
- persisting viewport camera, selection, hover, active tool, measurement, or UI
  panel state in `.wcad` or JSON;
- natural-language command entry;
- hosted collaboration;
- production MCP auth/permission systems.

## Major V10 Decisions

### 1. V10 Is A Modeling-Depth Release

V10 should deepen existing CAD behavior before chasing breadth. The release
should improve the edit/rebuild/reference lifecycle for existing sketches,
extrudes, booleans, revolve, hole, chamfer, and fillet before adding many new
feature families.

Adding one narrow new modeling command is acceptable only if it exercises the
same source, rebuild, reference, and diagnostics contract. It should not become
the center of the release.

### 2. Stable References Start From Source Semantics

V10 should not promote OCCT face indexes, edge indexes, mesh IDs, selection
buffer colors, GPU buffer offsets, or renderer object IDs into public CAD
references.

The preferred stable-reference evidence order is:

1. authored source role and feature lineage;
2. explicit generated/modified/deleted maps from a scoped kernel operation;
3. geometric signatures such as surface type, normal, axis, radius, center,
   area, length, and adjacency hints;
4. user/agent named reference intent;
5. structured ambiguity when the above evidence cannot prove identity.

If a result cannot prove identity, it must remain diagnostic-only for
command-ready references.

### 3. Feature Editability Is A Query Before It Is A Control

Before the UI offers edit controls, `cad-core` should answer:

```text
Is this feature editable?
Which source fields are editable?
Which downstream bodies/references depend on it?
What would a dry-run edit change, consume, replace, stale, or break?
Which diagnostics block commit?
```

The UI may format and route this state. It must not independently decide that a
feature field or generated reference is safe to edit.

### 4. Rebuild Is Transactional And Diagnosable

V10 edit commands should run through CADOps transactions and return semantic
diffs. A successful edit should report source edits, rebuilt features, replaced
bodies, survived references, stale references, deleted/consumed references, and
new diagnostics.

A failed edit should leave the document unchanged and return structured
diagnostics. A partially degraded rebuild should be allowed only if the source
state remains internally valid and all degraded references are explicitly
marked non-commandable or repair-needed.

### 5. Consumed And Replacement Bodies Are First-Class

Current target-consuming operations already create result bodies instead of
mutating source bodies in place. V10 should make that lifecycle explicit:

- active source body;
- consumed source body;
- result body;
- replacement body after edit/rebuild;
- stale result body when source dependencies changed but rebuild is not ready;
- failed result body when regeneration failed;
- suppressed feature output where suppression is implemented.

Queries, UI, agents, and MCP should all see the same lifecycle state.

### 6. Named References Need Health And Repair Semantics

Named references are important for humans and agents, but they should not hide
broken topology. V10 should make every named reference report health:

- active;
- resolved to replacement;
- stale;
- consumed;
- ambiguous;
- missing target;
- unsupported target;
- repair-needed;
- deleted.

Repair should be an explicit CADOps command where possible. Silent retargeting
is not acceptable.

### 7. Sketch Solver Expansion Serves Feature Editing

V10 should not attempt a complete professional sketch solver all at once. It
should add solver behavior only where it supports real edit/rebuild workflows.

Good V10 sketch-solver candidates:

- consistent solver status queries for current constraints and dimensions;
- broader dimension edit diagnostics;
- tangent/concentric/equal-radius constraints for circle-and-line workflows if
  implemented with clear conflict/underconstraint reporting;
- point/line/circle reference health after sketch edits;
- dry-run sketch edits that report affected profiles/features.

Deferred unless explicitly scoped:

- spline/ellipse solving;
- dragging solver UX;
- full automatic profile recognition;
- constraint inference from arbitrary imported geometry;
- sketch constraints created from measurement annotations.

### 8. New Source Data Requires A Schema Decision

V10 may introduce `web-cad.project.v17` only when a tranche adds new
source-of-truth data. Examples that likely require V17:

- persisted feature suppression state;
- new source constraint records;
- persisted feature dependency/source-link records that are not derivable;
- explicit reference repair records;
- new editable source fields for existing feature families.

Examples that should not require V17:

- query-only editability/readiness output;
- rebuild diagnostics;
- derived exact metadata snapshots;
- topology summaries;
- OPFS cache entries;
- viewport/session state;
- UI layout state;
- export artifacts;
- browser file handles.

Every V10 tranche must state whether it changes source truth. If it does not,
it must not introduce `web-cad.project.v17`.

### 9. B-Rep Checkpoints Stay Deferred By Default

The architecture allows future B-rep checkpoints as project source. V10 should
not add persistent B-rep checkpoints unless a tranche explicitly justifies the
format, identity rules, kernel-version compatibility, migration behavior, and
source/cache boundary.

The default V10 posture is deterministic rebuild from source feature intent,
with derived exact metadata and OPFS cache remaining rebuildable.

### 10. Viewport Integration Consumes Rebuild Semantics

V9 made the viewport semantic. V10 should keep that pattern:

```text
viewport selection -> semantic selection/reference query -> edit/rebuild
readiness -> command -> semantic diff -> viewport/inspector update
```

The viewport may show survived, stale, consumed, blocked, and repair-needed
states. It must not own feature editability, reference survival, or rebuild
authority.

### 11. Agents Need Dry-Run Rebuilds

Agent and MCP callers should be able to ask:

- what features are editable;
- what references depend on this feature;
- what a proposed edit would change;
- which references would survive or break;
- which diagnostics block the edit;
- what repair options exist.

Wrappers must remain thin over CADOps/query paths.

### 12. Release Samples Should Become Rebuild Samples

The current release samples and browser smokes already prove storage,
selection, viewport, and export workflows. V10 should add deterministic rebuild
samples that exercise:

- simple rectangle/circle extrude edit and reference survival;
- attached sketch on generated planar face followed by parent edit;
- named face/edge reference survival or structured repair-needed status;
- hole/revolve/chamfer/fillet edit where scoped;
- failed edit diagnostics without source mutation;
- `.wcad` round-trip preserving source state and reference health.

## Reference And Topology Model

V10 should distinguish these concepts explicitly.

### Source References

Source references point to authoritative source data:

- part;
- sketch;
- sketch entity;
- sketch point/line/circle target;
- feature;
- body source record;
- named reference record.

These may be public stable IDs when they are durable source IDs.

### Generated References

Generated references point to entities created by a feature from source intent:

- generated planar face;
- generated cylindrical face;
- generated cap face;
- generated linear edge;
- generated circular edge;
- generated axis;
- generated vertex where scoped later.

They are stable only when source role, feature lineage, and optional exact
evidence prove the identity.

### Result References

Result references describe entities on bodies created by operations that consume
or transform prior bodies:

- boolean add/cut result entities;
- hole result entities;
- revolve result entities;
- chamfer/fillet replacement entities.

V10 should support result references only where the operation can provide
defensible lineage and signatures. Otherwise the result body should remain
queryable but non-commandable for generated reference workflows.

### Replacement References

Replacement references map an old stable target to a new stable target after
edit/rebuild. They require explicit evidence. A replacement map may say:

- survived unchanged;
- replaced by new stable generated reference;
- consumed by feature;
- deleted by edit;
- ambiguous;
- stale because source changed;
- unsupported because no safe identity exists.

### Named References

Named references are user/agent labels over source or generated targets. A named
reference does not make an unstable target stable. It records intent and should
report health against the current resolved target.

## Rebuild Model

V10 should introduce or refine a rebuild model with these stages:

1. Validate source edit request.
2. Compute dependency impact.
3. Run dry-run rebuild for affected features where possible.
4. Compare generated references before/after.
5. Produce semantic diff and diagnostics.
6. Commit source edit and result updates transactionally.
7. Mark derived geometry/exact metadata/cache stale for regeneration.
8. Update UI/agent/MCP query surfaces from authoritative state.

The rebuild result should be structured:

```ts
FeatureRebuildResult {
  status:
    | "ready"
    | "noChange"
    | "partial"
    | "blocked"
    | "failed"
  editedFeatureIds: FeatureId[]
  rebuiltFeatureIds: FeatureId[]
  affectedBodyIds: BodyId[]
  referenceChanges: ReferenceChange[]
  diagnostics: RebuildDiagnostic[]
}
```

The exact shape can follow existing protocol conventions, but the concepts must
be typed and tested.

## Diagnostic Model

V10 should use structured diagnostic codes instead of vague strings.

Likely diagnostic families:

- feature not editable;
- field not editable;
- invalid edit value;
- dependency missing;
- dependency stale;
- dependency suppressed;
- consumed target;
- replacement ambiguous;
- generated reference unsupported;
- generated reference stale;
- generated reference consumed;
- named reference unresolved;
- repair required;
- sketch overconstrained;
- sketch underconstrained;
- sketch solve failed;
- profile invalid;
- profile not closed;
- rebuild failed;
- exact kernel unavailable;
- exact kernel result ambiguous;
- derived geometry stale;
- unsupported result topology;
- command would mutate source during dry-run;
- source identity mismatch.

Diagnostics should include enough structured context for UI, agents, MCP, and
tests to identify the blocking target without exposing raw OCCT, mesh,
renderer, selection-buffer, OPFS, or file-handle IDs.

## UI Model

V10 should improve product flow without adding viewport clutter.

### Feature Tree

The feature tree should expose:

- editable feature status;
- rebuild health;
- suppressed/active/failed state where implemented;
- dependency impact summary;
- selected feature edit entry point;
- structured diagnostics;
- references owned by or affected by the feature.

The feature tree should not become a full-screen debug console.

### Selection Tab And Inspector

Selection and Inspector should show:

- command-ready reference state;
- reference health after rebuild;
- repair-needed state;
- editability of selected feature/body/reference context;
- concise diagnostics with expandable detail.

### Modeling Workflow

Modeling panels should prefer source-backed edit commands over ad hoc UI state.
For example, editing a hole diameter or extrude depth should call a typed CADOps
feature edit command, not locally mutate React state and then infer geometry.

### Viewport

Viewport should stay direct and compact:

- selected/stale/consumed/repair-needed visual states;
- contextual "Edit feature" where query-proven;
- "Repair reference" where query-proven;
- no large diagnostic overlays blocking the model;
- no renderer-owned commandability.

## Agent And MCP Model

V10 should expose the same modeling-depth semantics to agents:

- `project.rebuildReadiness` or equivalent;
- `feature.editability` or equivalent;
- `feature.dependencies` or equivalent;
- `reference.health` or equivalent;
- dry-run feature edit;
- commit feature edit;
- reference repair command where scoped.

Adapters must stay thin. They should not implement their own dependency graph,
topology naming, or rebuild logic.

## Proposed Tranche Sequence

The tranche names are intentionally sequential but flexible. Each tranche should
be useful on its own, reviewed, tested, browser-smoked when UI changes, and
committed before moving on.

### Implemented Tranche A: Feature Edit And Rebuild Contract

Goal: add typed protocol/core support for feature editability, rebuild
readiness, reference-change diagnostics, and dry-run result shapes.

Implemented:

- `feature.editability` query types for editable field descriptors, proposed
  edit dry-run status, rebuild readiness, affected source/reference summaries,
  reference-change categories, and structured diagnostics;
- cad-core query behavior derived from authoritative document state and
  semantic generated/named references;
- supported editable status for active authored rectangle/circle `newBody`
  extrude depth/side via the existing `feature.updateExtrude` command path;
- structured blocked/unsupported/missing diagnostics for consumed bodies,
  primitive compatibility features, boolean-result topology ambiguity, and
  deferred revolve/hole/chamfer/fillet edit/rebuild semantics;
- non-mutating dry-run validation for proposed extrude edits;
- thin agent-adapter and MCP pass-through through the same CADOps query path;
- tests for protocol/query behavior, dry-run non-mutation, consumed/ambiguous/
  unsupported/missing diagnostics, reference-change summaries, source/derived/
  session separation, agent pass-through, and MCP pass-through.

Limitations:

- Tranche A does not add broad feature editing UI.
- Tranche A does not add new commit commands beyond the existing
  `feature.updateExtrude` path.
- Tranche A does not implement full transactional rebuild, named-reference
  repair, or stable topology for arbitrary boolean, revolve, hole, chamfer, or
  fillet outputs.
- Tranche A does not introduce `web-cad.project.v17` because it adds query-only
  derived contract data, not new source-of-truth document data.

### Implemented Tranche B: Dependency Graph And Reference Health Queries

Goal: make current feature/body/reference dependencies queryable and testable.

Implemented:

- `project.dependencyGraph` query types for source-derived graph nodes and
  edges covering sketches, sketch entities, features, bodies, generated
  references, named references, source edges, target edges, consumed-body
  edges, generated-reference edges, and named-reference edges;
- `reference.health` query types and cad-core behavior for all references or a
  focused body, generated reference, or named reference target;
- shared V10 reference-health categories aligned with Tranche A reference
  changes: active, replaced, stale, consumed, ambiguous, missing, unsupported,
  repair-needed, and deleted;
- cad-core graph/health derivation from authoritative document state only:
  current sketches/entities, authored feature records, generated body/reference
  semantics, named references, consumed target bodies, and existing source
  topology support boundaries;
- structured diagnostics for missing targets, stale generated/named references,
  consumed bodies, ambiguous boolean result topology, unsupported primitive
  bodies, and repair-needed deferred result references;
- thin agent-adapter and MCP pass-through for both new queries;
- tests for protocol/query behavior, rectangle/circle extrude source chains,
  consumed and ambiguous boolean add/cut topology, named-reference active/
  stale/missing/unsupported states, Tranche A editability consistency,
  source/derived/session separation, and agent/MCP pass-through.

Limitations:

- Tranche B is query-only. It does not implement transactional rebuild, feature
  edit commands, reference repair commands, broad UI health display, or
  generated result-reference expansion.
- Boolean, revolve, hole, chamfer, and fillet result bodies remain diagnostic
  or repair-needed unless existing source semantics can prove command-ready
  generated references.
- Renderer, mesh, OCCT, GPU, selection-buffer, OPFS, file-handle, export, and
  viewport/session identifiers remain outside public dependency/reference IDs.
- Tranche B does not introduce `web-cad.project.v17` because it adds derived
  query results, not new source-of-truth document data.

### Implemented Tranche C1: Authored Extrude Edit Commit And Rebuild Effects

Goal: make the first real V10 feature-edit commit path agree with Tranche A
editability and Tranche B dependency/reference-health semantics.

Implemented:

- `feature.updateExtrude` remains the CADOps commit path for supported authored
  rectangle/circle `newBody` extrude depth and side edits;
- dry-run reports modified feature/body IDs and does not mutate document state;
- commit mutates authoritative feature source state and marks the rebuilt body
  modified through the semantic diff;
- feature semantic diffs now include V10 `referenceEffects` for supported
  extrude edits, using the same active/stale/consumed/ambiguous/missing/
  unsupported/repair-needed/deleted category vocabulary as Tranche A/B;
- generated and named references on safe edited `newBody` extrudes remain
  active where source semantics prove stable generated reference roles;
- `feature.updateExtrude` now rejects boolean add/cut result extrudes instead
  of implying their result topology is command-ready;
- consumed target-body edits remain blocked and leave document source and
  reference health unchanged;
- transaction history preserves the new reference effects for UI, agent, and
  MCP callers through existing CADOps surfaces.

Limitations:

- C1 does not add broad feature editing UI.
- C1 does not edit revolve, hole, chamfer, fillet, or boolean result features.
- C1 does not implement full downstream transactional rebuild chains,
  named-reference repair commands, or arbitrary stable topology naming.
- C1 does not introduce `web-cad.project.v17` because it uses existing source
  fields and adds derived/audit semantic diff data only.

### Implemented Tranche C2: More Authored Feature Edit Commit Paths

Goal: make additional existing authored feature source records editable through
CADOps where current source semantics support it.

Implemented:

- added explicit CADOps command shapes for `feature.updateRevolve`,
  `feature.updateHole`, `feature.updateChamfer`, and `feature.updateFillet`;
- `feature.updateRevolve` supports authored `newBody` revolve angle edits;
- `feature.updateHole` supports authored hole `depthMode`, `depth`, and
  `direction` edits, including safe `blind`/`throughAll` validation;
- `feature.updateChamfer` supports authored chamfer `distance` edits when the
  original generated-edge target remains command-ready for that feature's own
  source edit;
- `feature.updateFillet` supports authored fillet `radius` edits under the same
  conservative source-edge rule;
- dry-run paths report modified feature/body IDs without mutating source;
- commit paths update authoritative feature source records through CADOps,
  produce feature/body semantic diffs, and preserve transaction-history
  summaries;
- `feature.editability` now exposes the C2 editable fields and proposed-edit
  dry-runs using the same command names and diagnostics as the commit paths;
- reference effects use the Tranche A/B/C1 vocabulary: supported edge-finish
  source references are active for their own parameter edits, consumed target
  references remain consumed, and revolve/hole/chamfer/fillet result-body
  generated topology remains repair-needed until a later stable-reference
  tranche proves stronger identity. E1 later proves a narrow authored hole
  result-reference subset;
- agent and MCP callers pass the new commands through existing `cad.batch`
  paths without gaining separate CAD authority.

Limitations:

- C2 does not add broad feature editing UI.
- C2 does not edit revolve axis/profile linkage, hole placement/circle source,
  chamfer/fillet target references, or boolean result feature parameters.
- C2 does not implement full downstream transactional rebuild chains,
  named-reference repair commands, or arbitrary stable topology naming.
- C2 does not introduce `web-cad.project.v17` because it uses existing feature
  source fields and derived/audit semantic diff data only.

### Implemented Tranche D1: Rebuild Plan And Body Lifecycle Contract

Goal: make body lifecycle and rebuild-plan state queryable and consistent
before adding broad rebuild execution.

Implemented:

- `project.rebuildPlan` protocol/core query types for active, source, consumed,
  result, modified, replacement, stale, failed, suppressed/deferred,
  repair-needed, unsupported, and derived-rebuild-pending body lifecycle state;
- rebuild affected summaries for sketches, sketch entities, features, bodies,
  generated references, named references, and derived artifact classes;
- structured rebuild diagnostics for blocked, stale, failed, unsupported,
  ambiguous, consumed, repair-needed, and derived-rebuild-pending states;
- cad-core rebuild plan derivation from authoritative project structure,
  existing dependency graph/reference-health results, consumed target bodies,
  result bodies, and latest V10 edit semantic diffs;
- lifecycle effects on `feature.updateExtrude`, `feature.updateRevolve`,
  `feature.updateHole`, `feature.updateChamfer`, and `feature.updateFillet`
  semantic diffs so history/UI/agents/MCP can see modified bodies and derived
  rebuild/readiness status;
- safe authored rectangle/circle `newBody` extrude edits mark the source body
  modified and derived-rebuild-pending while keeping generated/named reference
  health active where source semantics prove it;
- revolve/hole/chamfer/fillet edits mark result bodies modified,
  derived-rebuild-pending, and repair-needed/ambiguous; consumed target bodies
  remain explicit consumed lifecycle entries;
- non-mutating rebuild-plan query behavior and failed edit tests proving source,
  dependency graph, reference health, and JSON project source remain unchanged;
- thin agent-adapter and MCP pass-through through `project.rebuildPlan` and
  `cad.project_rebuild_plan`;
- tests for protocol/query shapes, active extrude lifecycle, consumed target
  plus repair-needed result chains, C1/C2 lifecycle effects after commits,
  failed/unsupported non-mutation, transaction history preservation, source/
  derived/session separation, and agent/MCP pass-through.

Limitations:

- D1 does not execute arbitrary downstream rebuild chains.
- D1 does not add broad UI rebuild controls.
- D1 does not expand stable topology for revolve, hole, chamfer, fillet, or
  boolean result faces/edges.
- D1 does not add named-reference repair commands.
- D1 does not introduce `web-cad.project.v17` because lifecycle/query results
  and semantic diff effects are derived/audit data, not new source-of-truth
  document fields.

### Implemented Tranche D2: Transactional Source-Model Rebuild For Scoped Chains

Goal: after supported source edits, execute the scoped source-model rebuild
transaction that the current document model can prove safely. This is not a
persistent B-rep or arbitrary topology replay; current Partbench source records
are authoritative, so the D2 execution slice validates and commits the source
edit, revalidates the direct downstream consumer from existing source records,
marks the affected result body, and leaves generated result topology
repair-needed unless a later stable-reference tranche proves stronger identity.
Tranche E1 later proved a narrow hole-result subset; the D2 limitation remains
active for cut/add, revolve, chamfer, and fillet result topology.

Implemented:

- `feature.updateExtrude` can now edit a supported rectangle/circle `newBody`
  source extrude even when that source body is consumed by one direct supported
  downstream feature whose existing source records can be revalidated in the
  same source-model transaction;
- the scoped downstream feature set is deliberately narrow: supported cut/add
  extrude chains, holes, chamfers, and fillets that already passed existing
  source/reference validation and whose result body is not itself consumed by
  another feature;
- successful scoped source edits keep existing source records and body IDs,
  mark the source target as consumed/source/modified, mark the direct
  downstream result as modified/replacement, and keep result topology
  repair-needed rather than claiming command-ready generated result references
  except for the later E1-supported authored hole result references;
- reference effects now distinguish the edited source feature
  (`sourceFeatureId`) from the revalidated downstream feature
  (`targetFeatureId`): consumed source references remain non-command-ready, and
  downstream result references are reported as replaced/repair-needed except
  for the later E1-supported authored hole result references;
- `project.rebuildPlan` now derives current lifecycle effects from the current
  transaction history instead of only the latest transaction, so pending or
  replacement state survives unrelated commits and follows undo/redo;
- failed scoped source-model rebuild leaves document source, transaction
  history, source identity, reference health, and rebuild plan unchanged, with
  structured diagnostics such as missing named-reference failures;
- `feature.editability`, `project.dependencyGraph`, `reference.health`, and
  `project.rebuildPlan` remain consistent before and after D2 commits.

Limitations:

- D2 does not execute arbitrary topology, persistent B-rep replay, or multi-hop
  rebuild graphs.
- D2 itself did not make downstream result faces/edges command-ready; E1 later
  adds a narrow authored hole result-reference subset.
- D2 does not add named-reference repair commands.
- D2 does not add UI, renderer, mesh, OCCT, OPFS, STEP, WebGPU, assembly, or
  storage behavior.
- D2 does not introduce `web-cad.project.v17` because it adds no new
  source-of-truth document data.

### Implemented Tranche E1: Source-Semantic Hole Generated References

Goal: add the first stable-reference expansion for result features without
claiming arbitrary topology naming.

Implemented:

- generated-reference protocol roles now distinguish extrude and hole roles
  instead of pretending hole result references are extrude faces or edges;
- authored `feature.hole` result bodies expose source-semantic generated
  references for the result body, cylindrical `holeWall` face, `startRim`
  circular edge;
- stable IDs are deterministic from public semantic source data: result body ID
  plus the hole role (`generated:face:<body>:holeWall`,
  `generated:edge:<body>:startRim`);
- geometric signatures come from authoritative source state: target body ID,
  source sketch/circle ID through the reference source fields, circle
  center/radius, sketch plane, sketch plane normal, hole direction axis,
  depth mode, and blind depth where present;
- `body.generatedReferences`, `body.resolveGeneratedReference`,
  `selection.referenceCandidates`, named-reference resolution,
  `reference.health`, `project.dependencyGraph`, `feature.editability`, and
  `project.rebuildPlan` now treat supported hole result references as active
  and commandable for selection/naming;
- consumed source target bodies remain consumed, and existing add/cut,
  revolve, chamfer, and fillet result topology remains diagnostic-only or
  repair-needed unless a later tranche proves stronger identity;
- source-analytic measurement and viewport refinement helpers explicitly guard
  their existing extrude-only math instead of interpreting hole roles as
  extrude depth/side topology;
- tests cover hole reference shape/count/stable IDs/signatures, resolve/select/
  health/named-reference behavior, source edit and delete/undo/redo behavior,
  public-ID hygiene, unchanged JSON/`.wcad` source schema, and unsupported
  result topology staying diagnostic-only.

Limitations:

- E1 does not add a generated axis reference because the current generated
  entity kind vocabulary has body, face, edge, and vertex but no axis kind.
- E1 does not make hole references eligible for attach-sketch, measurement,
  chamfer, or fillet commands; they are commandable for selection and naming.
- E1 exposes the deterministic start rim only; blind terminal rims and
  through-all exit rims stay deferred because target-body intersection topology
  is not proven from source circle/depth/direction alone.
- E1 does not add UI, new modeling commands, arbitrary topology naming, OCCT/
  mesh/renderer authority, storage behavior, or `web-cad.project.v17`.

### Implemented Tranche E2: Source-Semantic Axis References For Revolve And Hole

Goal: add explicit query-only generated axis references where authored source
semantics prove identity, without claiming stable result topology.

Implemented:

  `generated:axis:<bodyId>:holeAxis`;

## Validation Strategy

Every V10 tranche should include focused tests. Use browser smokes only when the
visible product surface changes.

Required validation categories:

- protocol shape tests;
- cad-core query/command tests;
- semantic diff tests;
- source/derived/session separation tests;
- import/export and `.wcad` round-trip tests where source data changes;
- agent/MCP pass-through tests where surfaces are added;
- UI helper/component tests where presentation changes;
- browser smoke for user-visible edit/rebuild workflows;
- regression tests proving unsupported/ambiguous topology remains diagnostic,
  not silently commandable.

The full release validation should run:

```sh
pnpm test
pnpm typecheck
pnpm lint
pnpm format:check
pnpm build
pnpm smoke:v7-release-samples
pnpm smoke:v9-viewport-workflow
```

If V10 adds a new smoke alias, it should cover the edit/rebuild workflow and may
reuse the existing compatibility browser workflow runner.

## Expected End State

At the end of V10, Partbench should still not be a complete CAD system. It
should, however, feel materially deeper:

- existing features are editable through source-backed commands;
- supported edits rebuild downstream bodies transactionally;
- reference health is explicit across feature edits;
- more post-feature geometry is command-ready where defensible;
- unsupported topology remains honestly diagnostic;
- named references can be repaired instead of silently drifting;
- agents can dry-run meaningful modeling edits;
- `.wcad`, JSON compatibility, OPFS cache, STEP export, viewport selection, and
  V9 interaction workflows continue to work.
