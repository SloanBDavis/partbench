# V9 Release Plan: Viewport-Native CAD Interaction

Status: active major-release plan as of 2026-06-15. Tranche A viewport
interaction contract/helpers, Tranche B body picking/selection routing,
Tranche C1 generated planar face picking/reference routing, and Tranche C2
generated edge picking/reference routing are implemented. Tranche D1 semantic
viewport visual-state inputs and Tranche E1 compact contextual command surface
are implemented. Tranche F1 single-target Measure/Inspect, Tranche F2
session-only two-target measurement, and Tranche G1 navigation/camera
ergonomics are implemented. Tranche H1 release smoke matrix expansion is
implemented. Tranche H2 responsive UX QA and diagnostic polish is implemented.

V9 follows the V8 Local CAD Foundation And Exact Interop release. V8 made
Partbench feel like a real local CAD application by adding `.wcad` project
files, File System Access open/save/save-as, OPFS-derived mesh cache behavior,
exact STEP export for supported bodies, agent/MCP package/export surfaces, and
release-grade smokes.

V9 should make Partbench feel like a direct CAD application instead of a panel
or tree-driven CAD debugger. The user should be able to point at real model
things in the viewport, understand exactly what they are selecting, and launch
valid commands from that selection without losing the architecture boundaries
that make future WebGPU, assemblies, STEP import, and broader topology work
possible.

The release center is:

```text
renderer-agnostic hit candidates -> semantic selection resolution ->
command-ready viewport selection -> compact contextual tools ->
exact/semantic inspect and measure -> release-grade interaction smokes
```

V9 is intentionally a large UI/interaction release, but it must not become a
renderer rewrite. The key decision is that viewport-native interaction belongs
above the renderer as typed semantic contracts. The current canvas renderer,
future WebGPU renderer, and future assembly renderer should all be able to feed
the same app/CADOps selection and command pipeline.

## Product Goal

The normal supported workflow should feel direct:

```text
create/open .wcad -> orbit/pan/zoom -> hover real bodies/faces/edges ->
click to select -> see concise command-ready state -> run valid command ->
measure or inspect exact/semantic data -> save/export with no viewport clutter
```

A user should not have to hunt through the feature tree for every action. The
tree remains important for structure, history, and recovery, but the viewport
becomes the primary interaction surface for supported geometry.

The product should feel like a focused CAD tool:

- select bodies, generated faces, generated edges, named references, and active
  sketch entities directly from the viewport where the current source semantics
  can support it;
- show hover/preselection, selected, command-target, consumed/stale/unsupported,
  and pending/failed derived-geometry states clearly;
- keep detailed diagnostics in the existing left selection/inspector surfaces,
  not as large overlays that cover the model;
- expose compact contextual actions only when they are command-ready;
- let users measure distance, angle, length, area, radius/diameter, and bounding
  values where exact/semantic data exists;
- make camera/orientation controls, fit, zoom, and active tool state reliable
  enough for daily modeling.

## Why This Does Not Need A Rewrite Later

V9 must be built around stable contracts, not around today's canvas details.

The renderer may produce hit candidates, display highlights, and camera state.
It must not decide whether a selected thing is stable, commandable, stale,
consumed, ambiguous, or a valid command target. Those decisions stay in
`cad-core` queries and CADOps command validation.

The app may hold transient UI state such as hover, active tool, selection set,
and drag gesture state. It must not persist that state in `.wcad`, project JSON,
OPFS cache, command history, agent summaries, or MCP responses unless a future
tranche explicitly adds source-of-truth data.

The future WebGPU renderer should replace only the rendering and hit-candidate
implementation. The semantic selection contract should remain:

```text
screen input -> renderer hit candidates -> semantic resolver query ->
selection.referenceCandidates -> command/action availability
```

The future assembly renderer should extend candidate context with an instance
path instead of replacing the interaction model:

```ts
ViewportHitCandidate {
  rendererHitId: string          // private/ephemeral
  displayEntityKind: "body" | "face" | "edge" | "vertex" | "sketchEntity"
  semanticHint?: CadSelectionReferenceInput
  instancePath?: readonly string[] // reserved for future assemblies
}
```

`rendererHitId`, mesh triangle IDs, OCCT indexes, selection-buffer colors, GPU
buffer offsets, and pixel IDs are private implementation details. They can be
used to route a current gesture, but they are not public stable CAD IDs.

## Architecture Alignment

V9 preserves these architecture decisions from `docs/architecture.md`:

- CADOps remains the center of CAD mutations.
- `cad-core` owns authoritative document state and command validation.
- Human UI, scripts, tests, agents, and MCP use the same command/query layer.
- Rendered meshes, exact metadata, topology summaries, hit buffers, hover
  state, selection state, and viewport state are derived/session data unless a
  tranche explicitly adds source data.
- React UI does not import geometry internals directly.
- Renderer, command engine, protocol, storage, geometry-worker, and MCP
  boundaries stay separate.
- Selection is typed and semantic. Face/edge/body selections must resolve to
  command-ready references before driving commands.
- Future large-assembly rendering needs culling, LOD, instancing, and ID buffer
  picking. V9 should define contracts compatible with that future without
  implementing it all now.
- Exact measurements and commandability must use semantic document state or the
  exact geometry boundary, not visualization meshes.

V9 deliberately does not contradict the architecture's WebGPU direction. It
prepares for it by making the renderer boundary explicit.

## Scope Summary

V9 is in scope for:

- typed viewport interaction protocol shapes for hit candidates, hover,
  selection set state, command targets, and measurement targets;
- a renderer/app boundary that lets the current canvas renderer produce
  private hit candidates without leaking renderer IDs into public CAD refs;
- body picking for currently visible supported bodies;
- generated face/edge picking for already defensible generated references from
  supported rectangle/circle `newBody` extrude bodies and existing V7/V8
  reference semantics;
- named-reference selection routing when a named reference resolves to a
  generated reference;
- semantic resolution through `selection.referenceCandidates` or focused
  cad-core query additions;
- hover and preselection visual state for supported entities;
- selected, command-target, consumed, stale, unsupported, ambiguous,
  non-commandable, pending, and failed visual/status states;
- compact contextual actions for existing supported commands, such as create
  sketch on generated planar face, name generated reference, chamfer/fillet
  supported edges, active sketch/entity commands, and exact STEP/export links
  where appropriate;
- viewport-native measurement/inspect tools over exact or semantic data where
  available, with approximate display-only measurements clearly marked if used;
- multi-select or selection sets only where current command contracts can
  consume them safely;
- camera/orientation/navigation controls and fit-to-selection behavior;
- keyboard/mouse interaction ergonomics for CAD basics: orbit, pan, zoom, fit,
  select, additive selection, cancel, and repeat last valid command where
  practical;
- focused accessibility and discoverability work for toolbar/tool state,
  inspector routing, and keyboard users;
- release smokes that exercise viewport pick -> semantic selection -> command
  action -> diagnostics -> save/export without console errors.

V9 is not in scope for:

- production WebGPU renderer replacement;
- large-assembly renderer, instancing, mates, LOD, or assembly document source;
- STEP import;
- broad arbitrary topological naming for every OCCT result shape;
- persistent selection or hover state;
- persistent view states, named cameras, or UI preferences in `.wcad`;
- broad sketch solver expansion or solver drag UX beyond interaction plumbing
  needed for current commands;
- new modeling commands such as shell, sweep, loft, mirror, patterns, direct
  edits, or broad booleans;
- storing mesh/selection-buffer/OCCT/renderer IDs as public stable IDs;
- `web-cad.project.v17` unless a tranche explicitly adds source-of-truth
  document data;
- production MCP auth, hosted collaboration, or natural-language command entry.

## Current Bottlenecks V9 Relieves

### Tree-First Modeling

V7 and V8 made the feature tree, selection tab, inspector, and Project/File
workflow coherent. The tradeoff is that many supported modeling actions still
begin in panels. V9 should let the viewport start those actions directly while
the panels remain structured support surfaces.

### Selection Is Semantic But Not Yet Physical

`selection.referenceCandidates` gives Partbench a shared command-ready reference
contract, but the current product mostly asks that question for tree and
inspector selections. V9 connects physical picking to the same semantic
contract.

### Viewport Status Is Too Thin For CAD Work

Users need to know what is hoverable, selected, commandable, stale, consumed, or
unsupported without reading debug walls. V9 creates a coherent viewport status
language backed by structured diagnostics.

### Measurements Are Not A First-Class Interaction

CAD users constantly inspect length, area, angle, radius, and distance. V9
should add a measurement tool surface that uses exact/semantic data where the
architecture supports it and honestly marks deferred/approximate paths.

### Future WebGPU Needs A Contract

The current renderer should not be overfit. V9 defines the interaction
interfaces WebGPU will implement later: visible entity inputs, hit candidates,
highlight layers, selection styles, and camera/tool events.

## Major V9 Decisions

### 1. Viewport Interaction Is A Protocol Surface

V9 should add typed protocol/app-facing shapes for viewport interaction where
they need to cross packages. The exact file/package placement can follow
existing patterns, but the concepts are stable:

```ts
ViewportPointerEventInput
ViewportHitCandidate
ViewportSelectionIntent
ViewportSelectionState
ViewportCommandTarget
ViewportMeasurementTarget
ViewportInteractionDiagnostic
```

These types describe intent and semantic candidates. They do not expose mesh
triangle indexes, OCCT topology indexes, GPU buffer offsets, selection-buffer
colors, or renderer object internals.

### 2. Hit Candidates Are Not CAD Authority

A renderer hit candidate can say:

```text
"The pointer hit display face candidate X for body Y."
```

It cannot say:

```text
"This is a stable CAD face reference and chamfer is allowed."
```

The app must resolve hit candidates through cad-core query behavior, usually by
turning a semantic hint into `selection.referenceCandidates`. Unsupported or
ambiguous candidates must return structured diagnostics.

### 3. V9 Selection State Is Session State

Hover, selected viewport entity, selected measurement targets, additive
selection sets, active tool, and drag gestures are app/session state. They are
not source data and are not stored in `.wcad`, JSON, command history, OPFS, MCP,
or agent responses.

No `web-cad.project.v17` is required for V9 unless a tranche adds source data
such as persisted named views or durable assembly instance references.

### 4. Use Existing Semantic References First

V9 should support the current defensible set before broadening topology:

- active source body references for authored rectangle/circle `newBody`
  extrudes;
- generated planar faces and generated edges already exposed through V7/V8
  semantic reference behavior;
- named references that resolve to those generated references;
- consumed/stale/unsupported/ambiguous diagnostics for everything else.

Boolean, revolve, hole, chamfer, and fillet result bodies should remain
unsupported or diagnostic-only unless their existing source semantics prove the
selection safely.

### 5. Context Actions Come From Commandability

Viewport context actions must be derived from query/command readiness, not from
hardcoded UI guesses. If `selection.referenceCandidates` says a target supports
`createSketchOnFace`, the contextual UI may offer that action. If not, it must
show no action or a structured diagnostic.

This keeps the viewport, tree, inspector, agents, and MCP aligned.

### 6. Measurements Must Declare Their Authority

Measurement results should report source:

```text
semantic document value
source analytic exact value
OCCT exact metadata value
display approximation
unsupported
```

Display approximations may be useful for rough feedback, but they must not be
presented as exact CAD measurements. Exact/semantic measurements should be
preferred wherever current cad-core or geometry-worker data can provide them.

### 7. Visual States Need A Small Vocabulary

V9 should use a restrained visual vocabulary:

- hover/preselection;
- selected;
- command target;
- additive selection member;
- consumed/stale/unsupported warning;
- derived geometry pending;
- derived geometry failed;
- hidden/suppressed/deferred where current data supports it.

No large viewport panels. Detailed information belongs in the Selection tab,
inspector, modeling workflow, or compact drawers.

### 8. Viewport Tools Are Modeful But Cancelable

V9 should make active tools visible and easy to exit:

- Select;
- Inspect;
- Measure;
- Sketch/Create on face where supported;
- Edge finish target selection where supported;
- Name reference where supported.

`Escape` should cancel transient tool state. Switching tree/selection/inspector
targets should update semantic selection without surprising panel changes.

### 9. Multi-Select Is Narrow And Command-Driven

V9 may add transient multi-select only for command workflows that already have
clear input semantics. It should not invent broad persistent selection sets or
assembly selection behavior before the document model supports them.

### 10. Future Assembly Compatibility Is Reserved

Viewport hit candidates and selection states should reserve an optional
`instancePath`/`assemblyPath` concept, but V9 should not implement assemblies.
Current single-part selections can leave this absent.

This lets future V10/V11 assembly work add instance context without replacing
the V9 interaction contract.

### 11. WebGPU Compatibility Is Required, WebGPU Implementation Is Not

The current canvas renderer can implement V9 picking with CPU-side projected
bounds, current render-scene metadata, or narrow hit maps. The future WebGPU
renderer can implement the same contract using ID buffers, BVHs, and exact
raycast refinement.

V9 should not couple React or cad-core to either strategy.

### 12. Browser Smokes Become Product Interaction Smokes

The V9 release smoke should cover actual user paths:

```text
create body -> pick body in viewport -> pick face -> create sketch on face ->
name reference -> pick edge -> inspect/diagnose commandability ->
measure -> save .wcad -> reload -> verify selection contracts still work
```

Smokes should verify no console errors and no viewport-obstructing panels.

## Proposed Tranche Sequence

### Tranche A: Viewport Interaction Contract

Status: implemented as the first V9 contract/helper slice.

Goal: define the renderer-agnostic interaction types and app helper state.

Scope:

- typed hit-candidate, pointer-intent, hover, selection, command-target, and
  measurement-target shapes;
- helper functions that map viewport intents to current semantic selection
  inputs without leaking renderer IDs;
- no picking implementation yet beyond current selection plumbing;
- tests proving source/session/renderer separation.

Definition of done:

- V9 interaction contracts are typed and tested;
- no `.wcad`, JSON, OPFS, agent, MCP, renderer, mesh, OCCT, or
  selection-buffer IDs become public stable refs;
- docs name the first supported target set and deferred cases.

Implemented boundaries:

- protocol shapes now cover pointer intent, private hit candidates,
  hover/selection state, command-target summaries, measurement targets, and
  structured viewport interaction diagnostics;
- app helpers map supported body, generated-reference, and named-reference
  semantic hints into `selection.referenceCandidates` query inputs;
- unsupported renderer-only, missing, stale, ambiguous, non-commandable,
  unsupported display-entity, and future assembly-instance cases remain
  structured diagnostics;
- full viewport body/face/edge picking is deferred to Tranches B and C.

### Tranche B: Body Picking And Selection Routing

Status: implemented as the first direct viewport-selection slice.

Goal: let users select supported visible bodies directly from the viewport.

Scope:

- current renderer/app hit candidate path for body-level picking, including
  object-backed body targets where the mapping is unambiguous;
- semantic resolution through the V9 helper path and
  `selection.referenceCandidates`;
- selected body routes to the same Selection tab/inspector/modeling state as
  tree selection;
- body hover and selected visual state;
- consumed/stale/unsupported diagnostics;
- browser smoke for viewport body pick -> command-ready reference.

Definition of done:

- supported bodies can be picked in viewport and resolve to command-ready
  semantic body references;
- unsupported picks return structured diagnostics;
- selection details do not cover the viewport.

Implemented boundaries:

- viewport clicks now produce private V9 body hit candidates and then resolve to
  semantic body selections before app selection state is updated;
- renderer hit IDs, mesh IDs, selection-buffer IDs, pixel data, and canvas
  internals do not become public stable CAD references;
- body picks preserve the user's current Tree/Selection tab preference instead
  of forcing the Selection panel open;
- the current canvas can show transient body hover/preselection and selected
  state, but face/edge/vertex picking remains deferred to Tranche C;
- viewport selection remains session state and does not change `.wcad`, JSON,
  OPFS cache, command history, or `web-cad.project.v17`.

### Tranche C: Generated Face/Edge Picking

Goal: pick supported generated references in the viewport.

Scope:

- generated planar face picking for supported rectangle/circle extrude source
  semantics;
- generated edge picking where current references are defensible;
- named reference routing when a name resolves to a generated reference;
- visual distinction between body, face, and edge hover/selection;
- structured unsupported/ambiguous/consumed/stale diagnostics.

Definition of done:

- create sketch on generated planar face can start from viewport selection;
- name generated reference can start from viewport selection;
- chamfer/fillet edge target affordances consume query-proven edge references;
- result bodies without stable references remain diagnostic-only.

Implemented C1 boundaries:

- the current canvas/app path can refine a selected supported rectangle/circle
  `newBody` extrude body pick into a private generated planar face hit
  candidate;
- generated face candidates resolve through the V9 interaction helper and
  `selection.referenceCandidates` before updating app selection state;
- the inspector and modeling workflow consume the same command-ready generated
  reference state used by tree/selection-panel reference lists;
- viewport face picking preserves the user's Tree/Selection tab preference and
  does not add a viewport detail overlay;
- exact face-level highlighting, generated edge/vertex picking, WebGPU,
  assemblies, persisted viewport state, and `web-cad.project.v17` remain
  deferred.

Implemented C2 boundaries:

- the current canvas/app path can refine a selected supported rectangle/circle
  `newBody` extrude body pick into a private generated edge hit candidate;
- rectangle cap edges (`start`/`end` `uMin`/`uMax`/`vMin`/`vMax`) and
  longitudinal rectangle corner edges are reconstructed from existing
  generated-reference profile/depth signatures for display-approximate picking;
- circle `start:circular` and `end:circular` cap edges are pickable where the
  existing generated edge references prove the semantic target;
- generated edge candidates resolve through the V9 interaction helper and
  `selection.referenceCandidates` before updating app selection state;
- the inspector and modeling workflow consume the selected generated edge state
  for name-reference and edge-finish affordances already governed by cad-core
  commandability;
- edge highlighting remains at the owning body because exact edge-level visual
  highlighting is not yet defensible in the current renderer;
- generated vertex picking, arbitrary result-body edge topology, WebGPU,
  assemblies, persisted viewport state, and `web-cad.project.v17` remain
  deferred.

### Tranche D: Viewport Visual State System

Status: D1 implemented as the first semantic visual-state input slice.

Goal: make hover, selection, command targeting, and diagnostics feel coherent.

Scope:

- restrained highlight layers for hover/preselection, selected, command target,
  warning/unsupported, pending, and failed states;
- renderer receives semantic display inputs only;
- no viewport debug wall or large overlay;
- mobile and desktop layout checks;
- visual QA and smoke assertions that the model remains visible.

Definition of done:

- supported interaction states are visible and consistent;
- the renderer remains a display consumer;
- no text or panels block normal modeling.

Implemented D1 boundaries:

- the app derives a single semantic viewport visual-state model from existing
  selection, hover/preselection, command-target, warning/blocked, pending, and
  failed status helpers;
- the renderer receives display-only visual inputs: render target id, display
  entity kind, and visual state. It does not decide commandability and does not
  receive generated stable face/edge ids as public refs;
- generated face and edge selections intentionally highlight the owning body
  while compact status text distinguishes the selected subentity. Exact
  face/edge canvas highlighting remains deferred until renderer data can prove
  it safely;
- the viewport status is a compact, pointer-transparent strip rather than a
  diagnostic panel. Detailed diagnostics remain in Selection, Inspector, and
  Modeling surfaces;
- no WebGPU, persistence, new project schema, new modeling command, OPFS,
  agent, MCP, or command-history behavior was added.

### Tranche E: Contextual Command Tools

Status: E1 implemented as the compact contextual command-surface slice.

Goal: make existing commands discoverable from viewport selections.

Scope:

- compact contextual action surface driven by command-ready query results;
- create sketch on selected planar face;
- name selected generated reference;
- edge-finish target workflows for supported generated edges;
- active sketch/entity actions where current command contracts support them;
- cancel/retry behavior and structured command errors.

Definition of done:

- viewport-selected targets expose only valid existing commands;
- commands still execute through CADOps;
- tree, inspector, modeling, agent, and MCP behavior do not regress.

Implemented E1 boundaries:

- the app derives a compact viewport-adjacent command surface from
  `selection.referenceCandidates`, `deriveModelingActions`, generated-reference
  selection state, and existing measurement/reference helpers;
- selected generated planar faces expose create-sketch-on-face, name, measure,
  and inspect affordances only when the current CADOps readiness surface proves
  them;
- selected generated edges expose name, chamfer, fillet, measure, and inspect
  affordances, with chamfer/fillet routed through the existing edge-finish form
  defaults and still blocked by the current rectangle-edge/new-body rules;
- selected bodies expose compact reference and measurement continuations, while
  face-specific sketch creation continues in Modeling until a face is selected;
- unsupported, stale, consumed, missing, ambiguous, and non-commandable
  selections stay concise in the viewport and keep detailed diagnostics in
  Selection, Inspector, and Modeling surfaces;
- the surface is transient UI state only: Escape and selection changes dismiss
  expanded name/measure/reference details, no Tree/Selection tab is
  auto-switched, and no `.wcad`, JSON, OPFS, command-history, renderer, mesh,
  OCCT, GPU, selection-buffer, agent, MCP, or `web-cad.project.v17` behavior was
  added.

### Tranche F: Measure And Inspect Tool

Status: F1 implemented as the single-target viewport Measure/Inspect contract.
F2 implemented as the session-only two-target measurement workflow.

Goal: add a practical viewport-native measure/inspect workflow.

Scope:

- measure selected body, face, edge, point/vertex where current semantics allow;
- distance between two supported targets;
- angle between supported linear/planar targets where exact/semantic data is
  available;
- radius/diameter for supported circular references;
- area/length/volume summaries where current exact/source data supports them;
- diagnostics for approximate, unsupported, ambiguous, or stale measurements.

Definition of done:

- measurements declare their authority source;
- exact/semantic values are preferred over display approximations;
- unsupported measurements are structured and readable.

Implemented F1 boundaries:

- compact viewport Measure/Inspect now covers one selected semantic CAD target
  at a time: selected bodies, generated planar faces, generated edges, and
  named references after they resolve to one of those generated references;
- values are derived from existing `body.measurements`,
  `body.generatedReferenceMeasurements`, and `selection.referenceCandidates`
  state, with explicit authority labels such as source-analytic exact or
  unsupported;
- body Measure shows volume, surface area, bounds, and centroid where current
  source-analytic measurements are available;
- generated planar face Measure shows area, center, surface, and normal where
  current source semantics prove the target;
- generated edge Measure shows length and circular radius/diameter where
  rectangle/circle extrusion semantics prove the value;
- compact Inspect shows semantic target identity, command readiness, authority,
  and structured diagnostics without exposing raw renderer, mesh, OCCT, GPU,
  pixel, selection-buffer, file-handle, or OPFS IDs;
- stale, consumed, ambiguous, missing, unsupported, and measurement-source
  unavailable states remain structured diagnostics;
- F1 does not add two-target distance/angle workflows, persisted measure state,
  new modeling commands, renderer authority, schema/storage changes, WebGPU,
  assemblies, STEP import, broad topology naming, agent/MCP changes, or
  `web-cad.project.v17`.

Implemented F2 boundaries:

- compact viewport Measure can start a two-target measure from the current
  supported semantic target, keep the first target in app/session state, preview
  the currently selected second target, and commit that second target for the
  active session;
- supported distance results use only source-backed body centroids, generated
  planar face centers, and generated edge centers or source endpoints from
  existing `body.measurements` and `body.generatedReferenceMeasurements` query
  data;
- supported angle results use only generated planar face normals and generated
  linear edge directions where source measurements prove those vectors;
- every result declares authority, and unsupported, missing, stale, consumed,
  ambiguous, non-commandable, and display-approximation-only states remain
  structured diagnostics;
- the workflow clears with Escape and after document sync, does not persist to
  `.wcad`, JSON, OPFS, command history, agents, or MCP, and does not expose raw
  renderer, mesh, OCCT, GPU, pixel, selection-buffer, file-handle, or OPFS IDs
  in public measurement output;
- radius/diameter remains the F1 single-target circular-edge measurement path.
  F2 does not add sketch constraints, measurement annotations, new modeling
  commands, renderer authority, schema/storage changes, WebGPU, assemblies,
  STEP import, broad topology naming, agent/MCP changes, or
  `web-cad.project.v17`.

### Tranche G: Navigation, Camera, And Tool Ergonomics

Status: G1 implemented as compact session-only navigation and camera
ergonomics.

Goal: make daily viewport use feel polished.

Scope:

- fit all and fit selection;
- orientation controls for standard views;
- consistent orbit/pan/zoom behavior;
- keyboard handling for select/additive-select/cancel/repeat where practical;
- tool state visible without taking space from the model;
- no persisted named views or preferences unless separately scoped.

Definition of done:

- a user can work mostly from viewport plus compact side panels;
- viewport interaction remains usable on supported desktop and mobile sizes;
- camera state remains session/display state.

Implemented G1 boundaries:

- compact viewport controls now route Fit all, Fit selected, Reset, Zoom, and
  standard Top/Front/Right/Isometric views through shared camera action helpers;
- Fit all uses visible render primitive/mesh/sketch-display bounds and resets
  to the default camera only when no visible geometry exists;
- Fit selected uses the selected render target when it has defensible display
  bounds. Generated face/edge selections continue to fit the owning body
  because exact subentity viewport bounds are deferred;
- standard views update only display yaw/pitch while preserving the current
  camera target and distance;
- Escape clears session-only viewport hover/pick diagnostic state and active
  two-target measurement state when focus is not inside an editable form field.
  Contextual Measure/Inspect/Name/Reference detail also clears through the same
  safe keyboard guard;
- no camera, hover, selection, active tool, `.wcad`, JSON, OPFS, command
  history, file-handle, agent/MCP, renderer, mesh, OCCT, GPU, pixel, or
  selection-buffer identifiers are promoted to project source or public CAD
  references.

### Tranche H: Release Smokes, UX QA, And Hardening

Status: H1 implemented as release smoke matrix expansion. H2 implemented as
responsive UX QA and diagnostic polish. H3 remains final release-hardening
work.

Goal: make V9 release confidence repeatable.

Scope:

- product browser smoke for body/face/edge picking, contextual commands,
  measurement, diagnostics, `.wcad` round-trip, STEP/GLB availability, and no
  console errors;
- release samples for viewport interaction expectations where non-browser tests
  can cover them;
- visual QA across desktop/mobile viewport sizes;
- docs cleanup and final scope audit.

Definition of done:

- `pnpm test`, `pnpm typecheck`, `pnpm lint`, `pnpm format:check`, and
  `pnpm build` pass;
- V8 `.wcad`, OPFS, STEP, GLB, JSON, agent/MCP, and V7 modeling workflows do
  not regress;
- V9 browser smokes cover direct viewport workflows;
- docs clearly state remaining deferrals.

Implemented H1 boundaries:

- the browser workflow smoke now covers direct viewport body picking for the
  supported circle `newBody` extrude path, then circle generated planar cap
  face and circular edge pick/reference routing through the same semantic
  `selection.referenceCandidates` path used by rectangle checks;
- circle generated circular edge coverage is recorded as command-ready routing
  when deterministic display picking succeeds. If this path regresses, the
  required smoke check fails rather than relying on renderer-private IDs;
- session-only ergonomics are covered by asserting that Escape outside editable
  controls clears active two-target measurement state and expanded contextual
  detail without changing CAD source;
- viewport obstruction checks now run after navigation/measurement and after
  project round-trip, and the smoke applies a narrow mobile-sized CDP viewport
  check to prove the model surface remains visible;
- required browser smoke check IDs and script tests were expanded so missing H1
  evidence fails deterministically;
- H1 changes release evidence only. It does not add modeling commands,
  generated vertex picking, exact subentity highlighting, persisted camera or
  selection state, WebGPU, assemblies, STEP import, project schema changes, or
  new production dependencies.

Implemented H2 boundaries:

- compact contextual command and viewport status text now wraps instead of
  clipping single-line labels, while measurement/detail sections keep internal
  scrolling and avoid horizontal viewport overflow;
- when status and contextual controls are both present, the contextual surface
  receives app-layer layout state so it can avoid overlapping the status strip
  without adding a viewport-blocking panel or persisted UI preference;
- blocked, consumed, unsupported, ambiguous, and missing viewport selection
  summaries prefer the primary structured diagnostic message over generic
  labels where current `selection.referenceCandidates` or viewport pick state
  provides one;
- the browser workflow smoke now requires desktop contextual/status readability
  and narrow/mobile clipping/scroll-trap checks in addition to the H1 model-area
  obstruction assertions. The validated H2 smoke evidence recorded
  `viewport-responsive-text-readability` with `contextual=280x190;
  status=210x45` and `viewport-narrow-scroll-readability` with
  `contextual=356x128; status=352x58`;
- H2 changes app-layer presentation, diagnostics, and smoke coverage only. It
  does not add modeling commands, generated vertex picking, exact subentity
  highlighting, persisted camera or selection state, renderer authority,
  WebGPU, assemblies, STEP import, project schema changes, agent/MCP behavior,
  or new production dependencies.

Remaining H3 release-hardening work:

- H3 should be the final V9 release audit and documentation gate: run the full
  validation suite, confirm V7/V8 workflows still pass, close stale release
  wording, and record final deferrals without expanding CAD behavior.

## Release Completion Criteria

V9 is release-ready only when:

1. Viewport body, generated face, and generated edge selections are semantic and
   command-ready for the supported target subset.
2. Unsupported picks and measurements return structured diagnostics.
3. Contextual viewport actions are driven by cad-core query/command readiness.
4. Measurements declare exact/semantic/display/unsupported authority clearly.
5. Renderer, mesh, OCCT, GPU, and selection-buffer identifiers remain private.
6. Selection, hover, active tool, and camera state remain session/display state.
7. No new project schema is introduced unless source data is added explicitly.
8. Existing V7/V8 modeling, `.wcad`, OPFS, STEP, GLB, JSON, agent, and MCP
   workflows still pass.
9. Browser smokes prove direct viewport workflows with no console errors.
10. The UI feels like a CAD workspace, not a debug console or obstructive panel
    stack.

## Explicitly Deferred Beyond V9 Unless Scoped Later

- Production WebGPU renderer replacement.
- Large assemblies, mates, LOD, and instancing.
- STEP import with healing and assembly reconstruction.
- Broad arbitrary topological naming for every OCCT result shape.
- Persistent B-rep checkpoints as source truth.
- Broad sketch solver expansion and drag-solving UX.
- New modeling commands such as shell, sweep, loft, mirror, patterns, direct
  edits, and broad booleans.
- Persistent view states, named cameras, UI preferences, or selection state.
- Hosted collaboration.
- Production MCP auth and permission administration.
- Natural-language command entry.

Implement V9 Tranche A: Viewport Interaction Contract

That tranche should add typed hit-candidate/selection-intent/helper behavior and
