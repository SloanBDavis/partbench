# V6 Target: Practical Solid Modeling On Exact Geometry

Status: planned active milestone.

V6 should be the first major usefulness jump after the V2/V3/V4/V5 foundation.
V5 made exact/topology status inspectable. V6 should use that foundation to make
Partbench capable of building more realistic mechanical parts with multiple
source-of-truth feature types, exact-kernel derived measurements where
available, and reference-consuming operations that fail honestly when topology
is ambiguous.

V6 is intentionally larger than V5. It is still implemented iteratively, but the
milestone scope should be broad enough that the design decisions are made now:

```text
constrained sketches -> extrude/revolve/hole/edge finish ->
derived exact geometry -> topology/measurement health -> usable UI and agents
```

## Product Goal

A user or structured agent can create a constrained sketch, build a part using
extrudes, revolves, circular holes, and simple edge finishing, then inspect the
result with clear topology/measurement health.

The user-facing result should feel like a small but real CAD workflow:

```text
sketch profile -> constrain/dimension -> extrude or revolve ->
attach sketch to face -> hole/cut -> chamfer/fillet selected edge ->
inspect measurements, topology health, and history
```

The agent-facing result should expose the same workflow through typed CADOps
commands and queries:

```text
query structure/health/topology -> resolve references -> dry-run feature ->
inspect semantic diffs and unsupported states -> commit through CADOps
```

## Architectural Rules

V6 must keep these rules:

- `cad-core` remains authoritative for source document state.
- All mutations go through CADOps.
- Geometry-kernel, geometry-worker, and OCCT remain derived execution layers.
- Rendered meshes, exact-kernel results, topology snapshots, and measurements
  are derived/cache data unless a phase explicitly adds persisted source records.
- React UI remains command/query-layer driven and does not import geometry
  internals.
- Agent and MCP adapters remain thin wrappers over CADOps and query paths.
- Generated references must not expose raw OCCT, mesh, or renderer indexes as
  stable IDs.
- Ambiguous topology fails with structured diagnostics instead of preserving the
  wrong face/edge/vertex.
- Browser E2E tests are optional manual verification only; durable confidence
  comes from unit/package tests.

## Design Decisions

V6 makes these decisions up front:

1. **V6 is a modeling milestone, not a storage or renderer milestone.**
   OPFS/File System Access, STEP import/export, WebGPU, assemblies, and hosted
   collaboration remain out of scope.

2. **New feature source records are persisted.**
   Each source-of-truth V6 feature that cannot be represented by the prior
   schema should introduce an explicit project format with migration coverage.
   V6 has introduced `web-cad.project.v14` for revolve,
   `web-cad.project.v15` for hole, and `web-cad.project.v16` for command-first
   chamfer/fillet source records.

3. **Exact geometry is still derived.**
   V6 should add kernel-derived metadata and measurements where practical, but
   it should not persist B-rep checkpoints or OCCT topology IDs in project JSON.

4. **Target-consuming features create result bodies.**
   Hole, chamfer, and fillet should follow the existing boolean add/cut model:
   they consume an active target body and create a new authored result body. This
   keeps history explicit and avoids mutating body identity in place.

5. **Stable references are only exposed where defensible.**
   Simple semantic references remain available for simple authored bodies.
   V6 result bodies may expose references only when role/lineage/signature
   matching is stable enough. Otherwise they must report `ambiguous`.

6. **UI must make workflows discoverable without becoming verbose.**
   V6 UI work should consolidate common modeling actions into compact tools:
   create sketch, choose profile, choose feature type, select reference/target,
   inspect health.

## V6 Scope

In scope:

- exact-kernel metadata and mass-property query paths where OCCT bindings allow;
- `feature.revolve` for supported sketch profiles around a sketch line axis;
- `feature.hole` for circular hole/cut workflows from planar sketches;
- first edge-consuming finishing features:
  - `feature.chamfer`,
  - `feature.fillet`;
- derived geometry and OCCT execution for those feature types;
- measurements, project extents, topology status, and dependency health for V6
  features;
- source format evolution through explicit schema versions when V6 feature
  records are persisted;
- agent-adapter and MCP pass-through for V6 commands and queries;
- compact UI workflows for revolve, hole, chamfer, fillet, and related health;
- broad unit/package coverage across protocol, cad-core, geometry, worker,
  adapter/MCP, UI helpers, serialization, undo/redo, and docs.

Out of scope:

- broad/general booleans beyond current add/cut plus V6 hole behavior;
- arbitrary topology naming across all possible feature edits;
- persistent B-rep checkpoints;
- STEP/IGES import/export;
- OPFS/File System Access;
- WebGPU renderer work;
- assemblies, mates, LOD, and instancing;
- shell, sweep, loft, pattern, mirror, direct modeling, or drawing generation;
- arbitrary parameter expressions;
- new broad sketch constraint families unless separately scoped;
- natural-language command parsing.

## Phase A: Exact Kernel Metadata And Measurement Upgrade

Goal: upgrade the derived geometry boundary so current and future V6 bodies can
return kernel-derived metadata where practical.

Planned deliverables:

- typed geometry-kernel/worker request and response shapes for exact body
  metadata:
  - bounds,
  - volume,
  - surface area,
  - centroid,
  - topology/entity counts where available,
  - status/diagnostics for kernel failure, empty result, invalid result, or
    unavailable binding;
- cad-core read/query plumbing that keeps these results derived;
- `body.topology` integration for optional derived exact metadata snapshots
  from app/adapter cache layers. Cad-core validates the snapshot against the
  current source-identity cache key, reports stale/kernel/unavailable statuses
  honestly, and does not persist the metadata;
- measurement confidence values that distinguish:
  - source-analytic,
  - kernel-derived,
  - unavailable,
  - ambiguous;
- tests proving source document JSON is unchanged by exact metadata queries;
- OCCT binding audit for mass properties, shape bounds, revolve, hole/cut,
  chamfer, and fillet APIs.

Initial Phase A progress:

- `geometry.exactBodyMetadata` now exists at the geometry-kernel/worker boundary
  for source-derived extrudes and the current narrow boolean-extrude sources;
- rectangle and circle extrude metadata is kernel-derived through OCCT bounds
  and mass-property APIs;
- current rectangle add/cut and circle-target cut sources can return
  kernel-derived boolean-result metadata through the geometry boundary;
- authored rectangle/circle revolve sources can return kernel-derived exact
  metadata through the same `geometry.exactBodyMetadata` boundary when the
  source profile, same-sketch line axis, angle, and placement frame are valid;
- supported authored hole result bodies can return kernel-derived exact
  metadata through `geometry.exactBodyMetadata` for rectangle/circle authored
  extrude targets and circular base-plane or resolved attached-sketch tools;
- supported authored chamfer/fillet result bodies can return kernel-derived
  exact metadata through `geometry.exactBodyMetadata` for the same narrow
  rectangle-target/generated-rectangle-edge subset proven by
  `geometry.edgeFinish`;
- the app geometry layer has a derived exact-metadata cache service with
  source-derived cache keys and stale async result protection, separate from the
  existing mesh cache;
- cad-core body topology/exact-metadata read paths can accept validated derived
  metadata snapshots while keeping metadata derived and non-persistent;
- `project.extents` can accept optional derived exact metadata snapshots as
  read-only query input and use matching, ready kernel bounds for V6 result
  bodies whose source-analytic extents are unavailable;
- metadata responses carry no mesh buffers and do not mutate or persist
  document source data;
- app UI and project JSON are intentionally not authoritative owners of this
  metadata boundary.

Acceptance:

- simple extrude metadata still works;
- boolean add/cut metadata is either kernel-derived or explicitly unavailable;
- authored revolve metadata is kernel-derived where OCCT can build the source
  shape, while generated references for revolve bodies remain unavailable;
- authored hole metadata is kernel-derived where OCCT can build the target and
  circular cutting tool, while generated references for hole result bodies
  remain unavailable;
- no mesh triangle data becomes authoritative measurement data;
- missing OCCT APIs are documented as binding gaps, not hidden behind fake data.

## Phase B: `feature.revolve` Core

Goal: add the first new major feature type since extrude.

Command shape decision:

```ts
feature.revolve {
  id?: FeatureId
  bodyId?: BodyId
  sketchId: SketchId
  entityId: SketchEntityId
  axis: { type: "sketchLine"; sketchId: SketchId; entityId: SketchEntityId }
  angleDegrees: number
  operationMode?: "newBody" | "add" | "cut" // add/cut rejected for now
  name?: string
}
```

Initial support:

- closed rectangle and circle sketch entities;
- axis from a sketch line in the same sketch;
- positive finite angle validation up to and including 360 degrees;
- `newBody` only.

Validation:

- source sketch exists;
- profile entity exists and is supported;
- axis entity exists and is a non-zero line;
- axis/profile self-intersection or invalid kernel result fails clearly;
- add/cut operation modes are rejected for revolve in V6 unless deliberately
  promoted later.

Acceptance:

- semantic diffs, undo/redo, batch dry-run/commit, import/export, transaction
  history, project structure/health, and adapter/MCP cover the command-first
  revolve source model;
- project format V14 stores revolve intent and preserves V1-V13 import
  compatibility;
- measurements/extents, derived geometry, generated references, and UI command
  helpers are deferred to later V6 slices;
- existing extrude behavior is unchanged.

Initial Phase B progress:

- `feature.revolve` exists as a typed CADOps command and persisted feature
  source record;
- rectangle and circle sketch profiles can create `newBody` revolve feature
  intent around a non-zero same-sketch line axis;
- add/cut revolve operation modes fail with structured unsupported errors;
- project structure lists authored revolve features and bodies;
- project health reports source dependency status and honest unsupported
  topology status for revolve bodies;
- transaction history, semantic diffs, undo/redo, batch dry-run/commit,
  project JSON V14 import/export, agent-adapter, and MCP batch pass-through are
  covered by unit/package tests;
- `geometry.revolveProfile` now exists at the geometry-kernel/worker boundary
  and can generate OCCT-derived mesh data for rectangle and circle profile
  revolves around a same-sketch line axis;
- authored revolve bodies now flow through the app derived-geometry cache and
  render from `geometry.revolveProfile` mesh results;
- stale async revolve mesh results are guarded by source cache keys, matching
  existing extrude/boolean behavior;
- the sketch feature panel can create `newBody` revolve features from
  rectangle/circle profiles using non-zero same-sketch line axes;
- authored revolve bodies can report derived exact-kernel metadata through
  `body.topology` when a matching snapshot is supplied;
- generated references, source-derived measurements, and extents for revolve
  bodies remain intentionally unsupported.

Planned deliverables:

- derived exact metadata source/cache support for authored revolve result

Acceptance:

- a user can create a sketch profile and axis, run revolve, select the result

## Phase D: `feature.hole` Circular Cut Workflow

Goal: add a practical face-based machining workflow without broadening general
booleans.

Command shape decision:

```ts
feature.hole {
  id?: FeatureId
  bodyId?: BodyId
  targetBodyId: BodyId
  sketchId: SketchId
  circleEntityId: SketchEntityId
  depthMode: "blind" | "throughAll"
  depth?: number
  direction?: "positive" | "negative"
  name?: string
}
```

Initial support:

- circle sketch entity as the tool profile;
- base-plane sketches and sketches attached to eligible planar generated faces;
- active authored target bodies where the kernel path can cut reliably;
- blind and through-all holes if through-all can be computed from target bounds
  without making source data ambiguous.

Validation:

- target body exists and is active;
- primitive-derived targets are rejected unless explicitly supported;
- circle entity exists and radius is positive;
- blind depth is positive finite;
- stale attached sketch references fail clearly;
- empty/full-removal kernel results return structured errors.

Acceptance:

- hole consumes the target body and creates an authored result body;
- measurements/extents/topology health are honest for the result;
- generated references are exposed only where stable, otherwise ambiguous;
- UI supports creating a hole from a selected circle on an eligible sketch.

Initial Phase D progress:

- `feature.hole` exists as a typed CADOps command and persisted feature source
  record;
- holes can reference circle sketch entities from base-plane sketches or from
  attached sketches whose planar generated-face reference still resolves;
- blind holes require positive finite depth, through-all holes omit depth, and
  direction defaults to `positive`;
- hole features consume one active authored target body and create a new
  authored result body, consistent with current add/cut behavior;
- project structure, project health, semantic diffs, undo/redo, batch
  dry-run/commit, transaction history, project JSON V15 source compatibility
  carried forward through current V16 import/export,
  agent-adapter, and MCP batch pass-through are covered by focused tests;
- `geometry.hole` now exists as a geometry-only kernel/worker request for
  feasibility testing. It can cut circular tools into rectangle extrude targets
  and circle extrude targets where the OCCT path succeeds, including blind depth
  and through-all depth derived from target bounds;
- the geometry-only hole path uses OCCT cylinder construction plus
  `BRepAlgoAPI_Cut` and returns serializable mesh data only. It does not mutate
  cad-core source data or project JSON;
- app derived-geometry wiring now builds `geometry.hole` mesh requests for
  supported authored hole result bodies, keeps consumed targets out of the
  render source list, and treats pending/error/unsupported hole results as
  mesh-only status instead of showing misleading primitive fallback;
- app derived exact-metadata wiring now builds `geometry.exactBodyMetadata`
  hole requests for the same supported authored hole result bodies. Hole exact
  metadata cache keys include the target body source, circular tool
  center/radius, resolved attached-sketch placement, depth mode/depth, and
  direction, and stale async results cannot overwrite newer source state;
- compact UI creation controls now support creating holes from selected circle
  sketch entities, eligible rectangle/circle newBody extrude targets,
  blind/through-all depth, and positive/negative direction;
- `body.topology` can report matching kernel-derived exact metadata snapshots
  for authored hole result bodies, with structured stale, unavailable-binding,
  kernel-failed, empty, or invalid-result issues. Generated references for hole
  result bodies, exact topology naming, source-derived hole measurements, and
  extents are still intentionally deferred.

## Phase E: Edge Finishing With Chamfer And Fillet

Goal: consume the generated-reference system for the first true edge operations.

Commands:

```ts
feature.chamfer {
  id?: FeatureId
  bodyId?: BodyId
  targetBodyId: BodyId
  edgeStableId?: string
  namedReference?: string
  distance: number
  name?: string
}

feature.fillet {
  id?: FeatureId
  bodyId?: BodyId
  targetBodyId: BodyId
  edgeStableId?: string
  namedReference?: string
  radius: number
  name?: string
}
```

Initial support:

- one edge reference per feature;
- generated or named edge references that resolve to stable semantic rectangle
  or circle newBody extrude edges;
- authored active target bodies;
- result body consumes target body;
- unsupported for boolean result bodies, revolve bodies, hole bodies,
  primitive-derived bodies, consumed targets, non-edge references, and stale
  references;
- current core support stores source-of-truth intent only;
- `geometry.edgeFinish` now proves a geometry-only mesh path for rectangle
  authored extrude targets and one generated rectangle edge stable ID. It maps
  semantic rectangle edge roles to transient OCCT edges by geometric endpoint
  matching, then executes chamfer/fillet with OCCT and returns serializable mesh
  data only;
- app derived-geometry wiring now builds `geometry.edgeFinish` mesh requests for
  supported authored chamfer/fillet result bodies. Supported rendering covers
  rectangle `newBody` extrude targets, generated rectangle edge stable IDs, and
  named edge references when the app can resolve the name to a current generated
  edge snapshot on the target body. Cache keys include the target source, edge
  stable ID, operation, and distance/radius, and stale async results cannot
  overwrite newer source state;
- app UI now exposes compact edge-finish controls from the generated-reference
  inspector for a selected generated rectangle edge, with optional use of a
  resolved name for that same edge. Unsupported targets/references and invalid
  distance/radius values remain disabled with explicit status text;
- authored chamfer/fillet result bodies in the rectangle-edge subset can now
  report derived exact metadata/topology status through the same read-only
  snapshot path as extrudes, booleans, revolves, and holes;
- circle target edge finishing, exact topology naming, and generated references
  for chamfer/fillet result bodies remain future work.

Validation:

- target body exists and is active;
- exactly one reference path is supplied: `edgeStableId` or `namedReference`;
- edge reference resolves to an edge and is eligible for the operation;
- chamfer distance / fillet radius is positive finite;
- primitive-derived and ambiguous topology targets are rejected.

Acceptance:

- command-first chamfer and fillet source records are covered through protocol,
  cad-core, adapter/MCP pass-through, import/export, undo/redo, project
  structure/health, transaction history, and docs;
- geometry-only `geometry.edgeFinish` can execute the narrow rectangle-edge
  chamfer/fillet feasibility path through geometry-kernel, geometry-worker, and
  occt-wasm without changing cad-core source authority or project JSON;
- supported authored chamfer/fillet result bodies render through derived
  `geometry.edgeFinish` meshes without double-rendering consumed target bodies
  or using primitive fallbacks for unsupported edge-finishing results;
- generated references for result bodies are ambiguous unless stable matching is
  explicitly implemented and tested.

## Phase F: V6 Topology, Measurements, And Reference Health

Goal: make V6 results inspectable without overstating topological naming.

Planned deliverables:

- topology/measurement health for revolve, hole, chamfer, and fillet results;
- named-reference stale/ambiguous handling after V6 features;
- project health entries that show consumed target chains and exact metadata
  confidence;
- project extents that use kernel-derived bounds where available;
- query tests for source edits, undo/redo, import/export, and stale async
  geometry results.

Acceptance:

- exact metadata improves measurements where reliable;
- ambiguous topology remains visible and structured;
- no generated reference silently points to the wrong V6 result entity.

Initial Phase F progress:

- project-level extents preserve primitive and simple authored
  rectangle/circle extrude behavior, while optionally using matching ready
  kernel-derived exact metadata snapshots for authored revolve, hole,
  chamfer/fillet, and boolean result bodies;
- missing, stale, unsupported, unavailable-binding, empty, invalid, and
  kernel-failed derived metadata snapshots produce structured extents warnings
  instead of mesh-derived or primitive fallback bounds;
- `body.measurements` remains source-derived for simple extrudes. V6 result
  body exact measurement health is surfaced through `body.topology` plus
  matching derived exact metadata snapshots.

## Phase G: Agent/MCP And UI Workflow Completion

Goal: make V6 usable by humans and agents through the same command/query layer.

Planned deliverables:

- agent-adapter and MCP pass-through for all V6 commands/queries;
- concise inspector/status UI for V6 feature bodies;
- feature creation controls that prioritize:
  - Extrude,
  - Revolve,
  - Hole,
  - Chamfer,
  - Fillet;
- filtering so unsupported targets are not presented as valid choices;
- clear feature history and health copy;
- helper tests for command building, filtering, status formatting, and project
  JSON preview behavior.

Acceptance:

- the app can demonstrate a representative workflow:
  constrained sketch -> extrude -> attached sketch -> hole -> edge chamfer or
  fillet -> measurement/topology inspection;
- the same workflow can be executed through CADOps batch where supported.

## Phase H: Stabilization And Completion

Goal: declare V6 complete only after the practical solid-modeling workflow is
coherent, tested, documented, and browser-verified.

Required review areas:

- protocol command/query shapes;
- cad-core source model, validation, semantic diffs, undo/redo, batch behavior;
- project JSON import/export/migration if introduced;
- geometry-kernel, geometry-worker, and OCCT derived execution paths;
- derived geometry cache keys and stale-result handling;
- project structure, health, measurements, extents, and transaction history;
- adapter/MCP pass-through;
- UI helper logic and compact workflow presentation;
- docs accuracy.

Required final checks:

- `pnpm test`
- `pnpm typecheck`
- `pnpm build`
- `pnpm lint`
- `pnpm format:check`
- manual browser verification of representative V6 workflows

V6 is complete when:

1. Revolve, hole, chamfer, and fillet are implemented for the supported scoped
   cases.
2. Unsupported geometry/topology cases fail with structured diagnostics.
3. Kernel-derived metadata is used where reliable and source-analytic fallback
   remains honest.
4. Project format changes, if any, are explicit and migrated.
5. Agent/MCP and UI surfaces expose the same behavior through CADOps.
6. Unit/package tests cover the milestone without relying on browser E2E tests.

## Deferred Beyond V6 Unless Explicitly Scoped

- full persistent B-rep checkpoints;
- broad topological naming across arbitrary feature edits;
- general boolean trees beyond the current scoped features;
- sweep, loft, shell, pattern, mirror, and direct modeling;
- STEP/IGES import/export;
- OPFS/File System Access;
- WebGPU renderer;
- assemblies/mates;
- natural-language command parsing.
