# V18 Release Plan: Precision CAD UI Overhaul

Status: **Planned and decision-complete; implementation has not started.**

V18 is a frontend-only release that replaces the accumulated prototype UI with
a coherent, mode-aware, power-user CAD workbench. It does not add modeling
commands, query shapes, document source, geometry recipes, topology support,
file formats, or agent/MCP behavior. V17 remains the complete capability and
support-matrix baseline.

The selected visual direction and five reference screens are in
`docs/v18-ui-concepts/`:

- [Sketch mode](v18-ui-concepts/f-mode-sketch.png)
- [Solid mode](v18-ui-concepts/f-mode-solid.png)
- [Active feature editing](v18-ui-concepts/f-mode-feature-edit.png)
- [Inspect mode](v18-ui-concepts/f-mode-inspect.png)
- [Project mode](v18-ui-concepts/f-mode-project.png)

The images define the spatial system, density, and interaction character. This
document defines the product contract. Where an image depicts a control outside
the completed V17 matrix, the V17 matrix wins.

When sources appear to disagree, implementation follows this order:

1. the frontend-only and non-goal boundaries in this document;
2. the completed V16/V17 command and support matrices;
3. the behavior, state, accessibility, and performance contracts below;
4. the reference images.

An illustrated control that is not present in the Slice A capability inventory
is omitted. Visual similarity never authorizes a capability or placeholder.

**Version naming caution:** product release V18 is not document schema
`web-cad.project.v18`, which already denotes the V13 topology-identity source
shape. V18 introduces **no schema version**. Minimum-triggered
`web-cad.project.v21`, `partbench.wcad.v2`, CADOps, and all public adapter
identifiers remain unchanged.

## Product Goal

V18 makes Partbench feel like one serious CAD application instead of a
collection of release-specific panels.

After V18, a user must be able to:

1. Understand the current document, mode, selection, and active operation at a
   glance.
2. Reach every completed V17 UI operation from a stable labeled ribbon, a
   searchable command surface, or an eligible contextual action.
3. Work in Project, Sketch, Solid, and Inspect modes without the document tree,
   selection semantics, or command model changing underneath them.
4. Create or edit a feature through an explicit
   select -> parameterize -> validate -> apply/cancel loop.
5. Use the sketcher as a dedicated precision workspace with geometry tools,
   dimensions, constraints, solver state, and inference.
6. Select bodies, supported faces, supported edges, sketch entities, features,
   and named references with clear visual and textual feedback.
7. Measure, inspect, name, and repair supported references without seeing raw
   implementation data by default.
8. Open, save, import, export, inspect parameters, and review human-readable
   history from a dedicated Project workspace.
9. Use keyboard search and documented shortcuts without losing mouse-driven
   discoverability.
10. Recover from blocked or failed operations through concise, actionable
    diagnostics adjacent to the affected action.
11. Reach a usable shell quickly and keep pointer, keyboard, selection, form,
    and layout interactions responsive while geometry work is pending.

The release demonstration is:

```text
Open or create a .wcad project.
Create and constrain a sketch in Sketch mode.
Finish the sketch and create an authored solid feature in Solid mode.
Select a feature in the tree, edit its supported fields, validate, apply,
undo, and redo.
Select supported geometry in the viewport and run contextual modeling,
measurement, naming, and inspection actions.
Inspect parameters and human-readable history in Project mode.
Save .wcad, verify STEP readiness, and repeat the V17 browser workflow.
```

## Why V18 Comes After V17

V7 through V17 deliberately added complete vertical slices. The UI accumulated
the controls required to prove each slice:

- a global quick-create toolbar;
- Tree and Selection tabs;
- a large general Inspector;
- separate composite feature panels;
- a broad Modeling actions panel;
- Project/JSON, Sketch, and History utility tabs;
- viewport contextual surfaces;
- release- and topology-oriented status text.

Those surfaces were appropriate while commands and support matrices were being
established. Together they now duplicate entry points, compete for viewport
space, expose implementation vocabulary, and make the user discover the
system's architecture rather than the next modeling action.

V17 provides enough stable modeling breadth to consolidate the product around a
durable workbench. V18 therefore changes presentation and workflow composition
only. It consumes the command/query contracts already proved by V17 instead of
reopening them.

## Frontend-Only Boundary

V18 production changes are limited to `apps/web`. Release scripts and
documentation may change to prove the frontend. Existing package tests may be
updated only when imports or frontend fixtures move; package behavior may not
change.

V18 must not require changes to:

- `packages/cad-protocol`;
- `packages/cad-core`;
- `packages/geometry-kernel`;
- `packages/geometry-worker`;
- `packages/occt-wasm`;
- `packages/sketch-solver`;
- `packages/renderer` or `packages/renderer-mesh-bridge`;
- agent or MCP adapters;
- project JSON, CBOR, `.wcad`, topology-checkpoint, or STEP formats.

If implementation discovers that a proposed workflow needs a new command,
query field, geometry request, selection primitive, renderer contract, or
source record, that workflow is out of V18. The UI must reflect the completed
matrix honestly rather than smuggling a backend expansion into a frontend
release.

V18 adds no production dependency by default. Icons use a small repository-local
SVG component set. CSS remains ordinary imported stylesheets. Any proposed
dependency requires separate approval and a documented bundle, maintenance, and
accessibility justification.

### Direct replacement and deletion rule

V18 is the replacement UI, not an alternate shell or staged cutover. There is
no runtime/build flag, parallel route, legacy fallback, or period in which two
shells are imported into the production bundle. Git is the rollback mechanism.

Each implementation slice has one owner for the surface it changes. The slice
is incomplete until it:

1. proves behavior parity for that surface;
2. makes the V18 surface the only rendered owner;
3. deletes the replaced component composition, shell state, imports, obsolete
   DOM tests/selectors, and CSS in the same change; and
4. retains tested command, query, storage, selection, and display helpers only
   while the replacement still imports them.

Cleanup targets obsolete composition and duplicate UI logic. It does not
justify rewriting stable package boundaries or deleting tested helpers that
still have a single clear consumer.

## Non-Negotiable Architecture

All existing architecture rules remain normative:

- CADOps owns every source mutation.
- The UI may collect drafts and gestures, but it must submit existing typed
  operations to the existing command executor.
- Queries, not component-local inference, determine commandability,
  editability, profile/path candidates, reference health, measurements, and
  topology-backed action readiness.
- `cad-core` remains the source of document, feature, sketch, parameter,
  constraint, reference, transaction, and diagnostic truth.
- Rendered meshes, sketch overlays, hover state, selection decoration, command
  previews, panel state, and layout preferences remain derived or UI-local.
- React does not import OCCT or geometry internals and does not read mesh
  triangles to decide CAD semantics.
- Renderer IDs are translated through the existing semantic pick-intent path.
  They never become command targets.
- Human UI, tests, and automation continue to exercise the same CADOps and query
  layer.
- Unsupported, stale, consumed, ambiguous, or unavailable states remain honest
  blocked states. Visual polish may not turn them into enabled controls.

### UI action registry is presentation, not a second command layer

V18 introduces one typed frontend action registry used by the ribbon, command
search, contextual strip, menus, and shortcuts. A registry item contains
presentation metadata and delegates to existing handlers:

```ts
type WorkbenchMode = "project" | "solid" | "sketch" | "inspect";

type UiActionAvailability =
  | { readonly status: "ready" }
  | { readonly status: "needs-selection"; readonly message: string }
  | { readonly status: "blocked"; readonly message: string };

interface UiActionDefinition {
  readonly id: string; // frontend identifier only
  readonly label: string;
  readonly group: string;
  readonly modes: readonly WorkbenchMode[];
  readonly aliases: readonly string[];
  readonly mutatesSource: boolean;
  readonly shortcut?: string;
  readonly getAvailability: (context: UiActionContext) => UiActionAvailability;
  readonly run: (context: UiActionContext) => void | Promise<void>;
}
```

Action IDs use the documented `<mode>.<action>` namespace, for example
`solid.extrude`, `inspect.measure`, and `project.save`. The registry covers
invokable workbench actions and navigation, not field changes, tree expansion,
camera movement, or transient sketch gestures. It consumes a memoized
`UiActionContext` assembled once per document revision, derived-result revision,
and semantic selection change. `getAvailability` must not run engine queries or
scan the document during React render.

The same definition and availability result is rendered by the ribbon, command
search, contextual strip, menu, and shortcut router. The registry may choose
which existing form or injected handler to open. It may not construct geometry,
replace a readiness query, invent a source ID, import the engine/executor into a
presentation component, or mutate a document directly. Every mutating `run`
path terminates in an existing tested CADOps builder and executor.

## Product Principles

### 1. Expert density, not minimalism

The application keeps complete tool groups visible for the active mode. It uses
compact labeled controls, predictable panel locations, and information
hierarchy. It does not replace tools with onboarding cards, a staged wizard, or
a single opaque recommendation.

### 2. Stable shell, adaptive content

The header, ribbon, document tree, central workspace, inspector, and status bar
stay in fixed regions. Modes change relevant tools and inspector content
without relocating the user's model or document context.

### 3. Progressive relevance, not capability hiding

Contextual actions accelerate selection-driven work. The full ribbon and
command search remain available. A user never has to guess which object to
select merely to discover that a tool exists.

### 4. Explicit draft and commit

Feature forms and destructive actions have visible Apply/Cancel controls.
Transient field edits are UI drafts. Source changes only when an existing
CADOps command commits.

### 5. Capability truth

Every enabled control corresponds to a currently supported command/query row.
Blocked controls explain the missing selection or unsupported condition in
human language. The UI never performs a hidden largest-profile, nearest-edge,
first-candidate, or kernel-success inference.

### 6. Human language by default

The primary surface says “Profile is open,” “Select a target body,” or
“Geometry preview is unavailable.” It does not lead with schema numbers,
stable IDs, diagnostic codes, checkpoint vocabulary, worker names, cache state,
JSON, or topology implementation details.

### 7. Keyboard speed and pointer discoverability

Common actions are searchable and shortcut-capable, but labeled controls remain
visible. No essential workflow depends on memorizing a gesture or shortcut.

## Stable Workbench Shell

The default desktop shell has six regions:

```text
┌──────────────────────────────────────────────────────────────────┐
│ Global document header                                          │
├──────────────────────────────────────────────────────────────────┤
│ Mode-aware labeled ribbon                                       │
├───────────────┬───────────────────────────────┬──────────────────┤
│ Document and  │                               │ Selection,       │
│ feature tree  │       Viewport or task        │ feature editor,  │
│               │          workspace            │ or mode details  │
├───────────────┴───────────────────────────────┴──────────────────┤
│ Status, selection filters, coordinates, zoom, units             │
└──────────────────────────────────────────────────────────────────┘
```

### Global document header

The header is the quiet global layer:

- Partbench mark and name;
- document name;
- saved/unsaved/local-file state;
- undo and redo;
- command search;
- help/shortcut entry;
- pending-command state.

Units, primitives, modeling features, diagnostics, and file conversion do not
live in the global header.

### Mode ribbon

The ribbon has:

- a visible current-mode selector;
- compact tool groups with icons and labels;
- consistent active, disabled, pending, and selected states;
- deterministic whole-group overflow under the breakpoint rules below;
- no horizontal page scroll.

Ribbon labels remain visible at widths of 1200px and above. The mode selector
and the mode's commit/finish group never overflow. Other groups retain their
declared order and move from the trailing edge into **More** until the ribbon
fits; a group is never split across the ribbon and overflow. Icon-only buttons
are allowed only for universally recognized global actions and still require
tooltips and accessible names.

### Left document tree

The left dock is persistent in Sketch, Solid, and Inspect modes. It shows
document order and ownership, not every diagnostic field.

Required groups:

- origin and base planes;
- parameters;
- sketches and authored features interleaved in source order, with sketch
  entities and owned result bodies nested beneath their owner on expansion;
- standalone primitives/scene objects that have no authored-feature owner;
- named references;
- imported features/bodies in their actual source and ownership position.

Every semantic object has one primary row; the tree must not repeat a body in a
parallel top-level Bodies group when ownership is known. Row click selects,
the disclosure control expands, and a dedicated visibility control appears
only for rows whose existing display operation is present in the Slice A
inventory. Rename, edit, delete, and contextual operations appear only when the
underlying command exists. Feature kind is conveyed through the icon; names
remain editable human labels. Healthy rows have no badge. Warning/error health
uses a compact badge and tooltip, not a paragraph or cluster of row icons.

Editing is communicated by the row's selected/editing treatment plus the
inspector and status text. V18 shows no rollback or insertion bar because it has
no rollback/suppression command or source state to back one.

### Central workspace

Sketch, Solid, and Inspect use one persistent existing viewport instance.
Changing among those modes must not remount the canvas or reset its camera.
Project mode uses a structured task workspace and does not trigger geometry
work merely to create a thumbnail. Entering Project hides rather than destroys
the viewport; returning preserves camera and semantic selection.

The viewport is always the largest region. Permanent UI may not float over its
center. Allowed overlays are:

- current sketch dimensions and constraints;
- direct-manipulation handles;
- selection/hover decoration;
- compact measurement annotations;
- a compact contextual action strip;
- transient numeric input adjacent to its manipulator;
- view controls and orientation indicator.

### Right inspector/editor

The right dock has one job at a time:

- selection inspector;
- active feature editor;
- sketch/constraint inspector;
- measurement/inspect results;
- project task details.

It does not stack the entire Project, Modeling, Inspector, and Sketch
applications in one scrolling column.

### Bottom status bar

The status bar is mode-specific:

| Mode | Required content |
| --- | --- |
| Sketch | Current instruction, pointer coordinates, zoom, read-only units, and solver summary. No Snap/Grid/Auto-constraints toggles. |
| Solid | Current instruction, active semantic selection filter, coordinates when meaningful, zoom, read-only units, and compact rebuild/pending state. |
| Inspect | Measurement/selection instruction, Body/Face/Edge filter, zoom, read-only units, and pending state. |
| Project | File, dirty/saved, readiness, and pending state only. No sketch, coordinate, filter, or zoom controls. |

Status text must be short enough to remain visible without increasing bar
height. Long text truncates visually and exposes the complete accessible text
through the associated detail surface. Passive coordinates are not focusable.
Full diagnostics open in the right dock.

### Layout sizing and resizing

The visual reference baseline is 1536x1024 CSS pixels. Release behavior is also
tested at 1280x720, 1024x768, and the existing 390x740 narrow workflow. The
latter is a compatibility target, not a mobile/touch-first product claim.

The implementation uses these exact responsive rules:

- left dock default 282px, resizable 220–380px;
- right dock default 350px, resizable 300–460px;
- at widths >=1200px both docks are inline and open by default;
- at 960–1199px the left dock remains inline and the right dock becomes an
  overlay drawer when opened;
- below 960px both docks are overlay drawers, only one may be open at a time,
  and the active editor drawer wins over passive document navigation;
- inline docks remain open only while at least 480px of viewport width remains;
- drawers overlay from an edge, contain focus while modal, close before other
  Escape-stack steps, and restore focus to their opener;
- the viewport is at least 480x320px at 1280x720 and at least 320x240px at
  390x740 when no drawer is open;
- header <=48px, ribbon <=108px, and status bar <=40px; header and status remain
  single-row;
- each dock scrolls independently;
- Project center content scrolls inside its workspace; and
- the page body has no horizontal or vertical scroll at any tested size.

Dock dividers are pointer- and keyboard-resizable. Docks may be collapsed
independently and restored without losing selection or drafts, and closing a
drawer does not unmount its active editor. Breakpoint-driven drawer/collapse
state is derived and is never written as the user's collapse preference;
returning to a wider width restores the prior explicit preference.

## Visual System

V18 uses one mixed-light workbench theme matching the reference screens:

- dark graphite global header;
- near-white ribbon and docks;
- neutral light viewport background;
- graphite text;
- cobalt blue for focus, selection, and primary actions;
- green for validated or completed success, never as the primary Apply color;
- amber for warning or provisional state;
- red for destructive actions and errors.

Required CSS token families:

- color: background, surface, elevated surface, border, text, muted text,
  accent, selection, success, warning, danger;
- spacing: 2, 4, 6, 8, 12, 16, 24px;
- control heights: 28, 32, 36px;
- typography: application, numeric/monospace, headings, captions;
- elevation: dock, popover, command search;
- focus ring and selection outline;
- panel widths and shell heights.

Slice A commits the actual token values before shell implementation. Controls
then use these fixed density rules: 32px standard buttons/fields, 28px dense
tree rows, 16px inline icons, 20px toolbar icons, 24px mode icons, 4px small
radius, and 6px popover/dialog radius. Normal ribbon and action targets are at
least 32px; only dense tree controls may use 28px. Docks use one-pixel borders,
not shadows. Elevation is reserved for popovers, drawers, dialogs, tooltips, and
command search. Production workbench surfaces use no decorative gradients.

Default body text is 13–14px. Dense captions may use 11–12px but must retain
contrast and line height. Numeric values align consistently and use tabular
figures where available. Numeric value and unit spacing is consistent across
forms, status, measurement, and Project pages.

V18 does not add theming or a full dark mode. The mixed theme is one tested
design system, not a preference matrix.

## Workbench State Model

V18 frontend state distinguishes document truth from UI session state.

```ts
interface WorkbenchUiState {
  readonly mode: "project" | "solid" | "sketch" | "inspect";
  readonly activeTool?: UiActionId;
  readonly activeEditor?: {
    readonly kind: FeatureEditorKind;
    readonly sourceId?: string;
  };
  readonly activeEditorDirty: boolean;
  readonly navigationIntent?: WorkbenchNavigationIntent;
  readonly projectPage?: ProjectPageId;
  readonly selectionFilter: "body" | "face" | "edge";
  readonly leftDockCollapsed: boolean;
  readonly rightDockCollapsed: boolean;
  readonly leftDockWidth: number;
  readonly rightDockWidth: number;
  readonly commandSearchOpen: boolean;
}
```

Actual form values and collector drafts live in the mounted editor's local
reducer, not in `WorkbenchUiState`. The editor reports only identity,
clean/dirty transitions, validity, pending state, and apply/discard callbacks
to the shell. Field typing therefore rerenders the editor, not the engine/query
root or viewport. Collapsing a dock preserves the mounted editor.

UI state may select existing source/query objects by ID for the duration of the
session. It does not create new public identity and is never serialized into
the CAD document or command history.

### Mode transitions

- **Solid** is the default modeling mode.
- Create Sketch first opens its existing plane/name collector. Apply submits
  exactly one existing sketch-create command and enters **Sketch** for the
  returned sketch; Cancel before Apply mutates nothing. Editing an existing
  sketch enters **Sketch** directly.
- Finish Sketch returns to Solid. It does not itself mutate source.
- Escape cancels the current sketch gesture; it does not roll back previously
  committed sketch commands. Undo handles committed work.
- Measure, mass-properties, or explicit Inspect selection enters **Inspect**.
- Project/File, Parameters, History, Import, or Export enters **Project**.
- Feature creation/editing is a Solid substate with the feature editor open,
  not a fifth top-level mode.
- Invoking Measure enters Inspect and uses the current eligible selection as
  target 1; with no eligible selection it starts first-target capture. Invoking
  mass properties enters Inspect without starting a two-target session.
- Inspect adopts the current eligible Body/Face/Edge selection as its filter;
  otherwise Body is the default. A collector may temporarily narrow eligible
  kinds without rewriting the user's filter.

If a draft differs from committed source, New, Open, undo, redo, mode change,
feature change, editor close, or command-search navigation opens one
Apply/Discard/Stay prompt. **Apply** submits and continues only after success;
**Discard** drops local draft state and continues; **Stay** cancels navigation.
Escape closes a clean editor immediately and opens this prompt for a dirty
editor. The explicit Cancel button discards only local draft state and closes.

While a mutation is pending, source-mutating and stale-draft-producing actions
are disabled and duplicate submission is impossible. Camera navigation, hover,
selection inspection, copy, and the navigation prompt remain responsive. A
pending operation is announced beside its editor and in the status bar.

### Editor state machine

Every create/edit flow uses the same states:

```text
closed
-> collecting required selections
-> editing clean or dirty draft
-> validating
-> ready or blocked
-> applying
-> success and close, or error with draft preserved
```

Opening an editor, collecting targets, and changing fields never mutate source.
Apply validates and submits exactly one existing command or existing atomic
batch. Apply failure preserves every field and collector choice. A collector
keeps the right dock in editor mode, visibly names accepted target kinds, and
rejects an ineligible pick with a short reason without losing the draft.

All feature, tree, and Delete/Backspace deletion entry points open the same
confirmation dialog naming the semantic target. Confirm submits the one
existing delete command; Cancel makes no mutation. No UI invents a dependency
count when an existing query does not provide one.

### UI preference persistence

V18 stores a versioned app-only local preference record for:

- left/right dock widths;
- dock collapsed state.

The record is not part of JSON, `.wcad`, CADOps, OPFS, or transaction history.
It may not contain document IDs, selected references, feature drafts, active
tools, file handles, measurements, or diagnostics. Malformed or unknown
versions are ignored and reset to defaults.

## Mode Contracts

### Project mode

Project mode replaces the current raw Project/JSON-first experience with a
professional document workspace.

Required pages:

1. **Overview** — document name, the existing unit update with **Relabel
   values** and **Convert size**, current file state, compact model summary, and
   health/readiness summary from existing queries. This is the sole
   units-editing surface; ribbon/status units are read-only and route here.
2. **Files** — New, Open, Save, Save As, `.wcad` upload/download
   fallback, STEP import, JSON import/export under Advanced Interchange.
3. **Parameters** — existing parameter values, expressions, evaluation state,
   descriptions, create/edit/delete actions.
4. **History** — human-readable transaction groups with actor and sequence,
   semantic summary, undo/redo, and optional technical diff disclosure.
5. **Export** — exact STEP readiness and execution plus existing visualization
   mesh/GLB export with an accurate format label.

Project mode consumes the existing file and capability boundaries. It does not
persist file handles into source, add recent-file storage, change native
packages, or broaden STEP.

Project has one workflow owner. The left dock becomes page navigation; the
center contains exactly one active Overview, Files, Parameters, History, or
Export page; the right dock is closed by default and is used only for details of
a selected row. File/export fields and execution controls live in their active
center page, not simultaneously in the ribbon or right dock. The ribbon opens
the owning page and focuses the action; it does not duplicate the form.

**Save** writes the current file handle when the existing capability permits
and otherwise routes to **Save As**. **Save As** chooses/downloads a new `.wcad`
target and makes it current according to the existing file workflow. V18 does
not add a separate Save Copy semantic.

Raw project JSON is not visible by default. It remains available in a clearly
labeled Advanced Interchange disclosure because it is a supported compatibility
workflow, not the primary product interface.

### Sketch mode

Sketch mode is a dedicated precision environment. Automatic camera orientation
is limited to sketches whose frame maps exactly to the existing Top, Front, or
Right standard views. Other attached-face sketches retain the current camera;
V18 does not invent an arbitrary-face camera contract. The document tree and
later feature context remain visible.

Required ribbon groups:

| Group | V18 actions |
| --- | --- |
| Create | Point, Line, Rectangle, Circle, Three-point Arc |
| State | Construction toggle, Delete |
| Constraint | Horizontal, Vertical, Fixed, Coincident, Midpoint, Parallel, Perpendicular |
| Dimension | Supported rectangle width/height, line length, circle/arc radius, arc sweep |
| Finish | Finish Sketch |

Loaded Tangent, Concentric, Equal Radius/Length, Angle, and Symmetry records
remain visible and retain the existing rename/delete behavior, but are not
offered for creation or structural editing. Creation is limited to Horizontal,
Vertical, Fixed, Coincident, Midpoint, Parallel, and Perpendicular through the
existing builders.

The sketch workspace must show:

- source geometry with semantic hover and selection;
- construction styling;
- conservative inference cues;
- current dimensions and constraint glyphs;
- under/fully/over-defined or conflicting solver state;
- degrees of freedom where the existing query proves it;
- selected entity or constraint properties in the right dock;
- profile/path candidate health when relevant to a downstream action.

The existing three-point arc gesture remains:

```text
click start -> click point on arc -> click end -> command commit
```

Escape cancels its session state without mutation.

There is no **Cancel Sketch** action: entity edits are already committed
commands, Finish Sketch only exits the mode, and Undo reverses committed work.

V18 does not add Trim, Extend, Offset, sketch Mirror/Pattern, general drag
support, new dimensions, or new constraints.

### Solid mode

Solid mode is the ordinary 3D modeling workbench.

Required ribbon groups:

| Group | Existing capability exposed |
| --- | --- |
| Create | Primitive menu (Box, Cylinder, Sphere, Cone, Torus), Sketch, Extrude, Revolve, Sweep, Loft |
| Modify | Transform, Hole, Fillet, Chamfer, Shell |
| Pattern | Linear Pattern, Circular Pattern, Mirror |
| Inspect shortcut | Measure |

Tool visibility is stable; availability is contextual:

- **ready** — run immediately or open a collector with sufficient context;
- **needs selection** — remains invokable and opens the editor/collector with an
  explicit selection instruction;
- **blocked** — remains keyboard-focusable and invokable only to open the exact
  query-derived reason and recovery action; it uses `aria-disabled="true"` but
  not native `disabled`, and its handled activation cannot execute the command;
  and
- **pending** — is disabled, announced, and cannot submit twice.

An action absent from the completed inventory is not registered and is omitted
from ribbon, search, Help, menus, and contextual surfaces. Runtime/environment
failure is **blocked**, not hidden. The same context must produce the same state
and message in every surface.

Selecting a supported face, edge, body, sketch, or feature updates the right
inspector and compact contextual strip. The strip contains only high-frequency
actions named by the Slice A inventory for that exact semantic selection, in
stable registry order. It shows at most four actions plus **More**. **More**
opens the remaining entries from the same registry; it is not a separate source
of behavior.

### Active feature editing

Every existing create/edit feature flow moves into the right dock. Editors use
a shared structure:

1. Feature name and kind.
2. Required selection collectors.
3. Operation mode where supported.
4. Numeric and enum parameters accepted by the existing command.
5. Readiness/validation summary.
6. A sticky footer with labeled cobalt **Apply** and neutral **Cancel** actions.
7. Destructive Delete in a separated danger area for existing features.

The footer remains visible while editor fields scroll. Apply shows
Ctrl/Cmd+Enter in its tooltip. Green is reserved for validated/completed status,
not the commit control.

Selection collectors accept eligible viewport and tree selections and display
human labels. Their stored submission value is the exact existing source or
query-returned reference.

Required editor parity:

- primitives and transforms;
- sketch create/edit and sketch-on-face;
- entity and composite-profile extrude;
- entity and composite-profile revolve;
- hole;
- chamfer and fillet;
- linear and circular pattern;
- mirror;
- shell;
- entity-profile sweep with supported line/arc path;
- loft within the V16 section matrix;
- named-reference create and repair.

Editors may reorganize fields and replace verbose helper copy, but they may not
drop a supported field, operation mode, explicit candidate choice, or blocked
diagnostic.

### Preview policy

V18 adds no exact pre-commit geometry API.

Allowed preview:

- sketch gesture and entity draft overlays;
- direction, axis, depth, angle, count, spacing, and transform manipulators
  that can be derived from the form without deciding CAD validity;
- profile/path highlighting from existing query candidates;
- readiness and dry-run validation from existing boundaries.

These overlays are provisional renderer/UI state. They never become source,
measurements, topology evidence, or proof of kernel success.

If an exact result cannot be produced before commit by an existing boundary,
the current committed body remains visible and the editor says “Ready to
apply.” V18 must not draw a convincing but geometrically invented result.

### Inspect mode

Inspect mode consolidates selection details, measurement, mass properties,
reference actions, and camera tools.

Required groups:

- Measure and two-target measurement;
- Mass properties when matching derived exact metadata is available;
- Name reference;
- Repair named/topology reference through the completed V17 repair action,
  rendered ready or blocked from its shared readiness result;
- Fit All, Fit Selection, Top, Front, Right, and Isometric views;
- project/body/reference health summaries.

Only Body, supported Face, and supported Edge filters are active in the 3D
viewport. Vertex is not offered. Sketch mode uses semantic sketch-entity
selection rather than pretending those entities are 3D edges.

The right dock shows:

- selection type and human name;
- owning part/body/feature when available;
- measurement values with units and confidence phrased for users;
- supported command actions;
- reference health and recovery actions;
- mass properties;
- Technical Details disclosure.

Two-target measurement remains session-only. Pinning or persisting a
measurement is not in V18.

## Selection and Viewport Interaction

V18 retains the current semantic selection pipeline:

```text
canvas hit
-> renderer-agnostic pick intent
-> supported generated body/face/edge candidate
-> selection.referenceCandidates / readiness queries
-> semantic UI selection
-> eligible UI actions
```

Required visual states:

- hover/preselection;
- selected;
- active collector target;
- command target;
- provisional preview;
- warning/blocked;
- pending.

Color is not the only differentiator; outline weight, fill treatment, icon, and
text state participate. Sketch geometry uses solver coloring only where an
existing query proves entity-level state; otherwise solver state is summarized
in the inspector/status bar. Selection always has a distinct outline/halo, and
construction geometry retains a dash treatment independent of color.

Ordinary click replaces selection. Generic multi-select is not added. Empty
click clears ordinary selection but not a dirty feature draft. Two-target
measurement owns an explicit first/second-target session rather than implying
general multi-select. An active collector temporarily restricts eligible target
kinds and keeps the editor in the right dock; an ordinary pick does not replace
that editor. Tree and viewport selection reconcile after apply, delete, undo,
redo, import, and open.

Escape uses a deterministic cancellation stack:

1. close a transient popover;
2. close the active overlay drawer and restore focus;
3. cancel an active viewport gesture;
4. clear a measurement's second-target capture;
5. close command search;
6. close a clean editor or prompt for a dirty editor;
7. clear selection.

No step mutates source except an explicit existing command such as Delete.

## Command Search and Shortcuts

Command search is an accelerator over the shared action registry, not a chatbot
or natural-language command interpreter.

Search matches:

- action label;
- documented aliases;
- mode/group;
- shortcut.

Search is case-insensitive normalized substring plus token-prefix matching over
those fields; it adds no fuzzy-search dependency. Results order by exact label,
label prefix, alias prefix, then stable registry order. Each result shows its
mode, shortcut, and current availability. Running **needs selection** opens its
collector. Running **blocked** opens its reason/recovery detail and does not
bypass readiness. A result in another mode navigates through the same dirty-
draft guard before opening its owner.

Minimum shortcuts:

| Shortcut | Action |
| --- | --- |
| Ctrl/Cmd+K | Open command search |
| Ctrl/Cmd+Z | Undo |
| Ctrl/Cmd+Shift+Z or Ctrl+Y | Redo |
| Escape | Cancellation stack |
| Ctrl/Cmd+Enter | Apply the active valid feature draft |
| Delete/Backspace | Open confirmation for the current deletable selection |
| F | Fit all, when focus is not in an editable control |

Single-key shortcuts do not fire from inputs, selects, textareas,
content-editable regions, or search. All shortcuts are listed in Help and in
action tooltips.

## Diagnostics and Internal Text Policy

V18 translates structured diagnostics into:

1. a short title;
2. one sentence describing the user-visible problem;
3. a concrete recovery action when one exists;
4. optional Technical Details.

Examples:

| Structured condition | Default user copy |
| --- | --- |
| `SKETCH_PROFILE_OPEN` | **Profile is open.** Connect the highlighted endpoints before creating this feature. |
| `SKETCH_PATH_JOIN_NOT_TANGENT` | **Sweep path has a sharp corner.** Use a tangent line/arc join for this sweep. |
| consumed target | **Target body is no longer available.** Select a current result body. |
| stale named reference | **Reference needs repair.** Review the suggested compatible targets. |
| geometry worker unavailable | **Geometry display is unavailable.** Your model source remains saved; retry display generation. |

The default product surface must not expose:

- raw stable IDs or generated-reference tokens;
- CADOps operation names;
- schema versions;
- diagnostic enum codes;
- checkpoint entity IDs or hashes;
- renderer, mesh, triangle, worker, cache, GPU, or selection-buffer IDs;
- raw JSON;
- local paths, OPFS paths, or file handles;
- internal support-matrix prose.

Technical Details may show relevant structured data behind an explicit
disclosure for debugging and agent handoff. It is closed by default, visually
subordinate, copyable, and never required to understand the recovery action.

Errors appear next to the affected field or action. A global error region is
reserved for document-wide failures. Successful routine operations update the
status bar or a brief non-blocking notice instead of adding permanent text.

## Empty, Loading, and Failure States

- **Empty document:** render the normal Solid workbench, tree, primitive/Sketch
  actions, and a short status prompt. Do not replace the workbench with an
  onboarding card.
- **No selection:** show a quiet document summary in the inspector or collapse
  it; never show a blank error surface.
- **Geometry generating:** keep shell, tree, Project pages, camera, and command
  search usable. Show pending feedback by the next animation frame.
- **Geometry display failure:** preserve source, tree, selection, and file
  actions; show the existing retry path adjacent to the failure.
- **Unavailable exact metadata or STEP:** show the query-derived blocked reason
  and recovery action; do not synthesize partial success.
- **Malformed open/import:** leave the current document untouched and show the
  failure beside the file action.

## Accessibility and Input Contract

V18 Must includes:

- complete keyboard navigation for header, ribbon, tree, right dock, command
  search, dialogs, and status controls;
- visible focus for every interactive element;
- focus restoration after dialogs, command search, and feature apply/cancel;
- accessible names for icon buttons;
- semantic tree, tab, dialog, toolbar, status, and live-region roles;
- no color-only status;
- text contrast meeting WCAG AA;
- reduced-motion support;
- logical tab order matching the visual shell;
- form errors associated with their fields;
- pointer targets at least 28px in dense desktop mode.

Keyboard mechanics are normative:

- ribbon/toolbars use roving tabindex and arrow-key navigation;
- the tree supports Arrow Up/Down, Left/Right, Home/End, F2 rename, and
  Shift+F10 context menu;
- dock resizers use `role="separator"`, orientation/value attributes, 8px Arrow
  steps, and 32px Shift+Arrow steps;
- tooltips open on focus and hover and are programmatically associated;
- dialogs/drawers trap focus while modal and restore it to the opener;
- opening an editor focuses its heading or first incomplete collector;
- Apply/Cancel returns focus to the originating tree/ribbon action when it
  still exists; and
- pending/success uses a polite live region while blocking errors use an
  assertive region.

Every viewport-only workflow also has a tree, inspector, ribbon, or search path
when the same semantic target can be selected without a viewport pick.

V18 does not claim a mobile or touch-first CAD experience. At narrow desktop
widths, docks collapse to drawers and ribbon groups move to overflow while the
viewport remains usable. Existing narrow browser-workflow checks must not
regress or introduce scroll traps.

## Performance and Load Contract

Performance is a Must gate, not final polish. Slice A records the clean V17
production baseline; the observed planning baseline is 2,083.93kB raw /
483.97kB gzip eager main JavaScript, 53.27kB raw / 9.34kB gzip CSS,
923.18kB raw / 194.13kB gzip command worker, 360.71kB raw / 81.84kB gzip
geometry worker, and 50,305.13kB raw / 13,808.54kB gzip OCCT WASM.
The committed baseline uses a script rather than these rounded prose values.

### Measurement profile

- production build, clean Chromium profile, 1536x1024, DPR 1;
- 4x CPU throttle, 4Mbps download, 20ms RTT for cold-load measurements;
- five cold runs with the median reported;
- DPR 2 for interaction sampling;
- fixed representative V17 model plus a UI-only stress fixture with 200 visible
  tree rows and 100 history rows; and
- raw/gzip assets measured deterministically with Node/zlib; network,
  performance marks, frames, and long tasks measured through the existing CDP
  harness.

Metrics are written as JSON release artifacts. A checked-in threshold or
baseline change requires an explicit V18 plan amendment; implementation may not
refresh the file merely to make a regression pass.

### Bundle and loading budgets

- critical-path UI JavaScript (entry plus static imports required for an empty
  Solid shell/viewport) <=400KiB gzip;
- critical-path CSS <=20KiB gzip;
- all non-worker UI JavaScript, eager plus lazy chunks, <=550KiB gzip;
- command worker, geometry worker, and OCCT artifacts do not grow from the
  committed Slice A baseline; and
- Project/Advanced Interchange and non-critical editor groups are absent from
  the initial eager graph and load on navigation/intent.

A fresh empty-project load requests no command worker, geometry worker, or OCCT
WASM. The command worker starts on the first source mutation and is disposed on
unmount/HMR. Geometry worker/WASM starts only when an existing derived-geometry,
import, or export path actually requests it. V18 removes the hard-coded 75ms
production command-worker delay; pending-state tests use injected delayed
fakes. The shell never waits for kernel initialization.

### Runtime budgets

- median cold `partbench:shell-ready` <=2.0s in the measurement profile;
- command-search keystroke-to-results p95 <=50ms;
- warm mode change, tree expansion, dock open, and semantic
  selection-to-tree/inspector agreement p95 <=100ms;
- pending feedback appears by the animation frame after invocation;
- hover, orbit, and dock-resize samples have p95 frame interval <=34ms and no
  main-thread task >50ms during a two-second interaction; and
- kernel/WASM execution is reported separately and is not charged to UI
  response, while its pending UI remains subject to the budgets above.

### Hot-path rules

- coalesce hover, orbit, and divider movement to at most one UI update per
  animation frame;
- notify React parents only when the semantic hover/selection target changes;
- update divider position through a CSS variable while dragging and commit
  reducer/local-preference state only on pointer/keyboard completion;
- compute query-backed projections once per document revision, derived-result
  revision, and semantic selection change; camera, hover, typing, and divider
  movement issue no readiness/document queries;
- form typing and dock movement do not rerender the viewport canvas;
- render tree children only for expanded branches; first test the 200-row
  fixture without virtualization and add repository-local windowing only if
  that fixed gate fails; and
- cap canvas backing DPR at 2.

Code splitting is budget-driven: keep the shell and viewport eager, use coarse
mode/editor chunks, and do not hand-author a chunk per control or add a bundling
framework, service worker, state library, fuzzy-search package, or virtualization
dependency.

1. **Minimal canvas with only contextual actions.** Rejected: hides capability
   from expert users and makes discovery selection-dependent.
2. **Guided Sketch -> Shape -> Refine wizard.** Rejected: parametric CAD is
   iterative, not a one-way phase sequence.
3. **Command search as the primary UI.** Rejected: useful accelerator, poor
   substitute for visible tools, selection collectors, and feature history.
4. **Reskin the existing panels without changing composition.** Rejected: the
   core problem is duplicated workflow ownership and permanent internal text,
   not color tokens.
5. **Clone one incumbent CAD application literally.** Rejected: Partbench needs
   its own command/query-aligned shell and cannot imply unsupported breadth.
6. **Implement every control shown in the concept images.** Rejected: mockup
   completeness cannot override the V17 capability contract.
7. **Put UI mode or layout state in the CAD document.** Rejected: presentation
   state is not model source and would trigger needless schema/collaboration
   consequences.
8. **Build an arbitrary docking framework.** Rejected: large scope and
   interaction risk without improving the selected stable-shell direction.
9. **Add a UI framework and icon library by default.** Rejected: existing React
   and CSS are sufficient; dependency cost is not justified for this tranche.
10. **Rewrite backend APIs to fit idealized forms.** Rejected: V18 is a
    frontend-only consolidation release.
11. **Keep old and new shells behind a feature flag.** Rejected: it duplicates
    ownership, bundle cost, tests, and cleanup. Each V18 slice replaces and
    deletes its legacy surface directly.

## Risk Controls

| Risk | Required control |
| --- | --- |
| A supported V17 workflow disappears during cleanup | Typed capability manifest plus parity test before same-slice legacy deletion |
| UI registry becomes a second command engine | Registry delegates to existing builders/handlers and reads existing readiness queries |
| Concept image implies future support | Capability-truth table and action inventory govern implementation |
| Dense UI recreates clutter | Stable regions, one right-dock task, compact hierarchy, no duplicated permanent panels |
| Contextual actions disagree with ribbon | Both render the same action definitions and availability results |
| Draft fields mutate source early | Explicit local draft state and command-spy tests for Apply/Cancel |
| Human copy hides necessary recovery evidence | Actionable default message plus optional Technical Details |
| Internal vocabulary leaks back in | Central diagnostic translator and visible-text blacklist tests |
| App remains a monolith | Container/projection boundaries and legacy component retirement gate |
| UI polish slows startup or pointer work | Immutable bundle/runtime budgets, lazy workers/modes, isolated hot-path state, and CDP performance gate |
| Responsive behavior steals the viewport | Dock min/max rules, overflow policy, and narrow browser assertions |
| Browser visual tests become brittle | Semantic/layout assertions; reference images remain human-review artifacts |
| Frontend work weakens geometry/storage behavior | Full V17 release workflow compatibility gate |

## Final Compatibility Statement

V18 changes how users reach and understand completed capabilities; it does not
change what Partbench models or stores.

V1–V17 documents and commands retain their current behavior. V17 arcs,
construction geometry, explicit composite profiles/paths, composite extrude
and revolve, curved sweeps, topology-backed downstream modeling, patterns,
mirror, shell, loft, parameter expressions, mass properties, STEP,
JSON/`.wcad`, agents, and MCP remain governed by their completed release
records.

The V18 promise is a professional frontend over those contracts: stable,
dense, explicit, keyboard-fast, selection-aware, and quiet about implementation
details until a power user deliberately asks to inspect them.
