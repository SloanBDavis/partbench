# V11 Planning Record: Full Sketch Solver And Parametric Sketch UX

Status: in progress as of 2026-06-19. V11 is the planned full sketch solver and
parametric sketch interaction release. Tranches A, B, C, D1, D2, D3A, and D3B
implement the first solver/status/source-contract surface, pure solver package
foundation, V17 advanced-constraint source migration, cad-core solver mapping,
core command/status alignment, parallel/perpendicular numerical support, and
concentric/equal-radius numerical support. It follows the completed V10
editable feature history and stable modeling references release.

V11 is a modeling-authority release, not a renderer or platform release. The
release center is:

```text
sketch source geometry -> constraints and dimensions -> solver result ->
profile validity -> feature rebuild impact -> reference health ->
viewport and inspector sketch editing that stay command-backed
```

## Product Goal

Partbench should feel like a real parametric CAD sketcher:

- users can draw and edit points, lines, rectangles, circles, arcs, and
  construction geometry in a sketch;
- constraints are inferred, displayed, added, edited, deleted, and diagnosed;
- dimensions are first-class driving source data, not labels layered on top of
  geometry;
- dragging sketch geometry previews a solved result before commit;
- the app explains under-defined, fully-defined, over-defined, conflicting, and
  redundant states;
- sketches produce valid profiles for downstream features;
- editing a sketch updates supported downstream features through the V10 rebuild
  and reference-health contracts;
- UI, agents, MCP, tests, and future viewport tools use the same CADOps path.

The user-visible difference should be direct sketch manipulation. A user should
be able to open a sketch, drag an endpoint, add a tangent or concentric
constraint, edit a dimension, and understand exactly why the sketch solves or
fails.

## Why This Is The Next Major Release

V7 made semantic references useful. V8 made local project files, cache, and
exact export real. V9 made viewport selection and compact CAD interaction
semantic. V10 made feature edits, rebuild impact, dependency health, and named
reference repair explicit.

The next bottleneck is sketch design intent. Current Partbench has source
sketches, some dimensions, and a limited constraint evaluator, but it does not
yet have a general solver. That limits:

- robust sketch editing after features exist;
- direct manipulation in the viewport;
- profile validity diagnostics;
- agent-authored parametric design;
- downstream rebuild reliability;
- future assemblies and feature families that depend on stable design intent.

V11 should remove that bottleneck without touching unrelated architecture
areas.

## Architecture Alignment

V11 follows `docs/architecture.md`:

- the command protocol remains the center of the system;
- `cad-core` remains the authority for document source, constraints,
  dimensions, solver status, profile validity, feature rebuild impact, and
  semantic diffs;
- the renderer and React UI remain consumers of display/session state and do
  not infer constraint validity or commandability;
- rendered meshes, hit candidates, selection-buffer IDs, viewport pixels, OPFS
  cache entries, STEP/GLB artifacts, and browser file handles never become
  source sketch or solver identity;
- solver output is exact/model semantic data first, mesh second;
- solver diagnostics are structured for humans and agents;
- MCP and agent adapters stay thin wrappers over CADOps;
- no GPL sketch solver dependency is introduced;
- the solver is custom and isolated behind a package/worker-ready boundary so
  implementation details can evolve without changing public CADOps semantics.

The architecture recommends a small custom 2D constraint solver. V11 should
implement that direction. If an early tranche uses TypeScript for iteration,
that code must sit behind the same solver package boundary and must not define
public APIs that prevent a later Rust/C++/WASM core.

## Implemented Tranche A Contract

V11 Tranche A adds the first shared solver vocabulary without introducing the
numerical solver, broad sketch editing UI, or `web-cad.project.v17`.

Implemented surface:

- `sketch.solverStatus` reports current solve/readiness status from
  authoritative sketch source;
- supported current source entities are summarized as solver participants for
  points, lines, rectangles, and circles;
- current driving dimensions and source-backed constraints are summarized with
  durable sketch entity/point/dimension/constraint target references;
- profile validity is reported for simple source-proven closed rectangle and
  circle profiles, while open profile entities receive structured diagnostics;
- tangent, equal length, distance, angle, and symmetry were reserved as V11
  contract cases in Tranche A; later D3 tranches move equal length and angle
  into numerical solver support while distance remains a contract placeholder;
- dry-run/preview result shapes are present but marked deferred;
- the source contract explicitly keeps query-only work on
  `web-cad.project.v16` and reserves `web-cad.project.v17` for the first tranche
  that commits new source-of-truth solver data;
- agent and MCP surfaces pass through the same cad-core query without adding
  separate solver authority.

Tranche A deliberately does not solve geometry numerically. It translates the
current source-backed sketch evaluator into the V11 contract so later solver,
migration, UI, and rebuild tranches have one vocabulary to build on.

## Implemented Tranche B Solver Package Foundation

V11 Tranche B adds `packages/sketch-solver` as the first pure numerical solver
boundary. It is intentionally not a cad-core source migration and does not
change saved project schema.

Implemented foundation:

- normalized `partbench.sketch-solver.v1` solve models with 2D point variables,
  scalar variables, source constraint records, driving dimension records,
  deterministic settings, and structured solve results;
- a dependency-free damped least-squares/Gauss-Newton style numerical core with
  finite-difference Jacobians;
- support for fixed point, coincident point, horizontal line, vertical line,
  midpoint, parallel, perpendicular, line-circle tangent, concentric, equal
  radius, equal length, angle, point-pair symmetry, point distance, line
  length, and circle radius residuals;
- deterministic statuses for converged, under-defined, over-defined,
  conflicting, failed, unsupported, and not-run cases;
- structured diagnostics for missing targets, invalid values, unsupported
  constraints, non-convergence, conflicts, under-defined sketches, and
  over-defined sketches;
- explicit structured diagnostics for unsupported target combinations and
  invalid solver targets;
- no React, renderer, geometry-worker, OCCT, OPFS, browser file, MCP, agent, or
  cad-core dependency.

Tranche B deliberately does not commit solved geometry into the document model,
does not replace Tranche A's current cad-core sketch source evaluator, and does
not introduce `web-cad.project.v17`. The solver package consumes normalized
solve models so later cad-core and worker tranches can map authoritative sketch
source into it without making the solver package a document authority.

## Implemented Tranche C V17 Solver Source Migration

V11 Tranche C introduces `web-cad.project.v17` only for real new solver-grade
sketch source data. Ordinary projects without V17 source records still export
as `web-cad.project.v16`.

Implemented source contract:

- `web-cad.project.v17` is accepted by the project JSON and `.wcad`
  read/write paths without changing `.wcad` package version
  `partbench.wcad.v1`;
- V17 preserves all V16 source data and adds advanced sketch constraint source
  records in the existing `document.sketchConstraints` collection;
- V17 advanced constraint records cover `tangent`, `concentric`,
  `equalLength`, `equalRadius`, `angle`, and `symmetry`;
- tangent constraints store `primaryTarget` and `secondaryTarget` curve
  targets with durable sketch entity IDs and explicit `line` or `circle`
  target kinds;
- concentric and equal-radius constraints store primary/secondary circle entity
  IDs;
- equal-length and angle constraints store primary/secondary line entity IDs,
  with angle storing a finite `angleDegrees` value greater than 0 and less than
  180;
- symmetry constraints store primary/secondary durable sketch point targets and
  a symmetry line entity ID;
- imported V17 advanced constraints are preserved through JSON, `.wcad`, source
  identity, semantic diff/reference summaries, solver status, project health,
  and sketch edit impact queries;
- V16 projects remain accepted and V16 projects containing advanced constraints
  are rejected with structured import diagnostics;
- `sketch.solverStatus` reports V17 advanced source records as source-backed,
  with D3D numerical support for line-circle tangent, concentric,
  equal-radius, equal-length, angle, and point-pair symmetry records; unsupported
  target combinations such as line-line tangent stay structured diagnostics.

Tranche C deliberately does not add UI or command creation for advanced
constraints, does not commit solved geometry, does not make advanced constraints
command-ready, and does not move solver authority into React, renderer,
geometry-worker, OCCT, agents, or MCP.

## Non-Goals

V11 should not implement:

- WebGPU production rendering;
- STEP import;
- assemblies, mates, large-model LOD, or instance selection;
- broad arbitrary topological naming;
- persistent exact B-rep checkpoints as project source;
- new solid modeling families such as shell, sweep, loft, pattern, mirror, or
  direct edit unless explicitly scoped later;
- hosted collaboration;
- natural-language command entry as a product surface;
- production MCP auth;
- persisted viewport camera, hover, selection, active tool, or measurement
  session state.

V11 may use existing viewport selection and sketch display surfaces, but it
should not turn the renderer into a sketch authority.

## Major V11 Decisions

### 1. V11 Is A Full Sketch Solver Release

V11 is not a narrow "add one constraint" milestone. It is the release that makes
sketch constraints, dimensions, solver state, profile validity, and direct
sketch editing one coherent CAD system.

The release can still land in tranches. Each tranche must be independently
useful and tested, but the end state should be a complete solver-backed sketch
workflow for the supported 2D entity and constraint set.

### 2. Source Data Lives In The Document

Sketch entities, construction flags, dimensions, constraints, constraint
labels, and persistent solver intent are source-of-truth document data.

Solver iteration state, Jacobians, residual caches, temporary drag previews,
hover state, selected handles, viewport pixels, and renderer hit IDs are
derived/session data.

### 3. V11 Will Need A Project Schema Migration

Current saved projects use `web-cad.project.v16`. V11 should introduce
`web-cad.project.v17` only when the first new source-of-truth solver data is
added. Query-only readiness tranches must not bump the schema.

Current V17 source additions:

- solver-grade advanced constraint records for tangent, concentric, equal
  length, equal radius, angle, and symmetry source intent.

Still-deferred possible V17-or-later source additions:

- construction geometry flags where not already source-backed;
- persistent constraint labels or dimension display names if they are part of
  design intent;
- solver settings that change source interpretation;
- sketch-level solve policy when it affects future rebuilds.

V17 must remain additive and migratable from V16. `.wcad` package version
`partbench.wcad.v1` does not need to change unless package layout changes; it
can contain a V17 document.

### 4. Constraint Identity Is Semantic

Constraint IDs must be stable source IDs. They must not encode:

- solver row indexes;
- matrix variable indexes;
- renderer IDs;
- canvas pixels;
- hit-test IDs;
- OCCT internal IDs;
- mesh buffer IDs;
- OPFS cache paths;
- file handles.

Constraint targets must use durable sketch entity or sketch point references.
Where an entity exposes multiple points, the target must identify the intended
role, such as line start, line end, circle center, arc start, or arc end.

### 5. The Solver Must Explain Its Result

A successful solve is not enough. Every solve response should report:

- status: solved, under-defined, fully-defined, over-defined, conflicting,
  redundant, failed, unsupported, or not-run;
- degrees of freedom where computable;
- constrained and unconstrained entity summaries;
- blocked constraints and dimensions;
- conflicting or redundant constraint groups;
- numerical convergence diagnostics;
- whether the resulting profile can drive features;
- downstream rebuild impact through V10 contracts.

The UI should show a compact summary and keep deeper diagnostics inspectable.
Agents and MCP should receive the full structured result.

### 6. Inference Is Helpful, Not Authoritative

The app may infer constraints while drawing or dragging, but inference becomes
source only after it is committed through CADOps. Preview inference is session
state.

Inference should start conservative:

- coincident endpoint snapping;
- horizontal and vertical line hints;
- midpoint hints;
- parallel and perpendicular hints;
- tangent and concentric hints when confidence is high.

The user should be able to accept, ignore, or delete inferred constraints.

### 7. Dimensions Are Driving Parameters

V11 dimensions should be source-backed, editable through CADOps, and integrated
with solver variables. They should not be measurement annotations.

Supported dimension classes:

- point-to-point distance;
- line length;
- horizontal distance;
- vertical distance;
- circle radius;
- circle diameter if represented as a radius-driving dimension;
- angle between lines;
- arc radius and arc angle once arcs are in the supported entity set.

Parameter expressions remain deferred unless explicitly scoped later. Numeric
dimension values are enough for V11.

### 8. Sketch Dragging Is A Dry-Run Then Commit

Viewport sketch dragging should use this flow:

```text
pointer drag -> private viewport handle -> sketch drag intent ->
solver preview dry-run -> compact diagnostics -> CADOps commit
```

Preview results may update temporary display geometry. They must not mutate the
authoritative document until commit. Failed previews should leave source
unchanged and return structured diagnostics.

### 9. Profile Validity Feeds Feature Rebuild

Sketch solve status and profile validity must feed V10 feature editability,
dependency graph, reference health, and rebuild plan behavior.

If a sketch edit makes a downstream feature invalid, the app should report that
as a source/rebuild problem, not as a renderer failure.

### 10. The UI Should Be CAD-Native And Compact

V11 should improve the product surface, but not with large panels or debug
walls. Sketch UI should prioritize:

- inline constraint glyphs;
- inline dimension labels;
- compact toolbars;
- clear selected-entity controls;
- restrained diagnostics;
- left panel Inspector/Selection integration;
- unobstructed viewport editing.

Deep solver diagnostics belong in inspectable disclosures, not blocking the
modeling view.

## Supported Source Entity Set

V11 should support these sketch entities as source-backed solver participants:

- point;
- line segment;
- rectangle as either an authored rectangle entity or a decomposed constrained
  line loop, depending on current source compatibility;
- circle;
- arc;
- construction line;
- construction circle;
- construction point.

Ellipse, spline, imported curves, projected model edges, and equation-driven
curves remain deferred unless explicitly scoped.

## Supported Constraint Set

V11 should target the architecture constraint set:

- coincident;
- horizontal;
- vertical;
- parallel;
- perpendicular;
- tangent;
- concentric;
- equal length;
- equal radius;
- distance;
- angle;
- midpoint;
- symmetry;
- fixed.

Current implemented constraints should be folded into the unified solver rather
than maintained as a parallel evaluator:

- horizontal/vertical line orientation;
- fixed point;
- coincident point;
- midpoint point;
- parallel line;
- perpendicular line.

The solver should return structured unsupported diagnostics for any target or
constraint combination outside the implemented subset during incremental
tranches.

## Solver Model

### Variables

The solver should model variables for supported source geometry:

- point x/y coordinates;
- line endpoint coordinates;
- circle center coordinates;
- circle radius;
- arc center coordinates, radius, start angle, and end angle;
- construction entity variables when they participate in constraints.

### Residuals

Each constraint should compile to one or more residual equations. Examples:

- coincident: point A minus point B;
- horizontal: y1 minus y2;
- vertical: x1 minus x2;
- parallel: cross product of normalized direction vectors;
- perpendicular: dot product of direction vectors;
- tangent: distance from circle center to line or circle center distance
  relation;
- concentric: center point difference;
- equal radius: radius difference;
- distance: distance minus target value;
- angle: angle between vectors minus target value;
- midpoint: point minus segment midpoint;
- fixed: variable minus source fixed value.

### Solving

The architecture calls for damped least squares / Levenberg-Marquardt style
solving. V11 should use that family of numerical method behind a deterministic
solver boundary.

Required behavior:

- deterministic output for the same input and initial guess;
- bounded iteration count;
- numeric tolerance constants documented in code and tests;
- clear convergence status;
- no dependence on render frame timing;
- no dependence on browser-only APIs in pure solver tests.

### Diagnostics

The solver should report:

- input validation failures;
- missing targets;
- unsupported entity or constraint types;
- unsatisfied residual count and max residual;
- iteration count;
- convergence state;
- degrees of freedom estimate where available;
- likely redundant constraints;
- likely conflicting constraint groups;
- under-defined entity summaries.

Diagnostics can improve over time, but V11 should not ship a solver that merely
says "failed."

## Protocol And CADOps Model

V11 should add or extend typed commands and queries around these surfaces:

- `sketch.solverStatus` or equivalent sketch solve query;
- `sketch.profileValidity`;
- `sketch.solvePreview` for non-mutating drag/edit preview;
- `sketch.constraint.create`;
- `sketch.constraint.update`;
- `sketch.constraint.delete`;
- `sketch.dimension.create`;
- `sketch.dimension.update`;
- `sketch.dimension.delete`;
- `sketch.entity.dragPreview`;
- `sketch.entity.dragCommit` or an equivalent typed update command;
- `sketch.entity.setConstruction`;
- `sketch.inferenceCandidates`.

Existing command names should be reused where they already exist and are
semantically correct. New commands should return semantic diffs and structured
diagnostics.

## Cad-Core Responsibilities

`cad-core` owns:

- source sketch records;
- source constraint and dimension records;
- schema migration;
- command validation;
- solve queries and solve previews;
- committed sketch edits;
- profile validity;
- feature rebuild impact summaries;
- reference health effects;
- semantic diffs;
- undo/redo;
- JSON and `.wcad` source round-trips.

`cad-core` may call a solver package, but the source model and command
semantics remain in `cad-core`.

## Solver Package Boundary

V11 should introduce or formalize a solver boundary, likely
`packages/sketch-solver`.

The package should be pure and testable:

- no React imports;
- no renderer imports;
- no geometry-worker dependency unless explicitly routed through a worker
  bridge;
- no OCCT dependency for 2D constraint solving;
- no browser file or OPFS dependency;
- no MCP-specific logic.

The public package inputs should be normalized sketch solve models, not full
Partbench documents. The outputs should be typed solve results and diagnostics.

## Worker Boundary

V11 can keep early solve calls synchronous if the supported sketches are small,
but the API must be worker-ready. Large sketches and drag previews should not
force React to own solver logic.

If a worker is added:

- app/cad-core still prepares authoritative solve requests;
- the worker returns derived solve results;
- the worker does not mutate document source;
- stale worker responses are ignored if the source identity or preview token no
  longer matches.

## UI Model

### Sketch Mode

V11 should make sketch mode an explicit product state:

- active sketch shown in the tree and viewport;
- visible sketch entities and constraints;
- current tool state session-only;
- escape behavior clears transient tool state before leaving sketch mode;
- source commits go through CADOps.

### Constraint Toolbar

The toolbar should expose common constraints with icons and tooltips:

- coincident;
- horizontal;
- vertical;
- parallel;
- perpendicular;
- tangent;
- concentric;
- equal;
- midpoint;
- symmetry;
- fixed.

Unavailable constraints should be disabled with structured reasons from
cad-core or helper validation, not hidden because React guessed.

### Dimension Editing

Dimension labels should be directly editable where the query says they are
driving dimensions. Editing a dimension should:

1. run a dry-run solve;
2. show compact success/failure state;
3. commit through CADOps;
4. update profile validity and downstream rebuild impact.

### Dragging

Dragging should prioritize:

- endpoint dragging;
- line dragging;
- circle center dragging;
- circle radius dragging;
- arc handles once arcs are supported.

The viewport may show preview geometry during drag, but the source document
changes only after commit.

### Diagnostics

The UI should show:

- sketch status: under-defined, fully-defined, over-defined, conflict, failed;
- compact constraint conflict rows;
- entity-level warnings;
- feature impact warnings;
- one-click access to the affected feature/body/reference where possible.

Diagnostics should not block half the viewport.

## Agent And MCP Model

Agents and MCP should use the same typed CADOps commands and queries:

- inspect solver status;
- list constraints and dimensions;
- propose constraints;
- dry-run dimension edits;
- dry-run drag or entity edits where useful;
- commit safe edits;
- inspect downstream feature/rebuild/reference impact.

Wrappers should remain thin. No agent/MCP-specific solver authority should be
introduced.

## Storage And Native Format

V11 source data must round-trip through:

- JSON import/export;
- `.wcad` packages;
- command history where current command history is authoritative;
- undo/redo.

OPFS may cache derived solve artifacts only if explicitly scoped. Such cache
entries must be rebuildable and must not be required to open a project.

STEP/GLB exports are unaffected unless a sketch edit changes the modeled body
they export.

## Performance Targets

V11 should be designed for:

- small sketches solving interactively during drag;
- medium sketches with dozens of entities and constraints producing clear
  diagnostics;
- stress fixtures with hundreds of constraints used as non-product benchmark
  cases;
- bounded solve time and deterministic failure modes.

Do not optimize before correctness and diagnostics are reliable.

## Validation Strategy

V11 needs more than unit tests for individual commands. It should add:

- pure solver fixtures for every supported constraint type;
- conflict and redundancy fixtures;
- under-defined and fully-defined fixtures;
- numeric tolerance tests;
- deterministic repeated-run tests;
- schema migration and round-trip tests;
- CADOps dry-run non-mutation tests;
- undo/redo tests;
- downstream feature rebuild impact tests;
- reference health tests after sketch edits;
- agent/MCP pass-through tests for any new surfaces;
- browser smokes for sketch edit workflows.

Browser smoke chains should cover combinations:

1. create sketch -> constrain -> dimension -> extrude -> edit dimension ->
   rebuild -> save/open `.wcad`;
2. create attached sketch -> infer constraints -> cut -> edit source sketch ->
   downstream rebuild diagnostics;
3. create conflicting constraints -> inspect diagnostics -> repair constraint
   -> verify feature/profile validity;
4. drag endpoint preview -> failed solve non-mutation -> successful commit ->
   undo/redo.

## Proposed Tranche Sequence

### Tranche A: Sketch Solver Protocol And Source Contract

Implemented as the first query-only `sketch.solverStatus` surface. It adds
typed sketch solve, constraint, dimension, profile-validity, preview, source
contract, and diagnostic shapes. It decides the likely V17 source schema
additions, but does not write V17 projects because no new source data is
committed.

Definition of done:

- protocol and cad-core query shapes are typed;
- source/session/derived boundaries are tested;
- unsupported constraints return structured diagnostics;
- no renderer/mesh/OCCT/viewport/OPFS/file-handle IDs leak into solver IDs;
- no schema bump for query-only work.

### Tranche B: Solver Package Foundation

Implemented as `packages/sketch-solver`. It creates the custom 2D solver package
boundary with deterministic solve inputs, variables, residuals, solve results,
tolerances, and diagnostics. The package supports fixed point, coincident,
horizontal, vertical, midpoint, point distance, line length, and circle radius
residuals. Advanced constraints remain structured unsupported/deferred cases.

Definition of done:

- pure solver tests cover coincident, horizontal, vertical, distance, radius,
  and fixed basics;
- repeated solves are deterministic;
- failures return structured diagnostics;
- no UI, renderer, OCCT, storage, or MCP dependency.

### Tranche C: V17 Constraint And Dimension Source Migration

Implemented as the first saved-source migration for solver-grade advanced
constraint intent. V17 is emitted only when advanced sketch constraint source
records are present; V16 remains the normal export schema for projects without
new solver source records.

Definition of done:

- JSON and `.wcad` round-trips preserve V17 advanced constraint source records;
- old V16 projects load safely and keep V16 export behavior;
- command history, undo/redo, and source identity stay deterministic;
- package version remains unchanged;
- advanced constraint records are preserved as source intent; later D3
  tranches add numerical support without changing the V17 source shape.

### Tranche D1: Cad-Core Solver Package Mapping And Core Solver Status

Implemented as the first cad-core integration with `packages/sketch-solver`.
`sketch.solverStatus` now builds a normalized solver-package model from
authoritative sketch source for the first safe subset: point, line, rectangle,
and circle variables; fixed, coincident, horizontal, vertical, and midpoint
constraints; and line length and circle radius dimensions. It reports whether
the model was built, whether the pure solver ran, the solver package model
version, numerical status, residual counts, and diagnostics.

The result is status-only. Solver output does not overwrite sketch geometry,
does not commit solved coordinates, does not introduce another schema, and does
not make React, renderer, geometry-worker, OCCT, OPFS, file handles, or viewport
state part of solver authority. Rectangle width/height dimensions remain
source-backed but not mapped to the numerical package in D1. Later D3 tranches
add numerical support for parallel/perpendicular and V17 advanced constraint
records without changing source schema.

Definition of done:

- cad-core can build normalized solver inputs for the supported current sketch
  subset;
- `sketch.solverStatus` includes structured numerical solver package status;
- unsupported/deferred constraints are explicit;
- query behavior is non-mutating;
- V16/V17 JSON and `.wcad` behavior remains unchanged.

### Tranche D2: Core Constraint Commands And Solver Status

Implemented as the command/status alignment tranche for the current core sketch
source subset. Existing CADOps paths for fixed point, coincident point-point,
horizontal line, vertical line, midpoint point-on-line, parallel line,
perpendicular line, line length dimension, and circle radius dimension now have
focused solver-status coverage through commit, dry-run, failure, undo/redo,
JSON/`.wcad` round-trip, agent, and MCP paths.

No solved geometry is committed by the numerical solver in D2. Cad-core remains
the source mutation authority: commands update authoritative sketch source and
derived entity geometry through the current CADOps/evaluator paths, then
`sketch.solverStatus` rebuilds the normalized solver-package model from source.
Parallel and perpendicular remain source-backed current constraints. D3A moves
them from solver-deferred status into real pure-solver line-pair residual
support without changing command source records. Point-distance dimensions
remain deferred because the current public source schema does not have a
durable point-distance dimension target.

Definition of done:

- commands go through CADOps;
- solver status updates after commits;
- dry-run and failed commands do not mutate source;
- feature editability/rebuild/reference health remain consistent;
- agent/MCP callers use the same `cad.batch` and `sketch.solverStatus` paths.

### Tranche D3A: Parallel And Perpendicular Numerical Solver Support

Implemented as the first D3 numerical solver expansion. The pure
`packages/sketch-solver` package now supports source-backed parallel and
perpendicular line constraints using normalized line-direction residuals.
Cad-core maps existing parallel/perpendicular source records into the solver
package model, and `sketch.solverStatus` reports them as numerically supported
rather than unsupported/deferred.

This remains status-only. Solver output does not overwrite sketch geometry, no
new CADOps mutation behavior is added, no project schema after
`web-cad.project.v17` is introduced, and `.wcad` package layout remains
unchanged. V17 advanced constraints beyond parallel/perpendicular remain
explicit unsupported/deferred cases until later D3 slices.

Definition of done:

- pure solver fixtures cover parallel, perpendicular, missing target,
  zero-length line, and conflicting line-pair cases;
- cad-core `sketch.solverStatus` maps source-backed parallel/perpendicular
  constraints into the numerical solver package;
- remaining advanced V17 constraints remain unsupported/deferred at the D3A
  boundary;
- source/session/derived boundaries and JSON/`.wcad` compatibility are
  unchanged.

### Tranche D3B: Concentric And Equal Radius Numerical Solver Support

Implemented as the second D3 numerical solver expansion. The pure
`packages/sketch-solver` package now supports source-backed concentric circle
constraints using center point residuals and equal-radius circle constraints
using scalar radius residuals. Cad-core maps V17 concentric/equalRadius source
records into the normalized solver package model, and `sketch.solverStatus`
distinguishes numerical solver support from direct evaluator support through
the constraint summary.

This remains status-only. Solver output does not overwrite sketch geometry, no
advanced constraint creation commands are added, no project schema after
`web-cad.project.v17` is introduced, and `.wcad` package layout remains
unchanged. Tangent, equal length, angle, and symmetry remain explicit
unsupported/deferred cases at the D3B boundary and are addressed by later D3
slices.

Definition of done:

- pure solver fixtures cover concentric, equal radius, missing target,
  non-positive radius, and conflicting circle-pair cases;
- cad-core `sketch.solverStatus` maps V17 concentric/equalRadius records into
  the numerical solver package;
- tangent, equal length, angle, and symmetry remain unsupported/deferred at the
  D3B boundary;
- source/session/derived boundaries and JSON/`.wcad` compatibility are
  unchanged.

### Tranche D3C: Equal Length And Angle Numerical Solver Support

Implemented as the third D3 numerical solver expansion. The pure
`packages/sketch-solver` package now supports source-backed equal-length line

### Tranche D3D: Tangent And Symmetry Numerical Solver Support

Implemented as the fourth D3 numerical solver expansion. The pure
`packages/sketch-solver` package now supports source-backed line-circle
tangent constraints using line distance to circle radius residuals and
point-pair symmetry constraints using mirror residuals across a source line.
Cad-core maps V17 line-circle or circle-line tangent records and V17 symmetry
records into the normalized solver package model, and `sketch.solverStatus`
reports supported target combinations as numerically supported while keeping
direct evaluator support separate.

This remains status-only. Solver output does not overwrite sketch geometry, no
advanced constraint creation commands are added, no project schema after
`web-cad.project.v17` is introduced, and `.wcad` package layout remains
unchanged. V17 tangent target combinations outside the supported line-circle
subset, such as line-line or circle-circle tangent, return structured
unsupported diagnostics rather than being silently ignored.

Definition of done:

- pure solver fixtures cover tangent, symmetry, missing target, invalid radius,
  zero-length line, and conflicting cases;
- cad-core `sketch.solverStatus` maps V17 line-circle tangent and symmetry
  records into the numerical solver package;
- unsupported target combinations return structured diagnostics;
- source/session/derived boundaries and JSON/`.wcad` compatibility are
  unchanged.

### Tranche E: Profile Validity And Feature Rebuild Integration

Implemented as the query integration slice between solver/profile status and
V10 rebuild/reference surfaces. Closed source rectangle/circle profiles can be
feature-ready when the numerical solver supports the sketch, even if legacy
direct evaluator support is still narrower. Stale, missing, or unsupported
source profile health now blocks feature editability, marks generated and named
references non-commandable, feeds dependency graph node status, and contributes
rebuild lifecycle effects without committing solved geometry.

Definition of done:

- valid and invalid profiles are typed and tested through
  `sketch.solverStatus`;
- feature editability, dependency graph, reference health, rebuild plan, and
  sketch edit readiness consume the same source-profile health;
- reference health after sketch edits is structured;
- no solved coordinates are written to source and no schema after
  `web-cad.project.v17` is introduced.

## Implemented Tranche F Sketch Editing Product UI

V11 Tranche F exposes the solver/status work in the existing product surface
without adding a viewport overlay, debug console, or new model authority.

Implemented UI behavior:

- the app reads `sketch.solverStatus` for current sketches through the cad-core
  query path and passes the structured response to UI consumers;
- the Sketches workspace panel extends the existing compact Evaluation section
  with solver status, numerical solve state, profile validity, and capped
  diagnostics behind a collapsed disclosure;
- sketch entity rows show concise dimension/constraint intent when an entity
  has source-backed driving data;
- the right-rail modeling context shows a compact Sketch status card for
  selected sketches and sketch entities, while keeping detailed solver output
  out of Project/File, viewport overlays, and debug surfaces;
- existing dimension, constraint, entity edit, and feature creation controls
  remain the command surfaces; React does not infer solve authority or mutate
  solved geometry.

Tranche F does not introduce new commands, drag-solve behavior, glyph overlays,
or `web-cad.project.v17` source changes. Solver status remains read-only UI
state derived from cad-core.

## Expected End State

At V11 completion, Partbench should have:

- a custom 2D sketch solver boundary;
- source-backed constraint and dimension records;
- V17 project schema only where new source data requires it;
- structured solver status and diagnostics;
- direct sketch entity and dimension edits through CADOps;
- compact, usable sketch constraint UI;

- profile validity feeding downstream feature rebuild and reference health;
- JSON and `.wcad` round-trip coverage;
