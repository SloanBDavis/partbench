# V3 Target: Parametric Sketch Editing

Status: active implementation target as of 2026-05-22. Phase A source
modeling, the first Phase B line-length evaluator target, initial
parameter/dimension UI, and Phase C rebuild propagation hardening are
implemented.

V3 should make Partbench meaningfully more useful without starting unrelated
long-term architecture work. V2 proved the feature/body foundation: sketches,
rectangle/circle extrudes, generated references, attached sketches, named
references, measurements, dependency health, and narrow rectangle-tool add/cut
boolean slices. V3 builds on that foundation by making sketches parametric.

## Product Goal

A user or structured agent can define and edit source-of-truth sketch dimensions
and parameters, rebuild downstream feature/body geometry through CADOps, and
inspect clear dependency health when something becomes invalid.

The user-facing outcome is simple:

```text
edit dimension -> sketch updates -> feature/body rebuilds -> measurements,
references, health, and derived mesh update coherently
```

The agent-facing outcome is equally important:

```text
query parameters/dimensions -> dry-run edits -> inspect semantic diffs and
health -> commit through the same CADOps path as the UI
```

## Architectural Rules

V3 must keep these rules:

- CADOps remains the shared mutation protocol.
- `cad-core` remains authoritative.
- Parameters, sketch dimensions, and solver status must be typed and queryable.
- React UI does not own hidden model state.
- OCCT remains outside `cad-core`.
- Renderer meshes remain derived views/caches.
- MCP and agents wrap CADOps.
- Project format changes happen only for persisted source-of-truth data.
- V1/V2 project compatibility remains importable through explicit migration.
- New model structures must have current callers and tests.

## V3 User Workflow Target

The V3 workflow should allow a user or agent to:

1. Open an existing V2 project or start a new one.
2. Create or select a sketch.
3. Add a rectangle or circle sketch entity.
4. Add or edit a driving dimension for the entity.
5. Optionally bind a dimension to a document parameter when scoped.
6. Extrude the driven sketch entity.
7. Edit the dimension or parameter later.
8. See the sketch, authored body, narrow boolean result where applicable,
   measurements, extents, references, and dependency health update coherently.
9. Save/load the V3 project data.
10. Query and mutate the same model through agent/MCP wrappers.

## Phase A: Parameter and Driving-Dimension Source Model

Goal: introduce the smallest source-of-truth model for parameters and sketch
dimensions.

Implemented deliverables:

- typed parameter IDs and records;
- typed sketch dimension IDs and records;
- commands for parameter create, update, rename, and delete;
- commands for sketch dimension create, update, rename, and delete;
- validation for names, unique IDs, finite positive values where needed, target
  entity existence, and supported target kinds;
- semantic diffs, undo/redo, batch dry-run/commit, and transaction summaries;
- `web-cad.project.v7` import/export with migration from V1 through V6;
- CADOps queries for listing parameters and sketch dimensions;
- agent/MCP pass-through for V3 commands and queries;
- focused unit/package tests for command/query contracts, migration, and rebuild
  invalidation.

Implemented first slice:

- document numeric parameters with optional descriptions;
- rectangle width and height dimensions;
- circle radius dimension;
- literal dimension values and parameter-bound dimension values;
- direct deterministic entity updates through CADOps, without a full solver.

Deferred from Phase A:

- parameter/dimension UI, because the first slice was protocol/core/storage first.

Non-goals:

- No full solver yet.
- No arbitrary expression language unless separately scoped.
- No tangent/coincident/equal/symmetry constraints.
- No dragging solver UX.

## Phase B: Minimal Sketch Solver/Evaluator

Goal: broaden the Phase A direct evaluator toward the smallest coherent sketch
solver/evaluator path without jumping to a general constraint system.

Expected deliverables:

- internal evaluator/solver structure in the correct package boundary;
- support for additional narrow driving cases only where endpoint/entity behavior
  is explicit;
- structured status for healthy, unsupported, under-defined, over-defined, and
  unsolved states where practical;
- no renderer or UI-owned solving;
- tests for deterministic update behavior and failure shapes.

This phase should be intentionally conservative. It is better to support a few
dimension types correctly than to introduce a broad solver that is hard to
reason about.

Non-goals:

- No general geometric constraint solver.
- No spline/NURBS sketch constraints.
- No broad profile solving or automatic profile recognition.

Implemented evaluator slice:

- line length dimensions are supported as the first post-rectangle/circle target;
- the deterministic update preserves the authored line midpoint and current
  direction, then recomputes both endpoints to match the requested length;
- zero-length lines reject line-length dimensions because direction is
  ambiguous.

## Solver Architecture Direction

The first V3 dimension work must set up future general sketch constraint solving
instead of becoming a one-off UI edit path.

The intended model is:

```text
Sketch source:
  entities
  parameters
  dimensions / constraints

Evaluation:
  read source records
  solve or evaluate supported relationships
  return solved geometry plus structured status

Downstream:
  features consume evaluated sketch geometry
  derived meshes rebuild from evaluated source
```

Required guardrails:

- Sketch entities remain source-of-truth authored data in `cad-core`.
- Parameters, dimensions, and constraints are source records with stable IDs.
- Solver/evaluator results are deterministic derived data, not hidden UI state.
- CADOps owns all parameter, dimension, and constraint mutations.
- Semantic diffs should identify changed source records and affected downstream
  sketch/entity/feature/body data where practical.
- UI, scripts, agents, and MCP wrappers inspect and mutate the same typed
  records.
- Invalid, unsupported, under-defined, or over-defined sketches must report
  structured status/errors instead of silently producing misleading geometry.
- Once solver behavior exists for a sketch path, downstream features should
  consume evaluated sketch geometry rather than bypassing the evaluator and
  reading stale raw fields.

Do not:

- store solver-only state in React;
- implement dimensions only as labels over existing numeric fields;
- let agent/MCP callers mutate solver state through a separate API;
- persist renderer meshes, OCCT topology, or solver cache outputs as source of
  truth;
- silently rebuild downstream features from invalid or ambiguous solved states.

## Phase C: Rebuild Propagation and Health

Goal: ensure parameter/dimension edits drive all current V2 downstream systems
coherently.

Implemented deliverables:

- source sketch entity values update through CADOps;
- authored extrudes rebuild from the updated sketch data;
- attached-sketch extrudes keep resolving through their generated face frames;
- narrow rectangle-tool add/cut results rebuild or report clear unsupported
  status;
- measurements, extents, generated references, named references, dependency
  health, and transaction history remain coherent;
- `project.health` includes parameter-bound sketch dimension entries, effective
  values, affected sketch entity references, affected authored features/bodies,
  and structured issues for missing/invalid dimension dependencies;
- derived geometry cache keys and stale async result handling include driven
  sketch changes.

Non-goals:

- No broad boolean expansion.
- No generated references for boolean result bodies unless specifically scoped.
- No exact B-rep topology persistence.

## Phase D: Human and Agent UX

Goal: make parametric editing understandable for users while keeping the model
agent-legible.

Expected deliverables:

- compact parameter and dimension UI;
- clear sketch/entity/dimension relationships;
- clear solver or dependency-health status;
- clear downstream impact for affected features/bodies;
- focused UI helper tests for command building, formatting, filtering, and
  status display;
- agent/MCP wrappers that expose V3 behavior as thin CADOps wrappers.

Non-goals:

- No natural-language command parser.
- No browser E2E expansion unless explicitly scoped.
- No UI-only model state.

## Phase E: Stabilization and Completion

Goal: declare V3 complete only when the parametric sketch workflow is coherent,
tested, and documented.

Expected deliverables:

- unit/package coverage for command/query contracts, solver/evaluator behavior,
  serialization, migration, rebuild propagation, derived geometry invalidation,
  adapter/MCP wrappers, and UI helpers;
- docs updated to mark V3 complete when acceptance criteria pass;
- implementation plan updated with the next target after V3.

## Project Format Direction

V3 Phase A introduced `web-cad.project.v7` because parameters and sketch
dimensions are persisted source-of-truth data. Query-only solver summaries or
derived health/status do not justify a new format version by themselves.

V7 includes:

- explicit saved parameter and dimension shapes;
- migration from `web-cad.project.v6`;
- tests proving V1 through V6 import compatibility still holds where practical;
- no persisted derived meshes, renderer state, or OCCT kernel indexes.

## Acceptance Criteria

V3 is complete when:

1. Parameters and sketch dimensions are source-of-truth data.
2. Current supported dimensions drive rectangle/circle sketch geometry through
   CADOps.
3. Editing dimensions or parameters rebuilds downstream authored extrudes,
   attached-sketch extrudes, narrow add/cut results, measurements, extents,
   generated references, named references, and dependency health where
   supported.
4. Unsupported or unsolved states fail with structured errors/status.
5. Project JSON import/export/migration is explicit and tested.
6. Agent/MCP adapters expose V3 commands/queries as wrappers over CADOps.
7. The UI clearly presents parameters, dimensions, solver status, and downstream
   impact without debug wording or hidden state.
8. Unit and package-level tests cover the workflow without relying on brittle
   browser E2E tests.

## Deferred Beyond V3 Unless Explicitly Scoped

- General sketch constraint solving.
- Full stable topological naming across broad feature edits.
- Generated references for boolean result bodies.
- General booleans/add, fillets, chamfers, shell, patterns, lofts, sweeps,
  revolve, and direct edits.
- STEP import/export.
- OPFS cache implementation and File System Access open/save.
- Native `.wcad` package implementation.
- Local launcher with cross-origin isolation headers.
- WebGPU production renderer.
- Assemblies, mates, large-model LOD, and instancing.
- Hosted collaboration.
- Production MCP auth/permission system.
- Natural-language command entry.
