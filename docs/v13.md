# V13 Planning Record: General Topology Identity And B-Rep Checkpoint Foundation

Status: in progress as of 2026-06-21. V13 is the first general topological
identity release. It follows the completed V10 editable feature history and
reference-health release, the completed V11 sketch solver release, and the
completed V12 stable boolean topology/result-reference release.

V13 is not another source-semantic special case release. It is the release that
starts the real, general machinery Partbench needs before it can honestly claim
fully-featured single-part modeling, robust imported geometry, future STEP
import, assemblies with reliable selections, and broad downstream feature
editing.

The release center is:

```text
source document + exact B-rep result ->
topology snapshot ->
stable topology anchors ->
checkpoint package payloads ->
topology matching ->
reference health and repair ->
command-ready references
```

## Product Goal

Partbench should stop treating topological identity as a collection of
per-feature exceptions.

After V13, the system should have a durable foundation for answering:

- What exact face, edge, or vertex did this generated reference originally
  point at?
- What source feature and exact B-rep checkpoint produced it?
- Did the entity survive a rebuild?
- Was it replaced, split, merged, deleted, or made ambiguous?
- Can a named reference be repaired explicitly?
- Can this face or edge be used by a downstream command safely?
- If not, which part of the topology identity chain failed?

The goal is not that every possible OCCT result is command-ready on day one.
The goal is that every exact topology result is represented by the same
identity/checkpoint/matching system instead of by ad hoc feature-specific
guesswork.

## Why This Is The Next Major Release

V12 made the first meaningful cut into stable result topology by proving
source-semantic roles for supported boolean cut/add results. That was the right
way to avoid fake stability, but it still leaves a fundamental ceiling:

- chamfer/fillet on result-body edges stays limited;
- arbitrary revolve, hole, chamfer, fillet, and future shell/sweep/loft/pattern
  result topology remains diagnostic-only or partly command-limited;
- imported bodies cannot have durable references without a checkpoint/matching
  system;
- future assemblies cannot make instance selections robust if the underlying
  part topology is unstable;
- agents cannot safely chain complex edits if generated references silently
  drift.

Because Partbench has no external users yet, now is the right time to build the
foundation correctly instead of layering more partial product behavior on top
of source-semantic exceptions.

V13 should be big enough to create the real subsystem, even if later releases
apply it to more modeling families and import workflows.

## Architecture Alignment

V13 follows `docs/architecture.md` directly:

- the command protocol remains the center of the system;
- `cad-core` remains authoritative for document source, public references,
  topology anchors, repair records, checkpoint metadata, reference health,
  commandability, transactions, and semantic diffs;
- geometry-worker/OCCT may produce exact B-rep payloads, topology snapshots,
  private kernel topology maps, and derived signatures;
- raw OCCT topology indexes, shape pointers, map keys, transient traversal
  order, mesh indexes, renderer object IDs, selection-buffer colors, GPU
  buffers, viewport pixels, OPFS paths, file handles, local paths, and export
  artifact IDs must never become public stable CAD IDs;
- renderer and React may display topology health, repair choices, and selected
  references, but they do not infer topology stability or commandability;
- agents and MCP remain thin wrappers over CADOps/query paths;
- unsupported, ambiguous, conflicting, stale, missing, or low-confidence
  topology states must return structured diagnostics instead of silently
  retargeting a reference.

V13 intentionally introduces source-of-truth topology identity data where
needed. That is different from V12, which avoided a schema bump because its
supported result references were query-derived from source semantics.

## Major V13 Decisions

### 1. V13 Is A General Topology Identity Release

V13 should build one shared identity layer for exact B-rep topology:

- topology snapshots;
- topology entity descriptors for bodies, faces, loops/wires, oriented edge
  uses/coedges, edges, vertices, axes, and adjacency;
- public stable topology anchors;
- private kernel topology mapping;
- signature extraction;
- checkpoint payloads;
- matching and confidence scoring;
- explicit repair records;
- reference-health integration;
- commandability gates.

Feature-specific semantic roles still matter. They are high-quality evidence
and should be used when present. But they must feed the shared topology identity
system rather than remain the only path to stable references.

### 2. Public IDs Are Stable Anchors, Not Kernel IDs

V13 public topology references should be stable CAD references controlled by
the document model. A public generated-reference ID can be deterministic for
source-semantic cases or minted as an authored topology anchor for arbitrary
exact topology.

A public topology reference may include:

- body ID or result body ID;
- source feature lineage;
- generated reference kind: body, face, edge, or vertex;
- a source-semantic role when one exists;
- a topology anchor ID;
- checkpoint lineage or source identity;
- an opaque signature hash.

A public topology reference must not include:

- OCCT face/edge/vertex index;
- OCCT shape pointer or transient shape hash;
- topology traversal ordinal unless the ordinal is explicitly source-defined;
- mesh or renderer ID;
- GPU/selection-buffer/pixel ID;
- OPFS path, file handle, local path, or export artifact identifier.

Private kernel topology IDs are allowed inside geometry-worker/checkpoint
payloads only. They are implementation details used to build snapshots and
matching candidates, never public reference IDs.

### 3. Checkpoints Are Source-Adjacent, Not Mesh Cache

V13 should introduce persistent exact B-rep checkpoints as source-adjacent
project data.

They are not rendered mesh caches. They are not viewport state. They are not
GLB/STEP export artifacts. They are not a replacement for the parametric source
graph.

A checkpoint records the exact B-rep result and topology snapshot for a body at
a particular source identity so future rebuilds can compare exact topology
instead of guessing from source semantics alone.

The source document should record checkpoint metadata and references to package
payloads. The payload bytes live in the native package. This keeps the
document graph auditable while avoiding huge exact payloads inside ordinary UI
state.

### 4. V13 Requires A Source Schema Decision

V13 should introduce `web-cad.project.v18` when it first persists any of:

- topology anchor records;
- checkpoint metadata records;
- topology repair records;
- exact checkpoint payload references;
- anchor-to-checkpoint matching policy;
- source-authored topology identity settings.

Query-only topology diagnostics do not require V18. The full V13 release does,
because a real topology identity foundation must persist more than derived
query output.

`web-cad.project.v18` is the next saved-project schema after V17. It must not
reuse old release numbers. The `web-cad.project.v13` schema identifier already
means the historical perpendicular-constraint storage shape.

### 5. V13 Should Introduce `.wcad` Package V2

The existing package version, `partbench.wcad.v1`, stores the source document
and command/history source entries. V13 checkpoint payloads are authoritative
project data for topology identity and should not be squeezed into cache-only
metadata.

V13 should introduce `partbench.wcad.v2` when checkpoint payloads become
losslessly persisted. Expected entries:

```text
manifest.json
document.cbor
commands.cbor
checkpoints/
  <checkpointId>.brep
  <checkpointId>.topology.cbor
  <checkpointId>.signature.cbor
```

The exact entry names can change during implementation, but the principles
cannot:

- checkpoint payloads are package entries with hash/byte-length validation;
- checkpoint entries are content-addressed or manifest-addressed;
- payloads are source-adjacent authoritative data, not OPFS cache;
- OPFS may cache unpacked or derived forms, but OPFS is never the only source;
- `.wcad` open/save must preserve checkpoint payloads or explicitly diagnose a
  lossy/debug path.

JSON remains useful as debug/interchange. If JSON cannot carry full checkpoint
payloads without unacceptable bloat, JSON export must clearly report that it is
source-graph-only or checkpoint-metadata-only for V18 projects. A JSON import
that lacks required checkpoint payloads should load the parametric source where
possible but mark affected topology anchors/checkpoints missing or
repair-needed. It must not silently pretend the checkpoint survived.

### 6. Matching Must Be Scored And Diagnosable

Topology matching is not boolean. V13 should model confidence and evidence.

The matcher should consider:

- source lineage and feature IDs;
- checkpoint source identity;
- previous and new topology entity kind;
- surface type and curve type;
- geometric signature hash;
- area, length, bounds, centroid, normal, axis, radius, curvature class;
- adjacency graph;
- neighboring anchored entities;
- kernel-generated modified/deleted/generated hints where available;
- source-semantic role matches where available;
- stable sketch/profile roles where available.

The matcher should return structured states:

- active;
- replaced;
- split;
- merged;
- stale;
- consumed;
- ambiguous;
- missing;
- unsupported;
- repair-needed;
- deleted;
- failed.

It must also expose why:

- exact signature match;
- source-semantic match;
- adjacency match;
- geometry match;
- low confidence;
- multiple candidates;
- no candidate;
- checkpoint missing;
- kernel snapshot unavailable;
- kind mismatch;
- consumed body;
- source changed beyond matching policy.

### 7. Repair Is Explicit, Not Silent

V13 must not silently retarget a named or generated reference when the match is
ambiguous.

The system may automatically keep a reference active only when confidence is
high enough and the match policy proves it. Otherwise the reference becomes
repair-needed or ambiguous.

Repair should be a CADOps command, likely extending or complementing
`reference.repairName`, with dry-run and commit behavior:

- show candidate replacement targets;
- explain confidence and evidence;
- reject kind-incompatible repair;
- record the repair in source history;
- preserve before/after semantic diffs;
- keep agents/MCP on the same path as the UI.

### 8. Command-Ready Requires Both Identity And Command Eligibility

A face or edge can have an identity and still not be command-ready for a
specific operation.

V13 should separate:

- identity status: can we track what entity this is?
- health status: active/replaced/stale/ambiguous/etc.;
- command eligibility: can this command safely consume it?

For example:

- a cylindrical face may be selectable, nameable, measurable, and inspectable
  but not eligible for sketch-on-face;
- an edge may be measurable but not eligible for chamfer/fillet until the
  command validator and geometry runtime accept topology-anchor targets;
- a low-confidence matched face may be inspectable as a candidate but not
  command-ready.

### 9. V13 Must Cover Existing Feature Families First

V13 should prove the identity system against the existing authored feature
families before adding broad new modeling features:

- rectangle/circle extrude newBody;
- direct cut/add extrude results;
- revolve;
- hole;
- chamfer;
- fillet.

The goal is not to make every generated entity from every family command-ready
immediately. The goal is that all existing exact result bodies can flow through
the same snapshot, checkpoint, identity, matching, health, and repair
vocabulary.

### 10. UI Must Reveal The Model Without Becoming The Model

V13 UI work should stay compact:

- selected topology reference health;
- command-ready actions when advertised by cad-core;
- repair-needed diagnostics and candidate repair choices;
- checkpoint/matching status in Project/File or Inspector when useful;
- no large viewport overlays;
- no dev milestone language;
- no React inference of topology stability.

The UI may help users select a replacement target, but cad-core must validate
and commit the repair.

## Source Model

V13 should add source records only where they are truly source-of-truth.

Candidate `web-cad.project.v18` records:

```ts
TopologyIdentitySettings {
  version: "partbench.topology-identity.v1"
  autoAnchorPolicy: "referenced-only" | "all-command-ready"
  matchThresholds: {
    active: number

  }
}

TopologyCheckpointRecord {
  id: string
  bodyId: string
  featureId?: string
  sourceIdentity: string
  kernelVersion: string
  units: "mm" | "cm" | "m" | "in"
  packageEntries: {
    brep?: string
    topology: string
    signature: string
  }
  contentHashes: {
    brep?: string
    topology: string
    signature: string
  }
  createdByTransactionId: string
  createdAt?: string
}

TopologyAnchorRecord {
  id: string
  kind: "face" | "edge" | "vertex" | "body"
  bodyId: string
  stableId: string
  sourceFeatureId?: string
  sourceReference?: {
    type: "generatedReference" | "namedReference" | "selection"
    stableId?: string
    name?: string
  }
  checkpointId: string
  checkpointEntityId: string
  signatureHash: string
  matchPolicy: "strict" | "sourceLineage" | "geometryAndAdjacency"
  label?: string
}

TopologyRepairRecord {
  id: string
  repairedReference: {
    type: "namedReference" | "topologyAnchor"
    name?: string
    anchorId?: string
  }
  previousAnchorId?: string

```

The implementation may refine names and shapes, but V13 must preserve the
conceptual separation:

- checkpoint records reference exact payloads;
- anchor records provide stable public identity;
- repair records record explicit human/script/agent intent;
- matching output remains query-derived and is not silently persisted as truth.

## Topology Snapshot Model

A topology snapshot is exact geometry metadata for one body checkpoint.

It should include:

- snapshot version;
- body/source identity;
- kernel version;
- units;
- entity descriptors for faces, edges, and vertices;
- loop/wire descriptors and oriented edge-use/coedge descriptors where the
  kernel exposes them or where they can be reconstructed safely;
- topology adjacency graph;
- geometric signatures;
- optional kernel local entity handles scoped to the checkpoint payload;
- validation diagnostics.

Face descriptor examples:

```ts
TopologyFaceDescriptor {
  checkpointEntityId: string
  surfaceType:
    | "plane"
    | "cylinder"
    | "cone"
    | "sphere"
    | "torus"
    | "nurbs"
    | "unknown"
  area?: number
  bounds?: { min: [number, number, number]; max: [number, number, number] }
  center?: [number, number, number]
  normal?: [number, number, number]
  axis?: [number, number, number]
  radius?: number
  adjacentEdges: string[]
  signatureHash: string
}
```

Edge descriptor examples:

```ts
TopologyEdgeDescriptor {
  checkpointEntityId: string
  curveType:
    | "line"
    | "circle"
    | "ellipse"
    | "spline"
    | "unknown"
  length?: number
  startVertex?: string
  endVertex?: string
  center?: [number, number, number]
  axis?: [number, number, number]
  radius?: number
  adjacentFaces: string[]
  signatureHash: string
}
```

Loop, wire, and oriented edge-use descriptors matter because many CAD
operations consume boundary structure rather than only unoriented edges. A face
with an inner hole boundary is not adequately described by a flat list of
adjacent edges. The snapshot model should therefore represent:

- outer loops and inner loops for faces;
- ordered oriented edge uses/coedges within each loop;
- orientation relative to the underlying edge;
- adjacent face relationships through paired edge uses;
- degenerate or seam edge uses where the kernel reports them;
- structured diagnostics when loop/coedge extraction is unavailable.

These descriptors may use checkpoint-local IDs. Public stable references still
come only from cad-core anchors and generated-reference contracts, not from
private extraction IDs.

The snapshot is not a renderer mesh. It is exact-topology metadata used for
identity, matching, measurement authority, diagnostics, and future exact
selection.

## Checkpoint Lifecycle

Each checkpoint must have explicit lifecycle rules:

- source identity: the exact document source, command state, units, and feature
  inputs that produced the body;
- transaction identity: the committed transaction that created or refreshed the
  checkpoint;
- kernel identity: OCCT/WASM binding version, tolerance settings, and topology
  extraction version;
- body scope: the body or result body the checkpoint covers;
- package identity: manifest entry path, byte length, content hash, and media
  type;
- validity status: active, stale, missing, incompatible, invalid, corrupt, or
  unsupported;
- rebuild policy: whether the checkpoint is required to resolve anchors,
  optional validation evidence, or safely discardable derived cache;
- migration policy: whether an older checkpoint payload can be loaded,
  upgraded, ignored, or requires repair.

Kernel version and tolerance changes must be visible. A checkpoint generated
with incompatible topology extraction settings should not silently validate a
new public reference.

## Storage And Package Model

V13 should treat storage as part of the foundation, not a later afterthought.

### JSON

JSON remains explicit debug/interchange.

Options for V18 JSON implementation should be decided in Tranche C:

1. Include only topology metadata and mark checkpoint payloads external or
   missing.
2. Include base64 checkpoint payloads for a complete but large debug artifact.
3. Offer two exports: source-only JSON and complete diagnostic JSON.

The release should choose one deliberate behavior. It must not silently drop
checkpoint payloads and then report topology anchors healthy.

### `.wcad`

`.wcad` becomes the lossless primary storage path for checkpointed projects.

`partbench.wcad.v2` should validate:

- manifest version;
- document schema;
- command history;
- checkpoint entry paths;
- byte lengths;
- hashes;
- source identity;
- kernel/checkpoint compatibility;
- topology snapshot validity;
- missing or stale checkpoint entries.

### OPFS

OPFS may cache:

- unpacked checkpoint payloads;
- derived topology indexes;
- matcher acceleration structures;
- tessellated meshes.

OPFS remains rebuildable browser-private cache. Clearing OPFS must not delete
authoritative checkpoint source from `.wcad`.

## Agent And MCP Model

Agents must see topology identity through compact structural queries, not
through raw B-rep or meshes.

Agent/MCP behavior should include:

- query topology anchor health;
- preview repair candidates;
- create anchors for selected topology references;
- repair named references explicitly;
- run feature edit dry-runs and inspect topology effects;
- reject low-confidence automatic edits;
- expose checkpoint readiness without leaking package paths or kernel IDs.

MCP remains a thin wrapper. It must not implement its own matching or repair
authority.

## UI Model

V13 UI should be functional but not dominant.

Primary surfaces:

- Selection tab: selected topology reference health, match confidence, eligible
  operations.
- Inspector/Modeling: command actions advertised by cad-core, repair affordance
  when needed.
- Feature tree: checkpoint/anchor status only where useful and compact.
- Project/File: checkpoint package health, lossy JSON warning if applicable.
- Viewport: selected/hovered owning entity display only; exact highlighting can
  remain a later renderer improvement unless a tranche explicitly adds it.

The UI should never imply a command is available unless
`selection.referenceCandidates` or a focused cad-core query advertises it.

## Proposed Tranche Sequence

### Tranche A: Topology Identity Protocol And Vocabulary

Status: implemented. Tranche A added the shared topology identity protocol
vocabulary and read-only `project.topologyIdentityReadiness` query. The query
reports the V13 contract version, planned V18 and `.wcad` v2 boundaries,
current source counts, capability readiness, and structured diagnostics. It is
query-only: it does not persist topology anchors, checkpoints, match results,
repair records, V18 document data, or `.wcad` v2 package entries.

Goal: define the shared language before implementing matching or persistence.

Scope:

- typed protocol shapes for topology snapshots, topology anchors, checkpoint
  metadata, topology match results, match evidence, confidence, repair
  candidates, and diagnostics;
- cad-core query stubs/readiness surfaces that report current support and
  deferred implementation state;
- agent/MCP pass-through only if new queries are exposed;
- tests proving public protocol shapes exclude renderer, mesh, OCCT, GPU,
  selection-buffer, OPFS, file-handle, path, viewport, and export IDs.

Non-goals:

- no checkpoint persistence;
- no schema migration;
- no broad UI;
- no matching engine;
- no new modeling commands.

### Tranche B: Exact Topology Snapshot Extraction

Status: implemented as a partial derived extractor. Tranche B added typed
`geometry.exactTopologySnapshot` support through geometry-kernel and
geometry-worker, plus OCCT-backed derived snapshots for supported exact body
sources. The current extractor reports body/solid/face/wire/edge/vertex counts
and local derived descriptors. Loop, coedge, axis, ordered adjacency, and
entity-level geometric descriptors remain structured as unavailable/deferred
evidence; these snapshots are not public stable identity and are not persisted
as checkpoints.

Goal: produce exact topology snapshots from the geometry boundary without
making them public identity.

Scope:

- geometry-kernel/geometry-worker snapshot request/response types;
- OCCT-backed extraction of supported body/solid/face/wire/edge/vertex counts
  and derived local signatures where bindings allow;
- structured unsupported diagnostics where bindings are insufficient;
- cad-core/app query consumption as derived evidence only;
- tests for source/derived boundary and no raw kernel ID leakage.

Non-goals:

- no persisted checkpoints yet;
- no stable public anchors;
- no UI commandability changes.

### Tranche C: V18 Source Contract And `.wcad` V2 Package Contract

Status: implemented as a contract tranche. Tranche C typed the
`web-cad.project.v18` topology identity source block, `.wcad` v2 checkpoint
manifest shape, checkpoint entry path rules, and v2 source identity behavior.
Existing projects still export as V16/V17 unless topology identity source
records are present, and current `.wcad` v1 read/write remains unchanged until
checkpoint persistence is implemented.

Goal: decide and type the source/persistence model for real topology identity.

Scope:

- `web-cad.project.v18` source contract for topology settings, checkpoint
  records, topology anchors, and repair records;
- migration and validation rules from V17;
- `.wcad` package v2 manifest and entry contract for checkpoint payloads;
- explicit JSON behavior for checkpointed projects;
- package/source identity rules proving checkpoints are included only when
  authoritative;
- docs and tests for storage boundaries.

Non-goals:

- no automatic checkpoint creation yet;
- no broad UI;
- no imported bodies.

### Tranche D: Checkpoint Write/Read And Validation

Goal: make exact topology checkpoints durable in project storage.

Status: implemented as package-level `.wcad` v2 checkpoint payload
preservation and validation. Cad-core can write/read caller-supplied checkpoint
B-rep, topology, and signature payload bytes, validate entry paths, hashes,
byte lengths, source identities, and topology/signature CBOR compatibility, and
keep `.wcad` v1 behavior unchanged. Actual B-rep checkpoint byte creation
through the geometry boundary remains deferred to a later tranche because the
current geometry boundary does not expose a B-rep writer.

Scope:

- preserve/read checkpoint payload bytes supplied by the topology/checkpoint
  layer through cad-core package metadata;
- write/read `.wcad` v2 checkpoint entries;
- validate hashes, byte lengths, package paths, source identities, and topology
  snapshot compatibility;
- preserve `.wcad` v1 read compatibility;
- mark missing/stale checkpoints structurally;
- focused Project/File status for checkpointed projects.

Non-goals:

- no broad matching engine yet;
- no synthetic or placeholder B-rep checkpoint bytes;
- no automatic repair;
- no STEP import.

### Tranche E: Topology Matching Engine

Goal: compare old and new topology snapshots and explain reference survival.

Status: implemented as a non-mutating `topology.matchSnapshots` query with
cad-core, agent-adapter, and MCP pass-through. The first matcher uses exact
topology snapshot entity kind, signature, checkpoint source identity, and
source-lineage evidence to report exact, replaced, split, merged, deleted,
ambiguous, low-confidence repair-needed, and kind-mismatch outcomes. Richer
geometry/adjacency scoring remains available to later snapshot-expansion
tranches when those descriptors exist.

Scope:

- matching engine in cad-core or a pure helper package, fed by topology
  snapshots and source context;
- scoring from source lineage, signatures, and available descriptor evidence;
- active/replaced/split/merged/deleted/ambiguous/repair-needed results;
- non-mutating dry-run matching query;
- tests over deterministic fixture snapshots including symmetric ambiguity,
  split/merge, deleted, low-confidence, and exact-match cases.

Non-goals:

- no silent source mutation;
- no UI repair flow yet;
- no renderer exact highlighting.
- no reference-health, rebuild, or command-eligibility consumption yet.

### Tranche F: Reference Health And Rebuild Integration

Goal: make existing reference-health/rebuild surfaces consume topology
matching.

Status: implemented as the first non-mutating topology match consumption slice.
`reference.health`, `project.dependencyGraph`, `project.rebuildPlan`, and
`feature.editability` now accept caller-supplied V13 topology match results and
consume persisted/imported V18 topology anchor records when present. Topology
anchors appear as first-class reference-health/dependency graph entries, body
lifecycle summaries include topology anchor/match state, and feature
editability reference effects include topology anchor IDs, checkpoint IDs, and
match confidence. Missing checkpoints, replaced matches, ambiguous/split/merged
matches, deleted matches, unsupported states, and repair-needed states remain
structured diagnostics. Agent and MCP wrappers pass this evidence through
without implementing matching or repair authority. This slice does not create
anchors, repair anchors, expand command eligibility, or persist query-derived
match results as source truth.

Scope:

- extend `reference.health`, `project.dependencyGraph`,
  `project.rebuildPlan`, `feature.editability`, and semantic diffs with V13
  topology match results;
- existing feature edits report topology anchor effects;
- named/generated references resolve through anchors where available;
- unsupported/missing checkpoints remain explicit.

Non-goals:

- no new modeling feature families;
- no arbitrary command eligibility expansion until validators support it.

### Tranche G: Topology Checkpoint, Anchor, And Repair Commands

Goal: let users, scripts, and agents intentionally create and repair topology
identity.

Status: implemented as the first CADOps command slice. Protocol, cad-core,
agent, and MCP batch paths now accept `topology.checkpoint.create`,
`topology.anchor.create`, and `topology.anchor.repair`. Checkpoint creation
records metadata-only V18 checkpoint source records with canonical `.wcad` v2
entry paths, caller-supplied source identity, and explicit active/stale/missing/
failed/unsupported status. Anchor creation records source-backed V18 topology
anchors against existing checkpoint records, and anchor repair appends explicit
repair records while retargeting the anchor to a replacement checkpoint entity.
All three commands support dry-run/commit through the existing batch pipeline,
produce structured semantic diffs, participate in undo/redo and
transaction-history summaries, and keep agent/MCP wrappers thin over CADOps.
This slice does not create B-rep/topology/signature payload bytes, parse
checkpoint payload entity tables, add UI repair affordances, expand command
eligibility beyond the Tranche I generated-reference-backed command slices, or
silently retarget references. Named-reference repair to topology-anchor targets
is implemented separately through the Tranche I command eligibility slice.

Scope:

- create checkpoint source records for supported bodies;
- create topology anchors from command-ready or selected exact topology
  candidates;
- repair named references or anchors to replacement candidates;
- dry-run/commit behavior with semantic diffs;
- undo/redo and transaction history;
- compact UI repair affordance;
- agent/MCP parity.

Non-goals:

- no automatic silent retargeting;
- no broad renderer rewrite.

### Tranche H: Apply Identity To Existing Result Feature Families

Goal: run the V13 system through the current exact feature families.

Status: first query slice implemented. `body.topologyIdentity` now derives
non-mutating generated-reference identity candidates from authoritative source,
V12/V13 generated-reference contracts, optional checkpoint metadata, and
caller-provided derived exact topology snapshots. Supported generated
references can bind to exact snapshot entities by deterministic semantic
signature evidence. Chamfer and fillet result bodies remain status-only until
their result topology is proven; the query does not invent generated face/edge
anchors for them.

Scope:

- extrude, cut/add boolean, revolve, hole, chamfer, and fillet result bodies
  produce checkpoints/snapshots where supported;
- supported faces/edges/vertices can be anchored, matched, named, measured, and
  diagnosed through one shared path;
- V12 source-semantic references become high-confidence evidence inside V13,
  not a separate identity subsystem;
- stable result topology for existing feature families is expanded where the
  matcher proves it.

Still deferred:

- checkpoint payload generation and validation against exact entity payload
  bytes;
- automatic anchor creation from identity candidates;
- mutation-command eligibility for topology-anchor targets;
- UI repair/diagnostic workflows for topology identity candidates.

Non-goals:

- no shell/sweep/loft/pattern/mirror/direct edit unless separately scoped;
- no STEP import.

### Tranche I: Command Eligibility For Topology Anchors

Goal: let existing commands consume topology-anchor references where validators
and geometry runtime can support them.

Status: first read-only, planar-face command, edge-finish command, named
reference repair, and downstream body-target command slices implemented.
Active topology anchors now report command eligibility through
`reference.health`, and stable generated-reference-backed anchors can resolve
through `selection.referenceCandidates` without exposing checkpoint-local
topology IDs. Stable face, edge, and vertex anchors can be measured/selected,
while active stable body or axis anchors can be selected only. Stable active
face anchors whose generated backing supports `feature.attachSketchPlane` can
now be used by `sketch.createOnFace`; the command stores the existing
generated-face sketch attachment shape and records the topology anchor in
transaction history. Stable active edge anchors whose generated backing
supports `feature.chamfer` or `feature.fillet` can now be used by those
edge-finish commands; the committed feature stores the resolved generated edge
plus topology-anchor provenance for transaction, health, dependency, import,
export, and UI helper surfaces. Existing named generated references can now be
repaired explicitly to stable active topology anchors through
`reference.repairName`; the command stores the resolved generated target plus
topology-anchor provenance in the named reference, semantic diff, transaction
history, reference health, dependency graph, import/export, agent, MCP, and app
helper surfaces. Stable active body anchors whose generated backing proves a
command-ready generated body reference can now be used as
`targetTopologyAnchorId` for `feature.extrude` add/cut. Cad-core resolves the
anchor to the existing generated `targetBodyId`, reuses the current conservative
add/cut target validator, and stores the resolved `targetBodyId` plus
topology-anchor provenance for transaction history, project structure, health,
dependency graph, import/export, agent, MCP, app helper, generated-reference
signature, and topology source-identity surfaces. Metadata-only anchors remain
health-only until a direct topology-anchor command target exists. Missing,
stale, replaced, ambiguous, deleted, unsupported, consumed, and repair-needed
anchors remain non-commandable with structured diagnostics. Automatic selection
routing, direct topology-anchor command targets without generated backing, and
UI repair flows still wait for explicit topology-anchor target validators.

Scope:

- sketch-on-face for anchored planar faces;
- chamfer/fillet for anchored supported edges;
- downstream cut/add targets where active result body topology and checkpoint
  health prove the target;
- exact measurement/inspect for anchored faces/edges/vertices;
- structured command-ineligible diagnostics for identities that are healthy but
  not valid command targets.

Non-goals:

- no new modeling families;
- no arbitrary command eligibility without exact validator support.

### Tranche J: Product Integration And Topology Diagnostics

Goal: make the topology system visible enough to be usable without making the
UI noisy.

Status: the first Project/File status slice is implemented. Cad-core
`project.topologyIdentityReadiness` now reports V18 source checkpoints and
anchors as public checkpoint metadata and anchor descriptors while keeping
checkpoint-local entity ids, renderer ids, mesh ids, OCCT ids, GPU ids,
selection-buffer ids, OPFS paths, file handles, and viewport state out of public
readiness output. The web Project/File panel consumes that query directly and
shows compact topology identity status, checkpoint/anchor counts, package
version transition, collapsed diagnostics, and a JSON debug/interchange warning
when topology checkpoint payload preservation requires `.wcad`. Selection,
Inspector, Modeling, viewport contextual-action, and browser repair/command
workflow integration remain later Tranche J work.

Scope:

- compact Selection/Inspector/Modeling surfaces for topology health, match
  evidence, repair candidates, and command eligibility;
- Project/File checkpoint status and lossy-export warnings;
- viewport contextual actions only for advertised operations;
- browser workflows for select/name/repair/command chains.

Non-goals:

- no viewport-blocking topology panels;
- no dev milestone language.

### Tranche K: Release Samples, Stress Fixtures, And Hardening

Goal: prove V13 with long deterministic and browser workflows.

Scope:

- deterministic source/query/package smokes for checkpoint creation, save/open,
  rebuild, match, repair, and command eligibility;
- fixture families for exact match, split, merge, delete, symmetric ambiguity,
  low confidence, and missing checkpoint payloads;
- browser smoke for result topology selection, explicit repair, and downstream
  commands;
- no raw ID leakage assertions;
- docs updated with completed support and remaining topology boundaries.

## Expected End State

At V13 completion, Partbench should have:

- a typed topology identity protocol;
- exact topology snapshot extraction through the geometry boundary;
- persisted checkpoint/anchor/repair source records in `web-cad.project.v18`;
- lossless checkpointed project storage through `.wcad` package v2;
- topology matching with confidence/evidence diagnostics;
- reference health, rebuild plan, dependency graph, and feature editability
  integrated with topology matching;
- explicit topology anchor and repair commands;
- current feature families flowing through the same topology identity system;
- command eligibility for supported topology-anchor references;
- compact UI and agent/MCP access to the same behavior;
- release smokes proving long topology identity workflows.

It should still not claim that every possible CAD model is perfectly
auto-repairable. The honest promise is that arbitrary topology now has a real
identity/checkpoint/matching/repair foundation instead of being postponed or
handled by raw kernel IDs.

## Deferred Beyond V13 Unless Explicitly Scoped Later

- broad new solid feature families such as shell, sweep, loft, pattern, mirror,
  and direct edit;
- STEP import and imported-body healing;
- assemblies, mates, large-model LOD, and instance selection;
- production WebGPU renderer and selection buffer;
- proprietary CAD import/export;
- parameter expression language;
- ellipse/spline/imported sketch solving;
- hosted collaboration;
- natural-language command entry;
- automatic low-confidence topology retargeting;
- treating renderer, mesh, OCCT, GPU, selection-buffer, OPFS, path, file-handle,
  or viewport identifiers as stable public CAD IDs.
